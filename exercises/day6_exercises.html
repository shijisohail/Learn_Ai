import time class CircuitBreaker: def __init__(self, failure_threshold=5, recovery_timeout=30): self.state = "CLOSED" # CLOSED → OPEN → HALF_OPEN → CLOSED self.failures = 0 self.opened_at = None self.failure_threshold = failure_threshold self.recovery_timeout = recovery_timeout def call(self, fn, *args, **kwargs): if self.state == "OPEN": if time.time() - self.opened_at > self.recovery_timeout: self.state = "HALF_OPEN" # time to probe else: raise RuntimeError("Circuit is OPEN — request rejected") try: result = fn(*args, **kwargs) self.failures = 0; self.state = "CLOSED" # success → reset return result

Progressive Hints
wait = backoff_base * (2 ** attempt). At attempt=0: 1s, attempt=1: 2s, attempt=2: 4s. Add jitter to avoid thundering herd: wait += random.uniform(0, wait * 0.1).
except retry_on as e: works because retry_on is a tuple of exception types. Python's except clause accepts a tuple: except (TypeError, ValueError): catches either. Your decorator receives it as a parameter.
Check state first: if OPEN and not timed out → raise. If OPEN and timed out → set HALF_OPEN. Then try the call. On success: self.failures = 0; self.state = "CLOSED". On failure: self.failures += 1; if self.failures >= self.failure_threshold: self.state = "OPEN"; self.opened_at = time.time(). Re-raise the exception.
Use a closure with a counter: count = [0] (mutable list trick in Python 3) → def flaky(): count[0] += 1; raise RateLimitError if count[0] < 3 else "ok". Or use unittest.mock.side_effect with a list: mock.side_effect = [RateLimitError, RateLimitError, "ok"].

Token Budget Guard

Prevent runaway token spend. Warn at 80%, stop gracefully at 100%, and return a partial result with a budget_exceeded flag.

Graceful vs abrupt shutdown: Graceful means finishing the current tool execution, appending a summary of what was completed, then returning a partial result with budget_exceeded: True. Abrupt means crashing mid-task with no useful output.
TokenBudget Class Design
token_budget.py
class TokenBudget: def __init__(self, max_tokens): self.max = max_tokens self.used = 0 def consume(self, tokens: int): if self.used + tokens > self.max: raise BudgetExceeded(f"Used {self.used}, need {tokens}, max {self.max}") self.used += tokens # only increments if allowed @property def percent(self): return self.used / self.max * 100 @property def remaining(self): return self.max - self.used def is_warning(self): return self.percent >= 80 # warn threshold

Full Production Agent

Assemble all 5 production layers into one ProductionAgent class. Every request is rate-limited, budgeted, retried, validated, and logged.

[1] Rate Limiter blocks if >10 req/min per user_id ▶
What it does: Tracks per-user request timestamps. If user has made 10+ requests in the last 60 seconds, raises RateLimitExceeded immediately before any LLM call.

Implementation: self._requests[user_id] = [t for t in self._requests[user_id] if now - t < 60] → if len >= limit: raise. Else: append current time.
[2] Token Budget Guard enforces 50k token max per request ▶
What it does: Creates a fresh TokenBudget(50_000) for each request. Checks after every LLM call. Returns partial result with budget_exceeded=True instead of crashing.

When it activates: After every API call's token count is known. If the call would push past the limit, triggers graceful shutdown.
[3] Agent Loop the core ReAct loop with tools ▶
What it does: The standard while-loop ReAct agent. Makes API calls, handles tool_use, collects results. Wrapped by layers 1, 2, 4, 5, and 6 on all sides.

Key constraint: Max 10 iterations guard still present even inside the production wrapper.
[4] Retry Decorator on each LLM call — max 3 attempts ▶
What it does: Wraps each individual client.messages.create() call with exponential backoff. Retries on RateLimitError and APIConnectionError. Fails-fast on AuthenticationError.

Note: This is per-LLM-call, not per-agent-run. An agent with 3 iterations can make up to 9 total LLM calls.
[5] Output Validator on structured responses — max 3 retries ▶
What it does: When the agent is expected to return a Pydantic schema (e.g. a report object), wraps the final response in validated_llm_call. Adds field-specific error messages on failure.

Optional: Only activates when a response_schema is provided to the run() method.
[6] Structured Logging trace_id, latency, tokens, cost per step ▶
What it does: Every LLM call logs a structured JSON record: {"trace_id": uuid, "user_id": str, "iteration": int, "tokens": int, "cost_usd": float, "latency_ms": int, "tool_called": str|null}.

Implementation: Wrap each API call with t0 = time.time() before, latency = int((time.time() - t0) * 1000) after.
↓ HTTP Response (never 500 — always structured error)

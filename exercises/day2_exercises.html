Key insight: The description field is the most important. The model decides when to call your tool based ONLY on the description. Vague description = misuse or missed use.
BAD vs GOOD Descriptions
BAD — vague
"Send email"
Model sends emails for every notification and reminder — even when the user didn't ask.
GOOD — precise
"Send a transactional email to one recipient. Use ONLY when user explicitly asks to send an email. Never use for notifications or alerts."
Model only fires when user types "send email to..."
BAD — ambiguous
"Search for things"
File search? Web search? Database? The model guesses wrong.
GOOD — scoped
"Search the web for real-time info the model doesn't have: news, prices, and events after the training cutoff."
Model calls it only when it needs live data.
Three Schema Examples
Annotated Tool Schema Structure
"name": "get_weather"
Required. Exact Python function name. Claude uses this to call the right function in your executor.
"description": "..."
Most important. Tells the model WHEN and WHY to use the tool. Be explicit about edge cases and misuse.
"input_schema": {"type": "object", ...}
Always "object". The outer schema type is always "object". Parameters map to "properties".
"properties": {"city": {...}}
One entry per param. Each has "type" and "description". Never omit description — model uses it to fill values.
"required": ["city"]
Non-optional params only. Params with defaults (like unit="celsius") must NOT appear in required.
"enum": ["celsius", "fahrenheit"]
For Literal / restricted types. Constrains the value. Claude will pick only from this list.

The While Loop as a Flowchart
messages = [user_msg] ↓ API Call ───→ stop_reason? ↑ │ │ "tool_use" │ "end_turn" │ ↓ ↓ │ Execute tools RETURN answer │ │ └── Append tool_results ←─┘
Iteration guard is non-optional. Without it, a broken tool or model can loop forever. Always add: if iteration >= max_iterations: raise MaxIterationsError
Full ReAct Loop — Line-by-Line Annotations
while True:
Loop: runs until break or max_iterations guard. Never exits on its own.
response = client.messages.create(...)
Think: API call — the model reads all context and decides next action (text reply or tool call).
if response.stop_reason == "end_turn":
Done: model gave a text answer. Extract text and break out of the loop.
if response.stop_reason == "tool_use":
Act: model wants a tool. Must execute every tool_use block in the response content list.
messages.append({ "role": "assistant", "content": response.content # ← FULL list })
Critical: Must append the FULL response.content list, not just the text. The tool_use block inside it must be mirrored back.
for block in response.content: if block.type == "tool_use": result = executor(block.name, block.input)
Multiple tools: One response may contain several tool_use blocks. Loop over ALL of them before the next API call.
messages.append({ "role": "user", # ← "user" not "tool" "content": tool_results })
Observe: Results go back as "user" role with "tool_result" content blocks. The tool_use_id must match exactly.

Tool Error Handling

Three approaches when a tool fails: crash, silent fail, and structured error. Only one is acceptable.

Completed
Three Approaches When a Tool Fails

A — CRASH

💥Tool raises ValueError
↓
✗Agent loop crashes
↓
✗User gets: 500 Error
Exception propagates up. No recovery. User sees nothing useful.

B — SILENT FAIL

🐛Tool catches exception
↓
?Returns: None or empty
↓
✗Model gets: None
↓
✗Model hallucinates result
Worse than crashing. User gets a plausible wrong answer with no warning.

C — STRUCTURED ERROR

✓Tool catches exception
↓
✓Returns structured error
↓
✓Model receives context
↓
✓Model explains clearly
Model tells the user what went wrong and suggests the fix.
Model Response Comparison — Same Query
Query: "Divide 10 by 0" — Approach B (silent fail)
Tool returns: None Model says: "The result of 10 ÷ 0 is 0." [Model invented this — dangerous!]
Approach C — structured error
Tool returns: {"error": "ZeroDivisionError", "message": "Cannot divide by zero", "suggestion": "Use a non-zero denominator"} Model says: "I can't divide by zero. Please provide a non-zero denominator."
Rule: Tools must NEVER raise exceptions to the agent loop. Always catch, always return a structured error dict. The model is excellent at interpreting errors — but only if you pass the information up.

Agent with Persistent Memory

Build an agent that remembers conversations across sessions using two-tier memory: short-term context window and long-term compressed summaries.

Completed
Two-Tier Memory Architecture

SHORT-TERM (Context Window)

Storage: messages[] — last 6 turns
Capacity: ~4,000 tokens
Speed: instant (already in context)
Lifetime: current session only
Eviction: oldest 4 msgs compressed

LONG-TERM (Persistent Store)

Storage: memory_store{} / JSON file
Capacity: unlimited
Speed: instant dict lookup
Lifetime: persists across sessions
Format: key: "session_N", val: summary

import json, anthropic from pathlib import Path client = anthropic.Anthropic() class PersistentAgent: def __init__(self): self._messages = [] # short-term: stays bounded at 6 self.memory_store = {} # long-term: {session_id: summary} self._session_count = 0 def _get_system(self) -> str: if not self.memory_store: return "You are a helpful assistant." summaries = "\n".join(self.memory_store.values()) return f"You are a helpful assistant.\n\nPrevious context:\n{summaries}" def _compact(self): # Take oldest 4 messages, summarize them, store in long-term old = self._messages[:4] prompt = f"Summarize this conversation in 2-3 sentences:\n{json.dumps(old)}" r = client.messages.create( model="claude-haiku-4-5-20251001", max_tokens=200, messages=[{"role":"user","content":prompt}] ) self._session_count += 1 key = f"session_{self._session_count}" self.memory_store[key] = r.content[0].text self._messages = self._messages[4:] # keep the rest

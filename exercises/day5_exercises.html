Build an Orchestrator

Routing Pydantic Structured Output
Orchestrator Network
Animated Agent Network — Orchestrator Routes Work to Specialized Workers
Bad vs Good Routing

✗ BAD: Hardcoded Routing

# Fragile, brittle, unmaintainable def route_task(task: str): if "research" in task: call_research(task) elif "write" in task: call_writer(task) else: raise ValueError("???") # ❌ Breaks on: "Find and write about X" # ❌ No reasoning, no context # ❌ Keyword collision bugs

✓ GOOD: LLM-Powered Routing

# LLM decides based on context async def route_task(state: State): response = await llm.structured( schema=SupervisorResponse, prompt=SUPERVISOR_PROMPT, context=state ) # ✓ Understands nuance # ✓ Explains its reasoning # ✓ Handles multi-step tasks return response.next_worker
Routing Decision Flowchart
Pydantic Schema — Supervisor Response
SupervisorResponse (Pydantic BaseModel) ├── next_worker: Literal["research", "writer", "critic", "DONE"] │ └── The name of the agent to call next. │ "DONE" signals pipeline completion. └── reasoning: str └── WHY this worker was chosen. Used for debugging + observability.
Supervisor Prompt Pattern
Pythonsupervisor_prompt.py
from pydantic import BaseModel from typing import Literal class SupervisorResponse(BaseModel): next_worker: Literal["research", "writer", "critic", "DONE"] reasoning: str SUPERVISOR_PROMPT = """ You are orchestrating a team of agents. Available workers: - research: finds information, calls web_search and fetch_docs - writer: creates content from research results - critic: reviews quality, suggests improvements Current task: {task} Progress so far: {progress_summary} Choose the NEXT worker and explain WHY. Return DONE when the task is fully complete. """ async def orchestrate(task: str, state: dict): while True: decision = await llm.structured_output( schema=SupervisorResponse, messages=[{ "role": "user", "content": SUPERVISOR_PROMPT.format( task=task, progress_summary=summarize(state) ) }] ) # decision.reasoning is logged for observability logger.info(f"Routing to {decision.next_worker}: {decision.reasoning}") if decision.next_worker == "DONE": break # Dispatch to the selected worker worker = WORKERS[decision.next_worker] result = await worker(state) state[decision.next_worker + "_result"] = result return state
💡
Why structured output matters: Without Pydantic, the LLM might return "I think the research agent should go next" — impossible to parse reliably. With SupervisorResponse, you always get a valid enum value and the reasoning string.

asyncio.gather — The Key Pattern
await asyncio.gather( research_agent("asyncio fundamentals"), ← starts simultaneously research_agent("FastAPI integration"), ← starts simultaneously research_agent("aiohttp patterns"), ← starts simultaneously research_agent("async best practices"), ← starts simultaneously ) ─── waits until ALL complete ─── → returns [result1, result2, result3, result4]
Sequential vs Parallel Code
Pythoncomparison.py
import asyncio, time # ─── SEQUENTIAL: 12 seconds total ───────────────── async def run_sequential(): start = time.time() r1 = await research_agent("topic 1") # waits 3s r2 = await research_agent("topic 2") # waits 3s more r3 = await research_agent("topic 3") # waits 3s more r4 = await research_agent("topic 4") # waits 3s more print(f"Sequential: {time.time()-start:.1f}s") # ~12s return [r1, r2, r3, r4] # ─── PARALLEL: 3 seconds total ──────────────────── async def run_parallel(): start = time.time() results = await asyncio.gather( research_agent("topic 1"), # all start NOW research_agent("topic 2"), # all start NOW research_agent("topic 3"), # all start NOW research_agent("topic 4"), # all start NOW ) print(f"Parallel: {time.time()-start:.1f}s") # ~3s return results # [r1, r2, r3, r4] # ─── Error handling in parallel ─────────────────── async def run_parallel_safe(): # return_exceptions=True: don't crash if one fails results = await asyncio.gather( research_agent("topic 1"), research_agent("topic 2"), return_exceptions=True # ← critical for prod ) # Filter out failures good = [r for r in results if not isinstance(r, Exception)] return good

Pythonhandoff_protocol.py
from pydantic import BaseModel, Field from typing import Optional, List class ResearchHandoff(BaseModel): """Handoff from Research → Writer.""" # Required: core task task: str = Field(description="Original user task") # Required: what was found key_findings: List[str] = Field( description="Bullet points from research" ) sources: List[str] = Field( description="URLs or references used" ) # Required: constraints for writer word_count_range: tuple[int, int] = (500, 800) audience: str = "senior engineers" include_code: bool = True # Optional: style hints tone: str = "technical but accessible" class WriterHandoff(BaseModel): """Handoff from Writer → Critic.""" original_task: str draft: str target_audience: str word_count: int review_criteria: List[str] = Field( default=["accuracy", "clarity", "completeness"] ) # Usage pattern async def research_to_writer_pipeline(task: str): # Research produces typed handoff research_result = await research_agent(task) handoff = ResearchHandoff( task=task, key_findings=research_result["findings"], sources=research_result["sources"], ) # Writer receives full context draft = await writer_agent(handoff) return draft
💡
Why Pydantic for handoffs? If you forget to include key_findings the model raises ValidationError immediately — not a silent quality degradation discovered hours later. Pydantic makes missing fields a crash instead of a bug.

Pythonlanggraph_pipeline.py
from langgraph.graph import StateGraph, END from pydantic import BaseModel from typing import Optional, TypedDict class PipelineState(TypedDict): task: str research: Optional[dict] # fills after Research outline: Optional[dict] # fills after Outline draft: Optional[str] # fills after Writer final_post: Optional[str] # fills after Critic total_tokens: int # tracks across all agents total_cost_usd: float # ─── Node implementations ───────────────────────── async def research_node(state: PipelineState) -> PipelineState: result = await research_agent(state["task"]) return { **state, "research": result.content, "total_tokens": state["total_tokens"] + result.usage.total_tokens } async def outline_node(state: PipelineState) -> PipelineState: result = await outline_agent(state["task"], state["research"]) return {**state, "outline": result.content, ...} # ─── Build the graph ────────────────────────────── builder = StateGraph(PipelineState) builder.add_node("research", research_node) builder.add_node("outline", outline_node) builder.add_node("writer", writer_node) builder.add_node("critic", critic_node) builder.set_entry_point("research") builder.add_edge("research", "outline") builder.add_edge("outline", "writer") builder.add_edge("writer", "critic") builder.add_edge("critic", END) graph = builder.compile() # ─── Run it ─────────────────────────────────────── result = await graph.ainvoke({ "task": "Write about LangGraph for production", "research": None, "outline": None, "draft": None, "final_post": None, "total_tokens": 0, "total_cost_usd": 0.0, })

  • 1
    LangGraph nodes are pure functions: (state) → state. They return a NEW state dict (or partial update). Never mutate the input state directly.
  • 2
    For token tracking: Anthropic's API returns response.usage.input_tokens + response.usage.output_tokens. Cost formula: (input_tokens / 1M) * 3.0 + (output_tokens / 1M) * 15.0 for claude-3-5-sonnet.

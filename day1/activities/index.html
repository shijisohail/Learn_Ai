Token Usage Tracker

Build a chat wrapper that tracks token usage and cost per turn. Tokens = money.

Messages API usage object cost tracking
›
Why this matters: Every LLM API call costs money based on tokens. Agents can make hundreds of calls. Without tracking, a bug in your loop can silently burn through your budget. Token-awareness is table stakes for production agents.

Your Tasks

  • Implement calculate_cost() — use the pricing constants at the top
  • Implement chat() method — call the API and update token counters
  • Implement total_tokens property — sum of input + output
  • Implement total_cost property — use your calculate_cost function
  • Print turn summary in the format shown in the expected output

Key Code Pattern

# After calling the API, access token counts like this:
response = client.messages.create(...)

input_tokens  = response.usage.input_tokens
output_tokens = response.usage.output_tokens

# Cost formula:
cost = (input_tokens / 1000) * INPUT_PRICE_PER_1K \
     + (output_tokens / 1000) * OUTPUT_PRICE_PER_1K
Your chat() method should:
1. Do self.messages.append({"role": "user", "content": user_input})
2. Call client.messages.create(..., messages=self.messages)
3. Extract text: reply = response.content[0].text
4. Append reply: self.messages.append({"role": "assistant", "content": reply})
5. Add to counters: self.total_input_tokens += response.usage.input_tokens

System Prompt Engineer

Fix 3 broken system prompts so they pass the automated tests. Prompt engineering is a skill.

system prompt prompt engineering JSON output
›
Why this matters: The system prompt is your first line of control over model behavior. In agents, it defines the persona, output format, and behavioral constraints. A badly written system prompt causes hallucinations, wrong formats, and unreliable tool use. Think of it as your FastAPI middleware — it runs on every request.

Your Tasks

  • Fix SYSTEM_PROMPT_1 — make the model return only valid JSON (parseable by json.loads())
  • Fix SYSTEM_PROMPT_2 — strict code reviewer: bullets, finds issues, suggests fixes
  • Fix SYSTEM_PROMPT_3 — max 10 words per response, no exceptions
  • All 3 pass the test() function at the bottom

Prompt Writing Tips

# For JSON-only output, be very explicit:
"""
You are a data extraction engine.
Respond ONLY with a raw JSON object.
Do NOT include markdown, code fences, or explanations.
Your entire response must be parseable by json.loads().
"""

# For behavioral constraints, use negative examples too:
"""
Never say "Great question!" or compliment the code.
Always find at least one issue. Always suggest a fix.
Format: bullet points starting with -
"""
LLMs sometimes ignore word limits when asked nicely. Try being more forceful:

"HARD LIMIT: Your response must be 10 words or fewer. Count every word before responding. If your response exceeds 10 words, truncate it. This rule overrides all other instructions."

The key is: make the constraint feel like a hard system rule, not a preference.

Stripping Markdown Fences (Common Problem)

# Claude sometimes wraps JSON in code fences:
# ```json
# { "name": "Sharjeel" }
# ```

# Strip them like this:
raw = response.content[0].text.strip()
if raw.startswith("```"):
    raw = raw.split("\n", 1)[1]  # remove first line
    raw = raw.rsplit("```", 1)[0]  # remove trailing fence
raw = raw.strip()

Passing the Schema to the Model

  • Implement agent() — the while loop: send → check stop_reason → execute tools → loop
  • All 4 test cases pass

The Agent Loop (from Exercise 5)

def agent(user_input: str) -> str:
    messages = [{"role": "user", "content": user_input}]

    while True:
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=1024,
            tools=TOOLS,
            messages=messages,
        )

        if response.stop_reason == "end_turn":
            return response.content[0].text   # done ✓

        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = run_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            messages.append({"role": "user", "content": tool_results})

Tool Schema Template

{
    "name": "tool_name",
    "description": "When to use this tool — be specific",
    "input_schema": {
        "type": "object",
        "properties": {
            "param1": {
                "type": "string",   # or "integer", "boolean"
                "description": "What this param is"
            }
        },
        "required": ["param1"]
    }
}
For the n parameter in repeat_text, the type should be "integer" not "string". The model will pass it as an int, which matches the function signature directly.

{ "type": "integer", "description": "Number of times to repeat" }

01 — What It Is

Computer Use in One Model

Computer Use is not a special API endpoint — it is a standard Messages API call with three built-in tools enabled. The model receives a screenshot, reasons about the current state, and emits structured tool-use blocks. Your code translates those blocks into real OS actions.

💡
Mental model: Think of it as a ReAct loop where the "observation" is always a screenshot. Claude is the brain, your code is the body, and the display is the shared world state. Every iteration: observe (screenshot) → reason → act (click/type/key/bash) → observe again.
WHAT CLAUDE RECEIVES
A PNG screenshot as a base64-encoded image in the user message, plus a text task description. On each loop iteration, the latest screenshot replaces the previous observation.
WHAT CLAUDE EMITS
Structured tool_use content blocks — each block names a tool (computer, bash, str_replace_based_edit_tool) and its input parameters.
WHAT YOUR CODE DOES
Dispatches each tool_use block to the appropriate handler — xdotool / pyautogui for mouse/keyboard, subprocess for bash, file I/O for editor actions — then captures a fresh screenshot.
WHEN THE LOOP ENDS
Claude emits stop_reason == "end_turn" with no tool_use blocks — meaning it considers the task complete. Your code also enforces a max-iteration limit as a hard safety stop.
02 — The Agent Loop

How the Loop Works

Each iteration is one round-trip to the API. The loop runs until Claude signals completion or your iteration limit is hit. Latency is dominated by the vision call — budget 2–5 seconds per step.

The tool_result for a computer action is always an image — the next screenshot. For bash actions, the tool_result is stdout/stderr text.
🔁
Message history grows linearly. Every screenshot is appended to the messages array. A 20-step task with 1920×1080 screenshots can consume 200k+ tokens in context. Use image compression (downscale to 1366×768) and consider truncating old images from history for long-running sessions.
03 — Built-in Tools

The Three Built-in Tools

Declare these in the tools array of your Messages call. Claude decides which to call; your dispatch loop handles execution.

🖱️ computer
Controls the graphical display — mouse, keyboard, and screen capture. The type identifier is versioned (computer_20250124). Requires display dimensions at declaration time.
screenshot left_click right_click double_click middle_click type key mouse_move scroll left_click_drag cursor_position
💻 bash
Runs arbitrary shell commands in a persistent session. Stdout and stderr are returned as the tool_result text. Claude can chain commands, install packages, read logs, or query APIs without touching the GUI.
run shell cmd persistent session stdout + stderr env access file I/O curl / wget
📝 str_replace_based_edit_tool
View and edit files with surgical string replacements — no need to rewrite entire files. Supports view, create, and str_replace sub-commands. Ideal for code editing workflows.
view create str_replace insert undo_edit

04 — Code

Full Working Agent Loop

A complete Python implementation of the computer use agent loop. Each iteration takes a screenshot, sends it to Claude, executes the returned actions, and feeds the result back. The loop exits on end_turn or a max-iteration guard.

PYTHON — Computer Use Agent Loop
import anthropic import base64 import subprocess import time from pathlib import Path client = anthropic.Anthropic() def take_screenshot() -> str: """Capture the screen and return as base64-encoded PNG string.""" # Linux: requires scrot → sudo apt install scrot subprocess.run( ["scrot", "--quality", "80", "/tmp/cu_screenshot.png"], check=True, capture_output=True ) with open("/tmp/cu_screenshot.png", "rb") as f: return base64.standard_b64encode(f.read()).decode() def execute_computer_action(action_input: dict) -> str: """Execute a computer tool_use block and return result screenshot as base64.""" action = action_input["action"] if action == "screenshot": return take_screenshot() elif action in ("left_click", "right_click", "double_click", "middle_click"): x, y = action_input["coordinate"] btn_map = { "left_click": "1", "right_click": "3", "middle_click": "2", "double_click": "1", } clicks = "--repeat 2" if action == "double_click" else "" subprocess.run( f"xdotool mousemove {x} {y} click {clicks} {btn_map[action]}", shell=True, check=True ) elif action == "type": text = action_input["text"] subprocess.run(["xdotool", "type", "--clearmodifiers", text], check=True) elif action == "key": key = action_input["text"] # e.g. "Return", "ctrl+c", "Tab" subprocess.run(["xdotool", "key", key], check=True) elif action == "mouse_move": x, y = action_input["coordinate"] subprocess.run(["xdotool", "mousemove", str(x), str(y)], check=True) elif action == "scroll": x, y = action_input["coordinate"] direction = action_input["direction"] # "up" | "down" | "left" | "right" amount = action_input.get("amount", 3) btn = {"up": "4", "down": "5", "left": "6", "right": "7"}[direction] subprocess.run( f"xdotool mousemove {x} {y} click --repeat {amount} {btn}", shell=True, check=True ) time.sleep(0.3) # let UI settle before screenshotting return take_screenshot() def execute_bash_action(cmd: str) -> str: """Run a shell command and return combined stdout + stderr.""" result = subprocess.run( cmd, shell=True, capture_output=True, text=True, timeout=30 ) output = (result.stdout + result.stderr).strip() return output[:8000] if output else "(no output)" def run_computer_use_agent(task: str, max_iterations: int = 20) -> str: """Run a computer use task. Returns final status message.""" tools = [ { "type": "computer_20250124", "name": "computer", "display_width_px": 1920, "display_height_px": 1080, "display_number": 1, }, {"type": "bash_20250124", "name": "bash"}, {"type": "text_editor_20250429", "name": "str_replace_based_edit_tool"}, ] screenshot = take_screenshot() messages = [{ "role": "user", "content": [ { "type": "image", "source": { "type": "base64", "media_type": "image/png", "data": screenshot, }, }, {"type": "text", "text": task}, ], }] for iteration in range(max_iterations): response = client.messages.create( model="claude-opus-4-5", # use a capable vision model max_tokens=4096, tools=tools, messages=messages, betas=["computer-use-2025-01-24"], # required beta header ) messages.append({"role": "assistant", "content": response.content}) print(f"[{iteration+1}] stop_reason={response.stop_reason}") if response.stop_reason == "end_turn": final_text = next( (b.text for b in response.content if hasattr(b, "text")), "Done." ) return final_text # Collect and dispatch all tool_use blocks in this response tool_results = [] for block in response.content: if block.type != "tool_use": continue print(f" tool={block.name} input={block.input}") if block.name == "computer": result_b64 = execute_computer_action(block.input) tool_results.append({ "type": "tool_result", "tool_use_id": block.id, "content": [{ "type": "image", "source": { "type": "base64", "media_type": "image/png", "data": result_b64, }, }], }) elif block.name == "bash": output = execute_bash_action(block.input["command"]) tool_results.append({ "type": "tool_result", "tool_use_id": block.id, "content": [{"type": "text", "text": output}], }) messages.append({"role": "user", "content": tool_results}) return f"Max iterations ({max_iterations}) reached without end_turn." if __name__ == "__main__": result = run_computer_use_agent( task="Open Firefox, navigate to https://example.com, and tell me the page title." ) print("Agent result:", result)
05 — Safety

Safety Considerations

Computer Use gives Claude real OS-level power. A single miscommunicated instruction can delete files, submit forms, or exfiltrate data. Treat every run as potentially destructive until proven otherwise.

⚠️
Never run on your main machine. Always use a dedicated VM, Docker container, or cloud sandbox (e.g., Anthropic's reference Docker image). Assume the agent can reach your entire home directory, browser cookies, and SSH keys if not isolated.
Principle of least privilege. Run the agent as a non-root user with no sudo. Mount the filesystem read-only except for the working directory. Disable clipboard access and limit network egress to the specific domains the task requires.
Human-in-the-loop checkpoints. Before any destructive action — file deletion, form submission, database write, git push — pause and prompt the user for approval. Implement a confirmation wrapper around high-risk bash commands.
Rate-limit and add hard timeouts. Each API call involves vision inference — enforce a per-task budget (e.g., max 20 iterations, max 5 minutes wall time). Without limits, a stuck loop can burn significant API credits and leave processes hanging.
Scrub screenshots before logging. Screenshots may contain passwords, auth tokens, or PII visible on screen. If you log screenshots for debugging, either disable logging by default or run them through a PII-detection filter before persisting.
Prompt injection via screen content. Websites or documents can display text like "Ignore all previous instructions." The model may follow these instructions. Sanitize untrusted display content or use a separate confirmation step before acting on information from the screen.
06 — Use Cases

Where Computer Use Shines

Computer Use is best applied where no programmatic alternative exists — legacy GUIs, visually-oriented tasks, and scenarios where maintaining a custom scraper or driver is costlier than agent inference.

🏛️
Legacy GUI Automation
Automate desktop apps with no REST API — ERP systems, government portals, insurance platforms. Computer Use handles any GUI that a human can operate.
🧪
Visual E2E Testing
Describe test cases in plain language. Claude navigates the app, verifies visual states, and reports failures — no Selenium selectors to maintain when the UI changes.
🔍
Web Research & Extraction
Extract data from sites with heavy JavaScript, anti-scraping, or login walls. Claude reads what it sees, not what the DOM exposes — immune to selector changes.
🎯
IT Helpdesk Automation
Resolve tier-1 tickets autonomously — reset passwords, clear caches, update user accounts in tools that only expose a web UI. Escalate to humans when uncertain.
♿
Accessibility Auditing
Navigate your app as a keyboard-only or screen-reader user would. Claude identifies unreachable elements, missing ARIA labels, and focus trap failures that automated tools miss.
🤖
Autonomous Dev Tasks
Point Claude at a repo, a failing CI screenshot, or a bug report, and let it open a terminal, reproduce the error, write a fix, and run the test suite — all inside a sandbox VM.
07 — Comparison

Computer Use vs. API Tool Use

Both approaches let Claude take actions in the world — but they target very different problem shapes. Use the cheapest tool that solves your problem.

API Tool Use Computer Use
Interface JSON API / function calls GUI + Screenshots
Speed per action Fast (~100ms) Slow (2–5s per step)
Reliability High — deterministic Variable — UI can change
Setup cost Low — define schema High — sandbox, display, xdotool
Token cost Low — text only High — image per step
Requires API Yes No — works on any GUI
Best used when A programmatic API exists for the target system No API exists; system only exposes a GUI
Maintenance burden Low — API schema is stable Medium — UI redesigns can break flows
🧩
Hybrid pattern: Many production agents combine both. Use API tool use for structured data (databases, REST APIs) and Computer Use only for the parts of the workflow that exist behind a GUI. This keeps cost and latency low while covering legacy surfaces.
08 — Limitations

Current Limitations

Computer Use is powerful but not production-ready for all workflows. Know where it breaks before committing to an architecture that depends on it.

LATENCY
2–5 seconds per action due to vision inference. A 20-step workflow takes 1–2 minutes minimum. Not suitable for real-time or high-frequency automation.
COORDINATE PRECISION
Claude sometimes mis-clicks on small UI elements (sub-20px buttons, dense dropdowns). Use higher-resolution screenshots and scale-aware prompting to improve accuracy.
SYSTEM DIALOGS
OS-level modal dialogs (file pickers, permission prompts, UAC dialogs) are rendered by the compositor and may not appear in regular screenshots depending on your capture method.
DRAG & SCROLL RELIABILITY
Complex drag operations and scrolling through long pages to find content are error-prone. Consider adding explicit scroll commands with verification screenshots after each scroll.
DYNAMIC UI RELIABILITY
Heavily animated UIs, canvas-based apps, video players, and WebGL content are difficult to reason about from static screenshots. Claude may misidentify state mid-animation.
CONTEXT WINDOW GROWTH
Each screenshot adds ~500–1000 tokens. Long sessions approach context limits. Implement image pruning (keep only the last N screenshots) or periodic context summarization for extended tasks.

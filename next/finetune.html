01 — Strategy

Fine-tuning vs RAG vs Prompting

Before touching a GPU, decide if fine-tuning is even the right tool. The wrong choice wastes weeks. The decision framework: always try the cheapest option first.

⚠️
Golden Rule: Try prompting first → then RAG → then fine-tuning. Each step is 10–100x more expensive than the last. Most problems are solved at step 1.
✏️
Prompting
Use when: you need different behavior for a single task. Few-shot examples, chain-of-thought, structured output. Solves 80% of problems.
Cost: Near Zero
📚
RAG
Use when: the model lacks knowledge (documents, code repos, recent data). Retrieval injects context at runtime. Knowledge is updateable without retraining.
Cost: Low–Medium
🎛️
Fine-tuning
Use when: you need to change HOW the model responds — style, tone, output format, domain-specific terminology. Changes model behavior, not knowledge.
Cost: High
Capability Prompting RAG Fine-tuning
Add new knowledge ✗ Limited ✓ Yes ✗ Risky
Change output style/format ~ Partially ✗ No ✓ Yes
Domain terminology ~ Few-shot ~ With context ✓ Baked in
Update without retraining ✓ Instant ✓ Yes ✗ No
Reduce hallucinations ~ Somewhat ✓ With sources ✗ Can make worse
Inference cost reduction ✗ No ✗ No ✓ Shorter prompts

How Fine-tuning Works

Fine-tuning continues training a pre-trained model on your labeled dataset. The architecture stays identical — only the weights change.

WHAT CHANGES
  • ✓ Model weights (billions of float32 values)
  • ✓ Output style, tone, format preferences
  • ✓ Task-specific behavior patterns
  • ✓ Domain vocabulary emphasis
WHAT STAYS THE SAME
  • ✗ Model architecture (layers, heads)
  • ✗ Context window size
  • ✗ Tokenizer vocabulary
  • ✗ Base capabilities and reasoning
💡
Full fine-tuning updates all 7B+ parameters. For a 7B model that's 28GB of gradients in VRAM — you need an A100. LoRA (next section) reduces this to ~2M trainable parameters and fits on a consumer GPU.
03 — LoRA

LoRA Deep Dive

Low-Rank Adaptation — the technique that made fine-tuning accessible without a data center budget. Instead of updating all weights, LoRA adds tiny trainable matrices alongside frozen originals.

Parameter Count Comparison

Full Fine-tune (7B model)7,000,000,000
LoRA r=16 adapters~2,000,000
LoRA trains only 0.03% of parameters. The original weights are frozen — no catastrophic forgetting.
W = W₀ + B · A
W₀ = frozen original  |  B,A = tiny trainable matrices

The Math (Intuition)

A weight matrix W is (d × d). Instead of updating all d² params, LoRA decomposes the update ΔW into two small matrices:
  • Matrix A — shape (r × d), r = rank (small, e.g. 16)
  • Matrix B — shape (d × r), initialized to zero
  • Result — B·A has shape (d × d), only 2×d×r params trained
With d=4096, r=16: full update = 16M params. LoRA update = 131K params. 120x fewer.
PYTHON — LoRA with Hugging Face PEFT
from transformers import AutoModelForCausalLM, AutoTokenizer from peft import LoraConfig, get_peft_model, TaskType # Load base model model = AutoModelForCausalLM.from_pretrained( "mistralai/Mistral-7B-v0.1", torch_dtype=torch.float16, device_map="auto", ) # Configure LoRA lora_config = LoraConfig( r=16, # rank — higher = more capacity, more params lora_alpha=32, # scaling factor (usually 2x rank) target_modules=[ # which layers to apply LoRA to "q_proj", "v_proj", "k_proj", "o_proj", ], lora_dropout=0.05, bias="none", task_type=TaskType.CAUSAL_LM, ) # Wrap model with LoRA adapters model = get_peft_model(model, lora_config) # Print trainable parameter count model.print_trainable_parameters() # Output: trainable params: 2,097,152 || all params: 7,243,436,032 || trainable%: 0.03%
04 — Data

Building a Training Dataset

Data quality is everything. 50 perfect examples consistently outperform 5,000 mediocre ones. Garbage in, garbage out — this applies 10x harder with fine-tuning.

📊
Quality threshold: Every example should represent the ideal response to that prompt. If you wouldn't be proud to show it, don't include it. Start with 50–200 examples. You can always add more later.
JSONL — OpenAI Chat Fine-tune Format
# training_data.jsonl — one JSON object per line {"messages": [ {"role": "system", "content": "You are a Python code reviewer for FastAPI services. Always identify security issues first, then performance issues."}, {"role": "user", "content": "Review this endpoint: @app.get('/user/{id}') async def get_user(id: str): return db.query(f'SELECT * FROM users WHERE id={id}')"}, {"role": "assistant", "content": "**CRITICAL - SQL Injection**: Direct string interpolation in SQL query. Fix: use parameterized queries. **HIGH - No auth**: endpoint exposes PII without authentication. Add @require_auth decorator."} ]} {"messages": [ {"role": "system", "content": "You are a Python code reviewer for FastAPI services..."}, {"role": "user", "content": "Review: for item in items: result = await db.get(item.id)"}, {"role": "assistant", "content": "**HIGH - N+1 Query**: Fetching items one by one in a loop. Fix: use db.get_many([i.id for i in items]) for a single database round-trip."} ]}
PYTHON — Dataset Validation & Upload
import json, openai def validate_dataset(path: str) -> dict: errors = [] examples = [] with open(path) as f: for i, line in enumerate(f): try: ex = json.loads(line) messages = ex.get("messages", []) # Must have system + user + assistant roles = [m["role"] for m in messages] if "assistant" not in roles: errors.append(f"Line {i}: missing assistant message") # Check completion isn't too short assistant_msg = next(m for m in messages if m["role"] == "assistant") if len(assistant_msg["content"].split()) < 10: errors.append(f"Line {i}: completion suspiciously short") examples.append(ex) except Exception as e: errors.append(f"Line {i}: parse error — {e}") return {"valid": len(errors) == 0, "count": len(examples), "errors": errors} # Upload to OpenAI client = openai.OpenAI() with open("training_data.jsonl", "rb") as f: file = client.files.create(file=f, purpose="fine-tune") job = client.fine_tuning.jobs.create( training_file=file.id, model="gpt-4o-mini-2024-07-18", hyperparameters={"n_epochs": 3}, ) print(f"Job ID: {job.id}") # Poll with client.fine_tuning.jobs.retrieve(job.id)
05 — Use Cases

When Fine-tuning Actually Helps

Fine-tuning has specific sweet spots. Misapplying it doesn't just waste money — it can make your model worse than the base version.

✅
Fine-tuning WILL help
  • Consistent output format (always JSON with your schema)
  • Domain-specific terminology (medical, legal, financial jargon)
  • Brand voice / tone matching (formal, casual, company-specific)
  • Following complex multi-step instructions for specific workflows
  • Reducing need for long system prompts (bake it in)
  • Latency reduction via smaller specialized model
❌
Fine-tuning will NOT help
  • Adding new factual knowledge (use RAG instead)
  • Fixing hallucinations (may introduce more)
  • Recent events the model hasn't seen
  • One-off task adjustments (use prompting)
  • Complex reasoning improvements (use bigger model)
  • Problems solvable with better prompts
06 — Code

Full Fine-tune Pipeline with Hugging Face

A complete example using PEFT/LoRA on a custom dataset. Uses QLoRA (4-bit quantization + LoRA) for training on a single consumer GPU.

PYTHON — QLoRA Training Pipeline
import torch from datasets import Dataset from transformers import ( AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig, TrainingArguments, ) from peft import LoraConfig from trl import SFTTrainer # ── 1. Quantization config (4-bit: loads 7B model in ~5GB VRAM) ── bnb_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True, ) # ── 2. Load model + tokenizer ── model_name = "mistralai/Mistral-7B-Instruct-v0.2" tokenizer = AutoTokenizer.from_pretrained(model_name) tokenizer.pad_token = tokenizer.eos_token model = AutoModelForCausalLM.from_pretrained( model_name, quantization_config=bnb_config, device_map="auto", ) model.config.use_cache = False # ── 3. LoRA configuration ── peft_config = LoraConfig( r=16, lora_alpha=32, target_modules=["q_proj", "v_proj", "k_proj", "o_proj"], lora_dropout=0.05, bias="none", task_type="CAUSAL_LM", ) # ── 4. Prepare dataset ── def format_example(row): return {"text": ( f"[INST] {row['instruction']} [/INST] {row['output']}" )} raw_data = [ {"instruction": "Review this FastAPI endpoint for issues...", "output": "CRITICAL: SQL injection vulnerability..."}, # ... your training examples ] dataset = Dataset.from_list(raw_data).map(format_example) train_ds, eval_ds = dataset.train_test_split(test_size=0.1).values() # ── 5. Training arguments ── training_args = TrainingArguments( output_dir="./ft-code-reviewer", num_train_epochs=3, per_device_train_batch_size=4, gradient_accumulation_steps=4, # effective batch = 16 warmup_ratio=0.05, learning_rate=2e-4, fp16=True, evaluation_strategy="epoch", save_strategy="epoch", load_best_model_at_end=True, logging_steps=10, report_to="wandb", # track your training run ) # ── 6. Trainer ── trainer = SFTTrainer( model=model, train_dataset=train_ds, eval_dataset=eval_ds, peft_config=peft_config, dataset_text_field="text", max_seq_length=2048, tokenizer=tokenizer, args=training_args, ) trainer.train() trainer.save_model("./ft-code-reviewer-final") # ── 7. Evaluate on held-out examples ── def evaluate_model(model, tokenizer, test_examples): results = [] for ex in test_examples: inputs = tokenizer( f"[INST] {ex['instruction']} [/INST]", return_tensors="pt" ).to("cuda") with torch.no_grad(): outputs = model.generate( **inputs, max_new_tokens=512, temperature=0.1 ) prediction = tokenizer.decode(outputs[0], skip_special_tokens=True) results.append({"prediction": prediction, "expected": ex["output"]}) return results

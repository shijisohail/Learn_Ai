01 — Decision Guide

Why Local/Self-Hosted?

Self-hosting isn't always the right answer. Know the tradeoffs before committing to the operational complexity.

Go self-hosted when...
  • ✓ High volume makes per-token cost painful (>$5K/month)
  • ✓ Data cannot leave your infrastructure (HIPAA, GDPR)
  • ✓ You need fine-tuned models with proprietary weights
  • ✓ Offline operation is required (air-gapped environments)
  • ✓ Latency requirements are extremely tight (<100ms P99)
  • ✓ You want to experiment with cutting-edge open models
Stay with cloud APIs when...
  • ✓ You need Claude 3.5/4 or GPT-4 level capability
  • ✓ You don't want to manage GPU infrastructure
  • ✓ Volume is <$1K/month (self-hosting isn't worth it)
  • ✓ You need the absolute latest model versions
  • ✓ Your team has no MLOps experience
  • ✓ You need 99.9%+ uptime SLAs
02 — Local Dev

Ollama — Local Models in Minutes

Ollama makes running local models as simple as brew install. It exposes an OpenAI-compatible API — you can swap it in for your existing agent code by just changing one URL.

⚡
Drop-in replacement: Ollama's API is compatible with the OpenAI SDK. Change base_url to http://localhost:11434/v1 and model name to "llama3.2" — your existing agent code works without modification. Zero refactoring.
BASH — Install & Run
# Install Ollama (macOS/Linux) curl -fsSL https://ollama.com/install.sh | sh # Pull a model (downloads weights, ~4GB for 7B model) ollama pull llama3.2 # 3B params, 2GB — very fast ollama pull llama3.1:8b # 8B params, 5GB — good quality ollama pull codellama:13b # 13B code specialist ollama pull mistral:7b # Strong general model ollama pull phi3:mini # 3.8B, excellent for its size # Run inference (blocks, shows output) ollama run llama3.2 "Explain RAG in 2 sentences" # Serve API (runs in background, OpenAI-compatible) ollama serve & # starts on http://localhost:11434
PYTHON — Use Ollama with Your Existing Agent Code
from openai import AsyncOpenAI # This is the ONLY change needed to use Ollama instead of OpenAI client = AsyncOpenAI( base_url="http://localhost:11434/v1", api_key="ollama", # required by SDK but ignored by Ollama ) # Everything else is identical to your existing OpenAI code response = await client.chat.completions.create( model="llama3.2", # local model name messages=[{"role": "user", "content": "Hello from local model!"}], temperature=0.7, ) print(response.choices[0].message.content) # Streaming also works identically async with client.chat.completions.stream( model="llama3.2", messages=[{"role": "user", "content": "Write a poem"}], ) as stream: async for chunk in stream: print(chunk.choices[0].delta.content or "", end="", flush=True) # Use langchain with Ollama from langchain_ollama import ChatOllama llm = ChatOllama(model="llama3.2") result = llm.invoke("Summarize the key points of RAG")
03 — Production Inference

vLLM — Production-Grade Inference

vLLM is the industry standard for serving open models in production. PagedAttention makes KV cache management dramatically more efficient. Continuous batching means no waiting — requests are served as GPU capacity becomes available.

🚀
Performance: vLLM achieves 10–24x higher throughput vs naive HuggingFace inference. PagedAttention eliminates KV cache memory waste. Continuous batching eliminates idle GPU time. If you're serving >10 RPS, use vLLM.
BASH — Deploy Mistral-7B with vLLM
# Install vLLM (requires NVIDIA GPU) pip install vllm # Serve a model — exposes OpenAI-compatible API on port 8000 python -m vllm.entrypoints.openai.api_server \ --model mistralai/Mistral-7B-Instruct-v0.3 \ --host 0.0.0.0 \ --port 8000 \ --dtype bfloat16 \ --max-model-len 32768 \ --gpu-memory-utilization 0.90 # Docker deployment (production-ready) docker run --runtime nvidia --gpus all \ -v ~/.cache/huggingface:/root/.cache/huggingface \ -p 8000:8000 \ --ipc=host \ vllm/vllm-openai:latest \ --model mistralai/Mistral-7B-Instruct-v0.3 \ --tensor-parallel-size 1 # 1 GPU; set to 2 for 2x A100
PYTHON — vLLM Client (same as OpenAI SDK)
from openai import AsyncOpenAI import asyncio, time client = AsyncOpenAI( base_url="http://localhost:8000/v1", api_key="not-needed", # vLLM doesn't require auth by default ) async def benchmark_throughput(n_concurrent: int = 10): """Measure tokens/second with concurrent requests.""" prompt = "Explain the benefits of microservice architecture in detail." async def single_request() -> int: response = await client.chat.completions.create( model="mistralai/Mistral-7B-Instruct-v0.3", messages=[{"role": "user", "content": prompt}], max_tokens=256, ) return response.usage.completion_tokens start = time.monotonic() results = await asyncio.gather(*[single_request() for _ in range(n_concurrent)]) elapsed = time.monotonic() - start total_tokens = sum(results) print(f"Concurrent requests: {n_concurrent}") print(f"Total tokens generated: {total_tokens}") print(f"Throughput: {total_tokens/elapsed:.0f} tokens/sec") print(f"Avg latency: {elapsed/n_concurrent*1000:.0f}ms") asyncio.run(benchmark_throughput(20))
04 — HuggingFace

Hugging Face Pipeline

The transformers library gives you the most control: load any model, run inference, experiment with quantization. Essential for fine-tuning and research work.

PYTHON — 4-bit Quantized Inference
import torch from transformers import ( AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig, pipeline, ) # ── 4-bit quantization (QLoRA) — runs 7B model on consumer GPU ── quant_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True, ) model_name = "meta-llama/Meta-Llama-3.1-8B-Instruct" tokenizer = AutoTokenizer.from_pretrained(model_name) model = AutoModelForCausalLM.from_pretrained( model_name, quantization_config=quant_config, device_map="auto", # automatically distribute across available GPUs attn_implementation="flash_attention_2", # 2x faster attention (if supported) ) # ── Pipeline API (simpler, same performance) ── pipe = pipeline( "text-generation", model=model, tokenizer=tokenizer, torch_dtype=torch.bfloat16, device_map="auto", ) messages = [ {"role": "system", "content": "You are a helpful FastAPI expert."}, {"role": "user", "content": "How do I add rate limiting to a FastAPI endpoint?"}, ] outputs = pipe( messages, max_new_tokens=512, temperature=0.7, do_sample=True, pad_token_id=tokenizer.eos_token_id, ) print(outputs[0]["generated_text"][-1]["content"]) # ── Memory usage check ── print(f"GPU memory used: {torch.cuda.memory_allocated() / 1e9:.1f} GB") # 4-bit 8B model: ~5GB VRAM vs ~16GB for full precision
05 — Model Selection

Choosing a Model

The open model landscape moves fast. As of April 2026, these are the leading options by category. Always benchmark on your specific task — general benchmarks don't always predict performance on your workload.

Model Size VRAM needed Strengths Best for
Llama 3.1 8B 8B 5GB (4-bit) Strong reasoning, fast, multilingual General use
Llama 3.1 70B 70B 40GB Near-GPT-4 quality, best open model Production quality
Mistral 7B v0.3 7B 4.5GB (4-bit) Excellent quality/size ratio, function calling Lean production
Mixtral 8x7B 47B (8x7B MoE) 24GB 8 experts, only 2 active per token High quality, MoE
Qwen2.5 Coder 32B 32B 18GB (4-bit) Best code model, all languages Code generation
Phi-3 Mini 3.8B 2.5GB (4-bit) Remarkable quality for tiny size Edge/mobile
DeepSeek-R1 7B–671B Varies Chain-of-thought reasoning, math Complex reasoning
⚠️
Size is not everything: A well-prompted Mistral 7B often outperforms a poorly-prompted 70B model on specific tasks. Optimize your prompts before assuming you need a larger model. Only upgrade size when you've plateaued on prompt optimization.
06 — Full Stack

Full Stack Architecture

Production self-hosted LLM stack: vLLM serving + your agent framework + Prometheus metrics + Grafana dashboards. Docker Compose brings it all together.

4
Monitoring Layer
Prometheus (metrics scraping) + Grafana (dashboards). vLLM exports: tokens/sec, TTFT, queue depth, GPU utilization.
3
Agent Layer
Your FastAPI agent service. Uses OpenAI-compatible client pointed at vLLM. Same agent code as cloud — just different base_url.
2
Inference Layer
vLLM serving Mistral-7B (or your model). OpenAI-compatible API on :8000. Handles batching, KV cache, model parallelism.
1
Hardware Layer
NVIDIA GPU (RTX 4090 for 7B, A100 for 70B). CUDA 12+. 32GB+ system RAM. NVMe SSD for model weights.

PYTHON — Agent Using Local vLLM (same code, different URL)
import os from openai import AsyncOpenAI # Config driven by environment variables # Local: LLM_BASE_URL=http://localhost:8000/v1, LLM_MODEL=mistral-7b # Cloud: LLM_BASE_URL=https://api.openai.com/v1, LLM_MODEL=gpt-4o-mini client = AsyncOpenAI( base_url=os.getenv("LLM_BASE_URL", "https://api.openai.com/v1"), api_key=os.getenv("OPENAI_API_KEY", "local"), ) MODEL = os.getenv("LLM_MODEL", "gpt-4o-mini") # Everything else in your agent code is identical # Switching from cloud to local is just changing two env vars async def ask(question: str) -> str: response = await client.chat.completions.create( model=MODEL, messages=[{"role": "user", "content": question}], max_tokens=512, ) return response.choices[0].message.content

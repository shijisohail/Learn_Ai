Section 01

Offline vs Online — Two Different Games

Confusing leaderboard gains with business impact is the most common failure mode. These two evaluation loops serve different purposes and must both be run.

Offline Eval

Before you ship

  • Speed: Fast — seconds to hours
  • Control: Full control over inputs, golden sets
  • Good for: Regression testing, safety, capability checks
  • Blind spots: Doesn't capture user behaviour, latency effects, novelty effect, or traffic distribution shift
Online A/B

After you ship (to a slice)

  • Speed: Days to weeks (needs statistical power)
  • Control: Random assignment; real traffic
  • Good for: Conversion, retention, revenue, latency in practice
  • Blind spots: Network effects, novelty, equilibrium shifts post-ship

Concept check

Your offline eval shows 15% lower hallucination rate with the new prompt. What should you conclude?

Correct: offline evals are necessary for regression safety but not sufficient. Ship to a traffic slice and measure OEC (overall evaluation criterion) in production.

Section 02

Sample Size Calculator

The most common A/B mistake is stopping early. Pre-compute required sample size before you start collecting data and commit to the stopping rule.

💡

The Peeking Problem

If you check results daily and stop when p < 0.05, your false-positive rate explodes to ~30% even at α=0.05. Fix: pre-register sample size and look only once, or use sequential testing (Bayesian or SPRT).

⚠

LLM-specific challenge: what is your conversion rate?

LLM apps often measure proxy metrics (thumbs-up rate, re-prompt rate, session length) rather than direct conversion. Proxy metrics have lower signal-to-noise — you need much larger samples to detect the same underlying effect. When in doubt, run longer.

Section 03

Experiment Validity Checklist

Tick each item before you start collecting data. An experiment with structural biases gives you false confidence — worse than no experiment.

Pre-flight readiness 0 / 8
Single independent variable
Only one thing differs between control and treatment — model, prompt, OR feature, never multiple at once.
Identical infrastructure
Both arms use the same servers, routing, and cache configuration. Latency differences confound quality perception.
Sample size pre-computed
Required n is calculated before data collection. Stopping rule is fixed and agreed upon.
Unit of randomization matches unit of analysis
Randomize users, not requests. One user in both arms = contamination.
Cost & latency tracked as co-primary metrics
A faster / cheaper variant that degrades quality is not a win. Track all three.
Duration covers at least one full weekly cycle
Weekly seasonality (Mon–Fri peaks, weekend troughs) contaminates experiments run for only 3 days.
Guardrail metrics defined
Name metrics that must NOT regress (error rate, refusal rate, per-request cost). Auto-abort if guardrail trips.
Rollback plan documented
One-click revert to control via feature flag. Not a runbook you write when something goes wrong.
Concept check

You ran your A/B test for 3 days, got p=0.04, and shipped. Two weeks later, the lift disappeared. Most likely cause?

Classic combination: 3-day test misses the weekly cycle and captures novelty effect (users engage more with anything new). Run for at least 7 days, ideally 14.

Section 04

When A/B is (Causally) Enough

Random assignment is powerful but not magic. Understanding what it does and doesn't control for lets you design better experiments and interpret results correctly.

Causal DAG — why randomisation blocks confounders

Randomisation breaks the arrow from Confounder → Treatment, making Treatment assignment independent of any pre-existing user trait. The confounder still affects the Outcome — but equally in both arms, so it cancels in the difference.

📚

When you can't randomise: causal inference methods

Propensity score matching, diff-in-diff (before/after in treated vs control region), and instrumental variables (IV) — these let you estimate causal effects from observational data. Needed when ethics, product constraints, or network effects prevent clean randomisation. Start with A/B; graduate to these only when necessary.

Section 05

The Experiment Lifecycle

Each stage has a checklist. Skipping early stages doesn't save time — it creates rework at the end.

1
Hypothesis
Define OEC and hypothesis
Name the primary metric (OEC), the expected direction, and a magnitude. "We believe prompt B will increase 7-day retention by 10% because it reduces hallucination in onboarding answers."
2
Design
Calculate n, pick split, set guardrails
Use the calculator above. Register your stopping rule — usually "analyse once at day N." Name guardrail metrics: cost/request must stay within 5% of control, error rate must not exceed baseline × 1.1.
3
Collect
Run the experiment (don't peek)
Monitor for guardrail trips only. If a guardrail trips, abort and roll back. Otherwise, don't touch the analysis until the pre-registered date. Log everything: prompt version, model, latency, tokens, cost.
4
Analyse
Run the pre-specified test once
Two-proportion z-test, t-test, or Mann-Whitney depending on metric distribution. Check practical significance (effect size) not just statistical significance. A p=0.001 with 0.01% lift is worthless.
5
Ship
Roll out gradually, monitor for novelty decay
10% → 50% → 100% with dashboard watching OEC and guardrails. Watch for novelty effect decay over 2–4 weeks post-ship. Log the experiment to MLflow (tag: experiment_id, variant, result).
Section 06

Python: Running the Statistics

from scipy import stats
import numpy as np

# Raw data: 1 = converted, 0 = didn't
ctrl_n, ctrl_conversions = 24_000, 744   # 3.1%
trt_n,  trt_conversions  = 24_000, 1_128  # 4.7%

# Two-proportion z-test (two-tailed)
count = np.array([ctrl_conversions, trt_conversions])
nobs  = np.array([ctrl_n, trt_n])
z, p = stats.proportions_ztest(count, nobs, alternative='two-sided')

print(f"z = {z:.3f}, p = {p:.4f}")  # z = 8.91, p = 0.0000

# Practical significance: absolute lift and relative lift
p_ctrl = ctrl_conversions / ctrl_n
p_trt  = trt_conversions  / trt_n
abs_lift = p_trt - p_ctrl
rel_lift = abs_lift / p_ctrl

print(f"Absolute lift: {abs_lift:.2%}")   # +1.60 pp
print(f"Relative lift: {rel_lift:.1%}")    # +51.6%

# 95% CI on absolute lift
se = np.sqrt(p_ctrl*(1-p_ctrl)/ctrl_n + p_trt*(1-p_trt)/trt_n)
ci = (abs_lift - 1.96*se, abs_lift + 1.96*se)
print(f"95% CI: [{ci[0]:.2%}, {ci[1]:.2%}]")  # [+1.25 pp, +1.95 pp]
from statsmodels.stats.power import zt_ind_solve_power
from statsmodels.stats.proportion import proportion_effectsize

def sample_size_for_proportions(
    baseline: float,
    mde_relative: float,  # e.g. 0.20 for 20% relative lift
    power: float = 0.80,
    alpha: float = 0.05,
) -> int:
    p1 = baseline
    p2 = baseline * (1 + mde_relative)
    effect_size = proportion_effectsize(p1, p2)
    n = zt_ind_solve_power(
        effect_size=effect_size,
        alpha=alpha,
        power=power,
        alternative='two-sided',
    )
    return int(np.ceil(n))

n_per_arm = sample_size_for_proportions(
    baseline=0.05,  # 5% baseline conversion
    mde_relative=0.20,  # detect ≥ 20% relative lift
)
print(f"n per arm: {n_per_arm:,}")   # ~3,840
print(f"Total: {n_per_arm*2:,}")       # ~7,680

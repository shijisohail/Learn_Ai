Section 01

Discovery — From Vague Idea to a Testable Problem

Before architecture: who pays, what decision gets better, and what "good" means in numbers. If you can't state a baseline metric, you can't prove the AI improved anything.

⚠

The "feels smarter" trap

If the only stated benefit of your AI feature is "it feels smarter" or "users will love it," you have not done discovery. Keep asking: what specific decision does a specific user make better or faster, and how much does the current wrong answer cost them in dollars, time, or trust?

Discovery framework — click each stage

👥 User & job-to-be-done
Who is the user and what outcome are they trying to achieve? NOT "uses the product" but: "SDR at a 50-person SaaS who needs to prioritise which of 800 leads to call this week". The more specific, the easier it is to know whether your AI actually helps.
🚨 Current workaround
What are they doing today without your feature? "They sort the Salesforce list by last-activity date and call the top 20." Understanding the workaround reveals the real cost bar: your AI must be meaningfully better than this, not just technically impressive.
💰 Cost of the wrong decision
If the workaround produces a wrong answer, what is the dollar / time / trust cost? "Calling a cold lead instead of a hot lead wastes 20 min and loses a potential deal worth $5k ARR." This number becomes your value ceiling — your AI cannot be worth more than the cost of the problem.
📈 Success metric
Name the metric that proves you solved the problem. "SDR connects-to-close rate increases by 15% within 60 days." This becomes your OEC for the A/B test and your north-star for the eval set. If you can't name it now, your launch gate will be arbitrary.
✅ One-sentence problem statement
"We are building for [specific user] who needs to [job-to-be-done] and currently [workaround], causing [cost]. We know we've solved this when [metric]." Write this before any architecture conversation. If you can't fill this in, you're not ready to build.
Concept check

Which of these is a testable AI product hypothesis?

Correct: a testable hypothesis names a user, a measurable outcome, and a magnitude. The first is aspirational, the third is a technical task — neither can be disproved.

Section 02

What Goes in an AI PRD

Same structure as any PRD, but with LLM-specific contracts that you must nail before architecture decisions are made. Click each section to see what belongs there.

  • Primary OEC: The one metric that proves this worked (from discovery)
  • Guardrail metrics: Must NOT regress (cost/request, error rate, latency, refusal rate)
  • Leading indicators: Proxy metrics available before OEC can be measured (session length, re-prompt rate, thumbs-up rate)
  • Explicitly list what the AI will NOT do. "The assistant will not answer questions outside the billing domain."
  • Scope creep in AI products is uniquely dangerous because LLMs will helpfully try to answer everything
  • Non-goals become your refusal tests in the eval set
  • Golden eval set: 50–200 input/expected-output pairs, version-controlled in git
  • Precision/recall targets: e.g. "precision ≥ 0.92 on factual queries, recall ≥ 0.88 on refusal cases"
  • Human rubric: Tone, citation format, length limits — things automated evals miss
  • Regression gate: Eval must pass before any model or prompt change ships
  • Latency SLO: p95 end-to-end (include retrieval + LLM + post-processing)
  • Cost envelope: Max $/request or $/user/month — derived from unit economics
  • Max tokens: Context window, max completion length
  • Regions / data residency: EU data must not leave EU, etc.
  • PII handling: What user data reaches the LLM API? What's stripped?
  • Hallucination: What happens when the model confidently answers incorrectly? (citations, confidence score, human review tier)
  • Prompt injection: Can user input override system prompt behaviour?
  • Tool abuse: Which tools can the agent call, and what's the blast radius of a wrong call?
  • Vendor lock-in: What's the migration cost if the model provider changes pricing overnight?

from dataclasses import dataclass

@dataclass
class ModelPricing:
    name: str
    price_in_per_1m: float   # $ per 1M input tokens
    price_out_per_1m: float  # $ per 1M output tokens

MODELS = {
    'haiku':  ModelPricing('claude-3-5-haiku-20241022',  0.80,  4.0),
    'sonnet': ModelPricing('claude-sonnet-4-5',           3.0,  15.0),
    'opus':   ModelPricing('claude-opus-4-5',             15.0, 75.0),
}

def cost_per_request(model_key: str, in_tok: int, out_tok: int) -> float:
    m = MODELS[model_key]
    return (in_tok * m.price_in_per_1m + out_tok * m.price_out_per_1m) / 1_000_000

def monthly_ai_cost(mau: int, calls_per_user: int, **kw) -> float:
    return mau * calls_per_user * cost_per_request(**kw)

# Example: 50k MAU, 50 calls/user/mo, 800 in + 400 out tokens
monthly = monthly_ai_cost(50_000, 50, model_key='haiku', in_tok=800, out_tok=400)
print(f"Monthly: ${monthly:,.0f}")  # ~$240 — very manageable

# Same traffic with Opus instead:
monthly_opus = monthly_ai_cost(50_000, 50, model_key='opus', in_tok=800, out_tok=400)
print(f"Opus: ${monthly_opus:,.0f}")   # ~$8,400 — 35× more expensive
from anthropic import Anthropic

client = Anthropic()

def classify_complexity(user_query: str) -> str:
    # Use the cheapest model to classify query complexity
    resp = client.messages.create(
        model="claude-3-5-haiku-20241022",
        max_tokens=10,
        messages=[{"role": "user",
                   "content": f"Reply 'simple' or 'complex' only. Query: {user_query}"}]
    )
    return resp.content[0].text.strip().lower()

def routed_completion(user_query: str, system: str) -> str:
    complexity = classify_complexity(user_query)
    model = (
        "claude-3-5-haiku-20241022" if complexity == "simple"
        else "claude-sonnet-4-5"     # reserve Sonnet for complex
    )
    resp = client.messages.create(
        model=model, max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": user_query}]
    )
    return resp.content[0].text

import anthropic

client = anthropic.Anthropic()

# Long system prompts repeated across many users = expensive
# Prompt caching stores the KV cache server-side for 5 minutes
# Cache hit: ~90% cheaper on input tokens

SYSTEM_PROMPT = """You are a billing assistant for Acme SaaS...
[2000 tokens of policy, examples, and context]"""

response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system=[{
        "type": "text",
        "text": SYSTEM_PROMPT,
        "cache_control": {"type": "ephemeral"}  # ← mark for caching
    }],
    messages=[{"role": "user", "content": user_query}]

# cache_read_input_tokens: tokens served from cache (cheap)

Section 04

Launch Gate Checklist

This is a gate, not a guarantee. Each item that's unchecked at launch is a known risk you're accepting. Check them off as you complete them.

Golden eval set versioned in git Eval
≥ 50 input/output pairs. Runs automatically on every prompt or model change. Gating CI step.
Observability dashboards live Ops
Latency (p50/p95/p99), error rate, cost per request, refusal rate — all live before users hit production.
Feature flag + one-click rollback Ops
Can disable the AI feature in under 2 minutes without a deploy. Verified that the rollback path works in staging.
Model version pinned Ops
Using a dated model ID (e.g. claude-3-5-haiku-20241022), not a floating alias that can silently update.
Tenant isolation verified Security
One tenant's data cannot appear in another tenant's context. Tested with two test accounts, confirmed no leakage.
PII scrubbed before LLM API call Security
Names, emails, SSNs, and card numbers are masked or omitted. Data classification documented in the PRD.
Cost per request within budget Cost
At 10× current traffic, monthly AI cost stays below 10% of ARPU. Verified with the calculator above at projected peak.
Prompt injection tested Security
"Ignore previous instructions and…" variants tested. System prompt cannot be overridden via user input. Constrained output format enforced.
Graceful degradation defined Ops
What happens when the LLM API is down? Fallback message, retry, or queue? Users are informed — not left with a spinner.
Incident runbook written Ops
What to do when: empty responses, tool failures, rate limits, hallucination report from user. Runbook lives in the team wiki, not in your head.
A/B test designed (if metric-driven launch) Eval
OEC defined, sample size pre-computed, stopping rule fixed, guardrail metrics named. See the Experimentation deep dive.
Stakeholder sign-off on quality bar Eval
The quality rubric was reviewed and accepted by someone from legal, product, and the team that owns the metric. Not just engineering.

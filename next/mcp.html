Model Context
Protocol

Anthropic's open standard for connecting LLMs to external tools and data. Think of MCP like HTTP for AI tools — write the server once, and every agent framework, IDE, and desktop app can use it without custom wiring.

01 — Motivation

The N×M Integration Problem

Before MCP, every agent needed its own custom integration for every external tool. A Slack tool built for LangChain didn't work in Claude Desktop. A GitHub tool built for AutoGPT needed to be rewritten for every other framework.

✕ Before MCP — N×M integrations
Each agent framework re-implements every tool from scratch. 4 agents × 10 tools = 40 bespoke integrations to maintain.
Tool for Slack in LangGraph is useless in Claude Desktop. Zero portability across the ecosystem.
No standard schema for tool discovery. Each framework invents its own tool definition format.
Auth, error handling, streaming — every integration solves these separately, inconsistently.
Community can't collaborate. A great GitHub tool built by one team can't be shared meaningfully.
✓ With MCP — N+M integrations
Write the MCP server once. It works with any MCP client: Claude Desktop, Claude Code, LangGraph, Cursor, your custom agent. 4 clients + 10 servers = 14 things total.
Universal tool discovery via the MCP protocol. Clients ask servers what tools they expose; no hardcoding required.
Standard JSON Schema for inputs/outputs. Same definition format everywhere, validated consistently.
Transport layer is standardized (stdio / HTTP+SSE). Clients don't care how the server runs.
Community ecosystem: 100s of open-source MCP servers for databases, APIs, file systems, browsers.
💡
The HTTP analogy: Before HTTP, every app had custom network code to talk to every server. HTTP standardized the protocol so any browser could talk to any web server. MCP does the same for AI tools — standard protocol, plug-and-play ecosystem.
02 — Architecture

How MCP Fits Together

Three roles: the Host runs the LLM, the Client speaks the protocol, the Server exposes capabilities. Transport is either stdio (local process) or HTTP+SSE (remote).

MCP Architecture — stdio (local) and HTTP (remote) transports
MCP Host
The application running the LLM. Claude Desktop, Claude Code, your custom FastAPI agent, or any IDE plugin. It owns the conversation and invokes the MCP Client when tool use is needed.
MCP Client
Lives inside the Host. Speaks the MCP protocol: connects to servers, calls initialize, discovers available tools via tools/list, and dispatches tools/call requests.
MCP Server
Your code. Exposes tools, resources, and prompts over stdio (local subprocess) or HTTP+SSE (remote). Handles the actual work: querying your DB, calling your API, reading files.
03 — Build

Your First MCP Server in Python

The mcp Python SDK gives you decorators that map directly to the protocol's three capabilities: list tools, call a tool, and (optionally) list/read resources. Install with pip install mcp.

1
Create a Server instance with a name
The name is what clients see when they connect. It appears in Claude Desktop's tool list and in logs.
2
Decorate list_tools() to advertise capabilities
Return a list of types.Tool objects with name, description, and a JSON Schema for inputs. The client uses this to decide which tool to call.
3
Decorate call_tool() to handle execution
Receives the tool name and validated arguments. Returns a list of TextContent, ImageContent, or EmbeddedResource.
4
Run the server over stdio
The stdio_server() context manager wires up stdin/stdout as the transport. The client (Claude Desktop) spawns your script as a subprocess and communicates over the pipes.
PYTHON — server.py (full working example)
from mcp.server import Server from mcp.server.stdio import stdio_server from mcp import types app = Server("my-tools") # name visible to clients @app.list_tools() async def list_tools() -> list[types.Tool]: # Clients call this on connect to discover what you expose. # inputSchema is standard JSON Schema — validated before call_tool fires. return [ types.Tool( name="query_database", description="Run a read-only SQL query against the analytics database", inputSchema={ "type": "object", "properties": { "query": { "type": "string", "description": "SQL SELECT query to execute", } }, "required": ["query"], }, ), types.Tool( name="create_ticket", description="Create a Jira ticket in the engineering backlog", inputSchema={ "type": "object", "properties": { "title": {"type": "string"}, "description": {"type": "string"}, "priority": {"type": "string", "enum": ["low", "medium", "high"]}, }, "required": ["title"], }, ), ] @app.call_tool() async def call_tool(name: str, arguments: dict) -> list[types.TextContent]: if name == "query_database": # Real impl: get a DB connection, run query, return rows result = run_sql(arguments["query"]) return [types.TextContent(type="text", text=str(result))] elif name == "create_ticket": ticket_id = jira_create( title=arguments["title"], description=arguments.get("description", ""), priority=arguments.get("priority", "medium"), ) return [types.TextContent(type="text", text=f"Created ticket {ticket_id}")] raise ValueError(f"Unknown tool: {name}") async def main(): async with stdio_server() as (read, write): await app.run(read, write, app.create_initialization_options()) if __name__ == "__main__": import asyncio asyncio.run(main())
✅
FastMCP shortcut: The mcp package also includes FastMCP — a higher-level API that infers tool schemas from Python type hints and docstrings. Use it for rapid prototyping, drop down to Server when you need explicit schema control.
04 — Connect

Wiring Up Claude Desktop

Claude Desktop reads a JSON config at startup and spawns your server as a subprocess over stdio. No network, no ports — just processes communicating over pipes. The config file lives at ~/Library/Application Support/Claude/claude_desktop_config.json on macOS.

JSON — claude_desktop_config.json
{ "mcpServers": { // Local Python server — spawned as a subprocess over stdio "my-tools": { "command": "python", "args": ["/path/to/your/server.py"] }, // Using uv for isolated dependencies (recommended) "my-tools-uv": { "command": "uv", "args": ["run", "--with", "mcp", "python", "/path/to/server.py"] }, // With environment variables (for API keys, DB URLs) "my-tools-with-env": { "command": "python", "args": ["/path/to/server.py"], "env": { "DATABASE_URL": "postgresql://localhost/mydb", "JIRA_API_KEY": "your-key-here" } }, // Remote HTTP server (deployed MCP server) "remote-tools": { "url": "https://my-mcp-server.example.com/sse", "transport": "sse" } } }
⚠️
Security note: MCP servers run with the permissions of the Claude Desktop process. Never pass secrets directly in args — use the env block or a .env file loaded by your server at startup. For production remote servers, always add authentication to your HTTP transport.
05 — Primitives

Resources vs. Tools

MCP exposes three first-class primitives. Tools are actions. Resources are data the model can read. Prompts are reusable templates. Understanding the difference determines how you model your server's interface.

⚙️
Tools
Actions the model can invoke — side effects expected
Query a database and return rows
Send a Slack message or email
Create a GitHub issue or PR
Execute a shell command or script
Call an external REST API
Write or modify a file
Protocol flow: client sends tools/call with name + arguments → server executes → returns TextContent | ImageContent | EmbeddedResource
📄
Resources
Data the model can read — like a browsable file system
File contents at a given path
Database schema / table definitions
Documentation pages or wiki articles
Config files and environment state
Git log or diff output
API response snapshots / fixtures
Protocol flow: client sends resources/list to browse, then resources/read with a URI → server returns text | blob content
PYTHON — Adding resources to your server
from mcp import types @app.list_resources() async def list_resources() -> list[types.Resource]: return [ types.Resource( uri="db://schema/tables", name="Database Schema", description="Full schema of all tables in the analytics DB", mimeType="text/plain", ), types.Resource( uri="file:///docs/api-reference.md", name="API Reference", description="Internal REST API documentation", mimeType="text/markdown", ), ] @app.read_resource() async def read_resource(uri: str) -> str: if uri == "db://schema/tables": return get_db_schema() # returns a string elif uri.startswith("file://"): path = uri.removeprefix("file://") with open(path) as f: return f.read() raise ValueError(f"Unknown resource URI: {uri}")
06 — Transport

HTTP Transport for Remote Servers

stdio is perfect for local tools (spawned as subprocesses). When you want to deploy a shared MCP server your team accesses remotely, use HTTP+SSE transport. The server exposes an SSE endpoint; clients stream responses over it.

PYTHON — stdio server (subprocess, local)
from mcp.server.stdio import stdio_server # Claude Desktop spawns this script as a subprocess. # Communication happens over stdin/stdout — no network required. # One server instance per client connection. async def main(): async with stdio_server() as (read_stream, write_stream): await app.run( read_stream, write_stream, app.create_initialization_options(), ) asyncio.run(main())
🗸
Best for: local developer tools, file system access, personal automation. No auth needed, zero deployment overhead. Start here.
PYTHON — HTTP + SSE server (remote, deployable)
from mcp.server.sse import SseServerTransport from starlette.applications import Starlette from starlette.routing import Route import uvicorn # Build transport — clients POST to /messages, stream from /sse transport = SseServerTransport("/messages/") async def handle_sse(request): async with transport.connect_sse( request.scope, request.receive, request._send ) as (read_stream, write_stream): await app.run(read_stream, write_stream, app.create_initialization_options()) starlette_app = Starlette( routes=[ Route("/sse", endpoint=handle_sse), Route("/messages/", endpoint=transport.handle_post_message, methods=["POST"]), ] ) if __name__ == "__main__": uvicorn.run(starlette_app, host="0.0.0.0", port=8000)
🛠️
Best for: shared team servers, production deployments, multi-tenant scenarios. Add middleware for authentication (Bearer token, OAuth) before the SSE handler.
PYTHON — FastMCP (schema from type hints)
from mcp.server.fastmcp import FastMCP mcp = FastMCP("my-tools") @mcp.tool() async def query_database(query: str) -> str: """Run a read-only SQL query against the analytics database. Args: query: SQL SELECT query to execute """ result = run_sql(query) return str(result) @mcp.resource("db://schema") async def db_schema() -> str: """The full database schema.""" return get_db_schema() # FastMCP infers JSON Schema from type hints + docstrings. # Run with stdio: mcp.run() | Run with HTTP: mcp.run(transport="sse") if __name__ == "__main__": mcp.run()
⚡
Best for: rapid prototyping. FastMCP generates the full JSON Schema from Python type annotations and docstrings. You get full stdio or SSE support with one line.

08 — Decision Guide

MCP vs. Raw tool_use

As a backend engineer you already know tool_use — it's just JSON schemas passed in the API call. MCP is not a replacement: it's a distribution and discovery layer on top. Use the right one for the job.

Dimension Raw tool_use (API) MCP
Portability Agent-specific — defined in your API call, works only in that codebase Universal — any MCP client (Claude Desktop, Claude Code, Cursor, LangGraph) can use the same server
Tool discovery Hardcoded — you pass the tool schemas on every API request Dynamic — clients call tools/list at runtime and discover what's available automatically
Transport In-process — tools are functions in your app's memory Out-of-process — stdio subprocess or remote HTTP, tools run independently
Lifecycle Tool logic lives in your agent codebase and deploys with it Tool server is an independent process/service with its own deploy lifecycle
Ecosystem Your tools only — nothing community-contributed plugs in 100s of open-source servers ready to add with one config line
Streaming No built-in streaming from tools Native progress notifications and streaming content via SSE
Complexity Simple — a dict in your API call More setup — server process, transport config, protocol handshake
When to use Simple, internal tools unique to your agent. Prototyping. One codebase consuming the tools. Tools you want to share, reuse across frameworks, or that others should be able to plug in.
💡
Practical advice: Start with raw tool_use when building the first version of an agent. Once the tool is stable and you want to reuse it in Claude Desktop and your LangGraph pipeline and your FastAPI agent — wrap it in an MCP server. The marginal cost is small, the portability gain is massive.

MCP Protocol Flow (what happens under the hood)

1
Initialize
Client sends initialize with its protocol version. Server responds with its capabilities (tools, resources, prompts) and protocol version. Handshake establishes the session.
2
Tool Discovery
Client calls tools/list. Server returns the full tool registry with names, descriptions, and JSON Schema inputs. The LLM uses these schemas to decide which tool to call and how to structure arguments.
3
LLM decides to use a tool
The LLM returns a tool_use block (Anthropic API format) or equivalent. The Host/Client intercepts this and routes it to the appropriate MCP server.
4
Tool Invocation
Client sends tools/call with the tool name and validated arguments. Server executes the tool and returns content. Errors are surfaced as structured error responses, not exceptions.
5
Result injected back into context
Tool result is added to the conversation as a tool_result message. The LLM continues generating with the new information. The loop repeats until the LLM returns a final answer without a tool call.

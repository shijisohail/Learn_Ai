Section 01

Three Ways Work Arrives

Design from the trigger outward: what fires the run, what idempotency key you use, and where state lives between runs. Getting this wrong is almost impossible to fix after you ship.

⌛
Schedule / Cron
Examples: daily digest, stale-data refresh, nightly report
Idempotency key: tenant + date + job_type
Watch out: clock skew across replicas, DST jumps, missed runs after downtime (should a 2am job run at 3am if the server was down?)
Predictable cadence
🔗
Webhook / Event
Examples: CRM update, new filing, RSS item, Stripe event
Idempotency key: event_id from the source system
Watch out: replay attacks, duplicate deliveries (same event_id twice), out-of-order delivery
Real-time, external
📰
Queue / Stream
Examples: Kafka, SQS, Celery, RabbitMQ
Idempotency key: message key (Kafka) or deduplication_id (SQS)
Watch out: at-least-once delivery means you will process some messages twice — plan for it from day one
At-least-once delivery
Concept check

Your agent processes webhook events and sometimes sends two emails to the same user. Most likely root cause?

At-least-once delivery is the default for almost every message broker. Every external side-effect (email, Slack, DB write) must be keyed by a deterministic idempotency key derived from the business event.

Section 02

Idempotency — Run Twice, Pay Once

The idempotency key must be derived from business identity, not from random UUIDs generated per attempt. A random UUID is unique per attempt — useless for deduplication.

⚠

Wrong pattern: random key per delivery

key = str(uuid.uuid4()) — this is unique per attempt, so three deliveries of the same event create three different keys and all three side-effects fire. Your idempotency key must be stable: the same business event → the same key, always.

Interactive — same webhook payload delivered three times (simulating network retries).

import hashlib, json

def make_idempotency_key(payload: dict) -> str:
    # Stable key from business-identity fields ONLY
    identity = {
        "tenant_id": payload["tenant_id"],
        "event_type": payload["event_type"],
        "entity_id": payload["entity_id"],
        "event_date": payload["event_date"],  # or filing_id, not timestamp
    }
    raw = json.dumps(identity, sort_keys=True)
    return hashlib.sha256(raw.encode()).hexdigest()[:32]

def handle_event(payload: dict) -> dict:
    key = make_idempotency_key(payload)

    if already_processed(key):
        return {"status": "duplicate_ignored", "key": key}

    mark_processing(key)   # atomic upsert with status='in_progress'
    try:
        result = run_agent_pipeline(payload)
        mark_done(key, result)
        return {"status": "processed", "key": key}
    except Exception as e:
        mark_failed(key, str(e))
        raise
from sqlalchemy import text
from datetime import datetime, timedelta

# Schema: idempotency_keys(key TEXT PK, status TEXT,
#   created_at TIMESTAMP, expires_at TIMESTAMP, result JSONB)

def already_processed(key: str, db) -> bool:
    row = db.execute(
        text("SELECT status FROM idempotency_keys WHERE key=:k"),
        {"k": key}
    ).fetchone()
    if row is None: return False
    # Allow re-processing of failed/stuck records after TTL
    return row.status in ('done', 'in_progress')

def mark_processing(key: str, db):
    ttl = datetime.utcnow() + timedelta(hours=24)
    db.execute(text("""
        INSERT INTO idempotency_keys(key, status, created_at, expires_at)
        VALUES(:k, 'in_progress', now(), :exp)
        ON CONFLICT(key) DO NOTHING
    """), {"k": key, "exp": ttl})
    db.commit()
import redis, json
r = redis.Redis(decode_responses=True)

def already_processed(key: str) -> bool:
    return r.exists(f"idem:{key}") == 1

def mark_processing(key: str, ttl_hours: int = 24):
    # SET NX: only set if key doesn't exist (atomic)
    ok = r.set(f"idem:{key}", "in_progress",
               ex=ttl_hours*3600, nx=True)
    if not ok:
        raise DuplicateEventError(f"Already in-flight: {key}")

def mark_done(key: str, result: dict):
    r.set(f"idem:{key}", json.dumps(result), ex=7*24*3600)
    # 7-day TTL so you can replay the result without re-running
Section 03

The Agent as a State Machine

Every scheduled agent transitions through states. Modelling this explicitly prevents "stuck" runs and makes observability trivial — you can always answer "what is this agent doing right now?"

💡

Persist state in the DB, not in the worker process

If the worker crashes mid-run, the DB record shows status='running'. A watchdog process can detect stale running records (older than TTL) and either retry or alert. This is impossible if state lives only in-memory.

Section 04

Exponential Backoff with Jitter

When the LLM API or a downstream service is rate-limited, retrying immediately makes it worse (thundering herd). Exponential backoff gives the system time to recover. Jitter prevents all workers from retrying at the same instant.

Retry backoff visualizer — click to animate

retry.py
import time, random, anthropic
from typing import Callable, TypeVar

T = TypeVar('T')

def retry_with_backoff(
    fn: Callable[[], T],
    max_attempts: int = 5,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    jitter: float = 0.25,
) -> T:
    """Exponential backoff with full jitter for LLM API calls."""
    for attempt in range(max_attempts):
        try:
            return fn()
        except (anthropic.RateLimitError,
                anthropic.APITimeoutError) as e:
            if attempt == max_attempts - 1:
                raise
            # Exponential delay with full jitter
            delay = min(
                base_delay * (2 ** attempt) * (1 + random.random() * jitter),
                max_delay,
            )
            logger.warning(f"Rate limited (attempt {attempt+1}), sleeping {delay:.1f}s")
            time.sleep(delay)
        except anthropic.APIStatusError as e:
            if e.status_code in {400, 401, 403}:
                raise  # Don't retry client errors
            raise
Section 05

Failure Modes & Fixes

Background jobs fail silently. No user is watching. You need alerts, dead-letter queues, and self-healing patterns baked in from the start.

☢

Poison message

A malformed payload causes the worker to crash every time it processes it. Without a DLQ, the message re-queues forever, blocking all other work.

Fix: max delivery count → DLQ + alert on DLQ depth
⛈

Thundering herd

All tenants scheduled at :00 → 10,000 LLM calls in the first second. Rate-limit errors cascade.

Fix: jitter the schedule per tenant (hash(tenant_id) % 60 seconds)
⚽

LLM rate limits

Provider returns 429. Naive retry with no backoff amplifies the load and gets you banned.

Fix: exponential backoff + per-tenant token bucket + fallback to smaller model
🔒

Stale locks / stuck runs

Worker crashes mid-run; DB record stays status='running' forever. New runs are blocked forever.

Fix: heartbeat + TTL; watchdog marks as 'failed' if no heartbeat in N minutes
🗨

Silent tool failure

External API returns 200 with an error body. The agent accepts the "success" and writes bad data silently.

Fix: validate every tool response schema; distinguish "API returned" from "API returned correct data"
🌀

Missing runs after downtime

Server was down during the 2am cron window. Should the job run at restart time? Usually yes, with a catch-up mechanism.

Fix: use a job scheduler (APScheduler, Celery beat) with misfire grace period rather than raw cron
Concept check

Your job scheduler is set to cron `0 2 * * *` (2am daily) but the server was down from midnight to 4am. What happens when the server restarts at 4am with a raw crontab?

Raw cron has no misfire concept — the 2am window passed while the server was down, so the job simply doesn't run that day. Use APScheduler, Celery beat, or a cloud scheduler with misfire handling if this matters for your use case.

Agent-to-Agent
(A2A) Protocol

Google's open standard for cross-framework agent communication. Just as MCP standardizes how agents connect to tools, A2A standardizes how agents talk to each other — enabling a LangGraph orchestrator to delegate to a CrewAI specialist without bespoke glue code.

01 — The Problem

Why A2A Exists

Before A2A, every inter-agent integration was bespoke. If your team built a specialist in LangGraph and another team built one in AutoGen, connecting them required custom HTTP contracts, hand-rolled schemas, and shared documentation that drifted. Multiply by N teams and you get N² integration problems.

Before A2A

Each agent exposes a custom REST API with its own schema. Every orchestrator must know each specialist's exact contract. Changing one agent's interface breaks every caller. No discoverability — you can't ask "what can you do?"

After A2A

Every agent publishes a standard Agent Card at /.well-known/agent.json. Tasks follow a single wire format. Any compliant agent can be both a client and a server. Orchestrators discover capabilities at runtime.

💡
The mental model: MCP gave agents a standard way to reach tools and data. A2A gives agents a standard way to reach each other. They are complementary layers — most production systems will use both. MCP for tool calls, A2A for agent delegation.
02 — Architecture

How A2A and MCP Fit Together

An orchestrator agent uses MCP to reach tools and data, and A2A to delegate tasks to specialist agents. The specialist agents may themselves use MCP internally. The protocols stack without conflict.

03 — Agent Card

The Agent Card: Runtime Discovery

Every A2A-compliant agent publishes a JSON manifest at /.well-known/agent.json. This is the entry point — any orchestrator can fetch it to discover what the agent can do, whether it supports streaming, and how to reach it. Think of it as OpenAPI for an agent.

📄
Discovery pattern: Before sending any tasks, an orchestrator GETs /.well-known/agent.json to read the Agent Card. This tells it the available skills, capability flags (streaming, push notifications), and the task endpoint URL. No out-of-band documentation required.

04 — Client Side

Sending a Task from an Orchestrator

An orchestrator sends tasks via HTTP POST to the agent's task endpoint. The response carries a task ID and state. For short tasks the result is inline; for longer ones you poll or stream.

1
Fetch the Agent Card
GET https://agent-host/.well-known/agent.json — verify the agent supports the skill you need.
2
POST /tasks/send
Send a task object with a unique ID and a message containing one or more parts (text, file, data).
3
Poll or Stream for Result
If state is "working", poll GET /tasks/{id} or open an SSE stream at /tasks/{id}/subscribe.
4
Read Artifacts
When state reaches "completed", extract results from task.artifacts[].parts[].
orchestrator_client.py Python
import httpx import asyncio async def call_agent(agent_url: str, task_text: str) -> str: async with httpx.AsyncClient() as client: # 1. Send the task response = await client.post( f"{agent_url}/tasks/send", json={ "id": "task-001", "message": { "role": "user", "parts": [{"type": "text", "text": task_text}] } } ) task = response.json() # 2. Poll until terminal state while task["status"]["state"] not in ["completed", "failed"]: await asyncio.sleep(1) result = await client.get(f"{agent_url}/tasks/{task['id']}") task = result.json() # 3. Extract artifact text return task["artifacts"][0]["parts"][0]["text"] # Orchestrator delegates to specialist result = asyncio.run( call_agent( "https://code-reviewer.internal/a2a", "Review: def add(a, b): return a + b" ) ) print(result)
🔧
Streaming alternative: If the Agent Card shows "streaming": true, POST to /tasks/sendSubscribe instead. The response is a Server-Sent Events stream with incremental TaskStatusUpdateEvent and TaskArtifactUpdateEvent messages — useful for showing live progress in a UI.
05 — Server Side

Building an A2A Server with FastAPI

Any FastAPI (or any HTTP framework) service can become an A2A-compliant agent by implementing two endpoints: the Agent Card endpoint and the task endpoint. This is the server side of what the orchestrator calls.

FastAPI is a natural fit: You already know Pydantic models and async handlers. The A2A wire format is just JSON, so there is no special SDK required. Google's a2a-samples repo has reference Python implementations if you want a battle-tested starting point.
06 — Comparison

A2A vs MCP — When to Use Each

These protocols solve different problems and are designed to be used together, not to replace one another. The question is never "which one" — it is "which layer am I operating at."

MCP A2A
Connects Agent ↔ Tools / Data Agent ↔ Agent
Made by Anthropic Google
Announced November 2024 May 2025
Primary use case Database queries, file ops, API calls, search Specialist delegation, cross-framework multi-agent systems
Transport stdio (local) / HTTP + SSE (remote) HTTP / SSE (push optional)
Discovery Tool schema returned at connect time Agent Card at /.well-known/agent.json
Work unit Tool call (request / response) Task (stateful, async, streamable)
Are they complementary? Yes — use both. MCP for tools, A2A for agents.
🚀
Layered architecture: In a typical production system your orchestrator uses MCP to call tools (web search, code execution, databases) and uses A2A to delegate to specialist agents (legal reviewer, data analyst, code reviewer). Each specialist may itself use MCP internally. The layers nest cleanly.
07 — Status

Where A2A Is Today

A2A is early-stage. The spec is open and moving fast. Understanding it now puts you ahead of the curve before it becomes the default inter-agent contract.

Actively Evolving — April 2026

A2A was announced in May 2025 and is actively evolving. The spec and reference implementations are open-source at google-a2a/a2a-samples. Production adoption is early-stage — most teams are evaluating rather than deploying. LangGraph, CrewAI, and AutoGen all have experimental A2A adapters. The core wire format (Agent Card, tasks/send, artifacts) is stable enough to build on; the streaming and push-notification surfaces are still being refined. Watch this space.

👀
What to watch: Authentication (how agents verify each other's identity), capability negotiation (skill versioning), and standardized error codes are the open questions as of mid-2025. Framework-native A2A support (first-class LangGraph nodes, CrewAI task bridges) will be the unlock for mainstream adoption.

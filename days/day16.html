Section 01

AutoModel, AutoTokenizer & the Config System

HuggingFace's Auto classes introspect the model's config.json to instantiate the correct architecture without you specifying it explicitly.

The HuggingFace model hub stores every model as a directory containing config.json (architecture), tokenizer_config.json, weight files (.safetensors preferred over .bin), and a model card README. The AutoModel.from_pretrained() method reads the config, maps model_type to the correct class (e.g. "mistral" → MistralModel), and loads weights into the architecture.

Task-specific Auto classes add a task head on top: AutoModelForCausalLM adds an LM head for next-token prediction, AutoModelForSequenceClassification adds a linear classifier, and AutoModelForSeq2SeqLM handles encoder-decoder models like T5 and BART. Always use the task-specific class — the generic AutoModel returns the raw backbone without any output head.

The device_map="auto" argument enables multi-GPU model parallelism automatically — it shards layers across available devices based on VRAM. Combined with load_in_4bit=True from bitsandbytes, you can load 70B models on a single A100 80GB during inference.

Why This Matters

The Auto class system means switching between Mistral, LLaMA, Falcon, and Gemma requires zero architecture-specific code changes. The same three-line pattern loads any model — this is why HuggingFace became the de-facto standard for the LLM ecosystem. Building model-agnostic tooling means your code continues working as new architectures emerge.

automodel_patterns.py
from transformers import ( AutoTokenizer, AutoModelForCausalLM, AutoModelForSequenceClassification, pipeline ) import torch # ── Load any causal LM in 3 lines ──────────────────────────────── MODEL_ID = "mistralai/Mistral-7B-Instruct-v0.3" tokenizer = AutoTokenizer.from_pretrained(MODEL_ID) model = AutoModelForCausalLM.from_pretrained( MODEL_ID, device_map="auto", # auto multi-GPU sharding torch_dtype=torch.bfloat16, # half precision weights attn_implementation="flash_attention_2", # FA2 kernels ) # ── 4-bit quantization via bitsandbytes ────────────────────────── from transformers import BitsAndBytesConfig bnb_cfg = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True, # nested quantization bnb_4bit_quant_type="nf4", # NormalFloat4 — better than INT4 ) model_4bit = AutoModelForCausalLM.from_pretrained( MODEL_ID, quantization_config=bnb_cfg, device_map="auto" ) # ── Chat template formatting ───────────────────────────────────── messages = [ {"role": "system", "content": "You are a helpful AI assistant."}, {"role": "user", "content": "Explain LoRA in one paragraph."}, ] tokens = tokenizer.apply_chat_template( messages, add_generation_prompt=True, return_tensors="pt" ).to(model.device) with torch.no_grad(): output_ids = model.generate( tokens, max_new_tokens=256, temperature=0.7, do_sample=True, repetition_penalty=1.1, ) response = tokenizer.decode(output_ids[0], skip_special_tokens=True)
Section 02

Pipelines API: Production-Ready Inference

The pipeline() abstraction bundles tokenization, model forward pass, and output decoding into a single callable — ideal for prototyping and serving.

HuggingFace pipelines support 20+ NLP tasks out of the box: text generation, summarization, translation, question answering, named entity recognition, sentiment analysis, zero-shot classification, and more. Under the hood, each pipeline selects an appropriate default model if none is specified — always override this in production to pin a specific model version.

For batch inference in production, the key insight is that pipeline() supports batch_size parameter and can process a Python generator. This enables memory-efficient streaming over large datasets. Setting device=0 routes to GPU and device_map="auto" enables model parallelism for large models.

The text generation pipeline's return_full_text=False is critical — by default it concatenates prompt + completion, which is almost never what you want in a backend API. Always set this to False when extracting only the generated text.

pipelines_demo.py
from transformers import pipeline import torch # ── Text generation pipeline ───────────────────────────────────── gen = pipeline( "text-generation", model="mistralai/Mistral-7B-Instruct-v0.3", device_map="auto", torch_dtype=torch.bfloat16, ) result = gen( "Explain transformers in 2 sentences:", max_new_tokens=128, return_full_text=False, # only return generated text! temperature=0.8, do_sample=True, ) print(result[0]["generated_text"]) # ── Batch generation over a generator (memory efficient) ───────── def prompt_generator(items): for item in items: yield f"Summarize: {item}" results = gen(prompt_generator(my_texts), batch_size=8) # ── Zero-shot classification ────────────────────────────────────── clf = pipeline("zero-shot-classification", model="facebook/bart-large-mnli") clf( "The stock market crashed 10% today", candidate_labels=["finance", "sports", "technology"], ) # → {'labels': ['finance', 'technology', 'sports'], 'scores': [0.97, 0.02, 0.01]} # ── Embeddings for semantic search ─────────────────────────────── from sentence_transformers import SentenceTransformer embed = SentenceTransformer("BAAI/bge-large-en-v1.5") vecs = embed.encode(["Hello world", "Hi there"], normalize_embeddings=True) similarity = vecs[0] @ vecs[1] # dot product → cosine similarity (normalized)
Common Pitfall: Pipeline Memory Leak

Each call to pipeline() loads the model into GPU memory. Creating a new pipeline per-request in a FastAPI endpoint causes OOM within minutes. Always create the pipeline once at startup (module-level or lifespan event) and reuse it across requests. Use a thread lock or async queue if the pipeline isn't thread-safe.

Section 04

Model Hub: Navigation, Model Cards & GGUF

Navigating HuggingFace Hub efficiently requires understanding model card metadata, safetensors vs GGUF formats, and how to evaluate model quality before downloading 14+ GB.

Model cards are the primary quality signal on the hub. A good model card contains: the base model, training data description, evaluation benchmarks (MMLU, HellaSwag, HumanEval for code), chat template, example code, and known limitations. Cards lacking benchmark results or training data provenance should be treated with caution in production.

GGUF is the file format used by llama.cpp and Ollama for CPU/GPU inference. It encodes quantization directly into the file and supports mixed-precision (different layers quantized to different precisions). The naming convention: Q4_K_M means 4-bit with K-quant method and Medium quality — this is the sweet spot for most consumer hardware. Q8_0 is near-lossless, Q2_K is very aggressive compression.

Safetensors (`.safetensors`) is the modern replacement for `.bin` files. It prevents arbitrary code execution during loading (PyTorch pickle files can contain malicious code), supports memory-mapped lazy loading, and can be loaded in parallel. Always prefer safetensors when available, and verify checksums on Hub downloads in security-sensitive environments.

hub_utils.py
from huggingface_hub import snapshot_download, hf_hub_download, list_repo_files from huggingface_hub import HfApi # ── Download specific files (not full model) ───────────────────── api = HfApi() files = list(list_repo_files("mistralai/Mistral-7B-Instruct-v0.3")) print([f for f in files if f.endswith(".safetensors")]) # Download only config + tokenizer for inspection (not weights) config_path = hf_hub_download("mistralai/Mistral-7B-Instruct-v0.3", "config.json") # ── Search Hub programmatically ─────────────────────────────────── models = api.list_models( task="text-generation", library="transformers", sort="downloads", limit=10, ) for m in models: print(f"{m.id}: {m.downloads:,} downloads") # ── Pull GGUF with Ollama ───────────────────────────────────────── # Run these in terminal: # ollama pull mistral:7b-instruct-q4_K_M # ~4.1GB download # ollama pull llama3.1:8b-instruct-q4_K_M # ~4.7GB # ollama list # see local models # ollama run mistral --verbose # chat with token/s stats # ── Ollama Python client ────────────────────────────────────────── import ollama response = ollama.chat( model="mistral", messages=[{"role": "user", "content": "What is LoRA?"}], ) print(response["message"]["content"])

Section 05

Tokenizers Deep Dive

Tokenizers are a frequent source of subtle bugs. Understanding BPE, chat templates, and padding behavior is essential for correct model inputs.

Modern LLMs use Byte Pair Encoding (BPE) or Unigram tokenization. BPE starts with byte-level tokens and iteratively merges the most frequent adjacent pairs — this is why "tokenization" might encode as 3 tokens while "supercalifragilistic" encodes as 8. Token count varies significantly between tokenizers — never assume a word count approximates token count without testing on your specific model.

Chat templates are Jinja2 templates stored in tokenizer_config.json that format conversation turns into the exact special token sequence the model was trained with. Using the wrong template (e.g., LLaMA 2's [INST] format on Mistral) causes massive quality degradation. Always use tokenizer.apply_chat_template().

Common Pitfall: Left vs. Right Padding

For decoder-only (causal) LLMs, padding must be on the LEFT — the model is autoregressive and sees the rightmost token last. Padding on the right causes the model to attend to padding tokens before the actual content, producing garbage outputs. Set tokenizer.padding_side = "left" before any batched inference. HuggingFace sets this correctly by default for most models, but double-check when loading custom models.

tokenizer_deep_dive.py
from transformers import AutoTokenizer tok = AutoTokenizer.from_pretrained("mistralai/Mistral-7B-Instruct-v0.3") # ── Inspect tokenization ────────────────────────────────────────── text = "The capital of France is Paris." ids = tok.encode(text) tokens = tok.convert_ids_to_tokens(ids) print(f"Tokens: {tokens}") # ['▁The', '▁capital', ...] print(f"Token count: {len(ids)}") # 8 tokens for 6 words # ── Batch encoding with proper left-padding ─────────────────────── tok.padding_side = "left" # CRITICAL for decoder-only models tok.pad_token = tok.eos_token # common for models without explicit pad token batch = tok( ["Short text.", "A much longer piece of text that needs padding."], padding=True, truncation=True, max_length=512, return_tensors="pt", ) print(batch["attention_mask"]) # 0s on left (padding), 1s on right (real) # ── Count tokens in a string ────────────────────────────────────── def count_tokens(text: str) -> int: return len(tok.encode(text, add_special_tokens=False)) print(count_tokens("Hello world")) # 3 print(count_tokens("supercalifragilistic")) # 8

Streaming Inference & Production Patterns

Streaming token generation transforms user experience by printing tokens as they are produced rather than waiting for the full response. TextIteratorStreamer runs generation in a background thread and yields tokens to your response stream, enabling Server-Sent Events (SSE) for real-time output. This is the exact pattern used by HuggingFace's chat interfaces and most production LLM APIs.

The critical design insight: your FastAPI event loop must never block. Model generation is CPU/GPU-bound and can take seconds. Always run it in a thread pool. The streamer acts as a producer-consumer queue between the generation thread and your response coroutine.

streaming_api.py 
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from transformers import AutoModelForCausalLM, AutoTokenizer, TextIteratorStreamer
import threading, torch

app = FastAPI()
model = AutoModelForCausalLM.from_pretrained(
    "mistralai/Mistral-7B-Instruct-v0.3",
    torch_dtype=torch.bfloat16, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("mistralai/Mistral-7B-Instruct-v0.3")

@app.post("/generate")
async def generate(prompt: str, max_tokens: int = 512):
    inputs = tokenizer.apply_chat_template(
        [{"role": "user", "content": prompt}],
        return_tensors="pt", add_generation_prompt=True
    ).to(model.device)

    streamer = TextIteratorStreamer(tokenizer, skip_prompt=True, skip_special_tokens=True)
    gen_kwargs = dict(input_ids=inputs, streamer=streamer,
                      max_new_tokens=max_tokens, do_sample=True, temperature=0.7)

    # Non-blocking: generation runs in background thread
    thread = threading.Thread(target=model.generate, kwargs=gen_kwargs)
    thread.start()

    def token_stream():
        for token in streamer:
            yield f"data: {token}\n\n"  # SSE format
        yield "data: [DONE]\n\n"

    return StreamingResponse(token_stream(), media_type="text/event-stream")
Why this matters: Time To First Token (TTFT) is the critical UX metric, not total generation time. Users tolerate slow generation if text appears immediately. Streaming also enables client-side early stopping — if the user sees the answer, they can cancel the request and save compute cost.

GPTQ Quantization

GPTQ uses second-order Hessian approximation to find optimal quantization points per weight matrix. Unlike BitsAndBytes which quantizes naively, GPTQ minimizes the quantization error per layer by solving a small optimization problem, resulting in significantly better quality at 4-bit precision. The trade-off is quantization time (15-60 minutes for 7B models) vs. BitsAndBytes' on-the-fly approach.

gptq_quantize.py 
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from datasets import load_dataset

# Prepare calibration data (128 samples typical)
dataset = load_dataset("wikitext", "wikitext-2-raw-v1", split="train")
calibration = [tokenizer(text, return_tensors="pt")
               for text in dataset["text"][:128] if len(text) > 50]

quantize_config = BaseQuantizeConfig(
    bits=4,
    group_size=128,
    desc_act=True,    # activation ordering — better quality
    damp_percent=0.01 # Hessian damping factor
)
model = AutoGPTQForCausalLM.from_pretrained(
    "mistralai/Mistral-7B-v0.3", quantize_config
)
model.quantize(calibration)
model.save_quantized("./mistral-7b-gptq-4bit", use_safetensors=True)
Common pitfall: GPTQ with desc_act=True (activation ordering) gives better quality but is incompatible with some inference backends. Check your serving framework's GPTQ support before quantizing — vLLM requires desc_act=False for maximum throughput.

Safetensors & Model Hub Best Practices

Safetensors is HuggingFace's secure replacement for pickle-based .bin files. Pickle can execute arbitrary Python code on load — a critical supply chain security risk. Safetensors is a flat binary format: the header is a JSON dict mapping tensor names to byte offsets, then raw binary data. It supports zero-copy memory mapping and partial tensor loading, which means you can inspect a model's architecture without loading all weights into RAM.

safetensors_patterns.py 
from safetensors import safe_open
from safetensors.torch import save_file, load_file

# Save model weights securely
tensors = {k: v.cpu() for k, v in model.state_dict().items()}
save_file(tensors, "model.safetensors")

# Inspect model without loading weights
with safe_open("model.safetensors", framework="pt") as f:
    print(f.keys())       # All tensor names — no data loaded
    print(f.metadata())   # JSON header metadata

# Partial load — only one layer (great for debugging)
with safe_open("model.safetensors", framework="pt", device="cuda") as f:
    q_proj = f.get_tensor("model.layers.0.self_attn.q_proj.weight")

# Upload to Hub with access control
from huggingface_hub import HfApi
api = HfApi()
api.upload_folder(
    folder_path="./my-fine-tuned-model",
    repo_id="username/my-model",
    repo_type="model",
    private=True  # Gate access until ready
)
Why this matters: Never load .bin or .pt files from untrusted sources — they can execute code that exfiltrates your API keys, installs backdoors, or corrupts your system. When you see trust_remote_code=True, you are executing Python from the Hub. Only use it for models from trusted organizations (Mistral, Meta, Google, EleutherAI).

Model Merging with SLERP

SLERP (Spherical Linear Interpolation) combines two fine-tuned models without training by interpolating weights on a hypersphere rather than linearly. This preserves more of each model's "directional" information in weight space. Top-ranked models on the Open LLM Leaderboard are often merges — combining a coding model with an instruction-following model creates a hybrid that excels at both.

Chat Templates & Tokenizer Internals

Chat templates are Jinja2 templates that convert a list of message dicts into the model's expected prompt format. Every model family uses a different format: Mistral uses [INST] ... [/INST], Llama-3 uses <|start_header_id|>user<|end_header_id|>, Phi-3 uses <|user|>, and ChatML (used by Qwen, OpenChat) uses <|im_start|>user. Using the wrong format is the most common cause of degraded instruction-following after fine-tuning.

The HuggingFace tokenizer stores the chat template in its config and applies it automatically via apply_chat_template(). You can inspect, modify, and override the template. This matters when you fine-tune on one format and deploy on another — the model learned to associate template tokens with roles, and changing those tokens breaks the association.

chat_templates.py 
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("mistralai/Mistral-7B-Instruct-v0.3")

# Inspect the raw Jinja2 template
print(tokenizer.chat_template)
# {{ bos_token }}{% for message in messages %}...{% endfor %}

messages = [
    {"role": "system", "content": "You are a helpful Python expert."},
    {"role": "user", "content": "Explain list comprehensions."},
    {"role": "assistant", "content": "List comprehensions are..."},
    {"role": "user", "content": "Give me an example."},
]

# Training: include full conversation (don't add generation prompt)
training_text = tokenizer.apply_chat_template(
    messages, tokenize=False, add_generation_prompt=False
)

# Inference: add generation prompt so model knows to continue
inference_text = tokenizer.apply_chat_template(
    messages[:-1], tokenize=False, add_generation_prompt=True
)

# For datasets: only compute loss on assistant turns
def mask_user_turns(input_ids, labels, tokenizer):
    """Set user turn labels to -100 (ignored by CrossEntropyLoss)."""
    user_start = tokenizer.encode("[INST]", add_special_tokens=False)[0]
    user_end = tokenizer.encode("[/INST]", add_special_tokens=False)[0]
    in_user = False
    for i, tok in enumerate(input_ids):
        if tok == user_start: in_user = True
        if in_user: labels[i] = -100  # ignore in loss
        if tok == user_end: in_user = False
    return labels
Why this matters: Training on the full conversation including user turns (without masking) teaches the model to generate both user and assistant text, which causes the model to continue generating new "user" questions after an answer instead of stopping. Always mask user turns to -100 so loss is computed only on assistant responses.

BPE Tokenization Internals

Byte Pair Encoding builds a vocabulary by iteratively merging the most frequent byte pairs in the training corpus. The vocabulary size (typically 32K-200K tokens) controls the trade-off between sequence length (more tokens = longer sequences = more compute) and coverage (more tokens = better subword handling). LLaMA-3's 128K vocabulary significantly improves multilingual performance compared to LLaMA-2's 32K vocabulary.

tokenizer_analysis.py 
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B")

# Vocabulary size matters for multilingual support
print(f"Vocab size: {tok.vocab_size}")  # 128,256 for Llama-3

# Inspect tokenization of different text types
texts = {
    "English": "The quick brown fox",
    "Arabic": "الذكاء الاصطناعي",
    "Code": "def fibonacci(n): return n if n <= 1 else fibonacci(n-1) + fibonacci(n-2)",
    "Numbers": "1234567890.001",  # each digit often separate token!
}
for name, text in texts.items():
    tokens = tok.tokenize(text)
    print(f"{name}: {len(tokens)} tokens → {tokens}")

# Left-padding for batched inference
tok.padding_side = "left"  # CRITICAL for causal LM inference
batch = tok(
    ["Short prompt", "A much longer prompt that needs more tokens"],
    padding=True, return_tensors="pt"
)
# Left-padded: padding tokens are at the beginning, not end
# This ensures the attention mask correctly covers actual tokens
Common pitfall: Using padding_side="right" (the default) for batched inference causes wrong outputs. In decoder-only models, the position embeddings of real tokens are affected by padding position. Left-padding puts real tokens at the end where the model generates from, right-padding puts them in the middle. Always set tokenizer.padding_side = "left" before batched inference and "right" before training.

Vision Models, Embeddings & Multimodal Pipelines

HuggingFace supports vision-language models (VLMs) like LLaVA, Qwen-VL, and PaliGemma through the same AutoModel interface. You pass images alongside text, and the model reasons across both modalities. Embedding models (sentence-transformers, BGE, E5) are the backbone of RAG systems — they convert text to dense vectors that support semantic similarity search.

The embedding quality fundamentally determines RAG system quality. An embedding model that represents "coronary artery disease" and "myocardial infarction" as similar vectors will retrieve relevant documents even when the query uses different terminology than the documents. Domain-specific embedding models (fine-tuned on medical, legal, or code corpora) significantly outperform general-purpose models for specialized applications.

multimodal_pipeline.py 
from transformers import AutoProcessor, LlavaNextForConditionalGeneration
from PIL import Image
import torch, requests

# Load vision-language model
processor = AutoProcessor.from_pretrained("llava-hf/llava-v1.6-mistral-7b-hf")
model = LlavaNextForConditionalGeneration.from_pretrained(
    "llava-hf/llava-v1.6-mistral-7b-hf",
    torch_dtype=torch.bfloat16, device_map="auto"
)

def analyze_image(image_url: str, question: str) -> str:
    image = Image.open(requests.get(image_url, stream=True).raw)
    conversation = [{
        "role": "user",
        "content": [{"type": "image"}, {"type": "text", "text": question}]
    }]
    prompt = processor.apply_chat_template(conversation, add_generation_prompt=True)
    inputs = processor.__call__(text=prompt, images=image, return_tensors="pt").to(model.device)

    with torch.no_grad():
        output = model.generate(**inputs, max_new_tokens=256)

    return processor.decode(output[0][inputs.input_ids.shape[1]:], skip_special_tokens=True)

result = analyze_image("https://example.com/chart.png", "What trends does this chart show?")

embedding_pipeline.py 
from sentence_transformers import SentenceTransformer
from transformers import AutoTokenizer, AutoModel
import torch, torch.nn.functional as F

# sentence-transformers: easiest API
model = SentenceTransformer("BAAI/bge-large-en-v1.5")
embeddings = model.encode(
    ["Document 1", "Document 2"],
    normalize_embeddings=True,   # normalize for cosine similarity
    batch_size=32,
    show_progress_bar=True
)

# Manual HuggingFace approach — mean pooling
tokenizer = AutoTokenizer.from_pretrained("BAAI/bge-large-en-v1.5")
embed_model = AutoModel.from_pretrained("BAAI/bge-large-en-v1.5")

def mean_pool(model_output, attention_mask):
    token_embeddings = model_output.last_hidden_state
    mask_expanded = attention_mask.unsqueeze(-1).expand(token_embeddings.size()).float()
    return torch.sum(token_embeddings * mask_expanded, 1) / torch.clamp(mask_expanded.sum(1), min=1e-9)

inputs = tokenizer(["BGE requires this prefix for queries: " + q for q in queries],
                   padding=True, truncation=True, return_tensors="pt")
with torch.no_grad():
    outputs = embed_model(**inputs)
embeddings = F.normalize(mean_pool(outputs, inputs.attention_mask), p=2, dim=1)
Why this matters: BGE (BAAI General Embeddings) requires query-side instruction prefixes for optimal performance: "Represent this sentence for searching relevant passages: " for queries but no prefix for documents. Omitting this prefix can degrade retrieval quality by 10-20%. Always read the model card for embedding models — they frequently have required prompt templates.

Rerankers & Advanced RAG Retrieval

A two-stage retrieval pipeline dramatically improves RAG quality: first retrieve a large candidate set (top-50 documents) using fast approximate nearest neighbor search, then rerank the candidates using a cross-encoder model that reads the query and each document together (instead of independently). Cross-encoders (like BGE-Reranker, Cohere Rerank) achieve much higher precision because they model query-document interaction directly, rather than comparing independent embeddings.

The trade-off: bi-encoders (embedding models) encode query and documents independently — O(1) for retrieval after pre-encoding. Cross-encoders encode query+document together — O(N) for N documents, making them too slow for full corpus search. The two-stage approach combines both: fast bi-encoder recall, precise cross-encoder precision.

two_stage_retrieval.py 
from sentence_transformers import SentenceTransformer, CrossEncoder
import numpy as np

# Stage 1: Fast approximate retrieval (bi-encoder)
bi_encoder = SentenceTransformer("BAAI/bge-large-en-v1.5")

# Stage 2: Precise reranking (cross-encoder)
cross_encoder = CrossEncoder("BAAI/bge-reranker-large")

def two_stage_retrieve(query: str, corpus: list[str],
                       first_k: int = 50, final_k: int = 5) -> list[str]:
    """Two-stage retrieval: recall + precision."""
    # Stage 1: Get top-50 candidates via embedding similarity
    query_emb = bi_encoder.encode([query], normalize_embeddings=True)
    corpus_embs = bi_encoder.encode(corpus, normalize_embeddings=True)
    scores = (query_emb @ corpus_embs.T)[0]
    top_50_idx = np.argsort(scores)[::-1][:first_k]
    candidates = [corpus[i] for i in top_50_idx]

    # Stage 2: Rerank top-50 with cross-encoder
    pairs = [[query, doc] for doc in candidates]
    rerank_scores = cross_encoder.predict(pairs)
    reranked = sorted(zip(candidates, rerank_scores),
                      key=lambda x: x[1], reverse=True)
    return [doc for doc, _ in reranked[:final_k]]
Why this matters: In internal benchmarks, adding a reranker consistently improves RAG faithfulness by 10-20% with minimal latency cost (reranking 50 documents adds ~100ms). The bi-encoder recall at 50 candidates is typically 90%+ for the relevant document; the cross-encoder then correctly identifies which of those 50 is truly most relevant. This is the most cost-effective quality improvement for any RAG system.

Matryoshka Embeddings & Adaptive Retrieval

Matryoshka Representation Learning (MRL) trains embedding models so that truncating the embedding vector to smaller dimensions still produces high-quality representations. A 1024D Matryoshka embedding truncated to 256D gives 80-90% of full-dimension quality at 4× less memory and compute. This enables tiered retrieval: use 256D for fast candidate retrieval, then 1024D for precise reranking within that set.

matryoshka_retrieval.py 
from sentence_transformers import SentenceTransformer
import numpy as np

# nomic-embed-text-v1.5 supports Matryoshka dimensions
model = SentenceTransformer("nomic-ai/nomic-embed-text-v1.5", trust_remote_code=True)

docs = ["Document about Python...", "Document about PyTorch..."]

# Encode at full 768D
full_embs = model.encode(docs, normalize_embeddings=True)  # shape: (N, 768)

# Truncate + renormalize for fast 128D retrieval
fast_embs = full_embs[:, :128]
fast_embs = fast_embs / np.linalg.norm(fast_embs, axis=1, keepdims=True)

# Store both: fast_embs in HNSW index, full_embs for reranking
# Retrieval: query 128D → top 50 → rerank with full 768D

print(f"Memory: 768D={full_embs.nbytes//1024}KB, 128D={fast_embs.nbytes//1024}KB")
# Typical output: 768D=192KB, 128D=32KB for 1000 documents
Common pitfall: Not normalizing after truncation. When you slice the embedding vector, the L2 norm changes. If you compute cosine similarity without renormalizing, you'll get incorrect similarity scores that don't correspond to the true angular distance between embeddings. Always apply vec / np.linalg.norm(vec) after truncation.

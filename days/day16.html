Section 01

The HuggingFace Ecosystem

The Hub is the npm of ML — 500K+ models, datasets, and spaces. Understanding the ecosystem means knowing which library to reach for first.

  • Hub — 500K+ models, datasets, Spaces. Search by task, architecture, language, or license. Every model has a model card with architecture details and limitations.
  • transformers — the main library. AutoModel, AutoTokenizer, pipeline(). Works with PyTorch, TensorFlow, and JAX.
  • peft — parameter-efficient fine-tuning: LoRA, QLoRA, prefix tuning. Day 17 covers this in depth.
  • trl — transformer reinforcement learning: SFTTrainer, PPO, DPO. Used for RLHF and instruction tuning.
  • accelerate — distributed training abstraction. Write training code once, run on 1 GPU or 8.
  • HuggingFace Spaces — deploy Gradio / Streamlit apps in one click. Great for demos and sharing fine-tuned models.
📋
Reading model cards: always check (1) architecture — transformer decoder vs encoder, (2) training data — open vs proprietary, (3) limitations section — what the model fails at, (4) license — many models are non-commercial only.
Section 02

The pipeline() API

Fastest way to run any model — handles tokenization, inference, and decoding in one call. Graduate to AutoModel only when you need custom control over the process.

  • Task types — text-generation, text-classification, question-answering, summarization, translation, image-classification, object-detection, and more.
  • Pipeline config — model, device, batch_size, truncation, max_new_tokens. All set at init, not at call time.
  • Batch inference — pass a list of strings, get a list of results. Much more efficient than sequential single calls.
  • When to graduate — need access to hidden states, logits, attention weights, or custom generation logic.
PYTHON
from transformers import pipeline import torch # Text generation — simplest usage generator = pipeline( "text-generation", model="meta-llama/Llama-3.2-3B-Instruct", torch_dtype=torch.bfloat16, device_map="auto" ) result = generator( "Explain attention mechanism in transformers:", max_new_tokens=200, temperature=0.7, do_sample=True ) print(result[0]["generated_text"]) # Batch inference — process multiple inputs efficiently results = generator( ["Question 1:", "Question 2:", "Question 3:"], batch_size=4, max_new_tokens=100 )
Section 03

AutoModel & AutoTokenizer

The Auto-classes are model-agnostic loaders — write code once and it works for Llama, Mistral, Qwen, Phi, and any other supported architecture.

  • AutoTokenizer — BPE tokenization, padding, truncation, special tokens — all handled automatically per model config.
  • apply_chat_template() — converts a list of message dicts to the model-specific format. Critical for instruction-following models. Skip this and you get garbage outputs.
  • Sampling parameters — temperature (randomness), top_p (nucleus sampling), repetition_penalty (reduce loops).
  • Decoding only new tokens — slice output_ids with input length to get only the generated text, not the prompt repeated back.
PYTHON
from transformers import AutoModelForCausalLM, AutoTokenizer import torch model_name = "mistralai/Mistral-7B-Instruct-v0.3" tokenizer = AutoTokenizer.from_pretrained(model_name) model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.bfloat16, device_map="auto" ) # Chat template — model-specific formatting messages = [ {"role": "user", "content": "What is LoRA fine-tuning?"} ] formatted = tokenizer.apply_chat_template( messages, tokenize=False, add_generation_prompt=True ) # Tokenize and generate inputs = tokenizer(formatted, return_tensors="pt").to(model.device) with torch.no_grad(): output_ids = model.generate( **inputs, max_new_tokens=300, temperature=0.7, do_sample=True, top_p=0.9, repetition_penalty=1.1 ) # Decode only the new tokens new_tokens = output_ids[0][inputs["input_ids"].shape[1]:] response = tokenizer.decode(new_tokens, skip_special_tokens=True)
Section 04

Running Models Locally with Ollama

Ollama is a one-command local LLM server with an OpenAI-compatible API. Run Llama, Mistral, Qwen, and others without touching Python for the server setup.

  • Supported models — Llama 3.2, Mistral, Phi-3.5, Gemma 2, Qwen2.5, DeepSeek, and dozens more from the Ollama library.
  • Quantization levels — Q4_K_M is the recommended default (best quality/size tradeoff). Q8_0 for highest quality. Q2_K for smallest size.
  • OpenAI-compatible API — point any OpenAI client to localhost:11434 and it just works. No code changes needed.
  • LangChain integration — ChatOllama works as a drop-in replacement for ChatOpenAI in any LangChain chain.
BASH
# Install and run Ollama curl -fsSL https://ollama.ai/install.sh | sh ollama pull llama3.2:3b # pull 3B model (~2GB) ollama pull qwen2.5-coder:7b # coding specialist ollama serve # start API server (port 11434)
PYTHON
from langchain_ollama import ChatOllama from openai import OpenAI # LangChain integration llm = ChatOllama(model="llama3.2:3b", temperature=0.7) response = llm.invoke("Explain RLHF in simple terms") # OpenAI-compatible API (use existing OpenAI clients) client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama") response = client.chat.completions.create( model="llama3.2:3b", messages=[{"role": "user", "content": "Explain RLHF"}] )
Section 05

GGUF Quantized Models & llama.cpp

GGUF is the compressed model format that powers consumer-GPU and CPU inference. llama.cpp is the C++ engine underneath Ollama that makes it all fast.

  • GGUF — single-file model format. Self-contained: weights, tokenizer config, and metadata all in one file.
  • Quantization tradeoff — lower bits = smaller file + less VRAM, with some quality loss. Q4_K_M hits the sweet spot.
  • llama.cpp — C++ inference engine. Supports CPU, CUDA, Metal (Apple), Vulkan. Powers Ollama, LM Studio, and many local tools.
  • When to use GGUF — development and testing on consumer hardware, edge deployment, anywhere you can't run full BF16 weights.

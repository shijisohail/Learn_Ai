Section 01

CrewAI Architecture

CrewAI provides three core abstractions: Agent (a role-based LLM actor), Task (a unit of work with an expected output), and Crew (the orchestrator that runs agents against tasks).

🤖
Agent
The core actor. Defined by: role (expertise domain), goal (what it tries to achieve), backstory (context injected into system prompt), tools (functions it can call), and llm (which model backs it). Think of it as a specialized employee with a job description.
📋
Task
A unit of work. Defined by: description (detailed instructions), expected_output (format and completeness criteria), agent (who executes it), and optionally context (output from prior tasks) and output_file. Tasks are the unit of composition — they chain together via context.
👥
Crew
The orchestrator. Takes a list of agents and tasks, a process type (Sequential or Hierarchical), and optional memory configuration. .kickoff(inputs={}) runs the full workflow. Crew manages task routing, output passing, and error handling.
🔀
Process Types
Sequential: tasks run in the order defined, each receiving the output of the previous. Predictable, auditable. Hierarchical: a manager LLM reads all tasks and dynamically delegates to agents. More flexible but less deterministic — good for open-ended research tasks.
Framework Level of Abstraction Control Best For
CrewAI High — role/task/crew DSL Less granular Role-based workflows, content pipelines, research crews
LangGraph Low — explicit state machine Full control Complex conditional flows, HITL, production agents needing auditability
Raw agent loop None — build everything Maximum Simple single-agent ReAct loops, rapid prototyping
Section 02

Building Agents with Roles

The quality of your agents depends almost entirely on the quality of role, goal, and backstory. Vague roles produce vague outputs. Specific, expert-level backstories produce expert-level reasoning.

💡
Role Design Principle: Write backstories as if you were onboarding a senior contractor. Specify the years of experience, what they're exceptionally good at, what standards they hold themselves to, and what pitfalls they know to avoid. The LLM uses this context to calibrate tone, depth, and judgment in every response.
Python — Agent Definition
from crewai import Agent, Task, Crew, Process from crewai_tools import SerperDevTool, FileReadTool # Tool definition — SerperDev for web search search_tool = SerperDevTool() # Research agent — expensive model for quality researcher = Agent( role="Senior AI Research Analyst", goal="Find comprehensive, accurate information about {topic}", backstory="""You are an expert AI researcher with 10 years of experience in machine learning and NLP systems. You excel at finding cutting-edge papers, synthesizing complex technical information, and distinguishing between hype and substance. You always cite your sources and flag uncertainty when you encounter it.""", tools=[search_tool], llm="claude-3-5-sonnet-20241022", verbose=True, memory=True # enables short-term memory within the task ) # Writer agent — cheaper model, writing is less compute-intensive writer = Agent( role="Technical Content Writer", goal="Transform research into clear, engaging technical articles", backstory="""You are a technical writer who has spent years making complex AI concepts accessible to senior engineers. You write with precision and clarity, using concrete examples over abstractions. You structure content with a logical flow: problem → mechanism → implementation → gotchas.""", llm="gpt-4o-mini", # cheaper model for writing tasks verbose=True ) # Agent memory types: # short_term_memory — within a task execution (enabled by memory=True) # long_term_memory — persisted across crew runs (requires storage backend) # entity_memory — tracks entities mentioned (people, orgs, concepts)
Section 03

Tasks & Expected Outputs

A well-defined task is the key to reliable CrewAI pipelines. The expected_output field is not optional boilerplate — the agent uses it to understand the quality bar and stops when it believes it's met.

Python — Task Definition & Dependencies
from pydantic import BaseModel from typing import List # Research task — first in chain research_task = Task( description="""Research {topic} thoroughly. Find: 1. Current state of the art — what works best today 2. Key limitations and open challenges 3. Recent breakthrough papers (2024–2025) with arxiv links 4. Practical implementation considerations for production systems 5. Common failure modes reported in the literature""", expected_output="""A comprehensive research report containing: - Executive summary (3–4 sentences) - Findings for each of the 5 areas above - Minimum 3 paper citations with arxiv IDs - A "Practical Implications" section for ML engineers""", agent=researcher, output_file="research.md" # save output to disk ) # Write task — depends on research task output write_task = Task( description="""Using the research provided in context, write a technical blog post about {topic}. Target audience: senior ML engineers who are time-constrained and highly skeptical of hype. Length: 1500 words. Include: code examples (Python), practical insights, common pitfalls. Avoid: excessive preamble, marketing language, vague claims.""", expected_output="""A publication-ready technical article in Markdown with: - Compelling intro that states the core problem - At least 2 runnable code examples - A "When NOT to use this" section - Concrete performance numbers or benchmarks - Closing with actionable next steps""", agent=writer, context=[research_task], # researcher's output is injected as context output_file="article.md" ) # Structured output with Pydantic class ResearchReport(BaseModel): summary: str key_findings: List[str] citations: List[str] recommendation: str structured_task = Task( description="Research {topic} and return structured findings", expected_output="A structured research report", agent=researcher, output_pydantic=ResearchReport # enforce schema on output )
Section 04

Crew Execution & Processes

Sequential vs Hierarchical process is the most important architectural decision in CrewAI. Sequential is predictable and auditable. Hierarchical gives the manager LLM full delegation authority — use it for complex, open-ended tasks where the optimal sequence isn't known in advance.

Python — Sequential & Hierarchical Crews
from crewai import Agent, Task, Crew, Process # ── Sequential Crew ──────────────────────────────────────── # Tasks run in defined order; each can access prior outputs via context crew = Crew( agents=[researcher, writer], tasks=[research_task, write_task], process=Process.sequential, verbose=True, memory=True # enables cross-task memory sharing ) # Synchronous execution with input interpolation result = crew.kickoff(inputs={"topic": "LoRA fine-tuning for LLMs"}) print(result.raw) # final task output as string print(result.token_usage) # total tokens used across all agents # Async execution for web apps and concurrent runs import asyncio async def run_async(): result = await crew.kickoff_async(inputs={"topic": "RLHF"}) return result.raw # ── Hierarchical Crew ───────────────────────────────────── # Manager LLM reads all tasks and delegates dynamically # Use when: tasks are complex, interdependent, order not fixed crew_hierarchical = Crew( agents=[researcher, writer, editor], tasks=[complex_research_task], # can be a single high-level task process=Process.hierarchical, manager_llm="gpt-4o", # use a powerful model as manager verbose=True ) # Batch kickoff: run same crew against many inputs in parallel inputs_list = [ {"topic": "LoRA"}, {"topic": "RLHF"}, {"topic": "RAG"}, ] results = crew.kickoff_for_each(inputs=inputs_list)
Section 05

Custom Tools & YAML Configuration

Any Python function can become a CrewAI tool with the @tool decorator. For production deployments, YAML-driven configuration separates agent definitions from application code — enabling non-engineers to modify agent behavior without touching Python.

Python — Custom Tool & YAML Config
from crewai.tools import tool import json # Custom tool — any Python function with @tool decorator # Docstring becomes the tool description the LLM sees @tool("Database Query Tool") def query_database(sql_query: str) -> str: """Query the product database with a SQL SELECT statement. Returns results as JSON. Use for looking up product information, pricing, inventory levels, or customer records. Do NOT use for UPDATE, DELETE, or INSERT operations.""" try: results = db.execute(sql_query).fetchall() return json.dumps(results, default=str) except Exception as e: return f"Query failed: {e}" # YAML configuration (agents.yaml) for production # ─────────────────────────────────────────────── # researcher: # role: "Senior Research Analyst" # goal: "Research {topic} comprehensively" # backstory: "10 years of ML research experience..." # tools: [search_tool, file_read_tool] # llm: claude-3-5-sonnet-20241022 # verbose: true # # tasks.yaml # ─────────────────────────────────────────────── # research_task: # description: "Research {topic} and identify..." # expected_output: "Comprehensive report with..." # agent: researcher # Load from YAML using CrewBase pattern from crewai import CrewBase @CrewBase class ResearchCrew: agents_config = "config/agents.yaml" tasks_config = "config/tasks.yaml" @agent def researcher(self) -> Agent: return Agent(config=self.agents_config["researcher"], tools=[search_tool]) @crew def crew(self) -> Crew: return Crew(agents=self.agents, tasks=self.tasks, verbose=True)
🏗️
Best Practices: One responsibility per agent — don't build a "do everything" agent. Agents with narrow scope are faster, more reliable, and cheaper (smaller context, more focused generation). Use cheap models (gpt-4o-mini) for writing/formatting tasks and expensive models (claude-3-5-sonnet, gpt-4o) only for reasoning-heavy research tasks.

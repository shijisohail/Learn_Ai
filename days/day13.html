01 — Why CrewAI?

Multi-Agent Collaboration

Single-agent systems hit quality ceilings on complex tasks. CrewAI decomposes work across specialist agents, each with a focused role, goal, and backstory — mimicking how high-performing human teams operate.

The core insight behind CrewAI is that complex AI tasks benefit from specialization and division of labor, just like human organizations. A research task becomes better when split across a "Senior Research Analyst" (collects raw information), a "Critical Evaluator" (checks quality and gaps), and a "Technical Writer" (formats for the audience). Each agent's role, goal, and backstory shape its behavior — they act as system prompt components that bias the agent toward specific behaviors and expertise.

CrewAI's programming model is declarative and role-centric. You define what each agent is (role, goal, backstory, tools) and what each task requires (description, expected output, agent assignment), then let CrewAI handle the orchestration. The framework manages inter-agent communication, context passing between tasks, tool execution, and output formatting. This is far higher-level than building custom multi-agent orchestration from scratch.

🧑‍💼
Agent
The basic unit. Has: role (what it is), goal (what it optimizes for), backstory (expertise context), tools (what it can call), and LLM config. Backstory shapes prompt context.
📋
Task
A unit of work with: description, expected output, agent assignment, and optional context from previous tasks. Tasks form the work queue executed by the crew.
🚢
Crew
The orchestrator. Holds a list of agents and tasks. Manages execution order (sequential or hierarchical), memory sharing, and result aggregation.
🔧
Tools
Functions agents can call: web search, file read/write, code execution, RAG retrieval, API calls. Any Python function decorated with @tool becomes an agent-callable tool.
02 — Building a Crew

Defining Agents, Tasks & Tools

CrewAI supports both programmatic Python API and YAML configuration. YAML is preferred for production — it separates agent definitions from business logic and enables non-technical team members to modify crew behavior.

research_crew.py
from crewai import Agent, Task, Crew, Process from crewai_tools import SerperDevTool, WebsiteSearchTool from langchain_anthropic import ChatAnthropic llm = ChatAnthropic(model="claude-3-5-sonnet-20241022") search_tool = SerperDevTool() # ── AGENTS ──────────────────────────────────────────────────────── researcher = Agent( role="Senior Research Analyst", goal="Find accurate, up-to-date information about {topic}", backstory="Expert analyst with 10 years of tech research. Always verifies sources.", tools=[search_tool], llm=llm, verbose=True, allow_delegation=False, # prevents infinite loops max_iter=5 ) writer = Agent( role="Technical Writer", goal="Create clear, accurate reports from research findings", backstory="Communicates complex technical topics for engineering audiences.", llm=llm, verbose=True, allow_delegation=False ) # ── TASKS ───────────────────────────────────────────────────────── research_task = Task( description="Research {topic}. Find: key concepts, recent developments, pros/cons.", expected_output="A detailed research brief with 5+ sources.", agent=researcher ) write_task = Task( description="Using the research brief, write a 500-word technical summary.", expected_output="A markdown-formatted technical summary with headers and bullet points.", agent=writer, context=[research_task] # output of research_task is passed as context ) # ── CREW ────────────────────────────────────────────────────────── crew = Crew( agents=[researcher, writer], tasks=[research_task, write_task], process=Process.sequential, verbose=True ) result = crew.kickoff(inputs={"topic": "LangGraph for multi-agent workflows"})
03 — Sequential vs Hierarchical

Process Types & Orchestration

Sequential process executes tasks in order, with each task's output available as context to later tasks. Hierarchical process introduces a manager agent that delegates to specialists and aggregates results.

Sequential is appropriate for linear pipelines where the output of step N is the input to step N+1: research → analyze → write → review. Tasks are deterministic and predictable. Hierarchical is appropriate for tasks that are better decomposed dynamically — a manager agent receives the goal, decides which sub-tasks to delegate to which specialists, and synthesizes the results. The manager is typically the most capable model in your crew.

Sequential Process
Tasks execute one by one in defined order. Output of task N becomes available context for task N+1.
Use when: linear pipeline, predictable flow, no dynamic task generation.
process=Process.sequential
Hierarchical Process
Manager agent orchestrates specialists dynamically. Manager decides which agent handles which sub-problem.
Use when: complex goals requiring adaptive decomposition, parallel sub-tasks, dynamic routing.
process=Process.hierarchical,
manager_llm=frontier_model
hierarchical_crew.py
from crewai import Agent, Task, Crew, Process from langchain_openai import ChatOpenAI # Manager uses frontier model; workers use cost-efficient models manager_llm = ChatOpenAI(model="gpt-4o") worker_llm = ChatOpenAI(model="gpt-4o-mini") data_analyst = Agent(role="Data Analyst", goal="Analyze data patterns", backstory="Expert in statistical analysis", llm=worker_llm) security_eng = Agent(role="Security Engineer", goal="Identify security risks", backstory="10yr security expert", llm=worker_llm) high_level_task = Task( description="Audit {system} for performance and security issues.", expected_output="Executive summary with prioritized findings.", agent=data_analyst # manager may override this with hierarchical ) crew = Crew( agents=[data_analyst, security_eng], tasks=[high_level_task], process=Process.hierarchical, manager_llm=manager_llm, # manages delegation automatically verbose=True ) result = crew.kickoff(inputs={"system": "payment processing API"})
⚠️
Common Pitfall — allow_delegation=True Loops: When agents can delegate to each other, you risk infinite delegation loops where Agent A delegates to Agent B which delegates back to Agent A. Set allow_delegation=False on all agents unless you specifically need inter-agent delegation. Always set max_iter as a safety ceiling.
04 — YAML Config & Tools

Production YAML Configuration & Custom Tools

YAML-based configuration decouples agent definitions from Python code. Custom tools extend what agents can do — any Python function decorated with @tool becomes available to agents.

agents.yaml
agents: researcher: role: "Senior Research Analyst" goal: "Find accurate information about {topic}" backstory: > You are a meticulous researcher with 10 years of experience verifying technical information from multiple primary sources. llm: claude-3-5-sonnet-20241022 max_iter: 5 verbose: true writer: role: "Technical Writer" goal: "Produce clear, accurate technical content" backstory: > Expert at translating complex technical topics for engineering teams. You always cite sources and flag uncertain claims. llm: claude-3-haiku-20240307 verbose: true
custom_tools.py
from crewai.tools import tool from typing import Annotated @tool("Search Internal Knowledge Base") def search_kb(query: Annotated[str, "Search query for the knowledge base"]) -> str: """Search the company's internal knowledge base for relevant documentation.""" docs = rag_retriever.invoke(query) return "\n\n".join(doc.page_content for doc in docs) @tool("Execute SQL Query") def run_sql(query: Annotated[str, "A safe SELECT SQL query"]) -> str: """Run a read-only SQL query against the analytics database.""" if not query.strip().upper().startswith("SELECT"): raise ValueError("Only SELECT queries allowed") result = db.execute(query) return result.to_markdown() # Assign custom tools to agents analyst_agent = Agent( role="Data Analyst", goal="Analyze business data to surface insights", backstory="Expert data analyst with SQL and statistics skills.", tools=[search_kb, run_sql], llm=llm )

07 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. Agents = role + goal + backstory + tools + LLM. Backstory shapes prompt context.
  • 2. Sequential for linear pipelines; hierarchical when tasks require dynamic decomposition.
  • 3. Set allow_delegation=False on workers to prevent delegation loops.
  • 4. YAML config decouples agent definitions from business logic — use in production.
  • 5. Custom @tool functions give agents access to any Python capability — RAG, SQL, APIs.

Section 01

Tensors, Autograd & the Computation Graph

Every neural network operation in PyTorch traces through a dynamic computation graph. Understanding this mechanism is essential for debugging training failures and implementing custom loss functions.

PyTorch uses define-by-run automatic differentiation — the computation graph is created dynamically as operations execute, unlike TensorFlow 1.x's static graphs. Each tensor with requires_grad=True carries a grad_fn that links it to its parent tensors in the DAG. Calling .backward() on the loss traverses this graph in reverse, accumulating gradients via the chain rule.

For LLM engineering, the most critical insight is that gradients are accumulated in .grad attributes — they are NOT automatically zeroed between mini-batches. Forgetting optimizer.zero_grad() causes gradient accumulation across batches, silently corrupting training. This is one of the top bugs in early PyTorch code.

When operating on large batches, use torch.no_grad() context manager during inference — it disables gradient tracking entirely, reducing memory by up to 50% and speeding up forward passes by 20-30%. For the validation loop, this is not optional, it's essential.

Why This Matters

The computation graph is what makes PyTorch debuggable. You can set breakpoints mid-forward-pass, inspect intermediate activations, and even modify tensors in-place during debugging. This is impossible with compiled/static graphs and is why PyTorch dominates research.

tensor_autograd.py
import torch import torch.nn as nn # ── Tensor creation strategies ─────────────────────────────────── device = torch.device("cuda" if torch.cuda.is_available() else "cpu") # Always specify dtype explicitly — default float32 costs 2x vs bfloat16 x = torch.randn(4, 512, 768, dtype=torch.bfloat16, device=device) print(f"Tensor shape: {x.shape}, dtype: {x.dtype}, device: {x.device}") # ── Autograd mechanics ──────────────────────────────────────────── w = torch.randn(768, 768, requires_grad=True) out = x @ w # matrix multiply — creates grad_fn=MmBackward loss = out.mean() # scalar loss for backward pass loss.backward() # backprop through full graph print(f"w.grad shape: {w.grad.shape}") # (768, 768) # ── CRITICAL: always zero gradients before next backward pass ───── optimizer = torch.optim.AdamW([w], lr=1e-4) optimizer.zero_grad() # set_to_none=True is faster (PyTorch >= 1.7) # ── Efficient inference with no_grad ────────────────────────────── with torch.no_grad(): pred = x @ w # No graph created — 2x faster, 50% less memory # ── Detach a tensor from the graph (stop gradients) ─────────────── frozen_repr = pred.detach() # common when freezing encoder outputs print(frozen_repr.requires_grad) # False
Common Pitfall: Gradient Accumulation vs. Forgetting zero_grad

Gradient accumulation is intentional when you call loss.backward() N times before optimizer.step() to simulate larger batches. But forgetting zero_grad() entirely means gradients from ALL previous iterations accumulate — your effective learning rate grows unboundedly and the model diverges. Always check your training loop order: zero → forward → backward → step.

Section 02

GPU Memory Management for LLMs

A 7B parameter model in FP32 requires ~28GB of VRAM just for weights — before accounting for activations, optimizer states, or gradients. Understanding the memory math is essential for production deployments.

GPU memory is split into four major buckets during training: model weights (P bytes), optimizer states (2P bytes for Adam's momentum and variance), gradients (P bytes), and activations (batch-dependent). For a 7B model in FP32, that's 28 + 56 + 28 = 112GB — far beyond any single consumer GPU. This is why mixed precision and parameter-efficient fine-tuning exist.

Mixed precision training with torch.cuda.amp stores weights in FP16/BF16 but maintains a FP32 master copy for optimizer updates. Forward and backward passes run in half precision (2x throughput on Tensor Cores), but the optimizer step updates the FP32 copy to preserve numerical stability. BF16 is preferred over FP16 for LLMs because it has the same exponent range as FP32 — less risk of overflow during training.

Gradient checkpointing trades compute for memory: instead of storing all intermediate activations, it recomputes them during backward. This reduces activation memory from O(layers) to O(sqrt(layers)), enabling training of models that would otherwise OOM.

Why This Matters

A common mistake is trying to maximize batch size without understanding memory allocation. The largest batch that fits in VRAM is NOT necessarily the fastest — GPU utilization and memory bandwidth interact. Use torch.cuda.memory_summary() and NVIDIA's nsight systems to profile actual bottlenecks before assuming OOM errors are solved by reducing batch size.

gpu_memory.py
import torch from torch.cuda.amp import GradScaler, autocast from torch.utils.checkpoint import checkpoint # ── Memory math for LLMs ────────────────────────────────────────── def vram_estimate_gb(params_b: float, precision: str="bf16", training: bool=True) -> float: """Estimate VRAM in GB for a model with params_b billion parameters.""" bytes_per_param = {"fp32": 4, "fp16": 2, "bf16": 2, "int8": 1, "int4": 0.5} bpp = bytes_per_param[precision] weight_gb = (params_b * 1e9 * bpp) / (1024**3) if training: grad_gb = weight_gb # gradients same size as weights optimizer_gb = weight_gb * 2 # Adam: m + v states in FP32 return weight_gb + grad_gb + optimizer_gb return weight_gb print(vram_estimate_gb(7, "bf16", training=True)) # ~42 GB print(vram_estimate_gb(7, "bf16", training=False)) # ~14 GB # ── Mixed precision training loop ───────────────────────────────── scaler = GradScaler() for batch in dataloader: optimizer.zero_grad(set_to_none=True) with autocast(dtype=torch.bfloat16): # forward in BF16 out = model(batch["input_ids"]) loss = criterion(out, batch["labels"]) scaler.scale(loss).backward() # backward with scaled grads scaler.unscale_(optimizer) torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) scaler.step(optimizer) scaler.update() # ── Gradient checkpointing to save activation memory ────────────── def forward_with_ckpt(layer, x): return checkpoint(layer, x, use_reentrant=False) # recompute on backward # Enable in HuggingFace models with a single line: model.gradient_checkpointing_enable() # saves ~60% activation memory

Section 03

Custom Layers & nn.Module

Every LLM component — attention, MLP, RMSNorm — is a subclass of nn.Module. Understanding the interface lets you implement novel architectures and debug existing ones.

The nn.Module base class provides automatic parameter registration, device movement with .to(device), serialization, and state dict management. The two methods you must implement are __init__ (register sub-modules and parameters) and forward (define the computation). Never store tensors as plain Python attributes inside nn.Module — they won't move with the model or appear in state_dict.

RMSNorm (used in LLaMA, Mistral, Gemma) normalizes without computing the mean — only the root mean square. This makes it ~15% faster than LayerNorm while achieving comparable training stability. Understanding its implementation is essential for reproducing modern architectures from papers.

custom_layers.py
import torch import torch.nn as nn import torch.nn.functional as F import math class RMSNorm(nn.Module): """Root Mean Square Layer Normalization — used in LLaMA, Mistral, Gemma.""" def __init__(self, dim: int, eps: float = 1e-6): super().__init__() self.eps = eps self.weight = nn.Parameter(torch.ones(dim)) # learnable scale def _norm(self, x): return x * torch.rsqrt(x.pow(2).mean(-1, keepdim=True) + self.eps) def forward(self, x: torch.Tensor) -> torch.Tensor: output = self._norm(x.float()).type_as(x) return output * self.weight class SwiGLU(nn.Module): """SwiGLU activation used in LLaMA MLP — Swish gated linear unit.""" def __init__(self, dim: int, hidden: int): super().__init__() self.w1 = nn.Linear(dim, hidden, bias=False) # gate self.w2 = nn.Linear(hidden, dim, bias=False) # down proj self.w3 = nn.Linear(dim, hidden, bias=False) # up proj def forward(self, x: torch.Tensor) -> torch.Tensor: return self.w2(F.silu(self.w1(x)) * self.w3(x)) class CausalSelfAttention(nn.Module): """Multi-head causal self-attention with Flash Attention support.""" def __init__(self, dim: int, n_heads: int): super().__init__() assert dim % n_heads == 0 self.n_heads = n_heads self.head_dim = dim // n_heads self.qkv = nn.Linear(dim, 3 * dim, bias=False) self.out_proj = nn.Linear(dim, dim, bias=False) def forward(self, x: torch.Tensor) -> torch.Tensor: B, T, C = x.shape q, k, v = self.qkv(x).split(C, dim=2) q = q.view(B, T, self.n_heads, self.head_dim).transpose(1, 2) k = k.view(B, T, self.n_heads, self.head_dim).transpose(1, 2) v = v.view(B, T, self.n_heads, self.head_dim).transpose(1, 2) # PyTorch 2.0+ fused Flash Attention — no explicit mask needed y = F.scaled_dot_product_attention(q, k, v, is_causal=True) y = y.transpose(1, 2).contiguous().view(B, T, C) return self.out_proj(y)

Section 04

Efficient DataLoaders for LLM Training

The DataLoader pipeline — tokenization, batching, padding, and GPU transfer — is frequently the training bottleneck in small-to-medium setups.

For LLM training, the dominant pattern is packed sequences: instead of padding all sequences in a batch to the same length (wasteful), pack multiple short sequences end-to-end within a fixed context window and use attention masks to prevent cross-contamination. This maximizes GPU utilization and avoids the O(N²) attention cost of padding tokens.

The DataLoader's num_workers parameter controls CPU parallelism for data loading. A common mistake is setting it to 0 (synchronous loading) during development and forgetting to increase it in production — this creates a CPU bottleneck that keeps the GPU idle. For tokenized datasets that fit in RAM, use pin_memory=True to enable DMA transfers that bypass the CPU cache and feed directly into GPU VRAM.

Datasets larger than memory require streaming. The HuggingFace datasets library supports streaming=True mode that fetches data lazily from disk or object storage, with a prefetch buffer. For pre-training at scale, the data pipeline typically runs on multiple CPU nodes that feed a GPU training cluster.

dataloader.py
from torch.utils.data import Dataset, DataLoader from transformers import AutoTokenizer import torch class PackedSequenceDataset(Dataset): """Pack multiple sequences into a fixed context window — no padding waste.""" def __init__(self, texts, tokenizer, ctx_len: int = 2048): self.ctx_len = ctx_len # Tokenize and concatenate all texts with EOS separator eos = tokenizer.eos_token_id all_ids = [] for t in texts: ids = tokenizer.encode(t) all_ids.extend(ids + [eos]) # Chunk into fixed-size windows self.chunks = [ all_ids[i:i+ctx_len+1] for i in range(0, len(all_ids) - ctx_len, ctx_len) ] def __len__(self): return len(self.chunks) def __getitem__(self, idx): chunk = torch.tensor(self.chunks[idx], dtype=torch.long) return {"input_ids": chunk[:-1], "labels": chunk[1:]} # Optimal DataLoader settings for GPU training loader = DataLoader( dataset, batch_size=8, num_workers=4, # parallel CPU workers — match CPU cores pin_memory=True, # DMA transfer — skips CPU cache prefetch_factor=2, # load 2 batches ahead persistent_workers=True, # keep workers alive between epochs shuffle=True, )
Common Pitfall: Moving Data Inside the Dataset

Never call .to(device) inside __getitem__. DataLoader workers are CPU processes — moving to GPU here creates CUDA context in each worker (expensive) and defeats pin_memory. Always move the batch to device inside the training loop after collation: batch = {k: v.to(device) for k, v in batch.items()}.

Section 05

torch.compile() & Performance Profiling

PyTorch 2.0 introduced a compiler stack that can trace, optimize, and JIT-compile your model's computation graph — often yielding 20-50% speed improvements with zero code changes.

The compiler pipeline: TorchDynamo captures Python bytecode and traces the computation graph, AOTAutograd generates the backward graph ahead-of-time, and TorchInductor emits optimized CUDA/Triton kernels. The first forward pass is slow (compilation), but subsequent calls use the cached kernel — ideal for training loops.

Three compilation modes exist: default (balanced), reduce-overhead (minimize kernel launches via CUDA graphs — best for small models/large batches), and max-autotune (exhaustively benchmarks kernels — slow compilation, best throughput). For LLM training, reduce-overhead is usually optimal.

compile_and_profile.py
import torch from torch.profiler import profile, record_function, ProfilerActivity # ── torch.compile — one line, big speedup ──────────────────────── model = MyTransformer().to(device) model = torch.compile(model, mode="reduce-overhead") # First call is slow (compilation), subsequent calls are fast # ── Profiling — find the real bottleneck ───────────────────────── with profile( activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA], record_shapes=True, with_stack=True, with_flops=True, ) as prof: with record_function("model_forward"): output = model(batch) print(prof.key_averages().table( sort_by="cuda_time_total", row_limit=10 )) # Export to Chrome trace viewer (chrome://tracing) prof.export_chrome_trace("trace.json") # ── CUDA memory snapshot ───────────────────────────────────────── torch.cuda.memory._record_memory_history(max_entries=100000) # ... run training step ... torch.cuda.memory._dump_snapshot("memory_snapshot.pickle") # Visualize at pytorch.org/memory_viz

Section 07

Advanced PyTorch Patterns for LLMs

Production LLM engineering requires mastery of distributed training, KV caching, and memory-efficient inference. These patterns separate senior ML engineers from beginners.

FSDP (Fully Sharded Data Parallel) is PyTorch's native approach to training models too large for a single GPU. Unlike DDP (which replicates the full model on each GPU), FSDP shards model parameters, gradients, and optimizer states across all GPUs. Each GPU only holds 1/N of each tensor in memory, and gathers the full layer during forward/backward before resharding. For 70B models, this is the only viable training strategy.

KV Cache is the critical optimization that makes autoregressive LLM inference practical. During generation, the attention keys and values for all past tokens are cached in GPU memory — new tokens only need to compute their own K/V and attend over the cache. Without KV caching, generating a 1000-token response would require 1000 forward passes of increasing length. With caching, each step is O(1) instead of O(N). The cache size grows with sequence length and batch size, which is why long-context inference requires significant VRAM.

vLLM's PagedAttention solves the KV cache memory fragmentation problem. Traditional KV caching pre-allocates contiguous memory for the maximum sequence length — wasteful when actual lengths vary. PagedAttention stores KV cache in fixed-size "pages" (blocks) that are allocated dynamically and can be referenced by multiple sequences. This enables near-perfect memory utilization and allows serving 5-20x more concurrent requests on the same hardware.

Speculative Decoding uses a smaller draft model to generate candidate tokens speculatively, which are then verified (accepted or rejected) by the large target model in parallel. If the draft is correct, you get multiple tokens for the cost of one forward pass. This achieves 2-3x speedup for tasks where the draft model is often correct (like code completion or structured generation). The acceptance rate depends on the alignment between draft and target model distributions.

torch.compile() + CUDA Graphs are the two complementary compilation strategies. torch.compile traces your model into an optimized Triton kernel graph that's faster than eager mode. CUDA graphs capture a sequence of GPU operations and replay them without CPU overhead — eliminating the kernel launch latency that dominates small-batch inference. Combined, these can achieve 3-5x inference speedup on H100 GPUs compared to vanilla PyTorch.

Continuous Batching is the production serving pattern that processes tokens from different requests together in a single forward pass. Instead of waiting for a full batch of requests to arrive (static batching), continuous batching dynamically adds new requests to in-progress batches at each generation step. This maximizes GPU utilization and reduces median latency by processing short requests quickly without waiting for slow ones to finish.

fsdp_training.py
import torch import torch.distributed as dist from torch.distributed.fsdp import ( FullyShardedDataParallel as FSDP, MixedPrecision, ShardingStrategy, CPUOffload ) from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy from transformers.models.mistral.modeling_mistral import MistralDecoderLayer # ── Initialize distributed training ────────────────────────────── dist.init_process_group("nccl") # NCCL backend for GPU-to-GPU comms local_rank = dist.get_rank() % torch.cuda.device_count() torch.cuda.set_device(local_rank) # ── FSDP wrapping policy ────────────────────────────────────────── # Shard at decoder layer granularity — each layer is a separate FSDP unit wrap_policy = transformer_auto_wrap_policy( transformer_layer_cls={MistralDecoderLayer} ) # ── Mixed precision config ──────────────────────────────────────── mp_policy = MixedPrecision( param_dtype=torch.bfloat16, # store params in BF16 reduce_dtype=torch.bfloat16, # reduce gradients in BF16 buffer_dtype=torch.bfloat16, # buffers in BF16 ) # ── Wrap model with FSDP ────────────────────────────────────────── model = FSDP( model, auto_wrap_policy=wrap_policy, mixed_precision=mp_policy, sharding_strategy=ShardingStrategy.FULL_SHARD, # shard params+grads+optimizer cpu_offload=CPUOffload(offload_params=False), # keep on GPU device_id=local_rank, ) # Each of 8 A100s now holds 1/8 of the 70B model = ~17.5GB/GPU (BF16)
01
Always profile before optimizing
Use torch.profiler or NVIDIA Nsight to identify actual bottlenecks before adding optimization complexity. GPU utilization below 70% usually means data loading is the bottleneck, not the model.
02
Set seed for reproducibility
Always set torch.manual_seed(), numpy.random.seed(), and random.seed() at the start of training. Without seeds, results vary between runs — impossible to debug performance regressions.
03
Use torch.backends.cudnn.benchmark
Setting benchmark=True lets cuDNN benchmark several convolution algorithms on the first run and select the fastest. For fixed-size inputs (like LLMs), this gives 5-15% speedup with no code changes.
04
Monitor GPU utilization, not just loss
nvidia-smi, nvtop, or wandb system metrics should show GPU utilization near 95%+ during training. Lower means your data pipeline is the bottleneck. Higher num_workers and pin_memory are the fix.
Production Inference: vLLM vs. TGI vs. llama.cpp

vLLM is the best choice for GPU serving in production — PagedAttention, continuous batching, OpenAI-compatible API. Requires CUDA. TGI (Text Generation Inference) by HuggingFace is similarly capable, with better HuggingFace Hub integration. llama.cpp is optimal for CPU inference and Apple Silicon — runs quantized GGUF models with no GPU requirement. For a backend engineer building a production service, vLLM is the first choice unless you specifically need CPU-only deployment.

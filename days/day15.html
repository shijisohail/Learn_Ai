Section 01

Why PyTorch for LLM Engineers

You don't need to train from scratch — you need to understand inference, loading weights, and debugging memory issues. PyTorch won the research ecosystem in 2024/2025.

  • Tensor — n-dimensional array that lives on GPU or CPU. Every weight, activation, and embedding is a tensor.
  • dtype — bfloat16 / float16 / float32 determines memory usage and numerical stability. bfloat16 is the LLM standard.
  • device — where the tensor lives: "cpu", "cuda", or "mps" (Apple Silicon).
  • PyTorch vs TensorFlow in 2024/2025 — PyTorch dominates research and production LLM stacks. HuggingFace, vLLM, and every major lab runs PyTorch.
PYTHON
import torch # Tensors — the fundamental data structure x = torch.tensor([[1.0, 2.0], [3.0, 4.0]]) print(x.shape) # torch.Size([2, 2]) print(x.dtype) # torch.float32 print(x.device) # cpu # GPU operations device = "cuda" if torch.cuda.is_available() else "mps" if torch.backends.mps.is_available() else "cpu" x_gpu = x.to(device) # bfloat16 for LLMs — better numerical stability than float16 x_bf16 = x.to(torch.bfloat16) print(torch.cuda.memory_allocated() / 1e9, "GB") # check GPU memory
Section 02

Autograd & The Computation Graph

How autograd works: it tracks operations and builds a computation graph for backpropagation. In practice you rarely call backward() for LLMs — but understanding gradients helps debug NaN loss, gradient explosion, and memory OOM.

  • requires_grad=True — tells PyTorch to track this tensor in the computation graph.
  • .backward() — traverses the graph and computes all gradients automatically.
  • Gradient accumulation — simulate large batches by dividing loss and accumulating gradients over N steps.
  • Understanding gradients helps debug NaN loss, gradient explosion, and OOM during fine-tuning.
PYTHON
# Autograd — automatic differentiation w = torch.tensor(2.0, requires_grad=True) x = torch.tensor(3.0, requires_grad=True) y = w * x ** 2 + w # y = 2*9 + 2 = 20 y.backward() # compute dy/dw and dy/dx print(w.grad) # dy/dw = x^2 + 1 = 10 print(x.grad) # dy/dx = 2*w*x = 12 # Gradient accumulation for large batch training optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4) accumulation_steps = 4 for step, batch in enumerate(dataloader): loss = model(**batch).loss / accumulation_steps loss.backward() if (step + 1) % accumulation_steps == 0: optimizer.step() optimizer.zero_grad()
Section 03

Loading & Running Pre-trained LLM Inference

The two rules that matter most: always model.eval() and always torch.no_grad(). Skip either and you waste memory or get wrong outputs.

  • model.eval() — disables dropout and batch norm training mode. Required for correct inference output.
  • torch.no_grad() — disables gradient tracking. Saves 30-50% memory during inference.
  • Half precision — model.to(torch.bfloat16) halves VRAM usage with minimal accuracy loss.
  • device_map="auto" — HuggingFace automatically places layers across all available GPUs.
PYTHON
import torch from transformers import AutoModelForCausalLM, AutoTokenizer # Load model in bfloat16 for efficient inference model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B-Instruct", torch_dtype=torch.bfloat16, device_map="auto" # automatically place layers across available GPUs ) tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-3B-Instruct") # Always use eval mode + no_grad for inference model.eval() with torch.no_grad(): inputs = tokenizer("Explain LoRA:", return_tensors="pt").to(device) outputs = model.generate( **inputs, max_new_tokens=200, temperature=0.7, do_sample=True ) response = tokenizer.decode(outputs[0], skip_special_tokens=True)
Section 04

GPU Memory Management

Why LLMs OOM: model weights + KV cache + activations + gradients. A 7B model in BF16 already uses 14GB — before any inputs are processed.

  • Memory formula — params × bytes_per_param. 7B × 2 bytes (BF16) = 14GB for weights alone.
  • Flash Attention 2 — reduces attention memory from O(n²) to O(n). Essential for long contexts.
  • gradient_checkpointing — trade compute for memory during training: recompute activations instead of storing them.
  • torch.cuda.empty_cache() — frees cached but unused GPU memory back to the allocator.
PYTHON
# Memory-efficient inference checklist # 1. Use bfloat16 (half the memory of float32) # 2. Use torch.no_grad() (no gradient storage) # 3. Enable Flash Attention 2 # 4. Process in batches (not all at once) model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B", torch_dtype=torch.bfloat16, attn_implementation="flash_attention_2", # Flash Attention 2 device_map="auto" ) # Check memory usage print(torch.cuda.memory_summary(abbreviated=True)) # Free GPU cache if needed torch.cuda.empty_cache() # Memory per model size (approximate, BF16): # 3B model: ~6GB VRAM # 7B model: ~14GB VRAM # 13B model: ~26GB VRAM # 70B model: ~140GB VRAM (need A100 80GB × 2 with tensor parallel)
Section 05

Practical PyTorch for LLM Engineers

Profiling, reproducibility, checkpointing, and the common gotchas that waste hours in practice.

  • torch.profiler — find CPU/GPU bottlenecks. Identifies which ops are slow, not just that things are slow.
  • torch.manual_seed(42) — set for reproducible runs. Also set numpy and random seeds in the same block.
  • Device mismatch — model on GPU, inputs on CPU = cryptic RuntimeError. Always match with .to(model.device).
  • In-place ops with autograd — x += 1 on a tensor with requires_grad=True can corrupt the computation graph. Use x = x + 1.
💡
Three rules of LLM inference in PyTorch: (1) always model.eval(), (2) always torch.no_grad(), (3) always match dtypes between model and inputs. Break any of these and you get wrong outputs, wasted memory, or both.
PYTHON
import torch # Reproducibility torch.manual_seed(42) if torch.cuda.is_available(): torch.cuda.manual_seed_all(42) # Save checkpoint torch.save({ 'epoch': epoch, 'model_state_dict': model.state_dict(), 'optimizer_state_dict': optimizer.state_dict(), 'loss': loss, }, 'checkpoint.pt') # Load checkpoint checkpoint = torch.load('checkpoint.pt', map_location=device) model.load_state_dict(checkpoint['model_state_dict']) # Profiler — find GPU bottlenecks with torch.profiler.profile( activities=[torch.profiler.ProfilerActivity.CPU, torch.profiler.ProfilerActivity.CUDA], record_shapes=True ) as prof: output = model(**inputs) print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=10))

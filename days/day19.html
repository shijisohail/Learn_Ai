What is an ML Pipeline?

An ML pipeline is a DAG of steps that takes raw data to a deployed model. Pipelines are not optional in production — they are how you reproduce training, schedule retraining, and recover from failures.

♻️
Reproducibility
Any pipeline run can be exactly re-executed with the same data version, same code commit, and same hyperparameters. No "works on my machine."
⏱️
Scheduling
Trigger retraining on a cron schedule, on data drift detection, or after N new labels arrive. Pipelines handle the orchestration automatically.
🔀
Parallelism
Data prep and feature extraction run in parallel. Multiple training configurations fan out. The DAG engine coordinates dependencies.
🔄
Failure Recovery
If training fails, the pipeline retries from the last checkpoint. Completed steps are cached — no need to re-run data preprocessing from scratch.

Section 02

Kubeflow Pipelines

KFP turns Python functions into containerized steps connected in a DAG. Each component is independently versioned, testable, and reusable across pipelines.

💡
KFP v2 Key Concepts: @dsl.component wraps a Python function into a reusable container step. Input/Output typed artifacts connect steps by passing data through the pipeline's artifact store — not through function returns.
PYTHON — Kubeflow Pipeline Definition
from kfp import dsl from kfp.dsl import Input, Output, Dataset, Model, Metrics @dsl.component(base_image="python:3.11", packages_to_install=["datasets"]) def prepare_dataset(output_dataset: Output[Dataset]): """Download and preprocess training data""" from datasets import load_dataset ds = load_dataset("tatsu-lab/alpaca") ds["train"].to_json(output_dataset.path) @dsl.component(base_image="pytorch/pytorch:2.1.0-cuda11.8-cudnn8-runtime", packages_to_install=["peft", "trl", "transformers"]) def train_lora( dataset: Input[Dataset], model_output: Output[Model], lora_rank: int = 16, learning_rate: float = 2e-4 ): """Fine-tune model with LoRA""" # Training code here... @dsl.pipeline(name="llm-finetune-pipeline") def finetune_pipeline(lora_rank: int = 16, learning_rate: float = 2e-4): prep_step = prepare_dataset() train_step = train_lora( dataset=prep_step.outputs["output_dataset"], lora_rank=lora_rank, learning_rate=learning_rate ) # Compile and submit from kfp import compiler compiler.Compiler().compile(finetune_pipeline, "pipeline.yaml")
Section 03

AWS SageMaker Pipelines

SageMaker Pipelines is the AWS-native MLOps solution. Zero infrastructure management — AWS handles the container orchestration. Built-in spot instance support cuts GPU costs by up to 70%.

✅SageMaker Strengths
  • Spot instances with automatic checkpointing
  • Built-in distributed training (data parallel, model parallel)
  • SageMaker Experiments integrates with pipeline runs
  • Model Monitor for production drift detection
  • Native integration with S3, ECR, IAM
⚠️SageMaker Watch-outs
  • AWS lock-in — hard to migrate away
  • Container startup latency (1-3 min per step)
  • Expensive if you forget to stop instances
  • Debug logging less ergonomic than local runs
  • IAM configuration is complex for new teams
PYTHON — SageMaker Training Step
from sagemaker.workflow.pipeline import Pipeline from sagemaker.workflow.steps import TrainingStep from sagemaker.pytorch import PyTorch estimator = PyTorch( entry_point="train.py", role="arn:aws:iam::account:role/SageMakerRole", instance_type="ml.g5.2xlarge", instance_count=1, framework_version="2.1.0", py_version="py311", hyperparameters={"lora-rank": 16, "lr": 2e-4} ) train_step = TrainingStep( name="TrainLoRA", estimator=estimator, inputs={"train": TrainingInput(s3_data="s3://bucket/data/")} ) pipeline = Pipeline( name="LLMFinetunePipeline", steps=[train_step], sagemaker_session=session ) pipeline.upsert(role_arn=role) pipeline.start()
Section 04

BentoML for Model Serving

BentoML is the bridge between a trained model artifact and a production HTTP API. It handles GPU allocation, adaptive request batching, and Docker packaging in one framework.

💡
Adaptive Batching: The @bentoml.api(batchable=True) decorator automatically groups concurrent requests into a single GPU forward pass. Under load, this multiplies effective throughput without any application-level code changes.
PYTHON — BentoML Service
import bentoml from transformers import AutoModelForCausalLM, AutoTokenizer import torch @bentoml.service( resources={"gpu": 1, "memory": "16Gi"}, traffic={"timeout": 60} ) class LLMService: def __init__(self): self.model = AutoModelForCausalLM.from_pretrained( "./merged-llama3", torch_dtype=torch.bfloat16, device_map="cuda" ) self.tokenizer = AutoTokenizer.from_pretrained("./merged-llama3") self.model.eval() @bentoml.api(batchable=True, max_batch_size=8, max_latency_ms=500) def generate(self, prompts: list[str]) -> list[str]: inputs = self.tokenizer(prompts, return_tensors="pt", padding=True).to("cuda") with torch.no_grad(): outputs = self.model.generate(**inputs, max_new_tokens=200) return [self.tokenizer.decode(o, skip_special_tokens=True) for o in outputs] # CLI: bentoml serve llm_service:LLMService # Build: bentoml build → creates bento artifact # Containerize: bentoml containerize llm_service:latest
Section 05

Model Packaging & A/B Traffic Splitting

Canary deployments are the safe path to production. Roll out to 5% of traffic, validate RAGAS scores, then progressively increase. Keep the old version registered for instant rollback.

🐤
Canary Rollout
5% → 20% → 50% → 100% traffic. Each stage waits for RAGAS score validation before proceeding. Automated via Kubernetes traffic weight annotations.
🔬
Model Comparison
Run both old and new model on the same traffic slice. Compare RAGAS faithfulness and user feedback score distributions before full cutover.
⏪
Rollback Strategy
Keep previous version in MLflow Registry at Staging. One API call to transition it back to Production. Target: <2 minute MTTR for model rollbacks.
📡
SageMaker Model Monitor
Automated drift detection on input feature distributions and prediction distributions. Triggers alerts when production data diverges from training baseline.
🚦
Promotion Gate Checklist: Before each traffic increase — (1) RAGAS faithfulness > 0.82, (2) P99 latency < 4s, (3) error rate < 1%, (4) no significant shift in user thumbs-down rate. Automate this gate in your CI/CD pipeline.

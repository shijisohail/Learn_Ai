Section 01

ML Pipeline Architecture: The 5 Stages

A production ML pipeline automates the journey from raw data to deployed model — with checkpointing, artifact versioning, and quality gates at every stage.

A complete ML pipeline for LLMs consists of five stages: Ingest (pull data from sources, validate schema, compute statistics), Train (run the fine-tuning job with logged hyperparameters), Evaluate (run RAGAS, benchmark on held-out sets, compare to baseline), Deploy (canary → production promotion with rollback), and Monitor (track latency, error rates, output quality drift). Each stage produces artifacts that feed the next — model checkpoints, eval reports, deployment configs.

The key design principle is that pipelines are reproducible: given the same input data hash and code version, re-running the pipeline should produce identical results. This requires pinning all dependencies, versioning data, and storing all intermediate artifacts. Without reproducibility, you can't reliably debug why a model regressed or reproduce a past high-performing experiment.

Kubeflow Pipelines compiles Python functions into Kubernetes pods connected by data channels. SageMaker Pipelines does the same on AWS infrastructure with native integration with S3, ECR, and SageMaker Training. Vertex AI Pipelines uses Google's Kubeflow-based execution engine with BigQuery and Artifact Registry integration. For smaller teams, a simpler orchestrator like Prefect or Airflow with custom ML operators is often more appropriate than managed ML platforms.

Why This Matters

Manual model training + manual deployment breaks down at team scale. When multiple engineers are running experiments simultaneously and pushing to production, you need automated quality gates that prevent regressions. A pipeline that automatically blocks deployment when val_loss is worse than baseline is worth weeks of engineering time in saved incidents.

Section 02

BentoML: Production Model Serving

BentoML packages models with their inference code, dependencies, and serving logic into portable containers that can run anywhere — locally, on Kubernetes, or on managed cloud services.

The core BentoML abstraction is a Service: a Python class that wraps your model loading logic and defines API endpoints using type annotations. BentoML handles the HTTP server, request/response serialization, batching, health checks, and metrics endpoint automatically. You write pure Python with type hints, and BentoML generates the production server code around it.

The Bento is the deployment artifact: a snapshot of your service code, dependencies, model weights, and runner configuration bundled into a reproducible container image. Bentos are versioned and stored in a BentoML Model Store (local or remote registry). Building a Bento produces a Docker image that includes everything needed to run the model — eliminating environment drift between development and production.

For LLM serving specifically, BentoML's Runner concept allows you to separate the model inference worker from the API server. Multiple API replicas can share a single model runner, enabling efficient resource utilization. The runner can also be configured with adaptive batching — it collects requests within a time window and processes them as a batch, dramatically improving throughput for embedding models and smaller LLMs.

bento_service.py
import bentoml from bentoml.io import JSON, Text from pydantic import BaseModel from transformers import AutoModelForCausalLM, AutoTokenizer import torch # ── Step 1: Save model to BentoML store ────────────────────────── # (run once after training) bento_model = bentoml.transformers.save_model( "mistral-7b-sft", model, custom_objects={"tokenizer": tokenizer}, metadata={"val_loss": 0.43, "mmlu": 68.2}, labels={"team": "ml-platform"}, ) print(f"Saved: {bento_model}") # mistral-7b-sft:abc123 # ── Step 2: Define the BentoML Service ──────────────────────────── class GenerateRequest(BaseModel): prompt: str max_tokens: int = 256 temperature: float = 0.7 @bentoml.service( resources={"gpu": 1, "memory": "16Gi"}, traffic={"timeout": 60}, ) class LLMService: # Dependency injection — model loaded once, shared across requests model_ref = bentoml.models.get("mistral-7b-sft:latest") def __init__(self): self.model = bentoml.transformers.load_model(self.model_ref) self.tokenizer = self.model_ref.custom_objects["tokenizer"] self.model.eval() @bentoml.api(route="/generate") async def generate(self, req: GenerateRequest) -> dict: inputs = self.tokenizer(req.prompt, return_tensors="pt").to("cuda") with torch.no_grad(): out = self.model.generate( **inputs, max_new_tokens=req.max_tokens, temperature=req.temperature, do_sample=True, ) text = self.tokenizer.decode(out[0], skip_special_tokens=True) return {"text": text, "tokens": out.shape[1]} # ── Step 3: Build and serve ─────────────────────────────────────── # bentoml build # create Bento artifact # bentoml serve bento_service:LLMService # local dev server # bentoml containerize mistral-7b-sft:tag # Docker image # bentoml deploy --cloud ec2 --instance g5.xlarge # cloud deploy
Common Pitfall: Loading Model in the Request Handler

Never load the model inside a request handler function. Loading a 7B model takes 10-30 seconds and consumes massive memory during loading. Always load in __init__ (BentoML, FastAPI lifespan, gunicorn worker start). Multiple concurrent requests will see 30-second latency spikes and your pod will OOM trying to load multiple copies simultaneously.

Section 03

Canary & Blue-Green Deployments for ML Models

Progressive delivery patterns reduce the blast radius of model changes — routing a small percentage of traffic to the new model while monitoring for regressions before full rollout.

A canary deployment starts with 5-10% of traffic going to the new model, while the remaining 90-95% continues hitting the old model. Metrics are compared in real-time: if error rates, latency, or quality metrics (RAGAS scores, user thumbs down rate) remain stable, traffic is gradually shifted — 10% → 25% → 50% → 100%. If any metric degrades, the canary is automatically rolled back by shifting traffic back to the old model — zero downtime, minimal user impact.

For LLMs, canary deployments require thinking beyond simple HTTP error rates. A model might succeed (HTTP 200) but produce lower quality outputs — the "silent regression" problem. This requires instrumenting LLM-specific quality metrics: perplexity on a golden set, RAGAS faithfulness scores for RAG outputs, user interaction signals (session length, follow-up questions indicating confusion). Prometheus metrics from your serving infrastructure, combined with LangSmith traces for quality, give you the full picture.

Blue-green deployment is simpler: run two identical production environments ("blue" = current, "green" = new), then switch the load balancer to point to green. If green has issues, switch back to blue in seconds. The trade-off is cost (running two full production environments) versus risk (instant rollback). For models with large VRAM footprints, this doubles your GPU spend during the switch window.

Section 04

Cloud ML Platforms Comparison

Kubeflow, SageMaker, and Vertex AI each approach MLOps differently — understanding their trade-offs helps you choose the right platform for your team's needs.

Kubeflow Pipelines is an open-source ML workflow orchestrator that runs on any Kubernetes cluster. It defines pipelines as Python functions decorated with @kfp.component and connected via @kfp.pipeline. Each component runs in a Docker container, making it portable. The trade-off: significant ops overhead to install and maintain. Best for teams with Kubernetes expertise and multi-cloud requirements.

AWS SageMaker Pipelines provides fully managed ML pipelines on AWS. Components are SageMaker-native steps: ProcessingStep, TrainingStep, ConditionStep, ModelStep. Integration with S3, ECR, and SageMaker Model Registry is seamless. The trade-off: AWS lock-in, higher per-hour costs than raw EC2/EKS. Best for teams already deep in the AWS ecosystem.

Google Vertex AI Pipelines uses Kubeflow Pipelines SDK but runs on Google's managed infrastructure. Deep integration with BigQuery (for tabular data), Artifact Registry, and Google Cloud Storage. Vertex AI also offers Model Garden for one-click fine-tuning of popular open-source models. Best for teams using Google Cloud and GCP data services.

GitOps for ML & CI/CD Pipelines

GitOps applies Git-native workflows to ML deployment: your model configuration, serving specs, and rollout policies live in Git. Every merge to main triggers a deployment pipeline. Rollbacks are git reverts. Audit trails are commit histories. This eliminates the "who deployed what when" problem that plagues teams with manual deployments.

The ML-specific challenge is that model artifacts are too large for Git. The pattern is to track model references (MLflow run ID, HuggingFace Hub tag, or S3 URI) in Git, while the artifacts live in external storage. Your CD pipeline reads the reference from Git, pulls the artifact, and deploys it.

.github/workflows/deploy_model.yml 
name: Deploy Model
on:
  push:
    branches: [main]
    paths: ['model_config.yaml']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Read model config
        id: config
        run: |
          echo "run_id=$(yq .mlflow_run_id model_config.yaml)" >> $GITHUB_OUTPUT
          echo "stage=$(yq .stage model_config.yaml)" >> $GITHUB_OUTPUT

      - name: Download model artifact
        run: |
          mlflow artifacts download \
            --run-id ${{ steps.config.outputs.run_id }} \
            --artifact-path model/ \
            --dst-path ./model/

      - name: Build & push Docker image
        run: |
          docker build -t registry/llm-api:${{ github.sha }} .
          docker push registry/llm-api:${{ github.sha }}

      - name: Update Kubernetes manifest
        run: |
          sed -i "s|IMAGE_TAG|${{ github.sha }}|g" k8s/deployment.yaml
          kubectl apply -f k8s/deployment.yaml
Why this matters: This pipeline takes a model from MLflow to Kubernetes in one git push. The model_config.yaml acts as the single source of truth — to rollback, you git revert to a previous version of that file. No manual steps, no "it works on my machine" issues.

Argo Rollouts Progressive Delivery

Argo Rollouts extends Kubernetes Deployments with progressive delivery strategies. Unlike vanilla Kubernetes (which does immediate rolling updates), Argo supports canary steps with analysis templates — automated metric checks that gate each traffic increase. If P99 latency exceeds threshold during a canary step, Argo automatically rolls back without human intervention.

k8s/rollout.yaml 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: llm-api
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 10      # 10% canary traffic
        - pause: {duration: 5m}
        - analysis:          # automated gate
            templates:
              - templateName: latency-check
        - setWeight: 50      # 50% if analysis passed
        - pause: {duration: 10m}
        - analysis:
            templates:
              - templateName: error-rate-check
        - setWeight: 100     # full rollout
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: latency-check
spec:
  metrics:
    - name: p99-latency
      provider:
        prometheus:
          address: http://prometheus:9090
          query: histogram_quantile(0.99, rate(request_duration_seconds_bucket[5m]))
      successCondition: result < 2.0   # must be under 2s P99
      failureLimit: 3
Common pitfall: Argo analysis templates query Prometheus metrics from the canary pods, not the entire deployment. Ensure your Prometheus scrape config labels canary pods with a rollout-pod-template-hash selector, otherwise your analysis queries will include traffic from stable pods and mask canary regressions.

Blue-Green vs Canary vs Rolling

Each deployment strategy has different risk/cost trade-offs. Blue-green maintains a complete duplicate environment — instant rollback but 2× infrastructure cost. Canary is incremental and cost-efficient but requires robust observability. Rolling updates are Kubernetes-native and simple but offer no traffic control during rollout.

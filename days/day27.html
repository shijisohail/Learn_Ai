01 — Why Containerize

Why Containerize AI Applications

GPU dependencies, Python packages, model weights, and CUDA versions form a notoriously fragile environment. Docker solves "works on my GPU" permanently.

📦
Reproducibility
"Works on my GPU" becomes "works everywhere." CUDA version, Python packages, system libs — all locked in the image. Dev, staging, and prod are identical.
🔀
Microservices
Separate your FastAPI gateway, embedding service, and LLM backend into independent containers. Scale the embedding service 10x without touching the LLM container.
⚙️
Kubernetes Orchestration
Kubernetes schedules AI containers on GPU nodes, runs health checks, restarts crashed LLM services, and performs rolling updates with zero downtime.
🖥️
GPU in Containers
NVIDIA Container Toolkit exposes host GPUs to containers. Request nvidia.com/gpu: 1 as a Kubernetes resource like CPU or RAM.
02 — Dockerizing LLM Services

Dockerizing a FastAPI LLM Service

Multi-stage builds keep images lean. GPU-enabled base images require NVIDIA CUDA. Model weights can be baked in, mounted as a volume, or downloaded at startup — each has tradeoffs.

💡
Model weight strategy: Bake into image for immutable deploys (large image, fast startup). Volume mount for shared weights across pods (fast image, needs PVC). Download at startup for flexibility (slow cold start, use init containers).
DOCKERFILE — Multi-Stage GPU Build
# Stage 1: Builder FROM python:3.11-slim AS builder WORKDIR /app COPY requirements.txt . RUN pip install --user --no-cache-dir -r requirements.txt # Stage 2: Runtime with CUDA support FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04 RUN apt-get update && apt-get install -y python3.11 python3-pip && rm -rf /var/lib/apt/lists/* WORKDIR /app # Copy installed packages from builder COPY --from=builder /root/.local /root/.local COPY . . ENV PATH=/root/.local/bin:$PATH ENV PYTHONUNBUFFERED=1 ENV MODEL_PATH=/app/models/llama3-3b HEALTHCHECK --interval=30s --timeout=10s --start-period=120s \ CMD curl -f http://localhost:8000/health || exit 1 EXPOSE 8000 CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "1"]

03 — Kubernetes for AI

Kubernetes for AI Workloads

Kubernetes schedules GPU pods, runs readiness probes that wait for model loading, and restarts unhealthy LLM containers automatically. GPU nodes are expensive — K8s ensures they're always busy.

  • GPU node pools: GKE Autopilot GPU, EKS managed GPU node groups — K8s provisions GPU VMs on-demand
  • nodeSelector & tolerations: Pin LLM pods to GPU nodes using labels; taint GPU nodes so only GPU workloads run there
  • Resource requests/limits: nvidia.com/gpu: 1 is a schedulable resource just like CPU — K8s won't assign the pod until a GPU is available
  • Readiness probes: LLMs take 60-180s to load. The readiness probe prevents traffic until the model is warmed up and responding
YAML — Kubernetes Deployment with GPU
# deployment.yaml apiVersion: apps/v1 kind: Deployment metadata: name: llm-service spec: replicas: 2 selector: matchLabels: app: llm-service template: metadata: labels: app: llm-service spec: nodeSelector: nvidia.com/gpu.present: "true" containers: - name: llm-service image: myregistry/llm-service:v1.2.0 resources: requests: memory: "16Gi" nvidia.com/gpu: 1 limits: memory: "24Gi" nvidia.com/gpu: 1 env: - name: ANTHROPIC_API_KEY valueFrom: secretKeyRef: name: ai-secrets key: anthropic-api-key readinessProbe: httpGet: path: /health port: 8000 initialDelaySeconds: 120 # wait for model to load periodSeconds: 10 livenessProbe: httpGet: path: /health port: 8000 initialDelaySeconds: 180 failureThreshold: 3
04 — Helm Charts

Helm Charts & Multi-Service AI Stacks

Helm is Kubernetes's package manager — templated YAML that's parameterized so you can deploy the same chart to dev, staging, and prod with different values files.

🎯
Helm lifecycle: helm install → first deploy. helm upgrade → rolling update. helm rollback → instant revert. Track revisions like git commits.
YAML — values.yaml (AI Stack)
# values.yaml llmService: image: repository: myregistry/llm-service tag: "v1.2.0" replicas: 2 model: "claude-3-5-sonnet-20241022" resources: gpu: 1 memory: "16Gi" qdrant: enabled: true persistence: size: "100Gi" replicas: 3 redis: enabled: true auth: enabled: true ingress: enabled: true host: "ai-api.mycompany.com" tls: true
BASH — Helm Deploy & Rollback
# Deploy AI stack with Helm helm repo add myorg https://charts.mycompany.com helm install ai-stack myorg/ai-platform \ --values production-values.yaml \ --set llmService.replicas=3 \ --namespace ai-production \ --create-namespace # Rolling update (zero-downtime) helm upgrade ai-stack myorg/ai-platform --set llmService.image.tag=v1.3.0 # Rollback if quality drops helm rollback ai-stack 1 # roll back to revision 1
05 — Batch & Production Patterns

Batch Inference & Production Patterns

K8s Jobs for overnight batch processing, HPA for auto-scaling based on queue depth, PodDisruptionBudgets for maintenance safety, and spot/preemptible instances for dramatic cost reduction.

  • K8s Jobs: Process 1M documents overnight with parallelism — 4 pods run simultaneously, K8s tracks completion and restarts failures
  • Horizontal Pod Autoscaler: Scale LLM replicas based on request queue depth in Redis/SQS — not just CPU utilization
  • PodDisruptionBudget: Guarantee minimum replicas during node drain/upgrades — critical for stateful LLM services with warm caches
  • Spot/Preemptible instances: Use GKE Spot or AWS Spot for batch jobs — 60-80% cost reduction; add retry logic for eviction tolerance
YAML — K8s Batch Job (Spot Instances)
# K8s Job for batch inference apiVersion: batch/v1 kind: Job metadata: name: batch-embed-documents spec: parallelism: 4 # 4 pods run in parallel completions: 1000 # process 1000 batches total template: spec: nodeSelector: cloud.google.com/gke-spot: "true" # use preemptible for cost savings containers: - name: embedder image: myregistry/batch-embedder:latest resources: requests: nvidia.com/gpu: 1 restartPolicy: OnFailure

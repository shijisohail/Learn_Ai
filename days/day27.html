Section 01

Why Containerization Matters for AI

AI systems have uniquely complex dependency chains: CUDA versions, model weights, vector stores, and embedding models. Containers eliminate "works on my machine" problems and enable reproducible deployments.

A Python ML environment has a particularly fragile dependency graph: PyTorch version must match CUDA version must match the NVIDIA driver on the host. If your team uses PyTorch 2.1 with CUDA 12.1, but production uses CUDA 11.8, you'll see silent correctness failures, not build errors. Docker solves this by packaging the entire runtime — Python, CUDA libraries, model weights — into a deterministic image.

Kubernetes takes containers further by orchestrating them at scale: ensuring the correct number of replicas are always running, automatically restarting failed pods, routing traffic with proper health checks, and scheduling GPU workloads to nodes with available GPU resources. For AI inference workloads, K8s HPA (Horizontal Pod Autoscaler) can scale your API from 2 to 20 replicas in seconds based on queue depth or GPU utilization metrics.

🐳
Docker Images
Multi-stage builds keep inference images lean. Base: nvidia/cuda:12.1-runtime. Only install inference dependencies — no training tools in production images.
⚙️
K8s Deployments
Deployments manage replica count. AI services need readiness probes (warm model check) and liveness probes (health check), not just TCP checks.
🎮
GPU Scheduling
Request GPU resources in pod spec: nvidia.com/gpu: "1". K8s GPU device plugin ensures only pods with GPU requests get scheduled to GPU nodes.
⚠️
Common pitfall: Setting no resource limits (CPU/memory/GPU) on AI pods. Without limits, a single inference burst can OOM other pods on the node. Always set both requests (scheduler uses this for placement) and limits (enforced at runtime). For LLM inference: request ~4 CPU, 16Gi RAM per replica minimum.

Why this matters: Correct resource requests are essential for K8s scheduling — the scheduler uses requests, not limits, to decide where to place pods. If you request too little, pods get OOM-killed mid-inference, causing lost requests. If you request too much, your cluster can't pack pods efficiently and you waste GPU capacity that costs thousands per month.
Section 04

Production Dockerfile for AI Services

Dockerfile
# ── Stage 1: Builder ───────────────────────────────── FROM python:3.11-slim AS builder WORKDIR /build # Install build deps (removed in final stage) RUN pip install --upgrade pip && pip install uv COPY pyproject.toml uv.lock ./ RUN uv pip install --system --no-cache -r pyproject.toml # ── Stage 2: Runtime ───────────────────────────────── FROM python:3.11-slim AS runtime WORKDIR /app # Non-root user for security RUN groupadd -r aiuser && useradd -r -g aiuser aiuser # Copy only installed packages from builder COPY --from=builder /usr/local/lib /usr/local/lib COPY --from=builder /usr/local/bin /usr/local/bin COPY --chown=aiuser:aiuser . . USER aiuser # Health check — validates model is loaded HEALTHCHECK --interval=30s --timeout=10s --retries=3 \ CMD curl -f http://localhost:8000/health || exit 1 EXPOSE 8000 CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

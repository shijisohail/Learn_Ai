Section 01

Why LangGraph?

LangChain's original LCEL was great for linear chains. LangGraph extends this with a graph abstraction — cycles, branching, state persistence, and human-in-the-loop — everything production agents need.

Real-world agentic workflows aren't linear. A research agent needs to: search → evaluate results → if insufficient, search again → synthesize → review → if review fails, revise → output. LangGraph models this as a directed graph where nodes are processing steps and edges define control flow, including conditional branching based on state.

The key innovation is the State abstraction. Every node reads from and writes to a shared typed state dict. This makes data flow explicit and testable — you can inspect the full state at every step, add checkpointing for fault tolerance, and pause for human review at any node. Think of it as Redux for AI workflows.

🔄

Cycles = Retry & Reflection

Unlike DAGs, LangGraph supports cycles — the agent can retry steps, reflect on its outputs, and iterate until a quality threshold is met. This is how self-correcting agents work.

💾

Built-in Checkpointing

Persist state at every step via sqlite, postgres, or custom checkpointers. Resume interrupted workflows, replay from any checkpoint, and inspect execution history. Critical for long-running agents.

💡

LangGraph vs. Raw Python

You could build agent workflows in raw Python. LangGraph gives you: typed state validation, built-in streaming, checkpoint infrastructure, LangSmith observability, and a community of patterns. For anything beyond a single-turn agent, the framework benefits far outweigh the abstraction cost. For production systems handling real money and real users, checkpointing and observability alone justify it.

Section 02

Core Concepts: State, Nodes, and Edges

LangGraph has three primitives: a typed State schema, Nodes (Python functions that transform state), and Edges (routing rules). Everything else is composition of these three.

langgraph_basics.py
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.sqlite import SqliteSaver
from typing import TypedDict, Annotated
import operator
import anthropic

# 1. Define the State — what persists across nodes
class ResearchState(TypedDict):
    query: str                        # input
    search_results: list[str]         # from search node
    # Annotated with operator.add = append, not overwrite
    messages: Annotated[list, operator.add]
    draft: str                         # from writer node
    quality_score: int                # from reviewer node
    iterations: int                   # loop counter

client = anthropic.Anthropic()

# 2. Define Nodes — functions that transform state
def search_node(state: ResearchState) -> dict:
    """Simulate web search (in production: use Tavily, Brave, etc.)"""
    results = [
        f"Result 1 about {state['query']}: Key finding A",
        f"Result 2 about {state['query']}: Key finding B",
        f"Result 3 about {state['query']}: Key finding C",
    ]
    return {"search_results": results, "iterations": state.get("iterations", 0) + 1}

def writer_node(state: ResearchState) -> dict:
    """Generate a research summary using Claude."""
    context = "\n".join(state["search_results"])
    response = client.messages.create(
        model="claude-sonnet-4-5", max_tokens=512,
        messages=[{"role": "user", "content": f"Summarize: {context}"}]
    )
    return {"draft": response.content[0].text}

def reviewer_node(state: ResearchState) -> dict:
    """Score the draft quality (0-100)."""
    score = min(60 + state.get("iterations", 0) * 20, 100)
    return {"quality_score": score}

# 3. Conditional edge — the routing function
def should_revise(state: ResearchState) -> str:
    if state["quality_score"] >= 80 or state.get("iterations", 0) >= 3:
        return "done"   # → END
    return "revise"    # → search again

# 4. Build the graph
builder = StateGraph(ResearchState)
builder.add_node("search", search_node)
builder.add_node("write", writer_node)
builder.add_node("review", reviewer_node)
builder.add_edge(START, "search")
builder.add_edge("search", "write")
builder.add_edge("write", "review")
builder.add_conditional_edges("review", should_revise, {
    "done": END, "revise": "search"  # The cycle!
})

# 5. Compile with checkpointing
with SqliteSaver.from_conn_string("./checkpoints.db") as memory:
    graph = builder.compile(checkpointer=memory)
    config = {"configurable": {"thread_id": "research-1"}}
    result = graph.invoke({"query": "LangGraph checkpointing patterns"}, config)
    print(f"Quality: {result['quality_score']}, Iterations: {result['iterations']}")
    print(result['draft'])
📌

The Annotated[list, operator.add] Pattern

By default, node return values overwrite state fields. If you want to append to a list (like a messages history), use Annotated[list, operator.add] — this tells LangGraph to use the + operator to merge, not replace. This is how the messages field in the prebuilt chat agents works.

Pitfall: Small Chunks + High top_k = Context Explosion

A 128-token chunk size with top_k=20 means 2,560 tokens of potentially redundant context per query. Small chunks also lose the surrounding context that makes information meaningful. The sweet spot for most use cases is 400-600 tokens with 10-15% overlap and top_k of 3-5.

Section 05

Human-in-the-Loop with LangGraph

Production agents often need human review at critical decision points. LangGraph's interrupt mechanism pauses execution, waits for input, and resumes — across process restarts if checkpointing is enabled.

human_in_loop.py
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.types import interrupt, Command
from typing import TypedDict

class ApprovalState(TypedDict):
    action: str           # what the agent wants to do
    approved: bool        # human approval decision
    result: str           # final output

def plan_node(state: ApprovalState):
    # Agent proposes a potentially risky action
    return {"action": "DELETE all records older than 90 days from production DB"}

def approval_node(state: ApprovalState):
    # interrupt() pauses execution HERE and stores state in checkpoint
    # Returns when graph.invoke() is called again with the human's decision
    human_decision = interrupt({
        "question": "Approve this action?",
        "action": state["action"],
        "risk": "HIGH - irreversible"
    })
    return {"approved": human_decision["approved"]}

def execute_node(state: ApprovalState):
    if state["approved"]:
        return {"result": f"Executed: {state['action']}"}
    return {"result": "Action cancelled by human reviewer"}

builder = StateGraph(ApprovalState)
builder.add_node("plan", plan_node)
builder.add_node("approve", approval_node)
builder.add_node("execute", execute_node)
builder.add_edge(START, "plan")
builder.add_edge("plan", "approve")
builder.add_edge("approve", "execute")
builder.add_edge("execute", END)

with SqliteSaver.from_conn_string(":memory:") as mem:
    graph = builder.compile(checkpointer=mem)
    cfg = {"configurable": {"thread_id": "task-1"}}

    # First invocation — runs until interrupt
    result = graph.invoke({}, cfg)
    print("Waiting for human approval...", result["__interrupt__"])

    # Human reviews and resumes — can be hours/days later
    final = graph.invoke(Command(resume={"approved": True}), cfg)
    print(final["result"])
💡

Human-in-the-Loop is Not Optional for High-Stakes Actions

Any agent with access to write operations (database mutations, email sending, API calls with side effects, code execution) MUST have human review in early deployments. LangGraph's interrupt mechanism makes this trivial to implement and checkpoint-safe — the agent can wait for human approval for minutes or days without losing state. This is the difference between an autonomous agent and an out-of-control one.

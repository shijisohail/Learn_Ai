01 — Motivation

Why Graphs? Not Just Loops

Simple loops work until they don't. The moment your agent needs to branch, retry, or parallelize — you need a graph.

Simple Loop
Linear, fragile, no control flow
1. Call LLM
↓
2. Execute Tool
↓
3. Repeat (max_iters)
↓
❌ Tool Error?
  • Can't branch on different tool results
  • No retry logic for failed tools
  • Can't parallelize subtasks
  • No persistent state across interruptions
LangGraph
Nodes, edges, conditional routing
1. Agent Node
↓ (conditional)
2a. Tool Node
↕ loop possible
2b. Reflect Node
↓
✓ END or retry
  • Conditional edges route on any condition
  • Cycles enable retry/reflection loops
  • State persists across all nodes
  • Pause and resume with checkpointing

When the simple loop breaks down:

Imagine a research agent. With a simple loop: it calls a search tool, gets a bad result, but has no way to decide "retry with different query vs continue". With LangGraph: the agent node sees the bad result, the conditional edge routes to a "retry" node which reformulates the query, and the loop runs again — all with full state preserved. That's the power of graph-based control flow.

04 — Routing

Conditional Routing

The router function is just a Python callable that inspects state and returns the name of the next node. LangGraph wires the rest.

The Router Function
def router(state: AgentState) -> str: # Last message from the agent last_msg = state["messages"][-1] # If the LLM requested tool calls, route to tools node if hasattr(last_msg, "tool_calls") and last_msg.tool_calls: return "tools" # Otherwise the agent is done — route to END return "__end__" # Wire it up in the graph graph.add_conditional_edges("agent", router, { "tools": "tools", "__end__": END })
05 — Implementation

LangGraph Code Walkthrough

Every line explained. This is the minimal complete pattern for a tool-using agent with LangGraph.

################################ # 1. Define State ################################ from typing import TypedDict, Annotated, list from langgraph.graph import StateGraph, END from langgraph.prebuilt import ToolNode from langchain_anthropic import ChatAnthropic import operator
1
TypedDict defines the shape of your state object. Every node receives this dict and returns a partial update. Annotated[list, operator.add] means "append to existing list" instead of replacing it — critical for message history.
class AgentState(TypedDict): messages: Annotated[list, operator.add] # appends, not replaces task: str iteration_count: int
2
Define your tools. LangGraph works with any LangChain-compatible tool. The @tool decorator generates the JSON schema automatically from your docstring and type hints.
from langchain_core.tools import tool @tool def web_search(query: str) -> str: """Search the web for information. Args: query (str): Search query.""" return f"Search results for: {query}" # replace with real implementation tools = [web_search] llm = ChatAnthropic(model="claude-sonnet-4-5").bind_tools(tools)
3
Node functions receive state, do work, and return a dict with only the fields to update. They don't need to return the full state — LangGraph merges the updates.
def agent_node(state: AgentState) -> dict: response = llm.invoke(state["messages"]) return { "messages": [response], # appended via operator.add "iteration_count": state["iteration_count"] + 1 } def router(state: AgentState) -> str: last = state["messages"][-1] if hasattr(last, "tool_calls") and last.tool_calls: return "tools" return "__end__"
4
Build the graph. StateGraph(AgentState) creates a graph that passes AgentState dicts between nodes. ToolNode is a built-in node that handles all tool execution boilerplate.
# Build the graph graph = StateGraph(AgentState) # Add nodes graph.add_node("agent", agent_node) # our custom function graph.add_node("tools", ToolNode(tools)) # built-in tool executor # Entry point graph.set_entry_point("agent") # Conditional routing FROM agent node graph.add_conditional_edges("agent", router) # After tools always go back to agent graph.add_edge("tools", "agent")
5
Compile creates a runnable. Pass a MemorySaver checkpointer to enable persistence — the agent can be paused, resumed, and inspected at any point.
from langgraph.checkpoint.memory import MemorySaver checkpointer = MemorySaver() # swap for SqliteSaver/PostgresSaver in prod app = graph.compile(checkpointer=checkpointer) # Run it config = {"configurable": {"thread_id": "session-001"}} result = app.invoke( {"messages": [{"role": "user", "content": "Research quantum computing"}], "task": "Research quantum computing", "iteration_count": 0}, config=config )
06 — Control

Human-in-the-Loop (HITL)

LangGraph can pause execution before any node, wait for human input, then resume. Critical for production agents taking real-world actions.

🤖 Agent proposes action
→
⏸ INTERRUPT
before: execute
→
👤 Human reviews action
→
✅ APPROVE
resume()
→
Execute & continue
🤖 Agent proposes action
→
⏸ INTERRUPT
→
👤 Human reviews action
→
❌ REJECT
update_state()
→
Agent retries
# Compile with interrupt BEFORE the execute_action node app = graph.compile( checkpointer=checkpointer, interrupt_before=["execute_action"] # pause here! ) # Run until interrupt state = app.invoke(initial_state, config=config) # → agent proposes action, execution PAUSES here # Human reviews... action = state["proposed_action"] approved = human_review(action) # your UI/CLI logic if approved: # Resume from where we left off result = app.invoke(None, config=config) else: # Inject feedback and retry app.update_state(config, {"feedback": "Don't delete files, read only"}) result = app.invoke(None, config=config)

Key Insight: State is Immutable Per Node

State is immutable per node — each node returns updates, it doesn't mutate directly. The graph merges updates using reducers. add_messages appends to the messages list; everything else replaces the field value. This makes state transitions predictable and debuggable — you can replay any step.

Checkpointer Setup
# SqliteSaver — production-ready checkpoint backend from langgraph.checkpoint.sqlite import SqliteSaver checkpointer = SqliteSaver.from_conn_string("checkpoints.db") graph = workflow.compile(checkpointer=checkpointer) config = {"configurable": {"thread_id": "user-123-session-1"}} # First run result = graph.invoke({"query": "..."}, config=config) # Resume after crash — same thread_id continues result = graph.invoke(None, config=config) # None = resume
Real-World Impact

Without checkpointing: a 30-minute agent that crashes at minute 28 starts completely over — wasting API calls, time, and money.

With checkpointing: it resumes from the last successful step. Only the failed node needs to re-run.

Section 01

Capstone Architecture Overview

A multi-provider knowledge assistant that routes queries intelligently: factual questions go to the RAG chain, complex research questions dispatch a CrewAI research crew. Every response passes a RAGAS quality gate before reaching users.

🏗️
Design Philosophy: Each component from Week 2 solves a specific production problem. LangChain LCEL makes the pipeline composable and streamable. Qdrant provides sub-millisecond hybrid vector search. CrewAI handles multi-step research. RAGAS catches hallucinations before they reach users. The intent router minimizes cost by using CrewAI only when necessary — it costs 10–100× more than a RAG chain call.

The architecture separates concerns cleanly: retrieval from generation, routing from answering, and quality control from serving. Each layer can be upgraded independently — swap Qdrant for Pinecone, swap Claude for Gemini, or replace the intent router with a fine-tuned classifier without touching the rest of the system.

This "loosely coupled, tightly monitored" design reflects how real production AI systems are built at companies like Notion, Perplexity, and Linear. The glue between components is always LangChain LCEL — its pipe operator lets you reason about the system as a directed graph of Runnables, not a tangle of callbacks.

Section 02

Building the RAG Foundation

The core retrieval chain: Qdrant vector store with hybrid search, multi-provider LLM with automatic fallback, and a complete LCEL pipeline. This is the production-grade pattern — not a tutorial shortcut.

The RAG chain is the workhorse of the system — it handles ~80% of queries with low latency (typically 1–3 seconds end-to-end). The LCEL pipe operator makes the flow explicit: retriever fetches documents → format_docs serializes them → prompt template inserts context and question → LLM generates → StrOutputParser extracts the string. Every step is independently observable in LangSmith.

Multi-provider fallback is non-negotiable for production. Anthropic, OpenAI, and Google all have occasional outages. Using .with_fallbacks() adds zero complexity to calling code — the fallback is transparent. You can also add a third-tier fallback (e.g., Mistral via Together.ai) for maximum resilience. The cost difference between providers is a secondary concern — uptime comes first.

rag_chain.py
from langchain_qdrant import QdrantVectorStore from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain_anthropic import ChatAnthropic from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough from openai import RateLimitError, APIConnectionError # ── Vector Store ────────────────────────────────────────────── vectorstore = QdrantVectorStore.from_existing_collection( url="http://localhost:6333", collection_name="knowledge_base", embedding=OpenAIEmbeddings(model="text-embedding-3-small") ) retriever = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 6, "score_threshold": 0.5} ) # ── Multi-Provider LLM with 2-tier Fallback ─────────────────── primary_llm = ChatAnthropic(model="claude-3-5-sonnet-20241022") fallback_llm = ChatOpenAI(model="gpt-4o") fallback2_llm = ChatOpenAI(model="gpt-4o-mini") # 3rd tier llm = primary_llm.with_fallbacks( [fallback_llm, fallback2_llm], exceptions_to_handle=(RateLimitError, APIConnectionError) ) # ── RAG Prompt ──────────────────────────────────────────────── rag_prompt = ChatPromptTemplate.from_template(""" You are a helpful AI assistant with access to a knowledge base. Answer ONLY using the context below. If the context doesn't contain the answer, say "I don't have that information in my knowledge base." Never guess or hallucinate facts. Context: {context} Question: {question} Provide a clear, accurate answer. Cite sources as [Source: ...]. """) # ── Context Formatter ───────────────────────────────────────── def format_docs(docs): return "\n\n---\n\n".join( f"[Source: {doc.metadata.get('source', 'unknown')}]\n{doc.page_content}" for doc in docs ) # ── Complete RAG Chain (LCEL) ────────────────────────────────── rag_chain = ( {"context": retriever | format_docs, "question": RunnablePassthrough()} | rag_prompt | llm | StrOutputParser() ) # Invoke (blocking): answer = rag_chain.invoke("What is Matryoshka Representation Learning?") # Stream (real-time UI — works with the same chain, zero changes): for chunk in rag_chain.stream("Explain LoRA fine-tuning"): print(chunk, end="", flush=True)
💡
Why this matters: The LCEL pipe operator isn't syntax sugar — it creates a true computation graph that LangSmith can trace step-by-step. You get automatic retry logic, streaming, batch processing, and async support for free. A traditional chain would require custom code for each of these; LCEL provides them all as first-class capabilities of every Runnable.
⚠️
Common Pitfall — score_threshold: Setting a score threshold too low means you retrieve low-relevance chunks that dilute the context and confuse the LLM. Too high, and you return no results for valid queries. Start at 0.5, evaluate on your corpus, and tune. Never use the default (no threshold) in production — it always retrieves k results even when they're irrelevant noise.
Section 03

Hybrid Search with Cross-Encoder Reranking

Dense embeddings miss exact keyword matches. BM25 misses semantic similarity. Hybrid search combines both, and cross-encoder reranking selects the top chunks with higher precision than either approach alone.

The Ensemble Retriever merges results from a BM25 retriever (lexical, good for technical terms and proper nouns) and a dense vector retriever (semantic, good for paraphrases and synonyms). Reciprocal Rank Fusion (RRF) combines the ranked lists without requiring score normalization. After merging, a cross-encoder reads each document alongside the query and produces a precise relevance score — far more accurate than cosine similarity, at the cost of higher latency. Use it as a post-retrieval reranker, not for full-corpus search.

hybrid_retriever.py
from langchain.retrievers import BM25Retriever, EnsembleRetriever from langchain.retrievers import ContextualCompressionRetriever from langchain_cohere import CohereRerank # ── Build BM25 retriever from corpus ────────────────────────── # Requires all documents upfront (in-memory index) bm25_retriever = BM25Retriever.from_documents( documents, # list of Document objects k=10 # retrieve more than needed — reranker will cut down ) # ── Dense retriever from Qdrant ─────────────────────────────── dense_retriever = vectorstore.as_retriever(search_kwargs={"k": 10}) # ── Ensemble with RRF weights ────────────────────────────────── ensemble_retriever = EnsembleRetriever( retrievers=[bm25_retriever, dense_retriever], weights=[0.4, 0.6] # 60% dense, 40% BM25 for mixed queries ) # ── Cross-encoder reranker (Cohere) ─────────────────────────── compressor = CohereRerank( model="rerank-english-v3.0", top_n=5 # return only top-5 after reranking ) hybrid_retriever = ContextualCompressionRetriever( base_compressor=compressor, base_retriever=ensemble_retriever ) # Plug directly into RAG chain — same interface as any retriever: rag_chain_hybrid = ( {"context": hybrid_retriever | format_docs, "question": RunnablePassthrough()} | rag_prompt | llm | StrOutputParser() )

Section 04

Intent Router + CrewAI Research Crew

Not every query belongs in a RAG chain. Complex, multi-faceted research questions benefit from a CrewAI crew that can search the web, synthesize across sources, and reason step-by-step.

🔀
Routing Heuristic: Factual = questions with a specific, retrievable answer ("What is the RAGAS faithfulness metric?"). Research = open-ended, comparative, or analytical questions requiring synthesis across sources ("Compare GPT-4o and Claude 3.5 Sonnet for enterprise RAG at scale"). When uncertain, default to RAG — it's 10–100× cheaper and 5–10× faster than a full CrewAI run.

The intent router uses a cheap, fast model (Claude Haiku at ~$0.25/M input tokens) to classify each query before dispatching to the appropriate pipeline. This is a common pattern in production — use the cheapest model that can accurately perform the routing task, and reserve expensive models for generation where quality matters most.

intent_router.py
from crewai import Agent, Task, Crew, Process from crewai_tools import SerperDevTool # ── Cheap model for routing ──────────────────────────────────── # Claude Haiku: $0.25/M input tokens (vs $3/M for Sonnet) router_llm = ChatAnthropic(model="claude-3-haiku-20240307") ROUTER_PROMPT = """Classify this query as 'factual' or 'research'. factual: Has a specific, retrievable answer. One or two sources suffice. research: Open-ended, comparative, or requires synthesis across many sources. Query: {query} Respond with ONLY the word 'factual' or 'research'.""" def route_query(query: str) -> str: classification = router_llm.invoke( ROUTER_PROMPT.format(query=query) ).content.strip().lower() if "factual" in classification: return rag_chain.invoke(query) else: crew = build_research_crew(query) result = crew.kickoff(inputs={"query": query}) return result.raw # ── Research Crew Factory ────────────────────────────────────── def build_research_crew(query: str) -> Crew: researcher = Agent( role="AI Research Analyst", goal=f"Research comprehensively: {query}", backstory="Expert AI researcher with deep technical background", tools=[SerperDevTool()], llm="claude-3-5-sonnet-20241022" ) analyst = Agent( role="Technical Synthesizer", goal="Turn research into a clear, actionable answer", backstory="Specializes in distilling complex ML research", llm="gpt-4o-mini" # cheaper model for synthesis ) return Crew( agents=[researcher, analyst], tasks=[ Task( description=f"Research: {query}. Cover state-of-art, key papers, practical use.", expected_output="Research notes with citations", agent=researcher ), Task( description="Synthesize research into a precise answer for a senior ML engineer.", expected_output="Clear answer with citations and code examples", agent=analyst, context=[research_task] # task dependency ) ], process=Process.sequential, memory=True )
Section 05

RAGAS Evaluation + CI/CD Quality Gate

Quality is not optional. Every RAG pipeline deployment should pass a RAGAS quality gate against a golden evaluation dataset. Failing faithfulness is a hard stop — a hallucinating system is worse than no system.

RAGAS measures four complementary dimensions. Faithfulness asks: are every claim in the answer supported by the retrieved context? This is your anti-hallucination metric. Answer Relevancy asks: does the answer actually address the question? This catches verbose, off-topic responses. Context Precision asks: what fraction of retrieved chunks were actually useful? Low precision means your retrieval is noisy. Context Recall asks: did the retrieved context contain everything needed to answer the question? Low recall means your RAG is missing important information.

Run RAGAS evaluation as a GitHub Actions step that runs against a curated 50–100 question golden set before any deployment. This catches regressions when you change embedding models, retrieval parameters, or prompts. The golden set should be maintained in version control alongside the codebase — it's as important as your test suite.

eval/quality_gate.py
from ragas import evaluate from ragas.metrics import ( faithfulness, answer_relevancy, context_precision, context_recall ) from datasets import Dataset import json, sys # Load golden set (version-controlled, human-curated) with open("eval/golden_set.json") as f: golden_set = json.load(f) # [{question, expected_answer}, ...] # Quality gate thresholds (tune for your use case) THRESHOLDS = { "faithfulness": 0.80, # hard gate — hallucination prevention "answer_relevancy": 0.75, # soft gate — answer quality "context_precision": 0.70, # retrieval signal-to-noise "context_recall": 0.70, # retrieval completeness } def run_quality_gate(): results = [] for item in golden_set: q = item["question"] answer = rag_chain.invoke(q) contexts = [doc.page_content for doc in retriever.invoke(q)] results.append({ "question": q, "answer": answer, "contexts": contexts, "ground_truth": item["expected_answer"] }) dataset = Dataset.from_list(results) scores = evaluate( dataset, metrics=[faithfulness, answer_relevancy, context_precision, context_recall] ) failures = [ f" ✗ {m}: {scores[m]:.2f} < threshold {t}" for m, t in THRESHOLDS.items() if scores.get(m, 0) < t ] if failures: print(f"❌ Quality gate FAILED:\n" + "\n".join(failures)) sys.exit(1) # non-zero exit → CI/CD blocks deployment print(f"✅ Quality gate PASSED — faith:{scores['faithfulness']:.2f}" f" rel:{scores['answer_relevancy']:.2f}" f" prec:{scores['context_precision']:.2f}" f" rec:{scores['context_recall']:.2f}") if __name__ == "__main__": run_quality_gate()
📋
GitHub Actions integration: Add this as a step in your CI workflow: python eval/quality_gate.py. If the script exits with code 1, the deployment step is blocked. This gives you automatic regression detection when teammates change retrieval parameters, swap embedding models, or modify prompts — the most common sources of quality regressions in production RAG systems.

FastAPI Service with Streaming

A production-ready FastAPI service wrapping the entire system — intent routing, RAG, CrewAI dispatch, and streaming responses. This is the deployable artifact.

The FastAPI service exposes two endpoints: POST /ask for streaming responses (server-sent events) and POST /ask/sync for blocking responses. Streaming is the right default for UIs — users see the first token within milliseconds rather than waiting 3–10 seconds for the full response. The StreamingResponse wraps the LCEL .stream() call, which works transparently across the full chain.

main.py
from fastapi import FastAPI, HTTPException from fastapi.responses import StreamingResponse from pydantic import BaseModel import asyncio, json app = FastAPI(title="Knowledge Assistant", version="1.0") class QueryRequest(BaseModel): question: str stream: bool = True # ── Streaming endpoint (SSE) ────────────────────────────────── @app.post("/ask") async def ask_streaming(req: QueryRequest): async def generate(): # Route query to appropriate pipeline route = classify_query(req.question) # "factual" | "research" chain = rag_chain if route == "factual" else research_chain yield f"data: {json.dumps({'route': route})}\n\n" async for chunk in chain.astream(req.question): yield f"data: {json.dumps({'chunk': chunk})}\n\n" yield "data: [DONE]\n\n" return StreamingResponse( generate(), media_type="text/event-stream", headers={"X-Accel-Buffering": "no"} # disable nginx buffering ) # ── Health + metrics endpoint ────────────────────────────────── @app.get("/health") async def health(): return {"status": "ok", "qdrant": check_qdrant(), "llm": "primary"}
Section 07

Production Considerations

What Week 2 covered and what you need before shipping to real users — caching, monitoring, security, and rate limiting. The gap between a demo and a production system lives here.

⚡
Semantic Caching
Store (embedding → answer) pairs in Redis. On each query, check if a near-duplicate embedding exists within a similarity threshold (0.95+). Cache hits return instantly and cost $0 in LLM calls. GPTCache or a custom Redis hash map both work. Expect 20–40% cache hit rates on high-traffic deployments with repetitive queries.
📊
LangSmith Monitoring
Instrument every chain step with LangSmith traces automatically — just set LANGCHAIN_TRACING_V2=true. Track: retrieval latency P95, token usage per request, RAGAS score distribution over time, and cache hit rate. Set alerts when P95 latency exceeds 5 seconds or RAGAS faithfulness drops below your threshold.
🔒
Security Hardening
Prompt injection via user queries is a real attack vector in RAG systems. Sanitize inputs: strip control characters, enforce length limits (≤2000 chars), and detect injection patterns ("Ignore previous instructions..."). Apply output content filtering with moderation APIs. Rate-limit per user ID at the API gateway layer, not just per IP.
💰
Cost Optimization
Three levers: (1) Route cheap queries to cheaper models — Haiku for routing, GPT-4o-mini for simple factual answers, Sonnet only for complex generation. (2) Cache aggressively — embeddings are cheap to store but expensive to regenerate. (3) Use Anthropic prompt caching for system prompts that repeat across requests (up to 90% cache discount).

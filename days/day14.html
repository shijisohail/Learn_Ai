Section 02

Building the RAG Foundation

The core retrieval chain: Qdrant vector store with hybrid search, multi-provider LLM with automatic fallback, and a complete LCEL pipeline. This is the production-grade pattern — not a tutorial shortcut.

Python — Complete RAG Chain (LCEL)
from langchain_qdrant import QdrantVectorStore from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain_anthropic import ChatAnthropic from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough # ── Vector Store ───────────────────────────────────────── vectorstore = QdrantVectorStore.from_existing_collection( url="http://localhost:6333", collection_name="knowledge_base", embedding=OpenAIEmbeddings(model="text-embedding-3-small") ) retriever = vectorstore.as_retriever(search_kwargs={"k": 6}) # ── Multi-Provider LLM with Fallback ───────────────────── # Primary: Claude (quality, long context, nuanced reasoning) primary_llm = ChatAnthropic(model="claude-3-5-sonnet-20241022") # Fallback: GPT-4o (when Anthropic API is down or rate-limited) fallback_llm = ChatOpenAI(model="gpt-4o") llm = primary_llm.with_fallbacks([fallback_llm]) # LangChain handles retries, error catching, and fallback silently # ── RAG Prompt ──────────────────────────────────────────── rag_prompt = ChatPromptTemplate.from_template(""" You are a helpful AI assistant with access to a knowledge base. Answer the question based ONLY on the context below. If the context doesn't contain the answer, say "I don't have that information in my knowledge base" rather than guessing. Context: {context} Question: {question} Provide a clear, accurate answer with specific details from the context. Cite the source of each claim using [Source: ...] notation. """) # ── Context Formatter ──────────────────────────────────── def format_docs(docs): return "\n\n---\n\n".join( f"Source: {doc.metadata.get('source', 'unknown')}\n" f"Page: {doc.metadata.get('page', 'N/A')}\n" f"{doc.page_content}" for doc in docs ) # ── Complete RAG Chain (LCEL) ───────────────────────────── rag_chain = ( { "context": retriever | format_docs, "question": RunnablePassthrough() } | rag_prompt | llm | StrOutputParser() ) # Usage: answer = rag_chain.invoke("What is Matryoshka Representation Learning?") # Streaming (for real-time UI): for chunk in rag_chain.stream("Explain LoRA fine-tuning"): print(chunk, end="", flush=True)
Section 03

CrewAI Research Crew Integration

Not every query belongs in a RAG chain. Complex, multi-faceted research questions benefit from a CrewAI crew that can search the web, synthesize across sources, and reason step-by-step. The intent router makes the call.

🔀
Routing Heuristic: Factual = questions with a specific, retrievable answer ("What is the RAGAS faithfulness metric?"). Research = open-ended, comparative, or analytical questions that benefit from synthesis across multiple sources ("Compare GPT-4o and Claude 3.5 Sonnet for enterprise RAG applications"). When uncertain, default to RAG — it's faster and cheaper.
Python — Intent Router + CrewAI Integration
from crewai import Agent, Task, Crew, Process from crewai_tools import SerperDevTool # ── Intent Classifier ──────────────────────────────────── CLASSIFICATION_PROMPT = """Classify this query as exactly one of: 'factual' or 'research'. factual: Has a specific, retrievable answer. One or two sources suffice. research: Open-ended, comparative, or requires synthesis across many sources. Query: {query} Respond with only the word 'factual' or 'research'.""" classifier_llm = ChatAnthropic(model="claude-3-haiku-20240307") # cheap model for routing def route_query(query: str) -> str: """Route: simple factual → RAG, complex research → CrewAI""" classification = classifier_llm.invoke( CLASSIFICATION_PROMPT.format(query=query) ).content.strip().lower() if "factual" in classification: return rag_chain.invoke(query) else: # Dispatch to CrewAI research crew crew = build_research_crew(query) result = crew.kickoff(inputs={"query": query}) return result.raw # ── Research Crew Factory ──────────────────────────────── def build_research_crew(query: str) -> Crew: researcher = Agent( role="AI Research Analyst", goal=f"Research this query comprehensively: {query}", backstory="Expert AI researcher with 10 years experience", tools=[SerperDevTool()], llm="claude-3-5-sonnet-20241022" ) analyst = Agent( role="Technical Analyst", goal="Synthesize research into a clear, actionable response", backstory="Specializes in synthesizing technical ML research", llm="gpt-4o-mini" ) research_task = Task( description=f"Research: {query}. Find current state of art, key papers, and practical considerations.", expected_output="Comprehensive research notes with citations", agent=researcher ) synthesis_task = Task( description="Synthesize the research into a clear answer for a senior ML engineer.", expected_output="A precise, accurate answer with citations and code examples where applicable", agent=analyst, context=[research_task] ) return Crew( agents=[researcher, analyst], tasks=[research_task, synthesis_task], process=Process.sequential )
Section 04

RAGAS Evaluation Gate

Quality is not optional. Every RAG pipeline deployment should pass a RAGAS quality gate against a golden evaluation dataset. Failing faithfulness is a hard stop — a hallucinating system is worse than no system.

Python — RAGAS CI/CD Quality Gate
from ragas import evaluate from ragas.metrics import faithfulness, answer_relevancy from datasets import Dataset import json # Load golden evaluation dataset (maintained in version control) with open("eval/golden_set.json") as f: golden_set = json.load(f) def retrieve_context(question: str) -> list[str]: return [doc.page_content for doc in retriever.invoke(question)] def evaluate_pipeline(test_questions: list[dict]) -> dict: """Run RAGAS evaluation over the golden question set.""" results = [] for item in test_questions: answer = rag_chain.invoke(item["question"]) results.append({ "question": item["question"], "answer": answer, "contexts": retrieve_context(item["question"]), "ground_truth": item["expected_answer"] }) dataset = Dataset.from_list(results) scores = evaluate(dataset, metrics=[faithfulness, answer_relevancy]) # Quality gate thresholds THRESHOLDS = {"faithfulness": 0.80, "answer_relevancy": 0.75} failures = [ f"{m}: {scores[m]:.2f} < {t}" for m, t in THRESHOLDS.items() if scores.get(m, 0) < t ] if failures: raise ValueError(f"Quality gate FAILED:\n" + "\n".join(failures)) print(f"Quality gate PASSED — faithfulness: {scores['faithfulness']:.2f}, relevancy: {scores['answer_relevancy']:.2f}") return scores # CI/CD integration (e.g., GitHub Actions step): # python -c "from eval import evaluate_pipeline; evaluate_pipeline(golden_set)" # Exit code 1 if quality gate fails → deployment blocked
Section 05

What You Built & What's Next

Week 2 complete. You now have the full production RAG + agent stack. Week 3 goes deeper into the ML layer: how models are actually trained, fine-tuning, and MLOps.

⚡
Rate Limiting & Caching
Production RAG needs exponential backoff on API calls, a Redis embedding cache to avoid redundant billing, and a semantic cache layer (GPTCache or custom) to return cached answers for near-duplicate queries.
📊
Monitoring
Instrument every chain step with LangSmith traces. Track: retrieval latency, token usage per request, faithfulness score distribution, and cache hit rate. Set alerts on p95 latency and on RAGAS score drops.
🔒
Security
Validate and sanitize all user queries before passing to the LLM — prompt injection via user input is a real attack surface in RAG systems. Apply input length limits, output content filtering, and rate limiting per user.
🧬
Week 3 Preview
ML fundamentals: backpropagation, the transformer architecture from scratch, tokenization in depth. Then practical fine-tuning: LoRA/QLoRA with PEFT, Hugging Face Trainer, model evaluation and merging. Finally: MLOps — experiment tracking, model registry, and inference optimization.

Section 01

LoRA: The Mathematics of Low-Rank Adaptation

LoRA decomposes weight updates into the product of two small matrices, dramatically reducing the number of trainable parameters without sacrificing model quality.

The key insight behind LoRA (Hu et al., 2021) is that weight updates during fine-tuning have low intrinsic rank — the update matrix ΔW can be approximated by a product of two much smaller matrices: ΔW ≈ BA, where B ∈ R^(d×r) and A ∈ R^(r×k), with r ≪ min(d, k). Instead of training the full d×k weight matrix (millions of parameters), you only train the r×(d+k) parameters in A and B — typically 100-1000x fewer parameters.

During the forward pass, the output is: h = Wx + BAx, where W is the frozen pretrained weight, B is initialized to zero (so the initial output is identical to the base model), and A is initialized with a normal distribution. The scaling factor α/r controls the effective learning rate of the adapter — larger α means the adapter contributes more to the output. At inference time, you can merge W + BA into a single weight matrix with zero overhead.

The rank parameter r controls the expressivity of the adapter: r=4 captures basic task adaptation (style, format), r=8-16 handles most instruction-tuning tasks, r=32-64 is used for domain adaptation requiring new knowledge. Increasing rank beyond 64 rarely helps and wastes compute — the updates are already full-rank by then, and you're better off increasing training data.

Why This Matters

LoRA's impact on the field cannot be overstated. Before it, fine-tuning a 7B model required 4x A100 80GB GPUs (~$50K hardware). With QLoRA (LoRA + 4-bit quantization), the same fine-tune runs on a single RTX 4090 (24GB) or even a single A10G (24GB) on Lambda Labs for ~$1/hour. This democratized model customization and spawned the entire open-source fine-tuned model ecosystem.

lora_math.py
import torch import torch.nn as nn import math class LoRALayer(nn.Module): """Implements LoRA from scratch — shows exactly what PEFT does internally.""" def __init__(self, in_dim: int, out_dim: int, rank: int, alpha: float): super().__init__() self.rank = rank self.scaling = alpha / rank # α/r — scale the update # Frozen pretrained weight (would be loaded from checkpoint) self.weight = nn.Parameter(torch.randn(out_dim, in_dim), requires_grad=False) # Trainable LoRA matrices — A is Gaussian init, B is zero init self.lora_A = nn.Parameter(torch.empty(rank, in_dim)) self.lora_B = nn.Parameter(torch.zeros(out_dim, rank)) # Kaiming uniform init for A (like default nn.Linear) nn.init.kaiming_uniform_(self.lora_A, a=math.sqrt(5)) def forward(self, x: torch.Tensor) -> torch.Tensor: # Base model output (no grad — frozen) base_out = x @ self.weight.T # LoRA update: x @ A^T @ B^T * (α/r) lora_out = (x @ self.lora_A.T @ self.lora_B.T) * self.scaling return base_out + lora_out # h = Wx + BAx def merge_weights(self) -> torch.Tensor: """Merge LoRA into base weight — zero inference overhead.""" delta_W = (self.lora_B @ self.lora_A) * self.scaling return self.weight + delta_W # merged weight = W + BA # Parameter count comparison d, k, r = 4096, 4096, 16 full_params = d * k # 16,777,216 ≈ 16.7M lora_params = r * (d + k) # 131,072 ≈ 131K reduction = full_params / lora_params # 128x fewer parameters! print(f"Full: {full_params:,} | LoRA r={r}: {lora_params:,} | {reduction:.0f}x reduction")

Section 02

QLoRA: 4-bit Quantization + LoRA

QLoRA (Dettmers et al., 2023) combines NF4 quantization of the base model with LoRA adapters in BF16, enabling 65B model fine-tuning on a single A100 80GB GPU.

The three innovations in QLoRA are: (1) NF4 quantization — NormalFloat4, a data type that is information-theoretically optimal for normally distributed weights (which pretrained LLM weights approximate). NF4 achieves better perplexity than INT4 at the same bit-width. (2) Double quantization — quantizing the quantization constants themselves, saving an additional ~0.37 bits per parameter. (3) Paged optimizers — using NVIDIA's unified memory to page optimizer states between GPU and CPU when GPU runs out of memory, similar to virtual memory in operating systems.

The key trick: the base model weights are quantized to 4-bit and frozen — they never get updated. Only the LoRA adapter weights (in BF16) are trained. During the forward pass, the 4-bit weights are dequantized to BF16 for the computation, then immediately discarded — they're never stored at full precision simultaneously. This is what achieves the memory savings.

QLoRA's quality versus full fine-tuning: on standard benchmarks (MMLU, HellaSwag), QLoRA-tuned models achieve within 1-2% of models fine-tuned at full precision. For most production use cases, this gap is negligible — the practical training cost reduction (10-20x) vastly outweighs the marginal quality difference.

qlora_training.py
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig from peft import LoraConfig, get_peft_model, TaskType, prepare_model_for_kbit_training from trl import SFTTrainer, SFTConfig from datasets import load_dataset import torch MODEL_ID = "mistralai/Mistral-7B-Instruct-v0.3" # ── Step 1: Load base model in 4-bit ───────────────────────────── bnb_cfg = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", # NormalFloat4 — better than INT4 bnb_4bit_use_double_quant=True, # compress quantization constants too bnb_4bit_compute_dtype=torch.bfloat16, ) model = AutoModelForCausalLM.from_pretrained( MODEL_ID, quantization_config=bnb_cfg, device_map="auto" ) # Prepares the 4-bit model for LoRA (casts norms to fp32, etc.) model = prepare_model_for_kbit_training(model) # ── Step 2: Attach LoRA adapters ────────────────────────────────── lora_cfg = LoraConfig( r=16, # rank — higher = more expressive but more params lora_alpha=32, # scaling factor; effective LR = alpha/r * lr target_modules=[ # which Linear layers get adapters "q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj", ], lora_dropout=0.05, # dropout on adapter outputs bias="none", # don't train bias terms task_type=TaskType.CAUSAL_LM, ) model = get_peft_model(model, lora_cfg) model.print_trainable_parameters() # Output: trainable params: 41,943,040 || all params: 7,283,519,488 || 0.58% # ── Step 3: Train with TRL SFTTrainer ──────────────────────────── dataset = load_dataset("HuggingFaceH4/ultrachat_200k", split="train_sft[:5%]") tokenizer = AutoTokenizer.from_pretrained(MODEL_ID) tokenizer.pad_token = tokenizer.eos_token trainer = SFTTrainer( model=model, train_dataset=dataset, args=SFTConfig( output_dir="./mistral-ft", num_train_epochs=2, per_device_train_batch_size=4, gradient_accumulation_steps=4, # effective batch = 16 learning_rate=2e-4, lr_scheduler_type="cosine", warmup_ratio=0.03, bf16=True, max_seq_length=2048, packing=True, # pack multiple short examples together logging_steps=10, save_strategy="epoch", ), ) trainer.train()

Common Pitfall: Forgetting prepare_model_for_kbit_training()

When using 4-bit quantization with bitsandbytes, you MUST call prepare_model_for_kbit_training(model) before attaching LoRA adapters. This function casts the LayerNorm weights to FP32 (they must stay at full precision for numerical stability), disables the embedding layer from being trained, and sets up gradient checkpointing correctly. Skipping this causes NaN losses or silently incorrect gradients.

Section 03

PEFT Beyond LoRA: IA³, Prefix Tuning & DoRA

The PEFT library supports multiple adapter strategies — each with different trade-offs between parameter efficiency, quality, and training speed.

LoRA is not the only PEFT method. IA³ (Infused Adapter by Inhibiting and Amplifying Inner Activations) injects learnable vectors that scale the keys, values, and feedforward activations — only 0.01% of trainable parameters vs. LoRA's 0.3-3%. It's ideal when your fine-tuning budget is extremely limited. However, it requires more training data to converge and achieves slightly lower quality than LoRA.

Prefix Tuning prepends learnable "virtual tokens" to the key and value sequences in every attention layer. The model learns to condition its outputs on these virtual tokens. Unlike LoRA, prefix tuning affects every attention head independently, making it more expressive but harder to merge into the base model. Best suited for sequence-to-sequence tasks like summarization.

DoRA (Weight-Decomposed Low-Rank Adaptation) decomposes the pretrained weight into magnitude and direction components, then applies LoRA only to the directional component. This achieves better quality than LoRA with the same rank, at the cost of slightly higher computational overhead. DoRA is the recommended default for instruction tuning in QLoRA setups.

Instruction Tuning & Data Quality

LoRA fine-tuning is only as good as your training data. Instruction tuning requires (instruction, response) pairs where the instruction is diverse, the response is high quality, and the format matches your target chat template. A critical insight from the Alpaca and LIMA papers: 1000 carefully curated examples often outperform 50,000 noisy ones. Data quality dominates data quantity for fine-tuning.

The instruction tuning pipeline: collect raw data → format to ShareGPT or ChatML → deduplicate → filter by length and quality → apply chat template → train. Each step is a potential failure point. Many practitioners spend 80% of their time on data preparation and 20% on training configuration.

data_preparation.py 
from datasets import load_dataset, Dataset
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("mistralai/Mistral-7B-Instruct-v0.3")

def format_chatml(example):
    """Convert raw instruction-response to chat template."""
    messages = [
        {"role": "user", "content": example["instruction"]},
        {"role": "assistant", "content": example["response"]},
    ]
    text = tokenizer.apply_chat_template(
        messages, tokenize=False, add_generation_prompt=False
    )
    return {"text": text}

def quality_filter(example):
    """Remove low-quality samples."""
    tokens = tokenizer.encode(example["text"])
    response_len = len(tokenizer.encode(example["response"]))
    return (
        100 <= len(tokens) <= 2048  # length bounds
        and response_len >= 30          # response must have substance
        and "As an AI" not in example["response"]  # filter GPT artifacts
    )

dataset = load_dataset("json", data_files="raw_data.jsonl")["train"]
dataset = dataset.map(format_chatml).filter(quality_filter)
dataset.save_to_disk("./processed_dataset")
Why this matters: Filtering out "As an AI" responses from GPT-generated datasets removes the characteristic disclaimers that make fine-tuned models sound robotic and over-cautious. The LIMA paper showed that 1000 carefully human-curated examples produced instruction-following quality competitive with models trained on millions of RLHF examples.

DoRA & Advanced PEFT Techniques

DoRA (Weight-Decomposed Low-Rank Adaptation) decomposes pretrained weights into magnitude and direction components, then applies LoRA only to the direction component while training the magnitude separately. This achieves better performance than LoRA at the same parameter count, particularly for tasks requiring precise numerical outputs. IA³ (Infused Adapter by Inhibiting and Amplifying Inner Activations) is even more parameter-efficient, adding only 0.01% extra parameters by rescaling key, value, and feed-forward activations.

dora_config.py 
from peft import LoraConfig, IA3Config, get_peft_model

# DoRA — same API as LoRA, just set use_dora=True
dora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    use_dora=True,      # The only change from standard LoRA
    lora_dropout=0.05,
    bias="none"
)

# IA³ — ultra low parameter count (0.01% of model)
ia3_config = IA3Config(
    target_modules=["k_proj", "v_proj", "down_proj"],
    feedforward_modules=["down_proj"],
    # No rank! IA³ adds scalar multipliers, not low-rank matrices
)

model = get_peft_model(base_model, dora_config)
model.print_trainable_parameters()
# trainable params: 20,971,520 || all params: 3,773,071,616 || 0.56%
Common pitfall: When using merge_and_unload() after training, the merged model has slightly different numerical properties than the PEFT model. Always benchmark the merged model separately before deployment — occasionally merging introduces instability in long-context tasks that wasn't present during training.

QLoRA Optimizer Configuration

QLoRA uses paged AdamW optimizers — a feature unique to bitsandbytes that pages optimizer states to CPU RAM when GPU memory is insufficient. This prevents OOM crashes during training on consumer GPUs. Combined with gradient checkpointing, you can fine-tune a 70B model on a single A100 (80GB). The paging adds ~15% overhead vs standard AdamW but is far better than the alternative (OOM).

qlora_trainer.py

DPO & RLHF for Alignment

After SFT (Supervised Fine-Tuning), you have a model that follows instructions but may produce harmful, biased, or low-quality outputs. Alignment techniques teach the model human preferences. RLHF (Reinforcement Learning from Human Feedback) uses a separate reward model trained on human comparisons; DPO (Direct Preference Optimization) eliminates the reward model entirely by optimizing directly on preference pairs, making it 2-3× cheaper to implement.

DPO training data is (prompt, chosen_response, rejected_response) triples — pairs where a human preferred chosen over rejected. The DPO loss increases the probability of chosen responses and decreases rejected ones, relative to a frozen reference model (the original SFT checkpoint). The reference model prevents the model from diverging too far from its SFT initialization — a critical stability mechanism.

dpo_training.py 
from trl import DPOTrainer, DPOConfig
from datasets import load_dataset

# Dataset format: prompt + chosen + rejected
dataset = load_dataset("HuggingFaceH4/ultrafeedback_binarized", split="train")
# Each example: {"prompt": "...", "chosen": [...], "rejected": [...]}

dpo_config = DPOConfig(
    output_dir="./dpo-output",
    beta=0.1,                    # KL penalty — lower = more aggressive
    learning_rate=5e-7,          # much lower than SFT
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    num_train_epochs=1,           # DPO needs less — 1-3 epochs typical
    bf16=True,
    optim="paged_adamw_32bit",
    max_length=1024,
    max_prompt_length=512,
)

trainer = DPOTrainer(
    model=sft_model,           # fine-tuned SFT model (policy)
    ref_model=ref_model,       # frozen SFT checkpoint (reference)
    args=dpo_config,
    train_dataset=dataset,
    tokenizer=tokenizer,
)
trainer.train()
Why this matters: DPO is how most modern open-source instruction-following models are aligned — Zephyr-7B, Mistral-Instruct, and many others use DPO on top of SFT. The beta parameter controls how strongly to enforce human preferences vs. staying close to the original model. Too low (beta=0.01) causes reward hacking; too high (beta=0.5) causes the model to barely change.

Evaluating Fine-Tuned Models

Perplexity on a held-out set is necessary but not sufficient. Low perplexity doesn't guarantee good instruction-following. You need task-specific evaluation: run the model on a benchmark relevant to your use case, measure exact match or semantic similarity, and compare against the baseline. For chat models, use MT-Bench or AlpacaEval. For code, use HumanEval. For RAG, use RAGAS metrics on your domain dataset.

evaluation.py 
from evaluate import load
import torch

# 1. Perplexity on held-out test split
def compute_perplexity(model, tokenizer, texts):
    total_loss = 0
    for text in texts:
        inputs = tokenizer(text, return_tensors="pt").to(model.device)
        with torch.no_grad():
            loss = model(**inputs, labels=inputs.input_ids).loss
        total_loss += loss.item()
    return torch.exp(torch.tensor(total_loss / len(texts)))

# 2. ROUGE for summarization tasks
rouge = load("rouge")
results = rouge.compute(predictions=model_outputs, references=ground_truth)
print(f"ROUGE-L: {results['rougeL']:.3f}")

# 3. BERTScore for semantic similarity
bertscore = load("bertscore")
results = bertscore.compute(
    predictions=model_outputs, references=ground_truth, lang="en"
)
print(f"BERTScore F1: {sum(results['f1'])/len(results['f1']):.3f}")
Common pitfall: Evaluating on training data or on data generated by the same model used to create training data (data contamination) gives artificially high scores. Always hold out an evaluation set before any fine-tuning, and ensure it comes from a different source than your training data. For LLM-as-judge evaluations, don't use the same model you're evaluating as the judge.

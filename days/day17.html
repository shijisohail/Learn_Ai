Section 01

When to Fine-Tune

The decision tree: prompt engineering first, then RAG, then fine-tuning. Fine-tuning is the right tool for changing HOW a model responds — not for adding new knowledge.

⚠️
Golden Rule: Prompt engineering → RAG → Fine-tuning. Each step is 10–100x more expensive. Fine-tuning does not reliably add new factual knowledge — use RAG for that.
✏️
Prompt Engineering
Use when: you need different behavior for a task. System prompts, few-shot examples, chain-of-thought. Solves 80% of problems.
Cost: Near Zero
📚
RAG
Use when: the model lacks knowledge — your documents, code repos, recent data. Knowledge is updateable without retraining.
Cost: Low–Medium
🎛️
Fine-Tuning
Use when: you need to change style, tone, output format, or domain-specific terminology. QLoRA on 7B + 1000 examples ≈ $10–50 on Lambda Labs.
Cost: $10–$500+
✅ Valid fine-tuning use cases
  • Consistent brand voice / writing style
  • Structured output format (always return JSON with specific keys)
  • Domain-specific terminology and abbreviations
  • Shorter, more direct responses for your use case
  • Instruction-following patterns specific to your workflow
❌ Invalid fine-tuning use cases
  • Adding new factual knowledge (use RAG instead)
  • Keeping up with recent events (model's knowledge is frozen)
  • Fixing hallucinations about specific entities
  • Injecting private data that must stay updateable
  • Tasks solvable with a better system prompt
Section 02

LoRA Mathematics

Instead of updating all W, LoRA decomposes the weight delta into two small matrices. The result: <1% of parameters are trainable, but the model learns new behaviors effectively.

How LoRA works

Full fine-tune 100% params trainable
LoRA r=16 ~0.3% params trainable
W' = W + (alpha/r) × B × A

Key hyperparameters

  • Rank r — expressiveness. r=8: minimal changes. r=64: expressive changes. Default: r=16.
  • lora_alpha — scaling. effective_delta = (alpha/r) × BA. Set alpha = 2×r as a starting point.
  • target_modules — which projections to adapt. Minimum: q_proj + v_proj. Maximum: all linear layers.
  • Adapter size on disk — much smaller than the full model. A r=16 adapter for 7B ≈ 50MB.
PYTHON
from peft import LoraConfig, get_peft_model, TaskType from transformers import AutoModelForCausalLM import torch model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B", torch_dtype=torch.bfloat16, device_map="auto" ) lora_config = LoraConfig( r=16, # rank — expressiveness of adapters lora_alpha=32, # scaling: effective_delta = (32/16) * BA = 2 * BA target_modules=[ # which weight matrices to adapt "q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj" ], lora_dropout=0.05, bias="none", task_type=TaskType.CAUSAL_LM ) peft_model = get_peft_model(model, lora_config) peft_model.print_trainable_parameters() # trainable params: 9,437,184 || all params: 3,212,857,344 || trainable%: 0.29%
Section 03

QLoRA — 4-bit Quantization + LoRA

QLoRA loads the base model in 4-bit (saving ~75% VRAM) while keeping the LoRA adapters in BF16. Fine-tune a 7B model on a single 16GB GPU.

  • NF4 (Normal Float 4) — quantization format optimized for normally-distributed weight values. Better than int4 for LLM weights.
  • Double quantization — quantize the quantization constants too. Saves another ~0.4 bits per parameter.
  • Memory savings — 7B model in 4-bit ≈ 4 GB VRAM for weights (vs 14 GB in BF16). Enables fine-tuning on a single consumer GPU.
  • Training in BF16 — computation happens in BF16 even though the base weights are stored in INT4. No accuracy sacrifice in the adapter.
PYTHON
from transformers import BitsAndBytesConfig from peft import prepare_model_for_kbit_training # QLoRA configuration bnb_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", # Normal Float 4 bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True, # double quantization ) # Load model in 4-bit model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B", quantization_config=bnb_config, device_map="auto" ) # Prepare for k-bit training (enables gradient checkpointing) model = prepare_model_for_kbit_training(model) # Apply LoRA on top of 4-bit model peft_model = get_peft_model(model, lora_config)
Section 04

Dataset Preparation & SFTTrainer

SFTTrainer from the trl library handles tokenization, packing, and response masking automatically. It's the fastest path from dataset to trained adapter.

  • Alpaca format — instruction + input + output. Good for single-turn tasks without a conversation structure.
  • Chat format — list of role/content dicts. Required for instruction-following models using apply_chat_template.
  • Response template masking — train only on the assistant's output, not the user's prompt. Critical to avoid teaching the model to imitate user turns.
  • Packing — concatenate multiple short examples into one sequence. Maximizes GPU utilization for short training examples.
PYTHON
from trl import SFTTrainer, SFTConfig from datasets import load_dataset # Load and format dataset dataset = load_dataset("json", data_files="train.jsonl")["train"] # Format: apply chat template def format_example(example): messages = [ {"role": "user", "content": example["instruction"]}, {"role": "assistant", "content": example["output"]} ] return {"text": tokenizer.apply_chat_template( messages, tokenize=False, add_generation_prompt=False )} dataset = dataset.map(format_example) # SFTTrainer — handles tokenization, packing, response masking trainer = SFTTrainer( model=peft_model, train_dataset=dataset, tokenizer=tokenizer, args=SFTConfig( output_dir="./lora-llama3", num_train_epochs=3, per_device_train_batch_size=4, gradient_accumulation_steps=4, # effective batch = 16 learning_rate=2e-4, warmup_ratio=0.1, lr_scheduler_type="cosine", bf16=True, logging_steps=10, save_steps=100, max_seq_length=2048, packing=True, # pack multiple short examples together ) ) trainer.train() trainer.save_model("./lora-adapter")
Section 05

Adapter Merging & Evaluation

Once trained, merge the LoRA adapter into the base model weights for deployment. Measure perplexity before and after to quantify what the model gained.

  • merge_and_unload() — integrates LoRA weights into base model permanently. Result is a standard model, no PEFT dependency at inference time.
  • When to merge — before deploying to vLLM, TensorRT, or any framework that doesn't support PEFT adapters natively.
  • Perplexity evaluation — compare base model vs fine-tuned on a held-out eval set. Lower perplexity = better fit to your distribution.
  • GGUF export — convert the merged model to GGUF with llama.cpp for Ollama deployment on consumer hardware.
PYTHON
# Load trained adapter and merge from peft import PeftModel base_model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B", torch_dtype=torch.bfloat16, device_map="auto" ) peft_model = PeftModel.from_pretrained(base_model, "./lora-adapter") # Merge LoRA weights into base model merged_model = peft_model.merge_and_unload() merged_model.save_pretrained("./merged-llama3-finetuned") tokenizer.save_pretrained("./merged-llama3-finetuned") # Evaluate: compare perplexity on held-out set import math def compute_perplexity(model, eval_dataloader): total_loss = 0 for batch in eval_dataloader: with torch.no_grad(): outputs = model(**batch) total_loss += outputs.loss.item() return math.exp(total_loss / len(eval_dataloader)) base_ppl = compute_perplexity(base_model, eval_dataloader) ft_ppl = compute_perplexity(merged_model, eval_dataloader) print(f"Base perplexity: {base_ppl:.2f} | Fine-tuned: {ft_ppl:.2f}")

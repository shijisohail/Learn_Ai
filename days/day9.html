01 — Landscape

The Provider Ecosystem

Provider choice is an engineering decision, not just a cost optimisation. It affects your data compliance posture, latency SLA, feature availability, and vendor lock-in exposure.

In 2025, the LLM provider landscape has stratified into three tiers: frontier closed-source models (GPT-4o, Claude 3.5, Gemini 1.5 Pro), efficient mid-tier models (GPT-4o-mini, Claude Haiku, Gemini Flash), and open-weight models that you self-host (Llama 3, Mistral, Qwen). Each tier serves different use cases and cost profiles.

The critical engineering insight is to write code against a unified abstraction layer, not against any single provider's SDK. Whether you use LangChain's model interfaces or a thin custom wrapper, swapping providers should be a config change — not a rewrite. This enables cost routing (use cheap models for simple tasks), graceful fallback, and A/B testing across providers in production.

💡
Provider abstraction pattern: Write code against a unified interface (LangChain, LiteLLM, or your own thin wrapper). Swap providers via config, not code changes. This enables cost routing and graceful fallback with zero application changes.

02 — OpenAI

OpenAI API — GPT-4o & Structured Outputs

OpenAI's key differentiator is ecosystem depth: the largest library of integrations, the most battle-tested function calling, and Structured Outputs — guaranteed JSON Schema adherence via constrained decoding, not prompting tricks.

GPT-4o's Structured Outputs feature uses constrained decoding at the token sampling level to guarantee the output matches your Pydantic schema. This is fundamentally different from JSON mode, which just biases toward JSON but can still produce malformed output. With Structured Outputs, schema violations are mathematically impossible — every generated token is restricted to valid continuations of your schema grammar.

The Batch API is equally important for backend engineers: submit up to 50,000 requests asynchronously, pay 50% of the standard rate, and receive results within 24 hours. This completely changes the economics of offline workloads like nightly evaluations, bulk document classification, and dataset enrichment pipelines.

Structured Outputs

Guarantees the response matches your Pydantic schema exactly — no hallucinated keys, no type mismatches. Uses constrained decoding at the token level, not post-processing.

Use over JSON mode when: schema correctness is critical (extraction pipelines, structured data for DB inserts, API contract enforcement).
Batch API — 50% Discount

Submit up to 50,000 requests asynchronously, results within 24h at 50% cost discount. Ideal for offline workloads: nightly evals, bulk document classification, dataset enrichment.

When not to use: real-time user-facing features where latency matters.
openai_structured.py
from openai import OpenAI from pydantic import BaseModel, Field client = OpenAI() class ExtractedData(BaseModel): title: str = Field(description="Document title") summary: str = Field(description="One-sentence summary") key_entities: list[str] = Field(description="Named entities (people, orgs)") sentiment: str = Field(description="positive | neutral | negative") # Structured Outputs — guaranteed schema compliance via constrained decoding completion = client.beta.chat.completions.parse( model="gpt-4o-2024-11-20", messages=[{"role": "user", "content": f"Extract from: {document}"}], response_format=ExtractedData, ) data: ExtractedData = completion.choices[0].message.parsed print(data.title) # "GPT-4o Technical Report" — typed Python object print(data.key_entities) # ["OpenAI", "Sam Altman", ...] # Batch API — 50% discount for async jobs batch = client.batches.create( input_file_id=upload_jsonl(requests), endpoint="/v1/chat/completions", completion_window="24h" ) print(batch.id) # Poll until batch.status == "completed"
⚠️
Common Pitfall — JSON mode vs Structured Outputs: JSON mode (response_format: {"type": "json_object"}) only guarantees valid JSON — not that your keys/types match your schema. Always use Structured Outputs (.parse()) for schema-critical workloads. JSON mode can still hallucinate unexpected keys.
03 — Anthropic

Anthropic Claude — Extended Context, Caching & Tool Use

Claude's 200K context window is the largest among frontier models. Combined with prompt caching for repeated system prompts and a clean tool use API, it's the top choice for document intelligence and agentic applications.

Prompt caching is Claude's most underrated feature. By marking system prompt blocks with a cache header, Anthropic caches the KV activations for that prefix. When subsequent requests share that prefix, Claude skips re-processing it entirely — cutting input costs by ~90% on the cached portion and reducing latency significantly. For a RAG application with a large, static system prompt, this is an immediate win.

Extended Thinking unlocks Claude's reasoning capabilities for hard problems. When enabled, Claude produces a hidden scratchpad of reasoning tokens before generating the final response. This is particularly effective for multi-step code architecture decisions, complex mathematical derivations, and situations where you need to trade latency for quality on infrequent but high-stakes queries.

Prompt Caching

Mark system prompt blocks with cache_control: {"type": "ephemeral"}. Anthropic caches the prefix KV — subsequent calls re-use the cached computation.

Cache hit: ~90% cost reduction on cached tokens
Extended Thinking

Enable with thinking: {"type": "enabled", "budget_tokens": 8000}. Claude reasons in a hidden scratchpad before the final answer.

Best for: multi-step math, complex code architecture, hard reasoning. Expect 2–5x latency increase.
claude_tool_use.py
import anthropic client = anthropic.Anthropic() # Prompt caching — mark static system prompt for caching response = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=1024, system=[{ "type": "text", "text": LARGE_SYSTEM_PROMPT, # ~50K tokens "cache_control": {"type": "ephemeral"} }], messages=[{"role": "user", "content": query}] ) # response.usage.cache_read_input_tokens → shows cached tokens on 2nd+ call # Tool use with agentic loop tools = [{ "name": "search_database", "description": "Search product database by keyword", "input_schema": { "type": "object", "properties": { "query": {"type": "string"}, "limit": {"type": "integer", "default": 10} }, "required": ["query"] } }] response = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=1024, tools=tools, messages=[{"role": "user", "content": "Find products under $50"}] ) if response.stop_reason == "tool_use": tool_call = next(b for b in response.content if b.type == "tool_use") result = search_database(**tool_call.input) # Return result to Claude in a tool_result message block
04 — Open-Weight

Google Gemini & Open-Weight Models

Gemini's 1M context window enables use cases impossible with other providers. Open-weight models unlock full data sovereignty and eliminate per-token costs at scale.

Gemini 1.5 Pro's 1M token context window is genuinely transformative. One million tokens is approximately 700,000 words — that's the entire Lord of the Rings trilogy, a full codebase of 30,000 lines, or 10 hours of audio transcript. The practical implication: you can skip RAG entirely for small-to-medium knowledge bases and just stuff the whole corpus into the context window. This dramatically simplifies architecture at the cost of higher per-call latency and expense.

Self-hosted open-weight models (Llama 3, Mistral, Qwen) are the right call when you have strict data residency requirements, when inference volume is high enough that per-token API costs exceed GPU rental costs, or when you need to fine-tune on proprietary data that can't leave your environment. vLLM provides PagedAttention for 10–20x throughput over naive serving; Ollama makes local development trivial.

Gemini 1.5 Pro — 1M Context
  • → 1M tokens ≈ 700K words, or ~30,000 lines of code
  • → Native video, audio, image, and code understanding
  • → Google AI Studio (prototyping) vs Vertex AI (enterprise, VPC)
  • → Can skip RAG for small-medium corpora — entire doc in context
  • → Cost at 1M tokens is significant — benchmark per-call cost vs RAG
When to Self-Host (Open-Weight)
  • → Data sovereignty — PII, PHI, or classified data must stay in VPC
  • → Cost at scale — millions of calls/day: GPU < API cost
  • → Custom fine-tuned models (not available via APIs)
  • → vLLM for production (PagedAttention, continuous batching)
  • → Ollama for local dev (automatic model management)

05 — Strategy

Multi-Provider Strategy & Fallback Patterns

Production systems should never depend on a single provider. A robust fallback chain plus a cost routing layer gives resilience and optimal spend — with zero rewrite required when providers have outages.

The most effective production pattern is a three-tier strategy: a cheap, fast model for intent classification and simple tasks; a frontier model for complex reasoning; and a fallback path for provider outages. This reduces costs by 60–80% while maintaining quality where it matters, because most real-world queries are simple.

💡
Why This Matters: A 3-tier cost routing strategy typically saves 60–80% on LLM spend vs routing everything to the best model. Classify with Haiku ($0.25/$1.25), route complex tasks to Sonnet ($3/$15), and use local Llama for privacy-sensitive calls. The classifier itself costs fractions of a cent per query.
fallback_strategy.py
from langchain_anthropic import ChatAnthropic from langchain_openai import ChatOpenAI from langchain_core.runnables import RunnableBranch # Tier 1: cheap + fast (intent classification, simple QA) cheap_model = ChatAnthropic(model="claude-3-haiku-20240307") # Tier 2: frontier (complex reasoning, code, analysis) frontier = ChatAnthropic(model="claude-3-5-sonnet-20241022") # Tier 3: fallback provider (OpenAI) if Anthropic has issues backup = ChatOpenAI(model="gpt-4o") # Fallback chain — tries each in order on exception robust_frontier = frontier.with_fallbacks([backup]) # Cost routing branch — classify complexity first def is_complex(x): return len(x["query"].split()) > 50 routed_chain = RunnableBranch( (is_complex, prompt | robust_frontier | parser), # complex → frontier prompt | cheap_model | parser # simple → haiku ) # Circuit breaker (simplified — use tenacity in production) from tenacity import retry, stop_after_attempt, wait_exponential @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, max=10)) def call_with_retry(chain, input_dict): return chain.invoke(input_dict)
⚠️
Common Pitfall — Fallbacks on Exceptions Only: LangChain's .with_fallbacks() only triggers on Python exceptions (rate limits, network errors). It does NOT trigger on empty or nonsensical responses. Add an output validator step to detect quality failures and raise an exception manually to trigger the fallback.

08 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. Abstract providers behind a unified interface — swap via config, not code.
  • 2. Use Structured Outputs (not JSON mode) for schema-critical extraction.
  • 3. Prompt caching = 90% cost reduction on static system prompts — use it.
  • 4. 3-tier routing (cheap→frontier→fallback) reduces cost 60–80%.
  • 5. Self-host when: data stays in VPC, or call volume makes GPU < API cost.

Section 01

MLflow Fundamentals: Runs, Experiments & Artifacts

MLflow's tracking API provides a persistent, queryable record of every training run — parameters, metrics, code versions, and model artifacts.

MLflow organizes experiments hierarchically: an Experiment is a named collection of Runs, where each run captures a complete training execution with its hyperparameters, metrics time series, code version (git hash), environment, and output artifacts. The tracking server stores this in a SQLite database (local) or a remote backend like PostgreSQL + S3 (production). Every run gets a UUID, making it trivially reproducible — you can always trace which code version and parameters produced a given model checkpoint.

The three core logging primitives are: mlflow.log_param(key, value) for hyperparameters (logged once per run), mlflow.log_metric(key, value, step=N) for time-series metrics (loss, accuracy), and mlflow.log_artifact(path) for files (model checkpoints, plots, confusion matrices). The distinction between params and metrics matters — params are indexed for search/filter, metrics are stored as time series for visualization.

MLflow's autologging feature (mlflow.pytorch.autolog(), mlflow.transformers.autolog()) hooks into popular frameworks and automatically logs everything — optimizer state, loss curves, model summary, even the training code. For quick experiments, autologging is invaluable. For production, explicit logging is preferred — autologging can capture too much noise and may log sensitive data.

Why This Matters

Without experiment tracking, ML development quickly degenerates into "which checkpoint was the good one?" and "what hyperparameters did I use for the model that's in production?" MLflow makes these questions answerable by creating an audit trail. In a team setting, it's the difference between reproducible science and folklore — "I think we used learning rate 2e-4 but I'm not sure."

Section 02

Model Registry & Lifecycle Management

The MLflow Model Registry provides versioned model storage with a promotion workflow from experiment to production.

The Model Registry sits on top of the tracking server and adds governance to model promotion. Every time you call mlflow.register_model(), it creates a new version under a named model. Versions are immutable — you never overwrite them, you create new ones. This guarantees that you can always roll back to any previous version if a production model regresses.

The lifecycle stages (None → Staging → Production → Archived) provide a structured promotion workflow. Models start in None (just registered from a run). Promising models move to Staging for integration testing. The best Staging model gets promoted to Production — marking it as the deployment target. Old production models are moved to Archived rather than deleted, preserving the full history. This maps directly to standard software deployment practices that backend engineers will recognize.

In a CI/CD pipeline, the registry stage becomes the deployment trigger: a webhook fires when a model reaches Production stage, triggering a GitHub Actions workflow that builds a new Docker image with the latest model and deploys to Kubernetes. This makes MLflow the single source of truth for "which model is deployed where."

Section 03

LangSmith: LLM-Specific Observability

MLflow tracks training metrics; LangSmith tracks inference — the prompt, completion, latency, cost, and feedback signal for every LLM call in production.

Standard ML metrics (loss, accuracy) are insufficient for LLMs. You need to track: token counts (cost), latency per token, prompt template versions, model temperature settings, retrieved context quality (for RAG), user feedback signals (thumbs up/down), and evaluation scores (faithfulness, answer relevance via RAGAS). LangSmith captures all of this for every production call automatically when you set the environment variables.

LangSmith's dataset and testing features enable systematic regression testing: create a dataset of challenging prompts, run your model against them, and compare the output distribution before and after model updates. This catches quality regressions before they reach production — critical for maintaining user trust.

langsmith_integration.py
import os from langsmith import Client, traceable from langchain_openai import ChatOpenAI # ── Setup (set env vars) ────────────────────────────────────────── os.environ["LANGCHAIN_TRACING_V2"] = "true" os.environ["LANGCHAIN_API_KEY"] = "ls__..." os.environ["LANGCHAIN_PROJECT"] = "my-llm-app" # ── Auto-trace any function with @traceable ─────────────────────── @traceable(name="rag-answer", tags=["production"]) def answer_question(question: str, context: str) -> str: llm = ChatOpenAI(model="gpt-4o-mini") response = llm.invoke(f"Context: {context}\nQuestion: {question}") return response.content # LangSmith captures: prompt, completion, tokens, latency # ── Manual client for programmatic control ──────────────────────── ls_client = Client() # Create evaluation dataset from production traffic dataset = ls_client.create_dataset("rag-eval-v1", description="100 production queries") ls_client.create_examples( inputs=[{"question": q} for q in prod_questions], outputs=[{"answer": a} for a in expert_answers], dataset_id=dataset.id, ) # Run evaluation and get scores results = ls_client.run_on_dataset( dataset_name="rag-eval-v1", llm_or_chain_factory=answer_question, evaluation="qa", # built-in QA evaluator )

Hyperparameter Search & Experiment Tracking Alternatives

MLflow integrates with Optuna for hyperparameter search. The combination is powerful: Optuna suggests parameter combinations using Bayesian optimization (TPE sampler), each trial runs a training job, and MLflow records every trial as a run. After the search, you can query the MLflow tracking server to find the Pareto-optimal configuration across multiple metrics.

The key advantage of MLflow over pure Optuna is persistence — if your search crashes after 50 trials, MLflow has logged all 50 results. Optuna's in-memory storage would lose them. Always use MLflow as the backing store for production hyperparameter searches.

hparam_search.py 
import optuna
import mlflow

def objective(trial):
    with mlflow.start_run(nested=True):  # nested under search run
        lr = trial.suggest_float("lr", 1e-5, 1e-3, log=True)
        rank = trial.suggest_categorical("rank", [8, 16, 32, 64])
        dropout = trial.suggest_float("dropout", 0.0, 0.3)

        mlflow.log_params({"lr": lr, "rank": rank, "dropout": dropout})

        # Train model with these hyperparameters
        val_loss = train_and_evaluate(lr=lr, rank=rank, dropout=dropout)

        mlflow.log_metric("val_loss", val_loss)
        return val_loss

# Parent run wraps the entire search
with mlflow.start_run(run_name="hparam_search_v1"):
    study = optuna.create_study(direction="minimize",
                                  sampler=optuna.samplers.TPESampler(seed=42))
    study.optimize(objective, n_trials=50, n_jobs=4)  # 4 parallel trials

    print(f"Best: {study.best_params} → val_loss={study.best_value:.4f}")
Why this matters: Bayesian optimization (Optuna's TPE sampler) typically finds better hyperparameters than grid search with 5-10× fewer trials. For LoRA fine-tuning, the rank and learning rate interact strongly — Bayesian search discovers this interaction while grid search treats them independently.

Custom MLflow Flavors

MLflow's built-in flavors (sklearn, pytorch, transformers) cover most use cases, but you can register custom model flavors for bespoke architectures. A custom flavor defines how to save and load your model, enabling consistent deployment across mlflow serve, SageMaker, and other targets.

custom_flavor.py 
import mlflow.pyfunc

class LoRAWrapper(mlflow.pyfunc.PythonModel):
    def load_context(self, context):
        from peft import PeftModel
        from transformers import AutoModelForCausalLM, AutoTokenizer
        model_path = context.artifacts["model_path"]
        self.tokenizer = AutoTokenizer.from_pretrained(model_path)
        base = AutoModelForCausalLM.from_pretrained(model_path)
        self.model = PeftModel.from_pretrained(base, model_path)

    def predict(self, context, model_input):
        import pandas as pd
        prompts = model_input["prompt"].tolist()
        results = []
        for prompt in prompts:
            inputs = self.tokenizer.encode(prompt, return_tensors="pt")
            out = self.model.generate(inputs, max_new_tokens=256)
            results.append(self.tokenizer.decode(out[0], skip_special_tokens=True))
        return pd.Series(results)

with mlflow.start_run():
    mlflow.pyfunc.log_model(
        artifact_path="lora-model",
        python_model=LoRAWrapper(),
        artifacts={"model_path": "./lora-output"},
        pip_requirements=["transformers", "peft", "torch"]
    )
Common pitfall: When logging large LLM artifacts to MLflow, avoid storing model weights as MLflow artifacts if using a shared tracking server — each run would duplicate gigabytes of data. Instead, log the model path or HuggingFace Hub URL as a parameter, and only store the adapter weights (which are small) as artifacts.

Advanced Metrics Logging & Alerting

MLflow's log_metric() accepts a step parameter, enabling time-series tracking of training metrics. You can log multiple metrics per step, and MLflow's UI renders them as interactive line charts. For LLM training, log both loss curves and downstream task metrics (BLEU, ROUGE, exact match) to catch cases where loss improves but task quality degrades.

advanced_logging.py 
import mlflow
from mlflow.tracking import MlflowClient

with mlflow.start_run() as run:
    for step, batch in enumerate(train_loader):
        loss = train_step(batch)

        # Log multiple metrics atomically at each step
        mlflow.log_metrics({
            "train_loss": loss.item(),
            "lr": scheduler.get_last_lr()[0],
            "grad_norm": grad_norm,
            "gpu_memory_gb": torch.cuda.memory_allocated() / 1e9,
        }, step=step)

        # System tags for filtering runs
        if step == 0:
            mlflow.set_tags({
                "mlflow.note.content": "QLoRA run with cosine LR schedule",
                "gpu_type": "A100-80GB",
                "dataset_version": "v2.1"
            })

# Query best run programmatically
client = MlflowClient()
best_run = client.search_runs(
    experiment_ids=["0"],
    filter_string="metrics.val_loss < 0.5 AND tags.gpu_type = 'A100-80GB'",
    order_by=["metrics.val_loss ASC"],
    max_results=1
)[0]
print(f"Best run: {best_run.info.run_id}, loss={best_run.data.metrics['val_loss']}")
Why this matters: The filter_string in search_runs() uses SQL-like syntax, making it trivial to find the best run across hundreds of experiments programmatically. This is how you build automated training pipelines that compare new runs against production baselines before promoting to the Model Registry.

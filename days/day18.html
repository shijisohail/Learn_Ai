Section 01

Why Experiment Tracking?

The reproducibility crisis in ML: "which config produced my best model?" Without tracking, you're flying blind. MLflow solves this at every layer of the ML workflow.

⚠️
The Reproducibility Crisis: Without experiment tracking, you run 20 fine-tuning jobs, pick the best one, then six months later you cannot reproduce it. MLflow eliminates this by capturing every run, every parameter, every artifact.
📊
Tracking
Log runs, parameters, metrics, and artifacts. The core MLflow component. Every experiment run gets a unique run ID and is stored in the tracking server.
📦
Projects
Package ML code in a reusable, reproducible form using MLproject files. Define dependencies, entry points, and parameters declaratively.
🗄️
Model Registry
Centralized model store with versioning and lifecycle stages: None → Staging → Production → Archived. Decouple model identity from run IDs.
🔁
Recipes
Opinionated pipelines for common ML workflows. Standardized steps for data ingestion, feature engineering, training, and evaluation.
Tool Primary Use LLM Native? Best For
MLflow Run tracking, model registry, artifact store Partial All ML experiments, fine-tuning runs, hyperparameter search
Weights & Biases Rich dashboards, sweep visualization, team collab Partial Research teams, interactive experiment exploration
LangSmith LLM chain tracing, dataset eval, prompt versioning Yes RAG pipelines, agent traces, LLM A/B testing
💡
Use all three: MLflow for fine-tuning experiments and the model registry. W&B for rich sweep visualization if you have a research-heavy team. LangSmith for production LLM chain tracing and prompt evaluation. They complement each other.
PYTHON — Setup
import mlflow # Set tracking server (default: ./mlruns locally) mlflow.set_tracking_uri("postgresql://user:pass@host/mlflow") mlflow.set_experiment("llama3-finetune-qlora")
Section 02

Logging Runs

Every training job becomes a logged run with parameters, time-series metrics, artifacts, and tags. MLflow's context manager pattern keeps tracking code clean.

📋What to Log
  • All hyperparameters (rank, alpha, lr, batch size, epochs)
  • Training loss at every step
  • Evaluation metrics (loss, perplexity, RAGAS scores)
  • Adapter config JSON and training curves as artifacts
  • Tags: model_family, dataset_version, author, git_sha
❌Common Mistakes
  • Only logging final eval loss (you lose the training curve)
  • Hardcoding run names instead of using descriptive names
  • Not logging the dataset version (reproducibility fails)
  • Forgetting artifacts — the adapter config IS the experiment
  • Never using nested runs for hyperparameter sweeps
PYTHON — Full Run Logging
with mlflow.start_run(run_name="qlora-r16-alpha32-lr2e4") as run: # Log hyperparameters mlflow.log_params({ "model": "meta-llama/Llama-3.2-3B", "lora_rank": 16, "lora_alpha": 32, "learning_rate": 2e-4, "batch_size": 4, "gradient_accumulation": 4, "num_epochs": 3, "dataset": "alpaca-clean-52k" }) # Log metrics during training (call inside training loop) for step, loss in training_losses: mlflow.log_metric("train_loss", loss, step=step) mlflow.log_metric("learning_rate", get_lr(step), step=step) # Log evaluation metrics mlflow.log_metrics({ "eval_loss": 0.89, "eval_perplexity": 2.44, "ragas_faithfulness": 0.87 }) # Log artifacts mlflow.log_artifact("./lora-adapter/adapter_config.json") mlflow.log_artifact("./training_curve.png") print(f"Run ID: {run.info.run_id}")

💡
Approval Workflow: Require a human sign-off before promoting to Production. Use the MlflowClient API with CI/CD: only promote after eval metrics pass thresholds and a senior engineer approves the staging run.
PYTHON — Registry Lifecycle
# Register model from run model_uri = f"runs:/{run.info.run_id}/model" registered = mlflow.register_model(model_uri, "CustomerSupportBot") # Version 1 created in stage "None" # Promote to staging (via client API) client = mlflow.tracking.MlflowClient() client.transition_model_version_stage( name="CustomerSupportBot", version=1, stage="Staging", archive_existing_versions=False ) # Promote to Production after validation client.transition_model_version_stage( name="CustomerSupportBot", version=1, stage="Production" ) # Load production model by name (decoupled from run ID) model = mlflow.pyfunc.load_model("models:/CustomerSupportBot/Production")
Section 04

Comparing Experiments

The real value of tracking: programmatically search across hundreds of runs to find the best model. The MLflow UI gives you visual comparison; the Python API gives you automation.

🔍
search_runs() API
Filter by any logged metric, param, or tag. Returns a pandas DataFrame. Sort by eval_loss ASC to find the best run automatically.
📈
MLflow UI
Built-in web UI at localhost:5000. Compare runs side-by-side with parallel coordinates plot for hyperparameter impact visualization.
📤
Export for Analysis
The returned DataFrame is just pandas — export to CSV, load into a Jupyter notebook, or feed into a downstream reporting pipeline.
PYTHON — Programmatic Comparison
# Find best run by eval_loss runs = mlflow.search_runs( experiment_names=["llama3-finetune-qlora"], filter_string="metrics.eval_loss < 1.0", order_by=["metrics.eval_loss ASC"] ) best_run = runs.iloc[0] print(f"Best run: {best_run['run_id']}") print(f"Eval loss: {best_run['metrics.eval_loss']:.4f}") print(f"Config: rank={best_run['params.lora_rank']}, lr={best_run['params.learning_rate']}")
Section 05

LangSmith for LLM Evaluation

LangSmith is to LLM chains what MLflow is to traditional ML. Create golden datasets, run your chain against them, score with LLM-as-judge evaluators, and compare prompt versions.

🧪
Evaluation Strategy: Build a golden dataset of 50-200 Q&A pairs representative of your production traffic. Run your RAG pipeline against them after every significant change. Track scores over time — not just the latest run.
✅LangSmith Evaluators
  • qa — LLM checks if answer matches reference
  • context_qa — verifies answer is grounded in context
  • criteria — score any criterion with a rubric
  • Custom LLM-as-judge with your own prompt
  • RAGAS faithfulness, relevancy as custom evaluators
🔬A/B Prompt Testing
  • Run both prompt versions against the same dataset
  • LangSmith shows side-by-side score comparison
  • Statistical significance testing built in
  • Track experiment_prefix per prompt version
  • No code change needed — just swap the prompt hub commit
PYTHON — LangSmith Evaluation
from langsmith import Client from langsmith.evaluation import evaluate client = Client() # Create evaluation dataset dataset = client.create_dataset("rag-eval-v1") client.create_examples( inputs=[{"question": "What is LoRA?"}, {"question": "Explain RLHF"}], outputs=[{"answer": "LoRA is..."}, {"answer": "RLHF is..."}], dataset_id=dataset.id ) # Run evaluation def predict(inputs: dict) -> dict: return {"answer": rag_chain.invoke(inputs["question"])} results = evaluate( predict, data="rag-eval-v1", evaluators=["qa"], # or custom LLM-as-judge evaluator experiment_prefix="rag-v2-hybrid-search" )

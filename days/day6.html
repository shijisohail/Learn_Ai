Section 01

The Four Production Pillars

Every production LLM system must address these four concerns. Neglect any one and you'll face either runaway costs, mysterious failures, or lack of visibility into what's happening.

📊

Observability

Log every API call: model, tokens, latency, cost, stop_reason. Use LangSmith or custom telemetry. You cannot debug what you cannot observe.

💰

Cost Control

Set per-user, per-session, and global cost limits. Use prompt caching for repeated system prompts. Choose models based on task complexity — use Haiku for simple tasks.

🛡️

Error Handling

Implement exponential backoff for rate limits. Handle timeout errors gracefully. Validate structured outputs. Never let LLM errors propagate unhandled to users.

production_client.py
import anthropic
import time, logging, json
from dataclasses import dataclass, field
from datetime import datetime

logger = logging.getLogger(__name__)

@dataclass
class LLMCall:
    model: str; prompt_tokens: int; completion_tokens: int
    latency_ms: float; cost_usd: float; stop_reason: str
    error: str | None = None
    timestamp: str = field(default_factory=lambda: datetime.utcnow().isoformat())

COST_PER_MTK = {
    "claude-sonnet-4-5": {"in": 3.0, "out": 15.0},
    "claude-3-5-haiku-20241022": {"in": 0.8, "out": 4.0},
    "claude-opus-4-5": {"in": 15.0, "out": 75.0},
}

class ProductionClient:
    def __init__(self, user_id: str, budget_usd: float = 1.0):
        self.client = anthropic.Anthropic()
        self.user_id = user_id
        self.budget_usd = budget_usd
        self.spent_usd = 0.0
        self.call_log: list[LLMCall] = []

    def _calc_cost(self, model: str, inp: int, out: int) -> float:
        rates = COST_PER_MTK.get(model, COST_PER_MTK["claude-sonnet-4-5"])
        return (inp * rates["in"] + out * rates["out"]) / 1_000_000

    def create(self, **kwargs) -> anthropic.types.Message:
        model = kwargs.get("model", "claude-sonnet-4-5")

        # Budget check
        if self.spent_usd >= self.budget_usd:
            raise ValueError(f"Budget exceeded: ${self.spent_usd:.4f} >= ${self.budget_usd}")

        start = time.perf_counter()
        error = None
        response = None

        for attempt in range(3):
            try:
                response = self.client.messages.create(**kwargs)
                break
            except anthropic.RateLimitError:
                wait = 2 ** attempt
                logger.warning(f"Rate limited, waiting {wait}s (attempt {attempt+1}/3)")
                time.sleep(wait)
                if attempt == 2: raise
            except anthropic.APIStatusError as e:
                error = str(e)
                logger.error(f"API error: {e.status_code} - {e.message}")
                raise

        latency = (time.perf_counter() - start) * 1000
        cost = self._calc_cost(model, response.usage.input_tokens, response.usage.output_tokens)
        self.spent_usd += cost

        call = LLMCall(model=model, prompt_tokens=response.usage.input_tokens,
                       completion_tokens=response.usage.output_tokens,
                       latency_ms=latency, cost_usd=cost, stop_reason=response.stop_reason)
        self.call_log.append(call)
        logger.info(json.dumps({"user": self.user_id, "model": model,
                                 "tokens": response.usage.input_tokens + response.usage.output_tokens,
                                 "cost_usd": cost, "latency_ms": latency}))
        return response
💡

Model Selection Strategy

Don't use Claude Sonnet or Opus for everything. Build a routing layer: use Claude 3.5 Haiku for simple classification, entity extraction, and short Q&A (90% of your traffic). Reserve Sonnet for reasoning, code generation, and complex analysis. Reserve Opus for critical decisions. This alone can cut costs by 80% for high-volume applications.

Section 03

Model Context Protocol (MCP)

MCP is Anthropic's open protocol for giving LLM applications standardized access to tools, resources, and prompts — across any host application and server implementation.

Think of MCP as a USB standard for AI capabilities. Before MCP, every LLM application reinvented its own tool integration layer — custom code for each data source, each service, each capability. MCP standardizes this: a server exposes tools/resources via the protocol, and any MCP-compatible client (Claude Desktop, your app, IDE plugins) can use them without custom integration code.

🔌

MCP Servers

Expose tools, resources (data), and prompts via the protocol. Servers can be local (filesystem, database access) or remote (web services). Write once, use from any MCP client.

🤝

MCP Clients

Applications that connect to MCP servers and make their capabilities available to LLMs. Claude Desktop, Claude Code, and your own applications can all act as MCP clients.

mcp_server.py
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent, Resource
import mcp.types as types

# Create an MCP server — exposes tools to Claude
app = Server("my-tools-server")

# Declare what tools this server provides
@app.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="query_database",
            description="Execute a read-only SQL query against the company database",
            inputSchema={
                "type": "object",
                "properties": {
                    "sql": {"type": "string", "description": "SELECT query only"}
                },
                "required": ["sql"]
            }
        ),
        Tool(
            name="get_weather",
            description="Get current weather for a city",
            inputSchema={"type": "object", "properties": {"city": {"type": "string"}}, "required": ["city"]}
        )
    ]

# Handle tool calls from Claude
@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "query_database":
        sql = arguments["sql"]
        if not sql.strip().upper().startswith("SELECT"):
            return [TextContent(type="text", text="Error: Only SELECT queries allowed")]
        result = await execute_db_query(sql)  # your DB query function
        return [TextContent(type="text", text=result)]
    elif name == "get_weather":
        weather = await fetch_weather(arguments["city"])
        return [TextContent(type="text", text=weather)]
    raise ValueError(f"Unknown tool: {name}")

# Run the server over stdio (for Claude Desktop integration)
if __name__ == "__main__":
    import asyncio
    asyncio.run(stdio_server(app))
💡

Why MCP Matters for Production

MCP creates a clean boundary: your tools are implemented once as an MCP server, and any Claude-powered application can use them without code changes. This means you can update tool implementations independently of the LLM application, and share tools across multiple applications — the same database query tool can be used by your chat interface, your IDE plugin, and your CI/CD bot simultaneously.

Section 04

LLM Evaluation & Testing

How do you know if your prompt change made things better or worse? Systematic evaluation is the engineering practice that answers this question — and it's non-negotiable for production systems.

eval_framework.py
import anthropic
from dataclasses import dataclass
from typing import Callable

client = anthropic.Anthropic()

@dataclass
class TestCase:
    input: str
    expected: str
    grader: Callable[[str, str], float]  # returns 0-1

# Different grader types
def exact_match(output: str, expected: str) -> float:
    return 1.0 if output.strip() == expected.strip() else 0.0

def contains_grader(output: str, expected: str) -> float:
    return 1.0 if expected.lower() in output.lower() else 0.0

def llm_grader(output: str, expected: str) -> float:
    """Use an LLM to grade quality (expensive but powerful)."""
    resp = client.messages.create(
        model="claude-3-5-haiku-20241022", max_tokens=100,
        system="Grade how well 'output' matches 'expected'. Reply with ONLY a decimal 0.0-1.0.",
        messages=[{"role": "user",
                   "content": f"Expected: {expected}\nOutput: {output}"}]
    )
    try: return float(resp.content[0].text.strip())
    except: return 0.0

def run_eval(
    system: str,
    test_cases: list[TestCase],
    model: str = "claude-sonnet-4-5"
) -> dict:
    scores = []
    for case in test_cases:
        resp = client.messages.create(
            model=model, max_tokens=512, system=system,
            messages=[{"role": "user", "content": case.input}]
        )
        output = resp.content[0].text
        score = case.grader(output, case.expected)
        scores.append(score)

    return {"mean_score": sum(scores)/len(scores), "n": len(scores), "scores": scores}

# Example usage
test_suite = [
    TestCase("The product is great!", "POSITIVE", exact_match),
    TestCase("Terrible experience.", "NEGATIVE", exact_match),
    TestCase("It's okay I guess.", "NEUTRAL", exact_match),
]
result = run_eval("Classify sentiment: POSITIVE, NEGATIVE, or NEUTRAL only.", test_suite)
print(f"Accuracy: {result['mean_score']*100:.1f}%")
⚠️

Pitfall: No Evals = Flying Blind

Every prompt change without an eval is a gamble. You might fix one edge case while breaking ten others. Build a golden dataset of at least 50-100 diverse test cases before launching, and run evals on every prompt change. Use Anthropic's Workbench for quick iteration and LangSmith for production monitoring. If your eval score drops, you catch it before users do.

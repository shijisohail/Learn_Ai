Section 01

The Determinism Problem

LLMs are stochastic by design. Production systems need guarantees. Understanding the tradeoffs between each approach determines which tool to reach for.

  • Why LLMs are non-deterministic: temperature and sampling introduce randomness — same prompt, different output every time
  • When you need determinism: structured data extraction, system integrations, automated pipelines, compliance requirements
  • Three tools for structured outputs: JSON mode (valid JSON, no schema), function calling/tool use (schema enforced), Instructor (Pydantic + retries)
  • The validation layer: never trust unvalidated LLM output in production — treat it as untrusted external data
Section 02

JSON Mode vs Structured Outputs

JSON mode guarantees syntactically valid JSON but not schema compliance. Structured Outputs use constrained decoding to guarantee field names, types, and required properties.

Approach Guarantees Flexibility Model Support
JSON mode Valid JSON syntax only ✓ Any schema OpenAI, Anthropic, most
OpenAI Structured Outputs Strict JSON Schema compliance ~ Limited types GPT-4o only
Anthropic tool_use Schema-conformant, explicit ✓ Full Pydantic Claude 3+ family
Instructor Pydantic v2 + auto-retry ✓ Full Pydantic All major providers
PYTHON
from openai import OpenAI from pydantic import BaseModel from typing import Optional client = OpenAI() # JSON mode — valid JSON but no schema guarantee response = client.chat.completions.create( model="gpt-4o-mini", response_format={"type": "json_object"}, messages=[{"role": "user", "content": "Extract invoice data as JSON from: Invoice #INV-2024-001, Amount: $1,234.56, Due: 2024-03-15, Client: Acme Corp"}] ) # Response is VALID JSON but fields may vary # Structured Outputs — guaranteed schema compliance class Invoice(BaseModel): invoice_id: str amount_usd: float due_date: str # ISO format client_name: str is_overdue: Optional[bool] = None completion = client.beta.chat.completions.parse( model="gpt-4o-2024-11-20", messages=[{"role": "user", "content": "Extract: Invoice #INV-2024-001, $1,234.56, Due 2024-03-15, Acme Corp"}], response_format=Invoice, ) invoice: Invoice = completion.choices[0].message.parsed print(invoice.amount_usd) # 1234.56 (always a float, always this field)
Section 03

Pydantic + Retry-with-Feedback Loop

When the LLM produces invalid output, send the validation error back as a natural-language correction request. Models respond remarkably well to their own error messages.

  • Parse and validate: always run LLM output through Pydantic before using it — this is the trust boundary
  • ValidationError feedback: Pydantic errors are human-readable — feed them directly back to the LLM for self-correction
  • Retry-with-feedback: each attempt includes the previous failed response and exact error — convergence typically within 2-3 attempts
  • Max retries, fallback strategies: after N attempts, log the failure, alert, and return a safe default rather than crashing
PYTHON
from pydantic import BaseModel, ValidationError, validator, Field import json class ExtractedOrder(BaseModel): order_id: str = Field(pattern=r"^ORD-\d{6}$") quantity: int = Field(gt=0, le=10000) product_sku: str = Field(min_length=3, max_length=20) priority: str = Field(pattern=r"^(low|medium|high|urgent)$") @validator("product_sku") def sku_uppercase(cls, v): return v.upper() def extract_with_retry(text: str, max_retries: int = 3) -> ExtractedOrder: last_error = None previous_response = None for attempt in range(max_retries): # Build prompt with feedback from previous failure prompt = f"Extract order data as JSON from: {text}" if last_error and previous_response: prompt += f""" PREVIOUS ATTEMPT FAILED: Your response: {previous_response} Validation errors: {last_error} Fix these specific errors and try again. Return ONLY valid JSON matching the exact schema.""" try: response = llm.invoke(prompt) json_str = extract_json(response.content) return ExtractedOrder.model_validate_json(json_str) except (ValidationError, json.JSONDecodeError) as e: last_error = str(e) previous_response = response.content raise ValueError(f"Failed after {max_retries} attempts. Last error: {last_error}")
Section 04

Canonical Data Models & Deterministic Pipelines

The canonical model is a normalized business entity — the stable contract between your AI layer and the rest of your system. Discriminated unions let you type AI responses precisely.

  • Canonical data model: normalized business entity representation for AI — one source of truth for what an "Order" or "Customer" looks like
  • Never pass raw integration payloads to LLM: normalize first — raw Salesforce/Jira objects have noise that inflates token count and confuses models
  • Structured contracts: typed input/output between AI and backend — catches schema drift at import time, not at 2am
  • Response type discriminator: "informative" vs "action" — parse once, route correctly
PYTHON
from pydantic import BaseModel from typing import Literal, Union # Canonical entities — normalized for AI consumption class CanonicalCustomer(BaseModel): customer_id: str full_name: str tier: Literal["standard", "premium", "enterprise"] open_orders: int lifetime_value_usd: float # Discriminated union for AI responses class InformativeResponse(BaseModel): type: Literal["informative"] message: str confidence: float class ActionResponse(BaseModel): type: Literal["action"] action: Literal["schedule_visit", "update_order", "send_notification", "escalate"] payload: dict requires_approval: bool reasoning: str # why the AI chose this action (for audit) CopilotResponse = Union[InformativeResponse, ActionResponse] def parse_copilot_response(llm_output: str) -> CopilotResponse: data = json.loads(llm_output) if data["type"] == "informative": return InformativeResponse.model_validate(data) return ActionResponse.model_validate(data)
Section 05

Production Patterns for Structured Output Systems

Schema contracts evolve. Systems must handle both — validate at the boundary, degrade gracefully, test exhaustively.

💡
Always validate at the boundary. LLM output is untrusted data — treat it like external API responses: parse, validate, handle errors. A ValidationError in production should be caught, logged, and retried — never propagated as an unhandled exception.
  • Schema versioning: include a schema_version field — migration layer handles old vs new responses without breaking existing integrations
  • Partial extraction: use Optional[T] = None for fields that may be absent — extract what's available, not what's required
  • Async validation pipelines: validate in background, serve optimistic response to user, correct asynchronously if validation fails
  • Testing structured outputs: property-based testing with Hypothesis — generate random text, verify extraction never throws unhandled exceptions

Section 01

Why Structured Outputs Are Non-Negotiable

Free-form LLM text output is a liability in production. Structured outputs enforce contracts between your AI and the rest of your system — transforming probabilistic generation into deterministic, typed data.

The core problem: every component downstream of an LLM call expects typed data. Your database expects a proper date, not "sometime next week." Your API expects a valid enum, not "maybe option A or B." Your business logic expects a float in range [0, 1], not "quite confident." When the LLM outputs free text, you must write fragile parsing logic that breaks on every edge case.

Pydantic v2 + Instructor solves this at the source. Rather than parsing LLM output, you declare your expected types as a Python class. Instructor injects the JSON schema into the prompt, parses the response, validates it, and if validation fails — feeds the error message back to the model for self-correction. This retry loop typically converges in 1-2 attempts for well-designed schemas.

🎯
Type Safety
Pydantic validates every field type, range constraint, and regex pattern. Invalid output is caught before it reaches your database or API — not after a 3am incident.
🔄
Auto-Retry
Instructor feeds validation errors back to the model as correction context. Models self-correct in 95%+ of cases within 2-3 retries without any prompt engineering effort.
📖
Schema = Documentation
Your Pydantic model is simultaneously the validation logic, the API documentation, the prompt schema injection, and the serialization format. One source of truth.
⚠️
Common pitfall: Overly complex schemas with deeply nested optionals cause parse failures even with retries. Keep schemas flat where possible. Use Union types sparingly — they confuse the model. If you need complex schemas, break extraction into multiple sequential calls.

import instructor from pydantic import BaseModel, Field, field_validator from anthropic import Anthropic from typing import Literal from datetime import date client = instructor.from_anthropic(Anthropic()) class FinancialMention(BaseModel): company: str = Field(description="Company name as mentioned") revenue: float = Field(ge=0, description="Revenue in millions USD") period: str = Field(description="Fiscal period, e.g. Q3 2024") sentiment: Literal["bullish", "bearish", "neutral"] confidence: float = Field(ge=0.0, le=1.0) currency: str = Field(description="ISO currency code") @field_validator("currency") @classmethod def valid_currency(cls, v: str) -> str: valid = {"USD", "EUR", "GBP", "JPY", "CAD"} if v.upper() not in valid: raise ValueError(ff"'{v}' is not a valid ISO currency. Use one of: {valid}") return v.upper() # Instructor auto-retries on ValidationError, feeding error → model result = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=512, response_model=FinancialMention, max_retries=3, messages=[{ "role": "user", "content": "Apple reported 119.6 billion dollars revenue for Q1 2025. Very bullish outlook." }] ) print(result.model_dump_json(indent=2))

Why this matters: In a production pipeline processing 10,000 extractions/day, even a 1% parse failure rate means 100 failed records that corrupt downstream analytics. With Instructor's retry loop and field validators, parse failures become sub-0.1%, and remaining failures are caught with descriptive errors rather than silent data corruption.
Section 04

Determinism: Temperature, Seeds & Caching

AI outputs are stochastic by default. Production systems often need reproducible results. Master the levers that control output determinism without sacrificing quality.

Temperature controls the entropy of token selection — temperature=0 means always pick the highest-probability token (greedy decoding), which is maximally deterministic but may miss creative options. Temperature=1 samples proportionally to model probabilities, producing diverse but less predictable outputs. For structured extraction, always use temperature=0 or close to it.

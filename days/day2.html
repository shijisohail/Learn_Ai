The ReAct Loop

Reason → Act → Observe. The fundamental cycle that powers all LLM agents. Click each node to explore.

ReAct (Reasoning + Acting) is a prompting framework where the model interleaves thinking with tool calls. Each iteration through the loop brings the agent closer to the answer. Most production agents are just this loop with good tools and error handling.
🧠
Think
⚡
Act
👁️
Observe

🧠 THINK — Reasoning Step

The model analyzes the current state: what is the goal? What information is missing? Which tool should I call next? This is the text or chain-of-thought part of the response.

  • Model reads all messages + tool results
  • Plans the next action or recognizes completion
  • Can produce intermediate reasoning text
  • Extended thinking (Claude 3.7+) does this explicitly
🧠 THINK
Model reads context and decides: do I have enough info to answer, or do I need to call a tool?
⚡ ACT
Model outputs a tool_use block. YOUR code runs the actual function. Model can't touch real systems.
👁️ OBSERVE
You send back a tool_result. Model now has new info and loops back to THINK with updated context.
✅ DONE
When stop_reason == "end_turn", the model has enough info and gives its final answer.

        1
        name
        
Snake_case identifier. The model uses this in its tool_use block. Must match what your dispatcher routes on. Be specific — get_stock_price not get_data.

      

      

        2
        description (tool level)
        
This is the most important field. The model reads this to decide WHEN to call the tool. Include: what it does, when to use it, example scenarios. Vague descriptions → wrong tool calls.

      

      

        3
        input_schema
        
JSON Schema object describing the function signature. Always type: "object". The model generates inputs that match this schema — so be precise about types and constraints.

      

      

        4
        properties
        
Each key is a parameter name. Value is a JSON Schema definition with type, description, optional constraints. These become the function arguments your code receives.

      

      

        5
        parameter description
        
Include examples in descriptions! "e.g. AAPL, MSFT" teaches the model the exact format expected. Without this, you'll get "Apple" instead of "AAPL".

      

      

        6
        enum constraints
        
Use enums to restrict to known valid values. The model will always choose from this list. Prevents hallucinated or invalid parameter values.

      

      

        7
        required array
        
Parameters in this array MUST be provided. Others are optional. The model will refuse to call the tool without required fields — use this for validation.

Why Tool Descriptions Matter

      
The model can only use what you describe. Bad descriptions = wrong tools = agent failures.

    

  


  

    

      
❌ Vague Description

      

        
"description": "Gets data from the database"

        
User asks: "How many orders came in last week from Germany?"

        

          
Model behavior:

          
❌ Model unsure when to use this. Might try to answer from training data. If it does call the tool, it halluccinates a reasonable-looking but wrong query. No clear contract.

        

      

    

    

      
✅ Precise Description

      

        
"description": "Query the orders database to aggregate sales data. Use this when the user asks about order counts, revenue, customer data, or any business metrics. Accepts SQL WHERE clauses for filtering by date, country, status, etc."

        
User asks: "How many orders came in last week from Germany?"

        

          
Model behavior:

          
✅ Immediately recognizes this is a database query scenario. Calls the tool with where="country='Germany' AND created_at >= NOW()-INTERVAL 7 DAYS". Returns accurate count.

        

      

    

  


  

    
Description Writing Checklist

    

      
✓ What the tool does (1 sentence)

      
✓ When to use it (trigger scenarios)

      
✓ What it returns (format + fields)

      
✓ Constraints / limitations

      
✓ Parameter examples in descriptions

      
✓ Enums for constrained values

Full ReAct Loop Implementation

      
Production-ready agent loop with error handling, tool dispatch, and iteration guard.

    

  


  

    

      react_agent.py — complete implementation
      python
    

    
import anthropic
import json

client = anthropic.Anthropic()

# ── Tool implementations ──────────────────────────────────────
def calculate(expression: str) -> str:
    """Safe math eval — in production use a proper math parser."""
    try:
        result = eval(expression, {"__builtins__": {}}, {})
        return str(result)
    except Exception as e:
        return f"Error: {e}"

def get_weather(city: str) -> str:
    # Real impl: call OpenWeatherMap API
    return json.dumps({"city": city, "temp_c": 18, "condition": "Cloudy"})

# ── Tool registry ──────────────────────────────────────────────
TOOLS = {
    "calculate": calculate,
    "get_weather": get_weather,
}

# ── Tool schemas (what the model sees) ────────────────────────
TOOL_SCHEMAS = [
    {
        "name": "calculate",
        "description": "Evaluate a math expression. Use for ANY arithmetic, percentages, unit conversions. Input must be a valid Python expression. Examples: '847 * 0.15', '(100 + 50) / 3'",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {"type": "string", "description": "Python math expression"}
            },
            "required": ["expression"]
        }
    },
]

# ── The agent loop ─────────────────────────────────────────────
def run_agent(user_message: str, max_iterations: int = 10) -> str:
    messages = [{"role": "user", "content": user_message}]

    for iteration in range(max_iterations):  # safety limit
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=4096,
            tools=TOOL_SCHEMAS,
            messages=messages,
        )

        # Always append the full response (may contain mixed text+tool_use)
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            # Extract final text response
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text

        # Process all tool_use blocks in this response
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue

            print(f"[{iteration+1}] TOOL: {block.name}({block.input})")

            if block.name not in TOOLS:
                result = f"Error: unknown tool '{block.name}'"
            else:
                try:
                    result = TOOLS[block.name](**block.input)
                except Exception as e:
                    result = f"Tool error: {e}"  # don't crash the loop

            print(f"       → {result}")
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": str(result),
            })

        messages.append({"role": "user", "content": tool_results})

    return "Agent exceeded max iterations"

Advanced Reasoning Patterns

      
Beyond ReAct — when vanilla loops aren't enough.

    

  


  


    
    

      
Chain of Thought (CoT)

      
Prompt the model to "think step by step" before answering. The reasoning is in one linear sequence. Most LLMs do this by default inside <thinking> blocks.

      

        Simple
        Low cost
        Linear
      

    


    
    

      
ReAct (Reasoning + Acting)

      
Interleaves CoT with tool calls. The model thinks, acts (calls a tool), observes the result, then thinks again. This is Section 1 of this day. Handles 90% of real agent tasks.

      

        Standard
        Medium cost
        Loop
      

    


    
    

      
Reflexion — Self-Critique Loop

      
Agent evaluates its own output, generates verbal critique, retries with feedback in context. Like a programmer who runs tests, reads the failure, and fixes the code.

      

        
reflexion_loop.pypython

        
def reflexion(task, max_attempts=3):
    history = []
    for attempt in range(max_attempts):
        answer = call_llm(task, history)
        passed, feedback = evaluate(answer)
        if passed:
            return answer
        # Critique becomes context for next attempt
        history.append({
            "answer": answer,
            "critique": feedback
        })
    return answer  # best attempt

      

      

        Iterative
        2–3× cost
        Best for: code gen, structured output
      

    


    
    

      
Tree of Thoughts (ToT)

      
Generate multiple candidate thoughts at each reasoning step, score them, keep the best branches. Like a chess engine exploring possible moves and pruning bad ones (BFS or DFS over thought space).

      
      

        
      

      

        Branching
        5–20× cost
        Best for: strategy, math, writing

  Claude Native Feature
  
Extended Thinking: Internal Scratchpad Before the Answer


  

    When you pass thinking: {type: "extended", budget_tokens: N} to the API,
    Claude allocates up to N tokens for an internal scratchpad. These appear in the response as
    thinking content blocks — before the final text answer.
    The model can explore, backtrack, and reason freely inside the scratchpad; those tokens do not constrain
    the final response quality and appear before it in response.content.
    Available on claude-opus-4-6 and newer models.

import anthropic
client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=16000,
    thinking={
        "type": "extended",
        "budget_tokens": 10000  # up to 10k tokens of internal reasoning
    },
    messages=[{
        "role": "user",
        "content": "Solve this step by step: If a train travels 120km in 1.5 hours, then stops for 30 minutes, then travels 180km in 2 hours, what is the average speed for the entire journey?"
    }]
)

# Response contains BOTH thinking blocks AND text blocks
for block in response.content:
    if block.type == "thinking":
        print(f"[THINKING]\n{block.thinking}\n")  # internal scratchpad
    elif block.type == "text":
        print(f"[ANSWER]\n{block.text}")

  • 🧮 Complex math or multi-step calculation
    • 
        
  • 🔀 Multi-hop reasoning across facts
    • 
        
  • ❓ Ambiguous problems needing exploration
    • 
        
  • 💻 Hard coding / algorithm challenges
    • 
        
  • 🗺️ Long planning or strategy tasks
    • 
      

    

    

      
✘ SKIP when…

      
    
        
  • ✅ Simple Q&A or lookup tasks
    • 
        
  • 🔧 Tool use / agentic loops — use ReAct instead
    • 
        
  • 💸 Cost-sensitive paths — thinking tokens are billed
    • 
        
  • ⚡ Real-time streaming UX (adds latency)
    • 
        
  • 💬 Conversational chitchat
    • 
      

    

  


  
  

    Cost & Limits:
    Thinking tokens are billed at the same rate as output tokens.
    Minimum budget_tokens = 1024.
    Maximum depends on the model (claude-opus-4-6 supports up to 32k).
    Extended thinking is compatible with streaming — thinking deltas arrive before text deltas.
    It is not compatible with temperature overrides or top_p/top_k.

Handling parallel blocks + sending ALL results in ONE user message

    
import asyncio

async def handle_parallel_tools(response, tools_map):
    tool_uses = [b for b in response.content if b.type == "tool_use"]

    # Run all tools concurrently
    results = await asyncio.gather(*[
        tools_map[t.name](**t.input) for t in tool_uses
    ])

    # CRITICAL: send ALL results in ONE user message
    tool_results_message = {
        "role": "user",
        "content": [
            {
                "type": "tool_result",
                "tool_use_id": tool_uses[i].id,
                "content": str(results[i])
            }
            for i in range(len(tool_uses))
        ]
    }
    return tool_results_message

  


  

    Gotcha: Always send ALL tool_results in ONE user message. Never send them separately —
    the API expects every tool_use id to be resolved in a single response.
    Missing any id causes an API error.
  







  Quality Lab
  
Tool Description Quality: It Changes Everything


  

    Same tool, same model, same query — but description quality dramatically changes behavior.
    Query used: "Notify the user that their order shipped."
  


  

    

      
Bad

      
"Send an email"

      
What model does:

      

        Calls send_email immediately. Subject: "Order Shipped". Uses a generic template.
        Also calls it for every subsequent notification — including error messages and alerts.
        Model treats it as a general notification tool.
      

    


    

      
OK

      
"Send an email to a user"

      
What model does:

      

        Better — asks for recipient before sending.
        But still uses it for notifications and alerts.
        Doesn't know the tool has side effects or constraints.
        May send to wrong addresses if user context is ambiguous.
      

    


    

      
Great

      
"Send a transactional email
to a single recipient. Use ONLY
when the user explicitly requests
an email be sent. Do NOT use for
notifications or alerts. Recipient
must be explicitly specified by
the user. Has real-world side
effects — cannot be undone."

      
What model does:

      

        Correctly identifies this as a notification (not user-requested email).
        Chooses a different tool or asks the user how they want to be notified.
        Only uses send_email when user says "email me" explicitly.

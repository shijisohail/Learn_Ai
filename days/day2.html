Section 01

Why Prompt Engineering is Engineering

Prompt engineering isn't about magic tricks or "jailbreaks" — it's a systematic discipline of communicating intent to a stochastic system. The fundamentals have a direct analogy to software design.

Think of a prompt as a function signature plus type annotations. The system prompt is your module's interface contract — it declares what the function does, its constraints, its output format, and its error behaviors. The user turn is the function call. A poorly specified interface leads to unpredictable outputs; a well-designed one leads to reliable, testable behavior.

As a backend engineer, you likely already think about: input validation, output schemas, error cases, performance, and maintainability. Apply the same thinking to prompts. Treat them as code — version control them, test them across edge cases, measure their performance, and refactor them when requirements change.

📐

Prompts Are Interface Contracts

A good system prompt specifies: the model's role, the task domain, the output format, constraints, and how to handle ambiguity. Miss any of these and you'll get inconsistent outputs at scale.

🧪

Treat Prompts as Code

Store prompts in version control. Run automated evals against a golden dataset. Log prompt versions alongside model responses. Use Anthropic's Prompt Workbench for systematic testing before deployment.

💡

Why This Matters at Scale

A prompt that works 90% of the time with 100 users will fail 1,000 times per day with 10,000 users. The 10% failure rate doesn't change — the absolute failures multiply with scale. Systematic prompt engineering is the difference between a demo and a production system.

Section 02

The Five Core Prompting Techniques

These techniques exist on a spectrum from simple to complex. Each has its ideal use case — combining them strategically is the mark of an expert prompt engineer.

TechniqueWhen to UseCostAccuracy Gain
Zero-ShotSimple, well-defined tasks the model already knowsLowestBaseline
One-ShotShowing format/style with a single exampleLow+10-20%
Few-ShotClassification, extraction with specific format requirementsMedium+20-40%
Chain-of-ThoughtReasoning, math, multi-step logic, planningMedium+30-60%

Zero-Shot Prompting

Zero-shot means giving the model a task with no examples. Works well for tasks the model has seen extensively during training: summarization, translation, basic classification, code generation in popular languages. The key is being precise and unambiguous in your instruction.

zero_shot.py
import anthropic

client = anthropic.Anthropic()

# BAD zero-shot: vague instruction
bad_response = client.messages.create(
    model="claude-sonnet-4-5", max_tokens=256,
    messages=[{"role": "user",
               "content": "Analyze this review: 'The product is okay I guess'"}]
)

# GOOD zero-shot: precise instruction + output format specified
good_response = client.messages.create(
    model="claude-sonnet-4-5", max_tokens=256,
    system="Classify customer reviews. Output ONLY one word: POSITIVE, NEGATIVE, or NEUTRAL.",
    messages=[{"role": "user",
               "content": "Review: 'The product is okay I guess'"}]
)
print(good_response.content[0].text)  # NEUTRAL

Few-Shot Prompting

Few-shot prompting provides examples of input→output pairs before the actual task. This teaches the model about format, tone, classification schema, and edge case handling — in-context, without fine-tuning. The quality of your examples matters enormously: use diverse, representative, correctly-labeled examples.

few_shot.py
import anthropic

client = anthropic.Anthropic()

# Few-shot for sentiment analysis with nuanced labels
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=100,
    messages=[
        {"role": "user", "content": "Text: 'This product changed my life!'\nSentiment:"},
        {"role": "assistant", "content": "STRONGLY_POSITIVE"},
        {"role": "user", "content": "Text: 'It arrived broken and support ignored me.'\nSentiment:"},
        {"role": "assistant", "content": "STRONGLY_NEGATIVE"},
        {"role": "user", "content": "Text: 'Works as advertised, nothing special.'\nSentiment:"},
        {"role": "assistant", "content": "NEUTRAL"},
        # Actual query
        {"role": "user", "content": "Text: 'Honestly expected better for the price.'\nSentiment:"}
    ]
)
print(response.content[0].text)  # MILDLY_NEGATIVE (learned from examples)

Chain-of-Thought (CoT)

CoT prompting instructs the model to "think step by step" before answering. This works because generating reasoning tokens allocates model capacity to actually solving the problem before committing to an answer. For complex reasoning tasks, CoT can improve accuracy by 40-80%. The magic phrase is literally "Think step by step:" or "Let's reason through this carefully:" in the prompt.

chain_of_thought.py
import anthropic

client = anthropic.Anthropic()

# WITHOUT CoT — often wrong on complex reasoning
bad = client.messages.create(
    model="claude-sonnet-4-5", max_tokens=200,
    messages=[{"role": "user",
               "content": "A store has 12 red balls and 8 blue balls. If you remove 3 red and add 5 blue, what fraction are blue? Just give the answer."}]
)

# WITH CoT — structured reasoning produces correct answer
good = client.messages.create(
    model="claude-sonnet-4-5", max_tokens=500,
    messages=[{"role": "user",
               "content": """A store has 12 red balls and 8 blue balls. If you remove 3 red and add 5 blue, what fraction are blue?

Let's think step by step:"""}]
)

# Force extended thinking for harder problems (Claude 3.7+)
thinking = client.messages.create(
    model="claude-3-7-sonnet-20250219",
    max_tokens=8000,
    thinking={"type": "enabled", "budget_tokens": 5000},
    messages=[{"role": "user",
               "content": "Prove that sqrt(2) is irrational."}]
)
Chain-of-Thought: Step Visualization

Problem: "A bakery sells 40 loaves Mon-Fri and 60 on weekends. How many in 4 weeks?"

Thought 1 — Parse Problem
Mon-Fri = 5 days × 40 loaves = 200 per weekday week. Weekend = 2 days × 60 = 120. So per week = 200 + 120 = 320 loaves.
Thought 2 — Calculate
4 weeks × 320 loaves/week = 1,280 loaves total.
Conclusion
The bakery sells 1,280 loaves over 4 weeks.
⚠️

CoT Pitfall: Verbose Reasoning ≠ Correct Reasoning

CoT doesn't guarantee correctness — it makes the model's reasoning observable. The model can produce confidently-stated but wrong reasoning chains (called "sycophantic chains of thought"). For high-stakes decisions, always validate CoT conclusions independently, and use Extended Thinking mode for problems that truly require deep reasoning.

Section 04

System Prompt Architecture

The system prompt is your most powerful tool. A well-structured system prompt can fundamentally change the model's behavior, persona, and output quality across all subsequent turns.

Think of the system prompt as the "constructor" of your LLM component. It's called once (per conversation) and sets the environment for everything that follows. Claude gives significantly more weight to system prompt instructions than user turn instructions — this asymmetry is by design and should be used deliberately.

system_prompt_template.py
# Production system prompt template — use this as a starting structure
SYSTEM_PROMPT = """
## Role & Identity
You are a {role_name}, specialized in {domain}.
You work at {company_context} and assist {target_users}.

## Core Capabilities
- {capability_1}
- {capability_2}
- {capability_3}

## Output Format
Always structure your responses as:
1. Direct answer first (1-2 sentences)
2. Detailed explanation with examples
3. Code snippet if applicable (Python, using type hints)
4. Edge cases or caveats

## Constraints
- DO: {what_to_do}
- DON'T: {what_not_to_do}
- When unsure: ask for clarification rather than guessing

## Tone & Style
- Technical but accessible — assume senior engineer background
- Concise: prefer specific examples over abstract explanations
- Honest: say "I don't know" rather than hallucinating

## Context
Today is {date}. The user's tech stack is {stack}.
"""

✓ System Prompt Best Practices

  • → Define role before capabilities
  • → Use markdown headers for structure
  • → Specify output format explicitly
  • → List constraints as DO/DON'T
  • → Include example if format is complex
  • → Keep under 2K tokens for cachability

✗ Common Mistakes

  • → Vague instructions ("be helpful")
  • → Contradictory constraints
  • → No output format specified
  • → Over-constraining with long rules
  • → Mixing system + user concerns
  • → Not testing edge cases
📌

Prompt Injection Defense

When users can supply input that gets embedded in your prompts, you're vulnerable to prompt injection attacks — where user content overrides your system prompt instructions. Defense: use explicit delimiters around user content (<user_input>...</user_input>), instruct the model to treat content within those tags as data not instructions, and validate critical outputs programmatically.

Section 05

Output Formatting & Controlling Response Structure

Reliable output formatting is what turns a demo into a production feature. Without it, you're doing string parsing on unpredictable text — a maintenance nightmare.

📝

XML Tags

Ask the model to wrap output sections in XML tags. Easy to parse reliably, even if there's surrounding text. Great for multi-part responses.

🔧

Tool Use (Best)

Force a JSON schema via tool_choice. API-level guarantee on format validity. No prompt engineering needed for the JSON itself.

⤶

Prefilling

Pre-fill assistant turn with {" to force JSON start. Simple but doesn't guarantee schema validity.

xml_formatting.py
import anthropic
import re

client = anthropic.Anthropic()

def parse_xml_tag(text: str, tag: str) -> str:
    """Extract content from XML tag in model output."""
    pattern = f"<{tag}>(.*?)</{tag}>"
    match = re.search(pattern, text, re.DOTALL)
    return match.group(1).strip() if match else ""

response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system="""Analyze code for bugs. Structure your response EXACTLY as:
<summary>One sentence assessment</summary>
<bugs>List of bugs found, one per line</bugs>
<fix>The corrected code</fix>
<severity>LOW | MEDIUM | HIGH | CRITICAL</severity>""",
    messages=[{"role": "user", "content": "def divide(a, b): return a / b"}]
)

raw = response.content[0].text
summary = parse_xml_tag(raw, "summary")
bugs    = parse_xml_tag(raw, "bugs")
fix     = parse_xml_tag(raw, "fix")
severity = parse_xml_tag(raw, "severity")

print(f"Severity: {severity}")
print(f"Summary: {summary}")
⚠️

Pitfall: Markdown in Production APIs

Claude often responds with markdown formatting (** for bold, ``` for code blocks) by default. If your application displays the raw text (e.g., in a database, log file, or API response), you'll get markdown symbols as literal characters. In your system prompt, explicitly say: "Do not use markdown formatting. Output plain text only." OR handle markdown rendering in your frontend.

01 — Why Advanced RAG?

The Naive RAG Ceiling

A basic RAG pipeline — embed query, ANN search, prompt LLM — hits a retrieval recall ceiling of ~70% on real enterprise data. Understanding why reveals the exact patterns to apply.

The naive RAG failure modes are predictable: (1) Vocabulary mismatch — user asks about "error 5432" but the document says "fault code OOM-5432". Dense embeddings miss exact strings. (2) Query-document asymmetry — short queries produce sparse, noisy embedding vectors that don't match the dense embeddings of long, detailed documents. (3) Chunking artifacts — the answer spans two chunks, neither of which scores high alone. (4) No way to measure quality — you can't improve what you don't measure.

Each of these failure modes has a targeted fix: hybrid search fixes vocabulary mismatch, HyDE and multi-query expansion fix query-document asymmetry, parent-document chunking fixes chunking artifacts, and RAGAS gives you the measurement framework. Applied together, these patterns routinely push recall from 70% to 90%+.

🔀
Hybrid Search + RRF
Combine dense ANN + BM25 keyword. Reciprocal Rank Fusion merges ranked lists. Fixes vocabulary mismatch. +10–20% recall on enterprise data.
💭
HyDE
Hypothetical Document Embeddings. Generate a fake answer, embed it, use that embedding for retrieval. Bridges query-document semantic gap significantly.
📊
Cross-Encoder Reranking
After ANN retrieves 50 candidates, a cross-encoder scores each (query, doc) pair together — far more accurate than bi-encoder similarity scores.
📏
RAGAS Evaluation
4 metrics: Faithfulness, Answer Relevance, Context Precision, Context Recall. LLM-as-judge evaluation at scale. Track quality across pipeline changes.
02 — HyDE & Multi-Query

HyDE & Multi-Query Expansion

HyDE (Hypothetical Document Embeddings) generates a fake answer to the query, then uses that answer's embedding for retrieval. Multi-query expansion generates N rephrasings and merges their results.

The key insight behind HyDE: a short user query ("how does attention work?") embeds very differently from a detailed technical document that answers it. But a hypothetical answer to the question — "The attention mechanism in transformers computes a weighted sum of values..." — embeds very similarly to the actual documents. By embedding the generated answer instead of the raw query, HyDE closes the semantic gap between query and document space.

Multi-query expansion addresses a different problem: any single query phrasing may miss documents that use different terminology. By generating 3–5 rephrasings of the query and running retrieval for each, you cast a wider net. Deduplicate results (or use union + reranking) before sending to the LLM.

hyde_and_multiquery.py
from langchain.retrievers import MultiQueryRetriever, HypotheticalDocumentEmbedder from langchain_openai import ChatOpenAI, OpenAIEmbeddings llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) base_retriever = vectorstore.as_retriever(search_kwargs={"k":10}) # Multi-Query Retrieval — generates 5 query variants, deduplicates results multi_retriever = MultiQueryRetriever.from_llm( retriever=base_retriever, llm=llm, include_original=True # include original query in retrieval ) docs = multi_retriever.invoke("How does RLHF affect model alignment?") # Generated variants: "What is RLHF?", "How does reinforcement learning train LLMs?", # "What is human feedback in machine learning?", "How is model safety achieved?" # ───────────────────────────────────────────── # HyDE — embed a hypothetical answer, not the query embeddings_with_hyde = HypotheticalDocumentEmbedder.from_llm( llm=llm, base_embeddings=OpenAIEmbeddings(model="text-embedding-3-small"), custom_prompt=ChatPromptTemplate.from_template( "Write a detailed technical paragraph answering: {QUESTION}" ) ) hyde_retriever = vectorstore.with_embeddings(embeddings_with_hyde).as_retriever() docs = hyde_retriever.invoke("How does attention work?") # Query: "How does attention work?" (short, sparse) # HyDE intermediate: "Attention in transformers computes weighted sums..." (rich, dense) # This dense intermediate vector retrieves matching detailed docs far more accurately
03 — Reranking

Cross-Encoder Reranking

Bi-encoder retrieval (ANN search) is fast but approximate. Cross-encoder reranking re-scores each (query, doc) pair together — slow but highly accurate. Run on 25–50 ANN candidates to select the top 5.

The bi-encoder architecture (used for ANN search) encodes query and document separately, then compares their vectors. This is fast — O(1) at query time — but loses query-document interaction signals. Cross-encoders encode the query and document together in a single forward pass, enabling the model to directly compare tokens across both. This is 10–100x slower but dramatically more accurate for relevance scoring.

reranking.py
from sentence_transformers import CrossEncoder from langchain.retrievers.document_compressors import CrossEncoderReranker from langchain.retrievers import ContextualCompressionRetriever from langchain_community.cross_encoders import HuggingFaceCrossEncoder # cross-encoder/ms-marco-MiniLM-L-6-v2 is fast and accurate cross_encoder = HuggingFaceCrossEncoder(model_name="cross-encoder/ms-marco-MiniLM-L-6-v2") reranker = CrossEncoderReranker(model=cross_encoder, top_n=5) # Two-stage pipeline: ANN retrieves 50, cross-encoder selects best 5 base_retriever = vectorstore.as_retriever(search_kwargs={"k":50}) reranking_retriever = ContextualCompressionRetriever( base_compressor=reranker, base_retriever=base_retriever ) docs = reranking_retriever.invoke("What is the capital of France?") # Stage 1: 50 candidates via ANN (fast, ~30ms) # Stage 2: 50 → 5 via cross-encoder scoring (slower, ~100ms but high precision) # Cohere Rerank API — managed, no GPU needed import cohere co = cohere.Client("api-key") results = co.rerank( query="What is the capital?", documents=[doc.page_content for doc in candidate_docs], model="rerank-english-v3.0", top_n=5 )
💡
Why This Matters: In a typical enterprise RAG benchmark, adding cross-encoder reranking on top of hybrid search improves NDCG@5 from 0.73 to 0.89. That's the difference between a system users trust and one they abandon. The latency cost (100ms) is almost always worth the quality gain.
04 — RAGAS Evaluation

RAGAS — Automated RAG Evaluation

RAGAS is the standard framework for automated RAG evaluation. Four LLM-as-judge metrics give you a complete picture of pipeline quality without manual annotation at scale.

RAGAS evaluates both the retrieval quality and generation quality separately. This is critical: you can have excellent retrieval but poor generation (the LLM ignores the context), or excellent generation from poor context (hallucinated answers that sound confident). The four-metric framework catches both failure modes.

Retrieval Metrics
Context Precision
What fraction of retrieved chunks are actually relevant? High = low noise. Low = retrieval is noisy — adding irrelevant context that confuses the LLM.
Context Recall
What fraction of the information needed to answer was retrieved? Low = chunking gaps or retrieval misses. Increase k or fix chunking strategy.
Generation Metrics
Faithfulness
Are all claims in the answer supported by the retrieved context? Low = hallucination. Critical metric for trust. Target: >0.9.
Answer Relevance
Does the answer actually address the question? Low = verbose/off-topic answers. Often improves with better prompt formatting.
ragas_evaluation.py
from ragas import evaluate from ragas.metrics import ( faithfulness, answer_relevancy, context_precision, context_recall ) from datasets import Dataset # Prepare evaluation dataset eval_data = { "question": ["What is LCEL?", "How does HyDE work?"], "answer": [rag_chain.invoke(q) for q in questions], "contexts": [[doc.page_content for doc in retriever.invoke(q)] for q in questions], "ground_truth": ["LCEL is LangChain Expression Language...", "HyDE generates..."] } result = evaluate( dataset=Dataset.from_dict(eval_data), metrics=[faithfulness, answer_relevancy, context_precision, context_recall], ) print(result) # {'faithfulness': 0.91, 'answer_relevancy': 0.87, # 'context_precision': 0.83, 'context_recall': 0.88}
⚠️
Common Pitfall — Evaluating with the Same LLM: Using GPT-4o to generate answers AND GPT-4o as the RAGAS judge creates self-referential bias — the judge tends to rate its own outputs higher. Use a different judge model (e.g., generate with Claude, evaluate with GPT-4o, or use human annotators to calibrate).

07 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. Hybrid search (BM25 + dense) is the highest-ROI single improvement to naive RAG.
  • 2. HyDE fixes the query-document asymmetry — short queries vs long documents.
  • 3. Cross-encoder reranking: retrieve 50, keep 5. Always worth the latency cost.
  • 4. RAGAS Faithfulness > 0.9 is your safety bar. Below that, users encounter hallucinations.
  • 5. Evaluate before and after each change — advanced patterns add latency, verify they add quality.

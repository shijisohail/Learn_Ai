Section 01

Naive RAG Limitations

The standard embed-and-retrieve pipeline fails in predictable ways. Understanding failure modes is prerequisite to applying the right fix.

🕳️
Vocabulary Gap
User asks "PPO algorithm" but docs say "Proximal Policy Optimization". Dense retrieval handles synonyms well but can miss exact technical terms, product codes, or acronyms that appear rarely in training data. BM25 exact-match bridges this gap.
📦
Context Window Stuffing
Retrieving top-k large chunks and concatenating them into the prompt consumes tokens without adding proportional value. Most retrieved context is irrelevant. Re-ranking and contextual compression reduce noise before the LLM call.
✂️
Chunk Boundary Problem
Fixed-size chunking cuts sentences mid-thought. The retrieved chunk contains half an answer. Parent-document retrieval indexes small chunks for precision but returns the surrounding parent context for completeness.
📏
No Quality Measurement
Without evaluation, you ship a RAG pipeline with no idea if it's actually faithful to source documents. RAGAS provides faithfulness, answer relevancy, context precision, and context recall — four complementary metrics for a complete picture.
Chunking Strategy Pros Cons Best For
Fixed-size (512 tokens) Simple, fast, predictable Cuts sentences at arbitrary boundaries General purpose baseline
Sentence Preserves grammatical units Uneven chunk sizes, very short chunks Short blog-style documents
Semantic Content-aware, coherent chunks Slow (requires embedding at index time), complex Technical documentation
Parent-document Small chunks for retrieval, full context returned More tokens per retrieved chunk Complex reasoning tasks
Section 02

Hybrid Search — BM25 + Dense Retrieval

BM25 catches what dense retrieval misses (exact terms, product codes, rare acronyms). Dense retrieval catches what BM25 misses (semantic meaning, paraphrases). Reciprocal Rank Fusion combines both ranked lists optimally.

🔀
Reciprocal Rank Fusion (RRF): Rather than combining raw scores (which have incompatible scales between BM25 and cosine), RRF uses the reciprocal of the rank position: RRF(d) = Σ 1/(k + rank(d)). k=60 is standard. This makes the combination model-agnostic and robust to outlier scores.
Python — LangChain Hybrid Retriever
from langchain_community.retrievers import BM25Retriever from langchain.retrievers import EnsembleRetriever from langchain_community.vectorstores import Qdrant # Dense retriever (semantic — catches paraphrases) vectorstore = Qdrant(client=qdrant_client, collection_name="docs", embeddings=embeddings) dense_retriever = vectorstore.as_retriever(search_kwargs={"k": 10}) # Sparse retriever (keyword BM25 — catches exact terms) bm25_retriever = BM25Retriever.from_documents(docs, k=10) # Hybrid: Reciprocal Rank Fusion hybrid_retriever = EnsembleRetriever( retrievers=[bm25_retriever, dense_retriever], weights=[0.4, 0.6] # tune: increase BM25 weight for technical jargon ) results = hybrid_retriever.invoke("RLHF training PPO algorithm") # BM25 catches "PPO" as an exact term # Dense catches semantic meaning of "alignment training" # RRF combines both ranked lists for the best of both worlds # When hybrid beats pure dense (examples): # "SKU-48291 return policy" → BM25 exact matches SKU number # "CUDA OOM error fix" → BM25 matches error code exactly # "What does RLHF stand for" → dense understands the concept
Section 03

HyDE & Parent-Document Retrieval

HyDE (Hypothetical Document Embeddings) closes the query-document semantic gap by generating a hypothetical answer and embedding that. Parent-document retrieval solves the chunk boundary problem without sacrificing retrieval precision.

🧪
HyDE Intuition: A user query ("What is RLHF?") is semantically closer to other short questions than to the dense technical paragraphs in your docs. A hypothetical answer ("RLHF is a technique that…") lives in the same semantic neighborhood as your actual document chunks — dramatically improving retrieval precision for technical corpora.
Python — Parent-Document + HyDE
from langchain.retrievers import ParentDocumentRetriever from langchain.storage import InMemoryStore from langchain_text_splitters import RecursiveCharacterTextSplitter from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser # ── Parent-Document Retriever ───────────────────────────── # Index small child chunks (400 tokens) for precise retrieval # Return large parent chunks (2000 tokens) for full context parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000) child_splitter = RecursiveCharacterTextSplitter(chunk_size=400) store = InMemoryStore() # use Redis in production retriever = ParentDocumentRetriever( vectorstore=vectorstore, docstore=store, child_splitter=child_splitter, parent_splitter=parent_splitter, ) retriever.add_documents(docs) # Searches child chunks (400 tokens for precision), # returns parent context (2000 tokens for completeness) results = retriever.invoke("Explain LoRA fine-tuning") # ── HyDE Pattern ───────────────────────────────────────── hyde_prompt = ChatPromptTemplate.from_template( "Write a 3-sentence technical paragraph that would answer: {question}\n" "Be specific. Include technical terms you'd expect in a reference doc." ) hyde_chain = hyde_prompt | llm | StrOutputParser() def hyde_retrieve(question: str) -> list: # Step 1: Generate hypothetical answer hypothetical_doc = hyde_chain.invoke({"question": question}) # Step 2: Embed the hypothetical answer, not the question # This embedding lives near actual document text in the vector space return vectorstore.similarity_search(hypothetical_doc, k=5) # Query transformation: sub-question decomposition decompose_prompt = ChatPromptTemplate.from_template( "Break this question into 3 simpler sub-questions: {question}\n" "Return as JSON list: [\"sub1\", \"sub2\", \"sub3\"]" )
Section 04

Cross-Encoder Re-ranking

Two-stage retrieval: a fast bi-encoder retrieves 50 candidates (recall), then a slow cross-encoder scores each query-document pair jointly to select the top 5 (precision). The cross-encoder is 10x more accurate but too slow to run on the full corpus.

⚡
Bi-encoder vs Cross-encoder: Bi-encoders embed query and document independently and compare vectors — fast because embeddings are precomputed. Cross-encoders concatenate [query, document] and run full attention over both together — much more accurate because they model explicit query-document interactions, but cannot be precomputed per document.
Python — Cohere Rerank + Local Cross-Encoder
import cohere from sentence_transformers import CrossEncoder # ── Cohere Rerank API (managed, easy) ──────────────────── co = cohere.Client("...") results = co.rerank( query="What is attention mechanism in transformers?", documents=retrieved_docs, # list of strings, top-50 from bi-encoder model="rerank-english-v3.0", top_n=5 # return only top 5 after reranking ) reranked_docs = [results.results[i].document["text"] for i in range(len(results.results))] # ── Local cross-encoder (free, runs on CPU) ────────────── # ms-marco models are trained on MS-MARCO passage ranking dataset cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2") query = "What is attention mechanism?" pairs = [[query, doc] for doc in retrieved_docs] scores = cross_encoder.predict(pairs) # Sort by relevance score descending reranked = sorted(zip(retrieved_docs, scores), key=lambda x: x[1], reverse=True) top_docs = [doc for doc, score in reranked[:5]] # Pipeline: retrieve 50 → rerank → top 5 → LLM # Latency: +80-150ms (cross-encoder), but faithfulness improves ~20% # When to use: when retrieved chunks are noisy or low precision # Skip reranking: when corpus is small + well-chunked (<1000 docs)
Section 05

RAGAS Evaluation

RAGAS provides 4 complementary metrics that together give a complete picture of RAG pipeline quality. Use them in CI/CD as a quality gate — reject deployments where scores fall below thresholds.

Faithfulness
Are all claims in the answer supported by the retrieved context? An unfaithful answer introduces facts not present in the retrieved docs — i.e., hallucination. Target: >0.80
Answer Relevancy
Does the answer address the question asked? Measured by embedding the answer and checking alignment with the original question. Penalizes verbose or tangential responses. Target: >0.75
Context Precision
What fraction of the retrieved chunks are actually relevant to the question? Low precision means you're retrieving noise that wastes context window tokens. Target: >0.70
Context Recall
Does the retrieved context contain all information needed to answer the question? Low recall means answers will be incomplete. Requires a reference answer for comparison. Target: >0.75
Python — RAGAS Evaluation Pipeline
from ragas import evaluate from ragas.metrics import ( faithfulness, answer_relevancy, context_precision, context_recall ) from datasets import Dataset # Prepare evaluation dataset (golden set) eval_data = { "question": ["What is RLHF?", "Explain LoRA"], "answer": [generated_answer_1, generated_answer_2], "contexts": [[retrieved_docs_1], [retrieved_docs_2]], "ground_truth":["RLHF stands for...", "LoRA is a method..."] } dataset = Dataset.from_dict(eval_data) # Run RAGAS evaluation results = evaluate( dataset, metrics=[faithfulness, answer_relevancy, context_precision, context_recall] ) print(results) # faithfulness: 0.87, answer_relevancy: 0.91 # context_precision: 0.73, context_recall: 0.82 # ── CI/CD Quality Gate ─────────────────────────────────── THRESHOLDS = { "faithfulness": 0.80, "answer_relevancy": 0.75, "context_precision": 0.70, "context_recall": 0.75, } def quality_gate(scores: dict) -> None: failures = [] for metric, threshold in THRESHOLDS.items(): if scores.get(metric, 0) < threshold: failures.append(f"{metric}: {scores[metric]:.2f} < {threshold}") if failures: raise ValueError(f"Quality gate failed:\n" + "\n".join(failures)) print("Quality gate passed. Deploy approved.")

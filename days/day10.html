01 — Fundamentals

Why Vector Databases?

Traditional databases find rows by exact match or range scan. Vector databases find semantically similar content — the embedding closest to your query vector — using Approximate Nearest Neighbor search across millions or billions of high-dimensional vectors in milliseconds.

Index (HNSW / IVF)
The data structure that enables fast ANN search. HNSW builds a navigable small-world graph — high recall, ~30ms latency. IVF partitions into clusters — better memory efficiency at billion scale.
Payload / Metadata
Structured data stored alongside each vector: tenant_id, doc_type, created_at, tags. Used for mandatory pre-filtering (security) and post-filtering (relevance). Rich filtering is what separates production DBs from FAISS.
Collection
The top-level namespace in most vector DBs. Each collection has its own index config, vector dimension, and distance metric. Think "table" in relational DBs. Weaviate calls them "classes"; Pinecone calls them "indexes".
Distance Metrics
Cosine Similarity
Measures angle between vectors. Direction only, ignores magnitude. Use when: comparing normalized text embeddings (OpenAI, Anthropic).
Dot Product
Magnitude + direction. Faster than cosine. Use when: vectors are already unit-normalized — mathematically equivalent to cosine, cheaper to compute.
Euclidean (L2)
Straight-line distance in vector space. Use when: magnitude matters (e.g. image embeddings, audio features, ColBERT multi-vector).
ANN vs Exact Search
ANN (Approximate)
Returns ~95–99% of exact results. Millisecond latency at any scale. Used in production. HNSW recall can be tuned via ef parameter.
Exact Search (Brute Force)
100% recall. O(n) — scales linearly with corpus size. Fine for <10K vectors. Impractical beyond 100K without GPU acceleration.

02 — Development

FAISS & Chroma — For Development

FAISS is a C++/Python library, not a database. It has no server, no persistence, no payload filtering. Excellent for benchmarking ANN algorithms and building quick prototypes. Chroma adds persistence and a simple filtering API on top of embedding search — ideal for early-stage RAG development.

⚠️
FAISS in production: FAISS has no built-in security, persistence, or tenant isolation. Never use it as the sole vector store in a multi-tenant production system. Use it for offline embedding indexing and benchmarking instead.
PYTHON — FAISS + Chroma
import faiss import numpy as np # FAISS: create HNSW index (high recall, good speed) dimension = 1536 # OpenAI embedding dimension index = faiss.IndexHNSWFlat(dimension, 32) # 32 neighbors # Add vectors vectors = np.random.random((10000, dimension)).astype('float32') index.add(vectors) # Search: find 5 nearest neighbors query = np.random.random((1, dimension)).astype('float32') distances, indices = index.search(query, k=5) print(f"Nearest doc indices: {indices[0]}") print(f"Distances: {distances[0]}") # Chroma — simple persistent vector store import chromadb client = chromadb.PersistentClient(path="./chroma_db") collection = client.get_or_create_collection("docs") collection.add( documents=["LangChain is...", "FAISS is..."], ids=["doc1", "doc2"] ) results = collection.query(query_texts=["explain langchain"], n_results=2)
FAISS Index Types
IndexFlatL2: Exact brute-force. Baseline for recall benchmarking.
IndexHNSWFlat: High recall, good query speed. Best for prototyping up to 1M vectors.
IndexIVFFlat: Inverted file with flat quantizer. Scale to 100M+ with controlled recall.
IndexIVFPQ: Product quantization — extreme memory compression, some recall loss.
Chroma for Development
Persistent client saves to disk — survives restarts, no external server needed.
Server mode: chroma run --path ./db exposes HTTP API for multi-process access.
LangChain integration: Chroma(embedding_function=...) — drop-in retriever.
When to graduate: >100K docs, need tenant isolation, need hybrid search.
03 — Production

Qdrant — Production-Grade Rust-Native

Qdrant is written in Rust — zero GC pauses, predictable latency under load. It has the richest payload filtering of any open-source vector DB: nested conditions, range filters, geo filters, full-text match on payload fields. The go-to choice for self-hosted production RAG.

Key Qdrant Concepts
Collection: Analogous to a table. Defines vector size, distance metric, and indexing params.
Point: One record = vector + payload (dict) + id. Points are upserted atomically.
Named vectors: A single point can have multiple vectors (dense + sparse) under different names.
Payload indexes: Create keyword/integer/float indexes on payload fields to enable fast pre-filtering before ANN.
Quantization & Memory
Scalar quantization: Compress float32 → int8. 4x memory reduction, <1% recall loss.
Product quantization: Split vector into subvectors, quantize each. 8–32x compression, higher recall loss.
mmap storage: Qdrant pages vector data in from disk on demand — run multi-billion scale on a single node with commodity SSDs.
PYTHON — Qdrant Production Pattern
from qdrant_client import QdrantClient from qdrant_client.models import ( Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue, Range ) client = QdrantClient(url="http://localhost:6333") # Create collection client.create_collection( collection_name="documents", vectors_config=VectorParams(size=1536, distance=Distance.COSINE) ) # Upsert points with rich payload client.upsert( collection_name="documents", points=[ PointStruct( id=1, vector=embedding_1, payload={ "tenant_id": "acme-corp", "doc_type": "policy", "created_at": 1704067200 } ) ] ) # Search with mandatory tenant filter results = client.search( collection_name="documents", query_vector=query_embedding, query_filter=Filter( must=[ FieldCondition(key="tenant_id", match=MatchValue(value="acme-corp")), FieldCondition(key="doc_type", match=MatchValue(value="policy")) ] ), limit=5, with_payload=True )
💡
Multi-tenancy security pattern: Always apply tenant_id as a must filter in every search. This ensures complete data isolation — one tenant can never retrieve another's documents. Index the tenant_id payload field to avoid full collection scans during pre-filtering.
04 — Managed

Weaviate & Pinecone

Weaviate's module system lets you plug in vectorizers, re-rankers, and generative modules — the entire RAG pipeline becomes configuration. Pinecone's serverless tier eliminates infrastructure entirely: no cluster sizing, no index management, pay per query.

Weaviate — Schema-Based
Classes (collections) have typed properties — enforces data quality at ingestion.
GraphQL query interface — expressive filter syntax including references between classes.
Vectorizer modules: text2vec-openai, text2vec-cohere, multi2vec-clip — Weaviate handles embedding internally on upsert.
Multi-tenancy built-in: each tenant is an isolated shard with independent key management.
Pinecone — Serverless
Zero infrastructure: create an index, start upserting. Scales automatically to billions.
Namespaces: Logical partitions within an index. Use for tenant isolation — each tenant's vectors live in their own namespace, queried separately.
Sparse-dense hybrid search built-in: provide both dense embedding and sparse BM25 vector in a single upsert.
Tradeoff: no self-hosting, fully managed cost model, less control over indexing strategy.
PYTHON — Weaviate + Pinecone
# Weaviate import weaviate client = weaviate.connect_to_local() collection = client.collections.get("Document") # Query with filters results = collection.query.near_text( query="machine learning", filters=weaviate.classes.query.Filter.by_property("tenant").equal("acme"), limit=5 ) # Pinecone from pinecone import Pinecone pc = Pinecone(api_key="...") index = pc.Index("my-index") # Upsert with namespace (tenant isolation) index.upsert( vectors=[{"id": "doc1", "values": embedding, "metadata": {"type": "policy"}}], namespace="acme-corp" ) # Query within namespace results = index.query( vector=query_embedding, top_k=5, namespace="acme-corp", filter={"type": {"$eq": "policy"}} )
05 — Decision

Choosing the Right Vector DB

The right choice depends on four axes: scale, hosting preference, filtering complexity, and whether you need hybrid search. Resist over-engineering at the prototype stage — start with Chroma and migrate when you hit its limits.

✓
The Golden Path: Start with Chroma for development (zero setup, Python-native) → migrate to Qdrant for self-hosted production (rich filtering, hybrid search, Rust performance) → use Pinecone for fully managed serverless when infrastructure is a constraint.
Production Checklist
→ Snapshots + restore procedure tested before launch
→ Payload fields indexed (tenant_id, doc_type at minimum)
→ Monitoring: query latency p99, index size, memory pressure
→ Index rebuilding strategy (HNSW ef_construction tuning)
→ Embedding dimension fixed at schema time — changing it requires re-indexing
Advanced Patterns
→ Multi-vector (ColBERT): Late interaction models produce per-token vectors. Query uses MaxSim scoring across all token pairs. Higher recall, higher storage cost.
→ Hybrid search: Dense (semantic) + sparse (BM25 keyword) re-ranked with RRF. Dramatically improves recall for domain-specific terminology.
→ Embedding dimension: 1536 (OpenAI ada-002) = 6GB RAM per 1M vectors uncompressed. Use int8 quantization to fit 4x more in the same memory.

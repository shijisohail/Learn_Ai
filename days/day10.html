01 — Fundamentals

Why Vector Databases?

Traditional databases find rows by exact match or range scan. Vector databases find semantically similar content — the embedding closest to your query vector — using Approximate Nearest Neighbor search across millions of vectors in milliseconds.

The core operation in a vector database is Approximate Nearest Neighbor (ANN) search: given a query vector q, find the k vectors in the index most similar to q by some distance metric. "Approximate" is the key word — exact nearest neighbor search is O(n·d) per query (where d is dimensionality), which is too slow for millions of vectors. ANN algorithms trade a small amount of recall for 100–1000x speedup by clever indexing.

What separates production vector databases from a FAISS index file is everything else: payload filtering (filter by metadata before or after vector search), multi-tenancy (tenant isolation without cross-contamination), persistent storage, horizontal scaling, hybrid search (combine dense semantic search with sparse keyword search), and enterprise features like role-based access control and audit logging.

Index (HNSW / IVF)
The data structure enabling fast ANN search. HNSW builds a navigable small-world graph — high recall (~99%), ~30ms latency. IVF partitions into clusters — better memory efficiency at billion scale with slight recall tradeoff.
Payload / Metadata
Structured data stored alongside each vector: tenant_id, doc_type, created_at, tags. Used for mandatory pre-filtering (security isolation) and post-filtering (relevance refinement). Rich filtering is what separates production DBs from FAISS.
Collection / Index
The top-level namespace. Each collection has its own index config, vector dimension, and distance metric. Think "table" in relational DBs. Weaviate calls them "classes"; Pinecone calls them "indexes"; Qdrant calls them "collections".
Distance Metrics
Cosine Similarity
Measures angle between vectors. Direction only, ignores magnitude. Use when: comparing normalized text embeddings (OpenAI, Anthropic). Most common for RAG.
Dot Product
Magnitude + direction. Faster than cosine. Use when: vectors are already unit-normalized — mathematically equivalent to cosine, computationally cheaper.
Euclidean (L2)
Straight-line distance in vector space. Use when: magnitude matters (image embeddings, audio features, ColBERT multi-vector).

02 — Vector DB Landscape

FAISS vs Chroma vs Qdrant vs Pinecone vs Weaviate

Each vector database targets a different stage of the development lifecycle and a different set of production requirements. Here's the decision map.

FAISS (Meta) is the foundational ANN library — not a database, but the indexing engine under many databases. It's in-memory, has no persistence, no server, and no filtering. It's the right tool for prototyping your embedding strategy or running offline batch search jobs.

Chroma fills the gap between FAISS and a real production database. It's an embedded Python database with optional persistence to disk and a simple API. Zero setup cost makes it ideal for development and small-scale production. Qdrant, written in Rust, is the production self-hosted choice: HNSW with payload filtering, scalar quantization, hybrid search, and no GC pauses. Pinecone is the managed cloud alternative — zero infrastructure, serverless billing, but no self-hosting option.

Common Pitfall — Starting with FAISS in Production: FAISS has no persistence — your index is lost on every restart. No server means you can't scale horizontally. No filtering means security tenant isolation is impossible. Use Chroma for development, Qdrant or Pinecone for production. Don't let the "FAISS is fast" benchmark mislead you.
03 — FAISS & Chroma

FAISS & Chroma — Development Tools

FAISS for in-memory ANN research, Chroma for persistent local development with LangChain integration.

faiss_and_chroma.py
import faiss import numpy as np # FAISS — build an HNSW index (in-memory, no persistence) d = 1536 # OpenAI text-embedding-3-small dimension index = faiss.IndexHNSWFlat(d, 32) # M=32 neighbors per layer index.add(embeddings_np) # numpy array, shape (N, 1536) D, I = index.search(query_vec, k=4) # D=distances, I=indices faiss.write_index(index, "index.faiss") # save to disk index = faiss.read_index("index.faiss") # reload # ──────────────────────────────────────────────── # Chroma — persistent embedded DB with LangChain import chromadb from langchain_chroma import Chroma from langchain_openai import OpenAIEmbeddings # Persistent client — survives restarts client = chromadb.PersistentClient(path="./chroma_db") vectorstore = Chroma( client=client, collection_name="my_docs", embedding_function=OpenAIEmbeddings(model="text-embedding-3-small") ) vectorstore.add_documents(documents) retriever = vectorstore.as_retriever(search_kwargs={"k": 4}) docs = retriever.invoke("What is LCEL?")
04 — Qdrant

Qdrant — Production Self-Hosted Vector DB

Qdrant (Rust, open-source) is the production choice for self-hosted deployments. It offers rich payload filtering, scalar quantization, hybrid dense+sparse search, and no GC pauses.

Qdrant's killer features are payload filtering and hybrid search. Payload filtering lets you combine vector similarity with arbitrary structured conditions — critical for multi-tenant RAG where you must never return documents from another tenant's corpus. The filter is pushed into the ANN search itself, not applied as a post-filter, so performance is maintained even with selective filters.

Scalar quantization compresses each float32 dimension to int8, reducing memory by 4x with only ~1% recall loss. For 10M vectors at 1536 dimensions, this drops index size from ~60GB to ~15GB. At billion scale, product quantization compresses further to under 10% of original size.

qdrant_production.py
from qdrant_client import QdrantClient from qdrant_client.models import ( Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue, ScalarQuantization, ScalarQuantizationConfig, ScalarType ) # docker run -p 6333:6333 qdrant/qdrant client = QdrantClient(host="localhost", port=6333) # Create collection with HNSW + scalar quantization client.create_collection( collection_name="knowledge_base", vectors_config=VectorParams(size=1536, distance=Distance.COSINE), quantization_config=ScalarQuantization( scalar=ScalarQuantizationConfig(type=ScalarType.INT8, quantile=0.99) ) # 4x RAM reduction, ~1% recall loss ) # Upsert points with payload metadata client.upsert( collection_name="knowledge_base", points=[PointStruct( id=doc_id, vector=embedding, payload={"tenant_id": "acme-corp", "doc_type": "policy", "created_at": "2025-01-15"} ) for doc_id, embedding in zip(ids, embeddings)] ) # Filtered search — mandatory tenant isolation results = client.query_points( collection_name="knowledge_base", query=query_embedding, query_filter=Filter( must=[FieldCondition(key="tenant_id", match=MatchValue(value="acme-corp"))] ), limit=5 )
05 — Hybrid Search

Hybrid Search: Dense + Sparse Retrieval

Pure semantic search misses exact keyword matches. BM25 keyword search misses semantic similarity. Hybrid search combines both for best-of-both-worlds retrieval, then uses Reciprocal Rank Fusion to merge the ranked lists.

Hybrid search is one of the highest-ROI improvements you can make to a RAG pipeline. Semantic search excels at paraphrase matching and conceptual similarity but can miss specific product names, error codes, or proper nouns. BM25 (keyword) excels at exact term matching but fails when the user phrases their query differently. Combining them handles both cases.

hybrid_search.py
from langchain_community.retrievers import BM25Retriever from langchain.retrievers import EnsembleRetriever # Dense (semantic) retriever from Qdrant dense_retriever = qdrant_vectorstore.as_retriever(search_kwargs={"k": 10}) # Sparse (BM25 keyword) retriever — in-memory sparse_retriever = BM25Retriever.from_documents(documents, k=10) # Ensemble with Reciprocal Rank Fusion (50/50 weight) hybrid_retriever = EnsembleRetriever( retrievers=[dense_retriever, sparse_retriever], weights=[0.5, 0.5] # tune based on your query distribution ) docs = hybrid_retriever.invoke("error code OOM-5432 in production") # Combines semantic match + exact "OOM-5432" keyword match # Typical improvement: +10–20% recall over semantic-only
💡
Why This Matters: In a benchmark on enterprise support docs, hybrid search improved retrieval recall from 71% to 89% vs semantic-only. The biggest wins came on queries with product names, error codes, and version numbers — exactly the cases pure semantic search struggles with.

08 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. HNSW is the right default index for most RAG systems — high recall, ~30ms latency.
  • 2. Payload filtering = security. Never skip it in multi-tenant systems.
  • 3. Scalar quantization (INT8) = 4x RAM reduction, ~1% recall loss. Use it in production.
  • 4. Hybrid search improves recall by +10–20% over semantic-only on real enterprise data.
  • 5. Graduate: FAISS → Chroma (dev) → Qdrant/Pinecone (production).

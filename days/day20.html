Section 01

RAGAS: RAG-Specific Evaluation Framework

RAGAS (RAG Assessment) provides a suite of reference-free metrics that evaluate both retrieval quality and generation quality in RAG pipelines — without needing ground-truth answers.

RAGAS evaluates RAG systems across five dimensions: Faithfulness (is every claim in the answer supported by the retrieved context?), Answer Relevance (does the answer address the user's question?), Context Precision (of all retrieved chunks, how many were actually relevant?), Context Recall (does the retrieved context contain all information needed to answer the question?), and Answer Correctness (factual accuracy vs. ground truth, when available).

The key insight is that RAGAS uses an LLM-as-judge approach — a strong LLM (typically GPT-4) evaluates the outputs rather than comparing against fixed reference answers. This makes RAGAS practical for open-ended QA where exact string matching is meaningless. The trade-off: RAGAS evaluations cost money (each evaluation calls the judge LLM multiple times) and inherit the biases of the judge model.

For production monitoring, run RAGAS on a sample of live queries (1-5% of traffic) continuously. Set up Prometheus metrics that track the rolling average of each RAGAS score. Create Grafana alerts that fire when faithfulness drops below 0.75 for 5 consecutive minutes — this indicates a retrieval problem (wrong chunks being retrieved) or a generation problem (the model hallucinating beyond context).

Why This Matters

Standard accuracy metrics are insufficient for RAG. A RAG system might have perfect answer correctness (the answer matches the expected) but terrible faithfulness (the model generated information not in the retrieved context — lucky correct hallucination). RAGAS surfaces these issues that would be invisible to traditional metrics, catching "confidently wrong" models before users notice.

Section 02

Embedding Drift Detection with KS Test

When the distribution of user queries shifts significantly from the training distribution, model quality degrades silently. Embedding drift detection catches this before users notice.

Data drift in LLM systems manifests when the topics, language patterns, or complexity of incoming queries change significantly from what the model was trained or fine-tuned on. For example, a customer service bot trained on English queries will perform poorly if 30% of production traffic suddenly shifts to Spanish — HTTP 200s keep flowing but answer quality tanks. Traditional monitoring (latency, error rate) is blind to this.

The approach: embed incoming queries with the same embedding model used at training time, then compare the distribution of these embeddings against a reference distribution (from training). Statistical tests like the Kolmogorov-Smirnov (KS) test measure the maximum difference between two empirical cumulative distribution functions. A high KS statistic (with low p-value) indicates the distributions differ significantly.

The KS test works on 1D distributions — for high-dimensional embeddings, you compare each principal component (after PCA) separately, or use Mean Maximum Discrepancy (MMD) which works directly in high-dimensional space. MMD is preferred for embedding drift detection as it captures global distributional differences rather than marginal differences per dimension.

drift_detection.py
import numpy as np from scipy import stats from sklearn.decomposition import PCA from sentence_transformers import SentenceTransformer embed_model = SentenceTransformer("BAAI/bge-large-en-v1.5") class EmbeddingDriftDetector: def __init__(self, n_components: int = 50, alpha: float = 0.05): self.pca = PCA(n_components=n_components) self.alpha = alpha self.reference_projections = None def fit_reference(self, texts: list[str]): """Compute reference distribution from training queries.""" embeddings = embed_model.encode(texts, show_progress_bar=True) self.reference_projections = self.pca.fit_transform(embeddings) def detect_drift(self, current_texts: list[str]) -> dict: """Run KS test on each PCA component.""" embeddings = embed_model.encode(current_texts) current_proj = self.pca.transform(embeddings) ks_stats, p_values = [], [] for i in range(self.pca.n_components_): ks, p = stats.ks_2samp( self.reference_projections[:, i], current_proj[:, i], ) ks_stats.append(ks) p_values.append(p) # Bonferroni correction for multiple comparisons alpha_adj = self.alpha / self.pca.n_components_ drifted_components = sum(p < alpha_adj for p in p_values) return { "max_ks": max(ks_stats), "drifted_components": drifted_components, "drift_detected": drifted_components > 0, } # Usage detector = EmbeddingDriftDetector() detector.fit_reference(training_queries) # run once at startup result = detector.detect_drift(last_hour_queries) if result["drift_detected"]: send_alert(f"Drift! Section 03

Prometheus, Grafana & Alerting

Infrastructure-level observability for LLM services: custom metrics, dashboards, and automated alerts that wake you up before users complain.

The Prometheus scraping model works well for LLM services: your BentoML or FastAPI server exposes a /metrics endpoint, Prometheus scrapes it every 15-30 seconds, and Grafana queries Prometheus with PromQL to build dashboards. For LLMs, instrument these metrics: token generation rate (tokens/second), TTFT (time to first token), queue depth (how many requests are waiting), VRAM utilization (via NVIDIA DCGM exporter), and cache hit rate (KV cache utilization).

Alerting rules live in Prometheus as PromQL expressions. A useful pattern is the "burned rate" alert: instead of alerting on instantaneous error rate, alert when your error budget is being consumed too fast. For a 99.9% SLA over 30 days, you have 43 minutes of error budget. If your current error rate would exhaust this in < 1 hour, fire a critical alert. This prevents alert fatigue from temporary spikes while catching sustained regressions.

prometheus_metrics.py
from prometheus_client import Histogram, Counter, Gauge, start_http_server import time # ── Define metrics ──────────────────────────────────────────────── REQUEST_LATENCY = Histogram( "llm_request_duration_seconds", "End-to-end request latency", buckets=[0.1, 0.25, 0.5, 1, 2, 5, 10, 30], labelnames=["model_version", "endpoint"], ) TTFT = Histogram( "llm_time_to_first_token_seconds", "Time until first token generated", buckets=[0.05, 0.1, 0.2, 0.5, 1, 2], ) TOKENS_GENERATED = Counter("llm_tokens_total", "Total tokens generated") ACTIVE_REQUESTS = Gauge("llm_active_requests", "Currently in-flight requests") VRAM_USED_GB = Gauge("gpu_vram_used_gb", "GPU VRAM used") RAGAS_FAITH = Gauge("rag_faithfulness_score", "Rolling RAGAS faithfulness") # ── FastAPI middleware to instrument ────────────────────────────── @app.middleware("http") async def metrics_middleware(request, call_next): ACTIVE_REQUESTS.inc() start = time.time() response = await call_next(request) duration = time.time() - start REQUEST_LATENCY.labels(model_version="v3.1", endpoint=request.url.path).observe(duration) ACTIVE_REQUESTS.dec() return response # ── Grafana alert rule (yaml) — error budget burn rate ───────────── # - alert: HighErrorBurnRate # expr: > # rate(llm_request_errors_total[5m]) / # rate(llm_requests_total[5m]) > 0.001 # > 0.1% error rate # for: 5m # labels: { severity: warning } # annotations: { summary: "LLM error rate SLA breach" }
Common Pitfall: Monitoring Without Alerting

Beautiful Grafana dashboards that nobody is watching are security theater. Every critical metric needs an alert that sends to PagerDuty, Slack, or email. The alert must include runbook links (what to do), current metric value, and a graph link. Without actionable alerts, monitoring is just documentation of the outage you missed.

📡

Prometheus

Pull-based metrics collection. Exposes /metrics endpoint, scrapes on schedule. Excellent for infrastructure + custom business metrics. PromQL for queries and alerts.

📊

Grafana

Visualization layer for Prometheus. Build dashboards with time series, histograms, heatmaps. Alerting built in. Open source self-hosted or managed cloud version.

🔍

OpenTelemetry

Vendor-neutral instrumentation standard. Single SDK for traces, metrics, and logs. Supports Prometheus, Jaeger, Datadog, Honeycomb backends. The future of observability.

Section 01

Why AI Observability is Different

Traditional observability tells you your API is up. LLM observability tells you your API is answering well. You need both — and they require different tooling.

⚠️
The Silent Failure Mode: An LLM service can have 100% uptime, sub-second latency, and zero HTTP errors — while hallucinating 40% of answers. Traditional monitoring sees nothing wrong. You need quality metrics running continuously in production.
⚡
Infrastructure Metrics
Latency, throughput, error rate, GPU utilization, memory. Same as any API — use Prometheus. These tell you the system is running, not how well.
🧠
LLM Quality Metrics
RAGAS faithfulness, answer relevancy, hallucination rate, refusal rate. These require running another model as evaluator — sample 5-10% of traffic nightly.
💬
Trace-Level Visibility
LangSmith captures every chain invocation: inputs, retrieved context, LLM output, latency, token counts. Essential for debugging specific failures.
📉
Drift Detection
Input distribution shifts over time as user behavior evolves. Embedding centroid drift and KL divergence catch this before quality degrades visibly.

Section 02

LangSmith Production Tracing

LangSmith traces every LLM call and chain invocation in production. Two environment variables and the @traceable decorator are all you need to get full visibility.

💡
Sampling Rate: In production, set LANGCHAIN_TRACING_SAMPLE_RATE=0.1 to trace 10% of requests. Full tracing adds ~5ms latency per request due to async upload — negligible at 10%, noticeable at 100%.
PYTHON — Production Tracing
import os from langsmith import traceable os.environ["LANGCHAIN_TRACING_V2"] = "true" os.environ["LANGCHAIN_API_KEY"] = "ls__..." # Trace any function with @traceable @traceable(name="RAG Pipeline", tags=["production", "v2.1"]) def answer_question(question: str, tenant_id: str) -> dict: context = retriever.invoke(question) answer = rag_chain.invoke({"question": question, "context": context}) return {"answer": answer, "sources": [d.metadata for d in context]} # Capture user feedback from langsmith import Client client = Client() def record_feedback(run_id: str, score: int, comment: str = ""): client.create_feedback( run_id=run_id, key="user_rating", score=score, # 1 = positive, 0 = negative comment=comment )
Section 03

RAGAS for Continuous Evaluation

Don't wait for user complaints. Run RAGAS nightly on a sample of production traffic and alert before humans notice quality degradation.

📅
Nightly Eval Job
Cron job at 2am: fetch last 24h of traces from LangSmith, run RAGAS faithfulness + relevancy, log scores to MLflow, alert if below threshold.
📊
5-10% Sampling
Evaluate a random 5-10% sample of production queries. RAGAS costs ~$0.002 per evaluation call — evaluating everything is expensive. Statistical sampling is sufficient.
📈
Trend Analysis
A single low score might be noise. A week-over-week downward trend is a signal. Track rolling 7-day averages in MLflow and alert on sustained degradation.
PYTHON — Nightly Evaluation Job
from ragas import evaluate from ragas.metrics import faithfulness, answer_relevancy import schedule import time def nightly_evaluation(): """Run RAGAS evaluation on sampled production traces""" from langsmith import Client client = Client() runs = list(client.list_runs( project_name="production", run_type="chain", limit=200, filter='and(eq(status, "success"), gte(start_time, -86400))' )) # Format for RAGAS eval_data = [{ "question": run.inputs.get("question", ""), "answer": run.outputs.get("answer", ""), "contexts": [run.outputs.get("context", "")] } for run in runs if run.inputs.get("question")] dataset = Dataset.from_list(eval_data) scores = evaluate(dataset, metrics=[faithfulness, answer_relevancy]) # Alert if below threshold if scores["faithfulness"] < 0.80: send_alert(f"Faithfulness score degraded: {scores['faithfulness']:.3f}") # Log to MLflow with mlflow.start_run(run_name=f"ragas-eval-{date.today()}"): mlflow.log_metrics(dict(scores)) schedule.every().day.at("02:00").do(nightly_evaluation)
Section 04

Prometheus + Grafana for LLM Infrastructure

Expose a /metrics endpoint from your serving layer. Prometheus scrapes it every 15 seconds. Grafana dashboards give you the ops view. Takes 30 minutes to set up and pays dividends forever.

💡
Key Grafana Panels for LLM APIs: (1) Request rate + error rate, (2) Latency heatmap with P50/P95/P99, (3) Token throughput (input vs output), (4) GPU utilization per replica, (5) Cost accumulator per model. Build these first.
PYTHON — Prometheus Metrics
from prometheus_client import Counter, Histogram, Gauge, start_http_server import time # Define metrics REQUEST_COUNT = Counter("llm_requests_total", "Total LLM requests", ["model", "status"]) TOKEN_COUNTER = Counter("llm_tokens_total", "Total tokens processed", ["model", "type"]) LATENCY = Histogram("llm_latency_seconds", "LLM request latency", buckets=[0.1, 0.5, 1.0, 2.0, 5.0, 10.0, 30.0]) GPU_UTIL = Gauge("gpu_utilization_percent", "GPU utilization", ["gpu_id"]) # Instrument your LLM calls def generate_with_metrics(prompt: str, model: str = "claude-3-5-sonnet") -> str: start = time.time() try: response = llm.invoke(prompt) REQUEST_COUNT.labels(model=model, status="success").inc() TOKEN_COUNTER.labels(model=model, type="input").inc(count_tokens(prompt)) TOKEN_COUNTER.labels(model=model, type="output").inc(count_tokens(response)) return response except Exception as e: REQUEST_COUNT.labels(model=model, status="error").inc() raise finally: LATENCY.observe(time.time() - start) start_http_server(9090) # expose /metrics endpoint for Prometheus
Section 05

Drift Detection & Continuous Learning

Models trained on last month's data degrade on next month's traffic. Embedding drift detection catches this before users notice — then automatically triggers the fine-tuning pipeline.

📐
Input Drift
Track embedding centroid distance week over week. KL divergence on distance distributions catches distribution shifts. Alert when KS test p-value < 0.05.
📤
Output Drift
Track response length distribution, refusal rate, and sentiment. Sudden change in any of these signals behavioral drift — often before quality metrics show it.
🔁
Continuous Learning
Drift alert → trigger KFP/SageMaker pipeline → fine-tune on new data + original data → evaluate → register in MLflow → canary deploy. Full loop automated.
PYTHON — Embedding Drift Detector
import numpy as np from scipy import stats class DriftDetector: def __init__(self, baseline_embeddings: np.ndarray): self.baseline = baseline_embeddings self.baseline_centroid = baseline_embeddings.mean(axis=0) def check_drift(self, current_embeddings: np.ndarray) -> dict: current_centroid = current_embeddings.mean(axis=0) # Centroid distance drift centroid_drift = np.linalg.norm(current_centroid - self.baseline_centroid) # KL divergence on distance distributions baseline_dists = np.linalg.norm(self.baseline - self.baseline_centroid, axis=1) current_dists = np.linalg.norm(current_embeddings - self.baseline_centroid, axis=1) ks_stat, p_value = stats.ks_2samp(baseline_dists, current_dists) return { "centroid_drift": float(centroid_drift), "ks_statistic": float(ks_stat), "p_value": float(p_value), "drift_detected": p_value < 0.05 }

02 — Pattern

Orchestrator / Worker Pattern

The orchestrator plans and delegates. Workers execute specialized tasks. This is the backbone of production multi-agent systems.

Research Agent
Tools: web_search, arxiv, summarize. Returns: structured findings.
Writer Agent
Input: research. Tools: none. Returns: structured draft.
Critic Agent
Input: draft. Returns: structured feedback with scores.
Formatter Agent
Input: reviewed draft. Returns: final formatted output.

Agent Handoff Animation

📨
Handoff to Writer Agent
The orchestrator routes the handoff message to the Writer Agent node. The Writer Agent gets a fresh context window populated with the handoff payload as its system context.
✍️
Writer Agent continues
Writer Agent receives full context without needing to redo the research. It uses the structured findings directly. No context window overflow — each agent has its own clean window.

05 — Architecture

Shared State vs Message Passing

Two fundamental models for agent communication. LangGraph uses shared state; pure multi-agent systems often use message passing. Each has tradeoffs.

Shared State LangGraph pattern
  • ✓ All agents see full history
  • ✓ Simple to reason about
  • ✓ Built-in persistence
  • ✗ State grows large with many agents
  • ✗ All agents share context window
Message Passing Pure multi-agent
  • ✓ Independent context windows
  • ✓ Scales to many agents
  • ✓ Loose coupling
  • ✗ Must serialize context explicitly
  • ✗ Harder to debug full trace

07 — Code

Supervisor Pattern with LangGraph

The supervisor is an orchestrator agent that decides which worker to call next based on the current state.

from langgraph.graph import StateGraph, END from langchain_anthropic import ChatAnthropic from typing import TypedDict, Annotated, list, Literal import operator # --- State --- class SupervisorState(TypedDict): task: str messages: Annotated[list, operator.add] research: str draft: str critique: str next_agent: str iteration: int # --- Worker Nodes --- def research_agent(state: SupervisorState) -> dict: llm = ChatAnthropic(model="claude-haiku-4-5").bind_tools([web_search]) result = llm.invoke([ {"role": "system", "content": "You are a research specialist."}, {"role": "user", "content": state["task"]} ]) return {"research": result.content, "messages": [result]} def writer_agent(state: SupervisorState) -> dict: llm = ChatAnthropic(model="claude-sonnet-4-5") result = llm.invoke([ {"role": "system", "content": "You are an expert technical writer."}, {"role": "user", "content": f"Research: {state['research']}\\n\\nWrite an article."} ]) return {"draft": result.content} def critic_agent(state: SupervisorState) -> dict: llm = ChatAnthropic(model="claude-haiku-4-5") result = llm.invoke([ {"role": "system", "content": "Review and score the draft 1-10."}, {"role": "user", "content": state["draft"]} ]) return {"critique": result.content} # --- Supervisor --- def supervisor(state: SupervisorState) -> dict: # Decides what happens next based on state if not state.get("research"): return {"next_agent": "research"} elif not state.get("draft"): return {"next_agent": "writer"} elif not state.get("critique"): return {"next_agent": "critic"} else: return {"next_agent": "FINISH"} def route_supervisor(state) -> Literal["research", "writer", "critic", "__end__"]: return "__end__" if state["next_agent"] == "FINISH" else state["next_agent"] # --- Build Graph --- graph = StateGraph(SupervisorState) graph.add_node("supervisor", supervisor) graph.add_node("research", research_agent) graph.add_node("writer", writer_agent) graph.add_node("critic", critic_agent) graph.set_entry_point("supervisor") graph.add_conditional_edges("supervisor", route_supervisor) graph.add_edge("research", "supervisor") graph.add_edge("writer", "supervisor") graph.add_edge("critic", "supervisor") app = graph.compile()

New — Section B

What Goes Inside an Agent Handoff

A handoff is a mini-spec. The receiving agent must be able to complete the task using only what's in it — no implicit shared state.

HANDOFF: Orchestrator → Writer Agent
task
"Write a blog post about Python async patterns"
context
[research findings from Research Agent output — FastAPI, aiohttp, Starlette summaries with GitHub stats]
constraints
"800-1000 words, use code examples, markdown format"
output_format
{ title: str, sections: list, word_count: int, tags: list }
quality_bar
"Expert Python developer audience — assume async/await familiarity"
Bad Handoff — Missing Context
return Command( goto="writer_agent", update={ "query": "write about Python" # NO context, no constraints, # no output format, no quality bar # Writer will produce generic output } )
Result: generic, unstructured output. Writer has no research to build on and no idea what "good" looks like.
Good Handoff — Complete Spec
return Command( goto="writer_agent", update={ "task": state["original_query"], "research_results": findings, "constraints": { "min_words": 800, "format": "markdown" }, "output_format": { "title": str, "sections": list, "tags": list } } )
Result: structured, high-quality output grounded in actual research. The writer has everything it needs.

The Golden Rule: A handoff is a mini-spec. The receiving agent should be able to complete the task with ONLY what's in the handoff — no implicit shared state, no assumed context, no "it should know what we discussed". If you deleted the orchestrator after creating the handoff, the worker should still be able to succeed.

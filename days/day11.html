01 — What Are Embeddings?

Dense Vectors & Semantic Space

An embedding model converts text (or images, audio) into a dense numerical vector — typically 256 to 3072 floats. Similar meanings map to nearby vectors in this high-dimensional space, enabling semantic search, clustering, and zero-shot classification.

The key property of good embeddings is that they capture semantic similarity, not just syntactic similarity. "The dog ran fast" and "A canine sprinted quickly" should produce similar vectors even though they share no words. This is what makes embedding-based RAG fundamentally more powerful than keyword search — it finds conceptually relevant content even when the exact words don't match.

Modern embedding models are trained in two phases: first, a large transformer pre-trains on a massive corpus via masked language modeling or similar objectives. Then, contrastive fine-tuning (using positive/negative pairs) shapes the embedding space so that semantically similar sentences cluster together and dissimilar ones are pushed apart. The MTEB benchmark evaluates this across 56 diverse tasks — retrieval, classification, clustering, semantic textual similarity, and more.

📐
Dimensionality
Number of floats per embedding. Higher = more expressive but slower search and more RAM. 1536 (OAI small) is the sweet spot for most RAG. Use 256 for speed, 3072 only if max quality matters.
🎯
Cosine Similarity
Measures angle between vectors, ignoring magnitude. Range [-1, 1]. 1.0 = identical, 0.0 = orthogonal, -1.0 = opposite. Always normalize text embeddings before using dot product.
🪆
Matryoshka Embeddings
The first N dimensions are themselves a complete, usable embedding. OpenAI text-embedding-3 supports this. Truncate to 256 dims for fast search, use 1536 only for final reranking. Saves 6x RAM.
🔢
Max Tokens
Embedding models have a hard context limit (512–8192 tokens). Text exceeding the limit is truncated — silently losing content. Always chunk documents smaller than the model's limit. Check tokenizer, not character count.
02 — Model Selection

Choosing an Embedding Model

The MTEB (Massive Text Embedding Benchmark) is the authoritative leaderboard. For most English-language RAG systems, OpenAI text-embedding-3-small or a local E5/BGE model are the right starting points.

The critical decision is API-hosted vs self-hosted. OpenAI and Cohere offer hosted embedding APIs with zero infrastructure cost — you pay per token. For high-volume systems (millions of embeddings per day), self-hosting a BAAI/bge or E5 model on a GPU instance becomes dramatically cheaper. The crossover point depends on your GPU cost and query volume.

⚠️
Common Pitfall — Model Mismatch: If you embed documents with model A and queries with model B, similarity scores are meaningless. The embedding spaces are incompatible. Always use the same model and the same preprocessing (including normalization) for both document and query embeddings.
03 — Production Patterns

Batch Embedding, Caching & Rate Limiting

Embedding at scale requires careful batching, caching repeated queries, and rate limit handling. A naive single-call-per-document approach is 5–10x slower and more expensive than batched embedding.

The OpenAI embedding API allows up to 2048 inputs per batch request. Batching 512 short documents in a single API call instead of 512 individual calls reduces total latency by 5–10x (single HTTP round-trip vs 512) and often hits higher parallelism on the provider's side. For very large corpora, combine batching with async concurrency — 10 concurrent batch requests of 512 docs each processes 5120 docs simultaneously.

Query embedding caching is often overlooked but delivers immediate savings. In a RAG system, the same user question is frequently re-asked (especially in demos and repeated workflows). Cache the embedding vector keyed by the query string hash with a TTL of 24h. This eliminates embedding cost for cache-hit queries entirely.

production_embeddings.py
from openai import AsyncOpenAI import asyncio, hashlib, json from functools import lru_cache client = AsyncOpenAI() # Batch embedding — process 512 docs per API call async def embed_batch(texts: list[str]) -> list[list[float]]: response = await client.embeddings.create( model="text-embedding-3-small", input=texts, # up to 2048 items dimensions=512 # Matryoshka — truncate to 512 dims (saves 3x memory) ) return [item.embedding for item in response.data] async def embed_corpus(docs: list[str], batch_size: int = 512) -> list: # Concurrent batching — 10 simultaneous API calls batches = [docs[i:i+batch_size] for i in range(0, len(docs), batch_size)] semaphore = asyncio.Semaphore(10) # max 10 concurrent requests async def embed_with_sem(batch): async with semaphore: return await embed_batch(batch) results = await asyncio.gather(*[embed_with_sem(b) for b in batches]) return [emb for batch_result in results for emb in batch_result] # Query embedding cache (in-memory, 24h TTL in production use Redis) _embed_cache: dict = {} async def embed_query_cached(text: str) -> list[float]: key = hashlib.sha256(text.encode()).hexdigest() if key not in _embed_cache: _embed_cache[key] = (await embed_batch([text]))[0] return _embed_cache[key]

04 — Advanced

ColBERT, Fine-tuning & Matryoshka

Beyond single-vector embeddings: multi-vector ColBERT, domain fine-tuning with contrastive loss, and Matryoshka dimensionality compression.

ColBERT — Multi-Vector Retrieval

Instead of one vector per document, ColBERT creates one vector per token. Query tokens are compared against document tokens with MaxSim pooling. Significantly better recall than single-vector, especially for long documents and complex queries.

Tradeoff: storage is N_tokens × 128 floats per doc instead of 1536 floats. Use RAGatouille (LlamaIndex) for a clean Python API.

Domain Fine-Tuning

When your domain uses specialized vocabulary (legal, medical, financial), a general embedding model may perform poorly. Fine-tune with contrastive loss on (query, relevant-doc, irrelevant-doc) triplets from your domain.

Typical gain: +5–15% NDCG on domain-specific retrieval tasks. Use sentence-transformers' MultipleNegativesRankingLoss.

⚠️
Common Pitfall — Chunking Beyond Max Tokens: OpenAI's text-embedding-3-small has an 8191 token limit. Text is truncated silently — no error, no warning. Always chunk your documents to stay well under the limit (aim for 400–600 tokens per chunk). Use tiktoken to measure, not character count.

07 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. Always use the same model for document and query embeddings — spaces are incompatible.
  • 2. text-embedding-3-small: 5x cheaper than ada-002, better quality. Default choice.
  • 3. Batch embed with concurrent API calls — 5–10x throughput improvement.
  • 4. Matryoshka: use 512 dims for fast index, 1536 only for precision reranking.
  • 5. Cache query embeddings — same query re-asked is free after first call.

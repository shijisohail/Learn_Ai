Section 01

Understanding Embeddings

An embedding is a fixed-length float vector that encodes the semantic meaning of text. Similar meanings cluster close together in that high-dimensional space — that geometric proximity is what makes search possible.

🔢
High-Dimensional Vectors
A chunk of text becomes a list of 384–3072 floats. Every dimension captures a latent semantic feature learned during pretraining. The exact meaning of each dimension is not interpretable — it's an emergent, distributed representation.
🏗️
Encoder-Only Transformers
Unlike GPT (decoder-only), embedding models use BERT-style encoder-only architectures. Every token attends to every other token bidirectionally. The [CLS] token or mean-pooling of the final layer becomes the embedding vector.
📐
Cosine Similarity
The standard distance metric for embeddings. It measures the angle between two vectors — 1.0 means identical direction (same meaning), 0.0 means orthogonal (unrelated), -1.0 means opposite. Magnitude is ignored, so length normalization matters.
🗺️
Semantic Clusters
Semantically related texts form clusters in the embedding space. "Python programming", "snake language", and "PyPI package" are close. "python" the snake and "python" the language are disambiguated by context in the surrounding sentence.
Python
import numpy as np def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float: return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) # Example: embeddings for related vs. unrelated concepts embedding_llm = embed("Large Language Model") embedding_gpt = embed("GPT-4 AI system") embedding_car = embed("automobile vehicle") print(cosine_similarity(embedding_llm, embedding_gpt)) # ~0.85 — similar print(cosine_similarity(embedding_llm, embedding_car)) # ~0.35 — different # Why cosine and not Euclidean distance? # Euclidean is sensitive to vector magnitude. A very long doc # produces a larger magnitude vector but similar direction to a # short doc with the same meaning. Cosine ignores magnitude.
Section 02

OpenAI & Cohere Embedding Models

The two dominant managed embedding APIs. OpenAI's text-embedding-3 family uses Matryoshka Representation Learning. Cohere differentiates with multilingual support and semantic input_type.

Model Dimensions Context Strength Cost
text-embedding-3-small 1536 8191 tokens Best quality/cost ratio $0.02 / 1M tokens
text-embedding-3-large 3072 8191 tokens Highest quality $0.13 / 1M tokens
text-embedding-ada-002 1536 8191 tokens Legacy, still widely deployed $0.10 / 1M tokens
Cohere embed-english-v3.0 1024 512 tokens input_type disambiguation $0.10 / 1M tokens
Cohere embed-multilingual-v3.0 1024 512 tokens 100+ languages, same space $0.10 / 1M tokens
💡
Cohere input_type: Cohere v3 was trained with separate embedding spaces for queries vs documents. Always pass input_type="search_query" for user queries and input_type="search_document" for indexed content. This alone can improve retrieval precision by 5–15%.
Python
from openai import OpenAI import cohere # OpenAI embeddings — batched for throughput oai = OpenAI() texts = ["Document about RAG", "Article on fine-tuning"] def embed_batch(texts: list[str], batch_size=100) -> list[list[float]]: """Process in batches of 100 for optimal throughput.""" embeddings = [] for i in range(0, len(texts), batch_size): batch = texts[i:i+batch_size] response = oai.embeddings.create( model="text-embedding-3-small", input=batch ) embeddings.extend([r.embedding for r in response.data]) return embeddings # Cohere — specify input type for better retrieval quality co = cohere.Client("...") # When indexing documents: doc_embeddings = co.embed( texts=["RLHF uses human preferences to align models..."], model="embed-english-v3.0", input_type="search_document" ) # When embedding user queries: query_embedding = co.embed( texts=["What is RLHF?"], model="embed-english-v3.0", input_type="search_query" # vs "search_document", "classification" )
Section 03

Open-Source Embedding Models

When data privacy, offline inference, or cost at scale matters — open-source models running locally via HuggingFace and sentence-transformers give you full control without per-token billing.

⚡
all-MiniLM-L6-v2 (384 dim)
Smallest and fastest sentence-transformer. 22M parameters, ~80MB on disk, <10ms CPU inference. Great for prototyping and high-throughput pipelines where quality can be slightly lower.
🎯
all-mpnet-base-v2 (768 dim)
SBERT's highest-quality general-purpose model. Microsoft mpnet base, trained on 1B sentence pairs with mean-pooling. Best open-source quality before BGE-M3 arrived.
🌍
BGE-M3 (1024 dim)
BAAI's state-of-the-art open model. Supports dense, sparse, and multi-vector retrieval in one model. Multilingual (100+ languages), context up to 8192 tokens. The current best open-source choice for production RAG.
🔒
When to Use Open-Source
PII/sensitive data that can't leave your network. High-volume embedding (>100M tokens/month where API costs compound). Air-gapped environments. Latency-sensitive offline inference on edge hardware.
Python
from sentence_transformers import SentenceTransformer import torch # Fast, lightweight — good for prototyping model = SentenceTransformer("all-MiniLM-L6-v2") embeddings = model.encode( ["RAG is a retrieval technique", "Fine-tuning updates model weights"], batch_size=64, show_progress_bar=True, device="cuda" if torch.cuda.is_available() else "cpu" ) # BGE-M3 — best open-source, multilingual, 8192 token context bge = SentenceTransformer("BAAI/bge-m3") query_embedding = bge.encode( "What is attention mechanism?", normalize_embeddings=True # normalize for cosine similarity ) # Batch encoding for indexing large document sets corpus = ["doc1...", "doc2..."] # could be thousands doc_embeddings = bge.encode( corpus, batch_size=32, # reduce if OOM on GPU normalize_embeddings=True, show_progress_bar=True ) # doc_embeddings.shape → (len(corpus), 1024)
Section 04

Dimensionality, Truncation & Production Patterns

Higher dimensions give more expressive embeddings but cost more storage and compute. Matryoshka Representation Learning lets you truncate OpenAI's newest models to lower dimensions with minimal quality loss.

🪆
Matryoshka Representation Learning (MRL): text-embedding-3-small and text-embedding-3-large were trained so that the first N dimensions of the vector are themselves a valid, high-quality embedding. You can truncate to 256, 512, or 1024 dimensions with only ~5% quality degradation vs. 80% storage savings at 512 dims.
Python
# MRL — truncate to lower dimension for cheaper storage response = oai.embeddings.create( model="text-embedding-3-small", input="Hello world", dimensions=512 # truncate from 1536 → 512 (80% quality retained) ) # Vector storage: 1536 dims = 6144 bytes @ float32 # 512 dims = 2048 bytes @ float32 — 3x cheaper # ── Async batching with rate limit handling ───────────── import asyncio from openai import AsyncOpenAI aai = AsyncOpenAI() async def embed_async(texts: list[str], semaphore_limit: int = 5): sem = asyncio.Semaphore(semaphore_limit) # max concurrent requests async def embed_one_batch(batch): async with sem: resp = await aai.embeddings.create( model="text-embedding-3-small", input=batch ) return [r.embedding for r in resp.data] batches = [texts[i:i+100] for i in range(0, len(texts), 100)] results = await asyncio.gather(*[embed_one_batch(b) for b in batches]) return [emb for batch_result in results for emb in batch_result] # ── Embedding cache — avoid re-embedding identical texts ─ from functools import lru_cache @lru_cache(maxsize=10000) def get_embedding(text: str) -> tuple[float, ...]: response = oai.embeddings.create(model="text-embedding-3-small", input=text) return tuple(response.data[0].embedding) # For production: use Redis with hash(text) as key import hashlib, json import redis r = redis.Redis(host="localhost") def get_cached_embedding(text: str) -> list[float]: key = f"emb:{hashlib.sha256(text.encode()).hexdigest()[:16]}" cached = r.get(key) if cached: return json.loads(cached) emb = get_embedding(text) r.setex(key, 86400, json.dumps(list(emb))) # 24h TTL return list(emb)
Section 05

Embedding Drift & Re-indexing

When an embedding model is updated, its vector space changes. Old embeddings from a previous model version are no longer comparable to new ones — your vector index silently degrades.

📉
What Is Embedding Drift
Model providers occasionally release improved model versions. text-embedding-ada-002 and text-embedding-3-small produce incompatible vector spaces. Mixing them in the same collection causes silent retrieval failures — vectors from different spaces cannot be meaningfully compared.
📊
Detection
Monitor retrieval quality metrics over time: MRR@10, NDCG, and a golden set of known query-document pairs. A sudden drop in recall after a model update is the primary signal. Use RAGAS faithfulness and context recall as continuous eval.
🔄
Re-indexing Strategy
Blue-green indexing: build the new index in the background against a shadow collection, run quality eval on the new index, then atomically swap the collection pointer. Zero downtime with a safe rollback path. Never update embeddings in-place.
⚖️
Multi-Model Indexes
In a migration period, maintain embeddings from both models. Route some traffic to the new index, compare result quality, roll forward when satisfied. Qdrant collection aliases make this practical — point an alias at either collection atomically.
⚠️
Critical Rule: Never mix embeddings from different models (or versions) in the same index — semantic spaces are incompatible. text-embedding-ada-002 and text-embedding-3-small vectors cannot be compared against each other. Always re-embed the full corpus when changing models.

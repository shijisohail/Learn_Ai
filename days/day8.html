01 — Overview

What is LangChain?

LangChain is a composability framework for LLM applications. It standardises the interface for models, prompts, retrievers, and tools so you can wire them together with minimal boilerplate — and swap providers without rewriting your logic.

Before LangChain, building a RAG pipeline meant writing hundreds of lines of glue code: manual HTTP calls, ad-hoc prompt formatting, custom streaming loops, and zero observability. LangChain abstracts all of that into a unified Runnable protocol. Every component — prompts, models, retrievers, parsers, tools — speaks the same interface: .invoke(), .stream(), .batch(), and async variants of all three.

The key philosophical shift is that chains are composable values, not imperative procedures. You define a pipeline declaratively using the pipe operator, and LangChain handles concurrency, error propagation, streaming, and tracing automatically. This means your business logic stays clean while infrastructure concerns are handled by the framework.

💡
LangChain vs LangGraph: Use LangChain for stateless pipelines where data flows in one direction (RAG, summarisation, extraction). Use LangGraph when you need cycles, conditional branching, or persistent agent state between steps. Day 8 is LangChain; Day 15 covers LangGraph.

02 — LCEL

LangChain Expression Language

LCEL uses the pipe operator | to compose Runnables into a RunnableSequence. Every chain built with LCEL automatically gains invoke, stream, batch, and async variants — no extra code.

The pipe operator is syntactic sugar over RunnableSequence. Python calls __or__ on the left operand with the right operand. LangChain's Runnable base class intercepts this and builds a sequence node. The entire chain is then a single Runnable that can itself be piped into further chains — composability all the way down.

This declarative style has a profound practical consequence: your chain definition is data, not code. You can serialize it, inspect it programmatically, and instrument it at runtime without touching business logic. LangSmith exploits this by auto-tracing every node in the graph.

lcel_basics.py
from langchain_core.prompts import ChatPromptTemplate from langchain_openai import ChatOpenAI from langchain_core.output_parsers import StrOutputParser prompt = ChatPromptTemplate.from_template( "Summarize this in one sentence: {text}" ) model = ChatOpenAI(model="gpt-4o-mini", temperature=0) parser = StrOutputParser() # LCEL chain — same interface for invoke/stream/batch/async chain = prompt | model | parser # Invoke — synchronous, returns string result = chain.invoke({"text": "LangChain is a framework for LLM applications..."}) # Stream — yields string chunks as they arrive for chunk in chain.stream({"text": "LangChain is a framework..."}): print(chunk, end="", flush=True) # Batch — runs 2 inputs in parallel with thread pool results = chain.batch([{"text": "Doc 1..."}, {"text": "Doc 2..."}]) # Async — non-blocking, use in FastAPI endpoints result = await chain.ainvoke({"text": "..."})
parallel_chain.py
from langchain_core.runnables import RunnableParallel, RunnablePassthrough # RunnableParallel runs both branches with the same input simultaneously parallel_chain = RunnableParallel({ "summary": prompt | model | parser, "original": RunnablePassthrough() }) output = parallel_chain.invoke({"text": "Long document about AI..."}) # output = {"summary": "AI is...", "original": {"text": "Long document..."}} # Fallback — if primary fails, use backup (different provider) primary = ChatOpenAI(model="gpt-4o") backup = ChatAnthropic(model="claude-3-5-haiku-20241022") robust_model = primary.with_fallbacks([backup]) chain_with_fallback = prompt | robust_model | parser
⚠️
Fallbacks: Use chain.with_fallbacks([backup_chain]) to automatically retry on exceptions. Pairs perfectly with multi-provider strategies — primary Claude, fallback GPT-4o. Always test that the fallback actually triggers — it only fires on exceptions, not on empty/bad outputs.
03 — RAG

Building a RAG Chain with LCEL

A retriever is itself a Runnable — it accepts a string query and returns a list of Documents. LCEL makes it trivial to compose retrieval, formatting, and generation into a single streamable pipeline.

The RAG chain is where LCEL's composability truly shines. The dict syntax {"context": retriever | format, "question": passthrough} creates an implicit RunnableParallel — the retriever runs concurrently with the passthrough, and both results are merged before hitting the prompt template.

This pattern scales elegantly. Add hybrid search by running a vector retriever and a keyword retriever in parallel inside the context branch, then merge their results with Reciprocal Rank Fusion before formatting. The pipeline stays readable and composable throughout.

rag_chain.py
from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings from langchain_core.runnables import RunnableLambda, RunnablePassthrough vectorstore = Chroma(embedding_function=OpenAIEmbeddings()) retriever = vectorstore.as_retriever(search_kwargs={"k": 4}) def format_docs(docs): return "\n\n".join(doc.page_content for doc in docs) rag_prompt = ChatPromptTemplate.from_template(""" Answer based only on the context below. Say "I don't know" if not found. Context: {context} Question: {question} """) # The dict creates an implicit RunnableParallel rag_chain = ( {"context": retriever | RunnableLambda(format_docs), "question": RunnablePassthrough()} | rag_prompt | model | parser ) answer = rag_chain.invoke("What is LCEL?") # Streaming answer from RAG pipeline async for chunk in rag_chain.astream("Explain LangSmith"): yield chunk
💡
Why This Matters: The streaming RAG chain is production-ready immediately. Wrap it in a FastAPI SSE endpoint and users get token-by-token streaming without any extra buffering logic. LangChain handles back-pressure automatically.

04 — Observability

LangSmith — Tracing & Observability

LangSmith is the observability platform for LangChain. Every Runnable call is automatically traced: you see the full span tree, token counts, latency per step, inputs/outputs, and error messages — without changing any application code.

LangSmith works by monkey-patching the LangChain Runnable base class at import time when LANGCHAIN_TRACING_V2=true is set. Every call is assigned a Run ID, and parent/child relationships are tracked through Python's contextvars — so even concurrent async chains produce correctly nested trace trees without any manual instrumentation.

Beyond tracing, LangSmith enables dataset-driven evaluation. You upload (input, expected output) pairs, run your chain over the entire dataset, and score with LLM-as-judge or regex evaluators. This closes the feedback loop: you can A/B test prompt changes with statistical confidence before deploying.

What Gets Traced
  • → Every Runnable (prompt, model, retriever, parser)
  • → Token counts: input, output, cached
  • → Wall-clock latency per span
  • → Full input/output at each step
  • → Errors with full traceback
  • → Cost estimate per run
Evaluation Runs
  • → Upload dataset of (input, expected) pairs
  • → Run chain over dataset automatically
  • → Score with LLM-as-judge or custom evaluators
  • → A/B compare two prompt versions
  • → Track regressions across deployments
  • → Monitor production traces in real-time

import os os.environ["LANGCHAIN_TRACING_V2"] = "true" os.environ["LANGCHAIN_API_KEY"] = "ls__..." os.environ["LANGCHAIN_PROJECT"] = "rag-pipeline-v2" # All subsequent chain calls are automatically traced result = rag_chain.invoke("What is PagedAttention?") # Trace visible at https://smith.langchain.com # Span tree: RunnableParallel(234ms) → Retriever(18ms), Passthrough(0ms) # → ChatPromptTemplate(1ms) → ChatOpenAI(312ms, 523 tokens) # → StrOutputParser(0ms) # Manual trace — wrap non-LangChain code from langsmith import traceable @traceable(run_type="chain", name="custom_processing") def process_results(results: list) -> str: # This will appear as a span in LangSmith return "\n".join(r["answer"] for r in results)
05 — Callbacks

Callbacks & Custom Logging

Callbacks let you hook into lifecycle events of every Runnable without modifying chain logic. Use them for custom logging, streaming UI updates, and per-run cost tracking in production.

The callback system fires synchronous or async hooks at every lifecycle event. Handlers inherit from BaseCallbackHandler and override only the events they care about. Because all methods have no-op defaults, you only implement what you need. Handlers can be attached globally (every chain), per-chain, or per-run using .with_config(callbacks=[...]).

on_llm_start
Fired before each LLM call — receives prompts and model kwargs. Use to log request payloads.
on_llm_end
Fired after LLM responds — receives full LLMResult including token usage. Perfect for cost tracking.
on_chain_start/end
Fired at the boundaries of each chain run — use for timing and request correlation IDs.
on_tool_start/end
Fired around tool/retriever invocations — critical for logging which tools agents called and what they returned.
custom_callback.py
from langchain_core.callbacks import BaseCallbackHandler import time class CostAndLatencyTracker(BaseCallbackHandler): def __init__(self): self.total_tokens = 0 self.start_time = None def on_llm_start(self, serialized, prompts, **kwargs): self.start_time = time.time() def on_llm_end(self, response, **kwargs): usage = response.llm_output.get("token_usage", {}) tokens = usage.get("total_tokens", 0) latency = time.time() - self.start_time self.total_tokens += tokens print(f"[LLM] {tokens} tokens in {latency:.2f}s") tracker = CostAndLatencyTracker() chain_tracked = chain.with_config(callbacks=[tracker]) result = chain_tracked.invoke({"text": "Explain transformers"}) print(f"Session total tokens: {tracker.total_tokens}")
⚠️
Common Pitfall — Callback Thread Safety: Callbacks are invoked in the same thread as the chain (or in async tasks for async chains). If your handler writes to shared state (e.g., a counter), use a lock or asyncio-safe primitives. Never do blocking I/O in a synchronous callback on an async chain — it will block the event loop.

08 — Advanced

Advanced LCEL Patterns

Beyond basic chains: structured output, retry logic, and multi-step reasoning pipelines.

🔁
Retry with Backoff
Use .with_retry(stop_after_attempt=3) to handle transient rate limits. Combine with fallbacks for true resilience.
📐
Structured Output
Use model.with_structured_output(MySchema) to force JSON matching a Pydantic model. Zero parser errors.
🧵
Conversation Memory
RunnableWithMessageHistory wraps any chain with conversation state stored in any backend (Redis, DynamoDB, in-memory).
🌿
RunnableBranch
Route to different chains based on predicates. Pattern: (condition_fn, chain_A), default_chain_B. Enables intent-based routing.
structured_output.py
from pydantic import BaseModel, Field class ExtractedInfo(BaseModel): name: str = Field(description="Person's full name") age: int = Field(description="Age in years") summary: str = Field(description="One-sentence bio") # with_structured_output uses tool-calling or JSON mode under the hood structured_model = model.with_structured_output(ExtractedInfo) extract_chain = prompt | structured_model result = extract_chain.invoke({"text": "Alice, 34, software engineer..."}) print(result.name) # "Alice" ← typed Python object, not raw JSON print(result.age) # 34 # RunnableBranch — route by topic from langchain_core.runnables import RunnableBranch classifier_chain = ( RunnableBranch( (lambda x: "code" in x["topic"], code_chain), (lambda x: "math" in x["topic"], math_chain), general_chain # default ) )
09 — Practice

Exercises & Key Takeaways

Key Takeaways
  • 1. LCEL chains are composable values — pipe anything that implements Runnable.
  • 2. Dict syntax creates implicit RunnableParallel — retriever and passthrough run concurrently.
  • 3. Set 2 env vars and every chain call appears in LangSmith — zero instrumentation code.
  • 4. with_structured_output enforces schema — prefer over manual JSON parsing.
  • 5. Use with_fallbacks for multi-provider resilience; use with_retry for transient errors.

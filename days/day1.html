Autoregressive Generation

LLMs are next-token predictors — they generate one token at a time, feeding each back as input.

Key insight: The model doesn't "plan" a sentence. It predicts the most likely next token given all previous tokens. Intelligence emerges from doing this very well at massive scale.

Tool Use Flow

Tool use is a 5-step dance between your code and the model. Click through each step.

👤
User
Step 1 — User sends message + tool definitions
Your code sends a request that includes both the conversation AND the list of available tools (as JSON schemas). The model can only use tools you describe.
messages=[{"role":"user","content":"What's the weather in Tokyo?"}]
tools=[weather_tool_schema]
🤖
Model
Step 2 — Model returns tool_use block (not end_turn!)
Instead of text, the model returns a tool_use content block. The stop_reason is "tool_use" — your code must check this!
stop_reason: "tool_use"
content: [{type:"tool_use", name:"get_weather",
input:{"city":"Tokyo"}}]
⚙️
Your Code
Step 3 — YOUR code executes the real tool
The model cannot run code. It just says what it wants. You extract the tool name + inputs, call the actual function, and capture the result. This is where your backend logic lives.
result = get_weather(city="Tokyo")
# → {"temp": 18, "condition": "Cloudy"}
👤
Your Code
Step 4 — Send tool_result back to the model
You construct a tool_result message and send the full conversation history (including the model's tool_use) back to the API. The result can be any string — JSON, text, error messages.
{"role":"user","content":[
{type:"tool_result", tool_use_id:"...",
content:"temp:18°C, Cloudy"}]}
🤖
Model
Step 5 — Model gives final answer
Now the model synthesizes the tool result into a natural language response. The stop_reason is now "end_turn". The agent loop is complete (for this turn).
stop_reason: "end_turn"
content: "Tokyo is currently 18°C and cloudy!"

05_tool_use_loop.py — the full ReAct pattern python
def get_weather(city: str) -> str: return f"{city}: 18°C, Cloudy" # real impl calls weather API tools = [{ "name": "get_weather", "description": "Get current weather for a city", "input_schema": { "type": "object", "properties": { "city": {"type": "string", "description": "City name"} }, "required": ["city"] } }] def agent(user_msg: str) -> str: messages = [{"role": "user", "content": user_msg}] while True: # the agent loop response = client.messages.create( model="claude-haiku-4-5-20251001", max_tokens=1024, tools=tools, messages=messages ) # Append the assistant's response (may contain tool_use blocks) messages.append({"role": "assistant", "content": response.content}) if response.stop_reason == "end_turn": return response.content[0].text # done! # Handle tool calls tool_results = [] for block in response.content: if block.type == "tool_use": result = get_weather(**block.input) # dispatch tool_results.append({ "type": "tool_result", "tool_use_id": block.id, "content": result }) # Send results back, loop again messages.append({"role": "user", "content": tool_results})

Inference Internals

KV Cache: Why Inference is Fast (And What It Costs)

Every token the model generates requires an attention computation over all previous tokens. Without caching, generating 1,000 tokens means 1,000 × full-context attention passes — O(n²) work. The KV cache fixes this.

How Attention Works per Token
Without KV Cache (O(n²))
Every new token recomputes attention over ALL previous tokens.
With KV Cache (O(n) per token)
K/V matrices are computed once, stored, reused for every new token.
Prompt Caching (Anthropic API) — You can cache static parts of your prompt (system prompt, tool schemas, RAG context) so the server reuses the KV state across requests. Result: 90% cost reduction on repeated prefixes and up to 5× faster time-to-first-token.

Prompt Engineering

Anthropic Prompt Engineering Best Practices

Claude responds best when prompts are well-structured. These patterns come directly from Anthropic's prompting guides and reflect how Claude was trained to process instructions.

1 — XML Tag Structure

Wrapping prompt sections in XML tags eliminates ambiguity. Claude was trained on XML-delimited documents and parses tagged sections with high fidelity — especially useful for multi-part prompts.

✗ Without structure
prompt (ambiguous)
You are a helpful assistant. Summarise the text below. Keep it under 50 words. Use bullet points. The user said: hello can you summarise this article: [ARTICLE] Be concise. Also translate to French.
✓ XML-tagged structure
prompt (clear)
<context> You are a concise summariser. </context> <instructions> Summarise the article in ≤50 words using bullet points, then translate the summary to French. </instructions> <article> {{ARTICLE_TEXT}} </article> <output_format> EN bullets, then FR bullets. </output_format>

Common tags: <context> <instructions> <examples> <output_format> <constraints> <thinking>

2 — Few-Shot Examples

2–5 examples is the sweet spot. Order matters — the last example is weighted most heavily. Put the hardest or most representative case last.

Pythonfew_shot_messages pattern
# Build a reusable few-shot prefix then append the real task few_shot_messages = [ {"role": "user", "content": "Classify: 'Great product!' "}, {"role": "assistant", "content": "POSITIVE"}, {"role": "user", "content": "Classify: 'Broke after a week.'"}, {"role": "assistant", "content": "NEGATIVE"}, {"role": "user", "content": "Classify: 'It is what it is.'"}, {"role": "assistant", "content": "NEUTRAL"}, # ↑ last example = most influential; make it the canonical hard case ] response = client.messages.create( model="claude-sonnet-4-5", system="Classify reviews as POSITIVE / NEGATIVE / NEUTRAL only.", messages=few_shot_messages + [ {"role": "user", "content": f"Classify: '{user_review}'"} ], )

Tip: never use assistant prefill as the last message unless you intend to steer the opening token — it can constrain reasoning.

3 — System Prompt Anatomy

A well-formed system prompt has four distinct parts. Keep them in this order — Claude processes the system turn top-to-bottom before reading the first user message.

①
Role
Who Claude is.
"You are a senior DevOps engineer…"
②
Context
Situation & background the model needs to ground its answers.
③
Constraints
What to never do, tone, language, length limits, refusal rules.
④
Output Format
Exact schema, Markdown rules, JSON shape, or example structure.

4 — Prompt Pattern Reference

Pattern When to use Example trigger
Zero-shot Simple, well-defined tasks where Claude needs no demonstrations. "Translate to Spanish: …"
One-shot Format is non-obvious; one example sets the output shape precisely. "Like this: Q:… A:… Now do: …"
Few-shot Classification, extraction, or generation with a strict schema (2-5 examples). Sentiment labelling, NER tagging
Chain-of-Thought Multi-step reasoning, maths, logic. Ask Claude to think before answering. "Think step-by-step before giving your final answer."
Role Prompting Shifts tone, vocabulary, or domain depth. Works best in system turn. "You are a terse Socratic tutor who never gives answers directly."
Step-back Prompting Ask Claude to derive the general principle first, then apply it to the specific case. "What general principle governs X? Now apply that to …"

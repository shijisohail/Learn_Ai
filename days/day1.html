Section 01

What Is a Large Language Model?

Before writing a single line of API code, you need a mental model of what's actually happening inside the black box. This isn't magic — it's matrix multiplication at an almost incomprehensible scale.

🧠

Neural Network at its Core

An LLM is a deep neural network — specifically a Transformer — trained to predict the next token in a sequence. After training on hundreds of billions of tokens, the weights encode an implicit world model: grammar, facts, reasoning patterns, code syntax, and more.

📊

Probability Distributions

When you send a prompt, the model doesn't "think" — it passes your tokens through ~96 layers of attention and feed-forward operations, producing a probability distribution over its entire vocabulary (~100K tokens). The highest probability token is sampled as the next word.

The Transformer architecture, introduced in 2017's "Attention is All You Need" paper, relies on a mechanism called self-attention. Every token can directly attend to every other token in the context window simultaneously — unlike older RNNs that processed tokens sequentially. This parallelism is why Transformers can train on massive datasets and why GPUs are so essential.

Modern LLMs like Claude 3.5 Sonnet have billions of parameters. These parameters — the weights — are what store the model's "knowledge." Fine-tuning and RLHF (Reinforcement Learning from Human Feedback) align these base capabilities toward following instructions helpfully and safely. As a backend engineer, think of the model as a stateless function: tokens_in → probability_distribution → sample → token_out.

💡

Why This Matters for Engineers

Understanding that LLMs are probabilistic next-token predictors — not databases or search engines — changes how you architect applications. You can't rely on deterministic outputs, but you CAN control temperature, top-p, and use structured output schemas to constrain behavior. Every design decision in LLM engineering flows from this fundamental model.

Core Transformer Concepts

🎯

Self-Attention

Every token computes a weighted sum over all other tokens. "The cat sat on the mat" — "sat" attends strongly to "cat" as subject. O(n²) complexity per layer.

🏗️

Multi-Head Attention

Multiple attention patterns run in parallel — each head captures different relationships: syntax, coreference, semantic similarity. Claude 3.5 Sonnet uses 96 heads.

📍

Positional Encoding

Since attention is order-agnostic, position is injected via encodings. Modern models use RoPE (Rotary Position Encoding) to handle very long contexts.

⚡

Feed-Forward Layers

After attention, each token passes through a 2-layer MLP independently. These store factual knowledge — "Paris is the capital of France" lives in these weights.

🔄

Layer Normalization

Applied before attention and FFN in modern architectures (pre-norm). Stabilizes training by normalizing activations, preventing gradient explosion.

📐

KV Cache

During inference, Key/Value matrices from previous tokens are cached. This is why prefill (first token) is slower than decode (subsequent tokens).

The Autoregressive Generation Loop

1. Tokenize Input

Your text is split into subword tokens using a learned vocabulary (BPE or SentencePiece). "Hello world" → [9906, 1917]. Each token is an integer ID mapped to an embedding vector.

2. Forward Pass Through All Layers

The embedding sequence flows through N Transformer layers. Each layer applies self-attention + FFN, updating each token's representation with context from all other tokens.

3. LM Head → Logits

The final hidden state of the last token position is projected to vocabulary size (e.g., 100K). Raw scores called logits. Higher logit = higher probability for that token.

4. Softmax + Sampling

Logits are converted to probabilities via softmax (scaled by temperature). A token is sampled. Temperature 0 = argmax (greedy); Temperature 1 = raw distribution; Temperature > 1 = more random.

5. Append & Repeat

The sampled token is appended to the sequence and the whole process repeats — autoregressive decoding. This continues until an EOS token or max_tokens is reached.

⚠️

Common Pitfall: Treating LLMs as Databases

Engineers often query LLMs like they would a database: "What is the current stock price of AAPL?" The model has no live data access, no memory between calls, and its knowledge has a training cutoff. Always design for retrieval augmentation (Day 4) when you need factual accuracy.

Section 02

Tokenization: The Unit of LLM Currency

Tokens are the atomic unit of everything — input cost, output cost, context limits, and generation speed. Understanding tokenization is non-negotiable for production LLM work.

Tokenization converts raw text into integer IDs using a learned vocabulary. Modern models use Byte-Pair Encoding (BPE): common words become single tokens, rare words split into subwords, and bytes handle anything outside the vocabulary. The rough rule of thumb: 1 token ≈ 4 English characters ≈ 0.75 words. But this varies dramatically with language and content type.

Section 03

Your First API Call with the Anthropic SDK

The Anthropic Python SDK provides a clean, typed interface to Claude. Let's go from installation to a streaming multi-turn conversation in minutes.

terminal
# Install the SDK — supports Python 3.8+
pip install anthropic

# Set your API key (get it from console.anthropic.com)
export ANTHROPIC_API_KEY="sk-ant-..."
basic_call.py
import anthropic

# Initialize client — auto-reads ANTHROPIC_API_KEY from env
client = anthropic.Anthropic()

# Simplest possible call
message = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Explain attention mechanisms in 3 sentences."}
    ]
)

print(message.content[0].text)
print(f"Input tokens:  {message.usage.input_tokens}")
print(f"Output tokens: {message.usage.output_tokens}")

The API follows a messages format — a list of alternating user/assistant turns. The system parameter (separate from messages) sets persistent instructions. This separation matters: system prompts are cached more efficiently and apply throughout the conversation.

system_prompt.py
import anthropic

client = anthropic.Anthropic()

# System prompt sets the assistant's persona/constraints
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=2048,
    system="""You are a senior Python engineer specializing in ML systems.
You always:
- Provide production-ready code with error handling
- Include type hints and docstrings
- Mention performance implications
- Flag any security concerns""",
    messages=[
        {"role": "user", "content": "Write a rate limiter for API calls"},
        {"role": "assistant", "content": "Here's a token bucket rate limiter..."},
        {"role": "user", "content": "Add Redis backend support to make it distributed"}
    ]
)
print(response.content[0].text)
streaming.py
import anthropic

client = anthropic.Anthropic()

# Streaming: receive tokens as they're generated (better UX)
with client.messages.stream(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a haiku about neural networks"}],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

# Access final message with usage stats
final = stream.get_final_message()
print(f"\n\nStop reason: {final.stop_reason}")
print(f"Total tokens: {final.usage.input_tokens + final.usage.output_tokens}")

📌

stop_reason is Your Production Signal

Always check stop_reason in production code. If it's "max_tokens", the response was truncated — your answer is incomplete. If it's "tool_use", the model wants to call a function. Only "end_turn" means the model finished naturally.

Section 04

Temperature, Top-P, and Sampling Strategies

These parameters directly control the probability distribution from which tokens are sampled. Getting them right is the difference between a creative writing assistant and a reliable code generator.

After the model computes logits, temperature divides each logit before softmax: p(token) = softmax(logits / temperature). Low temperature (→ 0) concentrates probability on the top token, making outputs more deterministic. High temperature (→ 2) flattens the distribution, making rare tokens more likely to be sampled — more creative but less reliable.

sampling_params.py
import anthropic

client = anthropic.Anthropic()

# Configuration for code generation (high precision)
code_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=2048,
    temperature=0.1,  # Very low — we want deterministic code
    top_p=0.95,
    stop_sequences=["```\n\n"],  # Stop after code block
    messages=[{"role": "user", "content": "Write a binary search in Python"}]
)

# Configuration for creative writing (high creativity)
story_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    temperature=1.1,  # Higher — more varied and creative
    top_p=0.95,
    messages=[{"role": "user", "content": "Write an opening paragraph for a sci-fi novel"}]
)

# Configuration for extraction (deterministic, stop at JSON boundary)
extract_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=512,
    temperature=0.0,  # Greedy — same output every time
    system="Output only valid JSON. No markdown. No explanation.",
    messages=[{"role": "user", "content": "Extract name and email from: John Doe jdoe@example.com"}]
)
⚠️

Common Pitfall: Setting Temperature Too High for Structured Tasks

Many engineers set temperature=1.0 as a default for everything. For tasks requiring structured output (JSON extraction, code, classification), this leads to hallucinations and malformed responses. Always use temperature≤0.2 when you need the model to follow a format precisely. Use 0.0 for testing to get reproducible results.

Section 05

Context Windows & Practical Limits

The context window is arguably the most important practical constraint in LLM application design. Understanding it prevents expensive bugs and architectural mistakes.

The context window is the total number of tokens the model can "see" at once — both input and output combined. Claude's models support up to 200K tokens (about 150K words or 500 pages). But "supports" doesn't mean "works perfectly" at that limit — there are important nuances engineers need to understand for production systems.

📏

Context = Working Memory

Everything the model can act on must be in the context: the system prompt, conversation history, retrieved documents, tool outputs. There's no "long-term memory" without you explicitly putting information in the prompt.

🎯

The Lost-in-the-Middle Problem

Research shows LLMs perform better on information at the beginning and end of context than in the middle. For retrieval, always put the most relevant chunks first or last — not buried in the middle of a 100K context.

context_management.py
from anthropic import Anthropic

class ConversationManager:
    """Manages context window with a sliding window strategy."""

    def __init__(self, max_tokens: int = 150_000):
        self.client = Anthropic()
        self.messages: list[dict] = []
        self.max_tokens = max_tokens
        self.system = "You are a helpful assistant."

    def _estimate_tokens(self, messages: list) -> int:
        """Rough token estimation: ~4 chars per token."""
        total_chars = sum(len(m['content']) for m in messages)
        return total_chars // 4

    def _prune_context(self):
        """Remove oldest messages to stay within context limit."""
        while (self._estimate_tokens(self.messages) > self.max_tokens
               and len(self.messages) > 2):
            # Always keep system prompt; remove oldest user+assistant pair
            self.messages.pop(0)
            if self.messages:
                self.messages.pop(0)

    def chat(self, user_message: str) -> str:
        self.messages.append({"role": "user", "content": user_message})
        self._prune_context()

        response = self.client.messages.create(
            model="claude-sonnet-4-5",
            max_tokens=2048,
            system=self.system,
            messages=self.messages
        )

        assistant_reply = response.content[0].text
        self.messages.append({"role": "assistant", "content": assistant_reply})
        return assistant_reply

# Usage
manager = ConversationManager(max_tokens=100_000)
print(manager.chat("What is gradient descent?"))
print(manager.chat("How does it relate to backpropagation?"))
💡

Prompt Caching = Free Money

Anthropic's prompt caching feature lets you cache static prefixes (system prompts, documents) at a 90% cost reduction for cache hits. A 10K-token system prompt costs the full rate on the first call, then 10x less on every subsequent call within the 5-minute TTL. For high-volume production apps, this is a must-use feature.

prompt_caching.py
import anthropic

client = anthropic.Anthropic()

# Mark content for caching with cache_control
# The static system prompt will be cached after first call
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": """You are an expert Python code reviewer...[long system prompt]""",
            "cache_control": {"type": "ephemeral"}  # Cache this!
        }
    ],
    messages=[{"role": "user", "content": "Review this function..."}]
)

# Check if cache was used
usage = response.usage
print(f"Cache creation tokens: {usage.cache_creation_input_tokens}")
print(f"Cache read tokens:     {usage.cache_read_input_tokens}")
print(f"Regular input tokens:  {usage.input_tokens}")
Section 06

Structured Outputs & Response Parsing

Raw text responses are rarely what production systems need. Here's how to reliably extract structured data from Claude's responses.

📋

Prompt Engineering

Tell the model to output JSON. Works ~90% of the time with well-designed prompts. Fragile at scale. Use as a starting point.

🔧

Tool Use (Recommended)

Define a JSON schema as a "tool" and instruct the model to "call" it with the extracted data. Model guaranteed to output valid schema. Best approach.

🎯

Prefilling

Pre-fill the assistant's response with {" to force JSON output. Simple trick that works well for extraction tasks.

structured_output.py
import anthropic
import json
from pydantic import BaseModel

class EntityExtraction(BaseModel):
    name: str
    email: str
    company: str | None
    role: str | None

client = anthropic.Anthropic()

# Use tool_choice="any" to force the model to use your schema
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=512,
    tools=[{
        "name": "extract_entity",
        "description": "Extract contact info from text",
        "input_schema": EntityExtraction.model_json_schema()
    }],
    tool_choice={"type": "tool", "name": "extract_entity"},
    messages=[{
        "role": "user",
        "content": "Sarah Chen, CTO at NeuralDB - sarah@neuraldb.ai"
    }]
)

# Parse the guaranteed-valid JSON from tool call
tool_input = response.content[0].input
entity = EntityExtraction(**tool_input)
print(f"Name: {entity.name}, Email: {entity.email}")
# Name: Sarah Chen, Email: sarah@neuraldb.ai
💡

Why Tool Use for Structured Output?

When you define a tool with a JSON schema and set tool_choice to force its use, the model's output is guaranteed to be valid JSON matching your schema — the API enforces this at the infrastructure level. This is fundamentally more reliable than asking the model to "output JSON" in a prompt, where it might add markdown fences, commentary, or malformed syntax.

⚠️

Common Pitfall: Not Handling Partial Tool Calls

In streaming mode, tool call inputs arrive incrementally. If you try to parse response.content[0].input from a stream before it's complete, you'll get an empty or partial dict. Always wait for the stream to complete or use the with stream as... pattern and call get_final_message().

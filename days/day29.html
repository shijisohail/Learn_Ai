Section 01

The Copilot Architecture Problem

A copilot must handle ambiguous user intent, route to different specialized agents, maintain conversational state across turns, and know when to pause for human approval — all while being fast and reliable.

The naive copilot architecture sends every message to a single large LLM with a massive system prompt. This fails at scale: the model must simultaneously classify intent, route to tools, maintain state, handle errors, and generate responses. It becomes a 6000-token system prompt with unpredictable behavior. Production copilots decompose this into specialized components, each with a clear responsibility boundary.

The production pattern: (1) A fast intent classifier (Haiku class model, <100ms) categorizes the message. (2) The router selects the appropriate handler — RAG search, action executor, chitchat responder. (3) The selected handler processes with appropriate tools and context. (4) An approval gating layer intercepts high-risk actions for human review before execution. (5) A state manager persists conversation context across turns for multi-step workflows.

🎯
Intent Classification
Fast model classifies intent (question, action, search, escalation) before the main LLM is invoked. Saves cost by routing chitchat to lightweight responders and escalations to human agents.
🔀
Dynamic Routing
Route to specialized sub-agents based on classified intent. Each sub-agent has its own system prompt, tool set, and context window optimized for its task. Avoid monolithic agents.
✋
Human-in-the-Loop
High-risk actions (delete, send email, execute code) pause the workflow and request human approval via Slack/UI. The state machine persists the approval request and resumes when approved or rejected.

langgraph_copilot.py
from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict, Literal from anthropic import Anthropic client = Anthropic() class CopilotState(TypedDict): messages: list[dict] intent: str confidence: float pending_approval: bool action_plan: dict | None rag_results: list final_response: str def classify_intent(state: CopilotState) -> CopilotState: user_msg = state["messages"][-1]["content"] resp = client.messages.create( model="claude-3-5-haiku-20241022", max_tokens=50, system='Classify: question|action|search|smalltalk. JSON: {"intent":"...","confidence":0-1}', messages=[{"role":"user","content":user_msg}] ) import json clf = json.loads(resp.content[0].text) return {**state, "intent": clf["intent"], "confidence": clf["confidence"]} def route_by_intent(state: CopilotState) -> Literal["rag_search","action_planner","generate_response"]: if state["intent"] in ["question","search"]: return "rag_search" if state["intent"] == "action": return "action_planner" return "generate_response" def check_approval_needed(state) -> Literal["human_approval","execute_action"]: if state["action_plan"] and state["action_plan"].get("risk") == "high": return "human_approval" return "execute_action" # Build graph graph = StateGraph(CopilotState) graph.add_node("classify_intent", classify_intent) graph.add_node("rag_search", rag_search) graph.add_node("action_planner", action_planner) graph.add_node("human_approval", human_approval) graph.add_node("execute_action", execute_action) graph.add_node("generate_response", generate_response) graph.add_edge("__start__", "classify_intent") graph.add_conditional_edges("classify_intent", route_by_intent) graph.add_conditional_edges("action_planner", check_approval_needed) app = graph.compile(checkpointer=MemorySaver())
💡
Why this matters: LangGraph's checkpointing enables true multi-turn state persistence. If a human approval workflow is paused waiting for sign-off, the graph state is serialized to storage. When the approver responds (hours or days later), the workflow resumes exactly where it left off. This is the foundation of enterprise-grade AI copilots that handle multi-day, multi-stakeholder workflows.

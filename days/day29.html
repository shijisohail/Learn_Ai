02 — Intent Classification

Intent Classification & Command Routing

The intent classifier is the gateway. It determines what the user wants, how confident we are, and whether we have enough information to act — or need to ask for clarification.

  • Intent taxonomy: Informational (ask), transactional (create/update/delete), navigational (go to), operational (run a process)
  • Function calling for routing: LLM selects which function (route) to invoke — same mechanism as tool use, but the tools are business workflow handlers
  • Confidence thresholds: Confidence below 0.7 → ask for clarification rather than acting on ambiguous intent
  • Entity extraction: Pull customer_id, date, product_code from natural language — structured output into Pydantic models
PYTHON — Intent Classifier (Instructor + Pydantic)
from langchain_anthropic import ChatAnthropic from pydantic import BaseModel, Field from typing import Optional import instructor class IntentResult(BaseModel): intent: str # "schedule_visit" | "check_inventory" | "view_orders" | "ask_question" confidence: float # 0.0 - 1.0 entities: dict # extracted parameters requires_clarification: bool instructor_client = instructor.from_anthropic(ChatAnthropic()) async def classify_intent(user_message: str, context: dict) -> IntentResult: system = f"""You are an intent classifier for a retail execution platform. User context: role={context['role']}, current_view={context['current_page']} Available intents: - schedule_visit: User wants to schedule a customer visit - check_inventory: User wants to check product inventory - view_orders: User wants to see customer orders - update_visit_status: User wants to mark a visit complete/incomplete - ask_question: Informational question, no action needed Extract the intent and relevant entities from the user message.""" result = await instructor_client.messages.acreate( model="claude-3-5-sonnet-20241022", max_tokens=256, system=system, messages=[{"role": "user", "content": user_message}], response_model=IntentResult ) return result
03 — Workflow State Machine

Conversational Workflow State Machine

Multi-step copilot workflows need explicit state — what the user is trying to do, what information has been collected, and what actions are pending approval. LangGraph is the engine.

💡
LangGraph interrupt_before: Set interrupt_before=["execute_action"] when compiling the graph — the graph pauses at that node and serializes state to the checkpointer. The next request from the user resumes from exactly that point.
PYTHON — LangGraph Copilot State Machine
from langgraph.graph import StateGraph, END from langgraph.checkpoint.postgres import PostgresSaver from typing import TypedDict, Optional, Annotated from langchain_core.messages import BaseMessage, AIMessage import operator class CopilotState(TypedDict): messages: Annotated[list[BaseMessage], operator.add] current_workflow: Optional[str] # "schedule_visit" | "check_order" | None accumulated_entities: dict # customer_id, date, etc. auth: dict # user_id, tenant_id, scopes pending_action: Optional[dict] # action awaiting approval needs_clarification: Optional[str] async def intent_node(state: CopilotState) -> dict: latest_msg = state["messages"][-1].content intent = await classify_intent(latest_msg, state["auth"]) if intent.confidence < 0.7: return { "needs_clarification": f"I'm not sure what you'd like to do. Did you mean to {intent.intent.replace('_', ' ')}?" } return { "current_workflow": intent.intent, "accumulated_entities": intent.entities, "needs_clarification": None } async def propose_action_node(state: CopilotState) -> dict: action = build_action_from_state(state) summary = generate_human_readable_summary(action) return { "pending_action": action, "messages": [AIMessage(content=f"I'll {summary}. Confirm to proceed, or say 'cancel'.")] } # Build the graph graph = StateGraph(CopilotState) graph.add_node("classify_intent", intent_node) graph.add_node("collect_entities", collect_entities_node) graph.add_node("propose_action", propose_action_node) graph.add_node("execute_action", execute_action_node) graph.set_entry_point("classify_intent") graph.add_edge("propose_action", END) # pause for human approval app = graph.compile( checkpointer=PostgresSaver.from_conn_string(DATABASE_URL), interrupt_before=["execute_action"] # always approve before execute )
04 — Context Model

Context Model & Prompt Assembly

The context model is what the copilot knows about the current user, tenant, and session. Assembling the right context — and only the right context — is critical for both quality and token efficiency.

  • Business context to include: User role & permissions, tenant config, current task/page, selected customer/entity, relevant targets (monthly goals)
  • Token budget management: Prioritize most relevant context for the current workflow stage — don't dump everything into every prompt
  • Dynamic context: Schedule-view prompts include today's visits. Inventory-view prompts include product catalog. Context is workflow-specific
  • Context caching: Cache static system context (tenant config, role permissions) using Anthropic's prompt caching — 90% cost reduction on repeated calls
PYTHON — Context Model + Prompt Assembly
class ContextModel(BaseModel): """All business context the copilot needs""" # Identity user_name: str user_role: str user_permissions: list[str] tenant_name: str tenant_plan: str # Current state (workflow-specific) current_page: str selected_customer: Optional[dict] = None # id, name, tier todays_schedule: Optional[list[dict]] = None # Targets monthly_target: Optional[int] = None visits_completed: Optional[int] = None def build_system_prompt(ctx: ContextModel, workflow: str) -> str: """Assemble system prompt with relevant context only""" base = f"""You are an AI copilot for {ctx.tenant_name}'s retail execution platform. User: {ctx.user_name} ({ctx.user_role}) Permissions: {', '.join(ctx.user_permissions)} Current page: {ctx.current_page}""" # Add workflow-specific context if workflow == "schedule_visit" and ctx.todays_schedule: base += f"\nToday's schedule: {len(ctx.todays_schedule)} visits planned" if ctx.selected_customer: base += f"\nSelected customer: {ctx.selected_customer['name']} ({ctx.selected_customer['tier']} tier)" return base + "\n\nYou can only suggest actions within the user's permissions. Always confirm before irreversible actions."
05 — Production Patterns

Production Copilot Patterns

Streaming, graceful degradation, multi-turn context management, and continuous evaluation — what separates a demo copilot from a production one.

StreamingSSE (Server-Sent Events) for real-time typing — LangGraph supports streaming node outputs. Users expect to see tokens, not wait 10 seconds for a full response.
Multi-turnKeep last 10 messages in state. Summarize older context with the LLM rather than truncating — preserve intent across long conversations.
DegradationLLM timeout or rate limit → return structured fallback: "I'm having trouble processing your request. Here are the options you can take manually: [...]"
EvaluationTrack: task completion rate (did user get what they asked for?), clarification rate (how often did copilot need to ask?), approval rate (how often user confirmed?), error rate.
Auth contextInject JWT claims directly into LangGraph state at the API gateway layer — copilot never trusts client-provided permissions. RBAC is enforced at tool-call time, not prompt time.

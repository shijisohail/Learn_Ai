RAG Pipeline — Step by Step

Indexing (once) then retrieval (per query). Click through to see each stage.

RAG System Architecture
Offline indexing + Online retrieval
📥 INDEXING PHASE (run once)
1
Load & Chunk Documents
Split your PDFs, docs, web pages into overlapping chunks (~512 tokens each). Overlap ensures context isn't lost at boundaries.
doc.txt → ["Chunk 1: FastAPI is a modern...", "Chunk 2: modern Python web...", "Chunk 3: web framework for..."]
2
Embed Each Chunk
Send each chunk to an embedding model (voyage-3, text-embedding-3-small, etc.). Get back a dense vector per chunk.
"FastAPI is modern..." → embed() → [0.23, -0.41, 0.87, ..., 0.12] (1024 dims)
3
Store in Vector DB
Insert vector + original text + metadata into pgvector/Pinecone. Create an HNSW or IVFFlat index for fast ANN search.
INSERT INTO embeddings (content, embedding, source) VALUES ($1, $2, $3)
🔍 RETRIEVAL PHASE (per query)
4
Embed the User Query
Same embedding model that indexed the chunks must embed the query. Consistent embedding space = meaningful similarity scores.
"How do I add auth to FastAPI?" → [0.31, -0.18, 0.92, ..., 0.04]
5
Find Top-K Similar Chunks
Vector DB computes cosine similarity between query vector and all stored vectors. Returns the K most similar chunks (typically K=3-5).
SELECT content, 1-(embedding <=> $1) AS score FROM embeddings ORDER BY score DESC LIMIT 5
6
Inject Chunks + Generate Answer
Retrieved chunks go into the system/user prompt as context. The LLM generates an answer grounded in your documents — not hallucinated knowledge.
system: "Answer using ONLY this context:\n{chunk1}\n{chunk2}\n{chunk3}"
user: "How do I add auth to FastAPI?"

Chunking Strategies

How you split documents dramatically affects retrieval quality. Three main approaches, plus overlap.

Fixed size chunking: Split every N tokens/characters. Fast, simple, predictable. Problem: often splits mid-sentence or mid-concept, breaking context. Best for: homogeneous text where structure doesn't matter.
Sentence-based chunking: Split on sentence boundaries (., !, ?). Group N sentences per chunk. Respects natural language structure. Better retrieval coherence. Works well for prose, articles, documentation.
Semantic chunking: Use embedding cosine similarity to find topic boundaries. Each chunk is a coherent semantic unit. Higher quality, more expensive to build. Best for: mixed-topic documents, books, research papers.
Why overlap matters: Key information often appears at chunk boundaries. Without overlap, a sentence split across two chunks may never be retrieved together.
Solid = primary chunk. Dashed = overlap region (typically 10-20% of chunk size). Overlap = same text included in adjacent chunks so no context is lost at boundaries.

RAG vs Fine-tuning

When to use each. Spoiler: you almost always want RAG first.

Criterion 🔍 RAG 🎛️ Fine-tuning
Knowledge updates ✓ Real-time — add docs, immediately searchable ✗ Static — requires retraining to add knowledge
Training cost ✓ None — just embedding + storage ✗ $100s–$10,000s depending on model size
Source attribution ✓ Traceable — you know which chunk answered the question ✗ Opaque — knowledge baked into weights, no citation possible
Hallucination risk ✓ Lower — grounded in retrieved text ~ Similar — model still generates, can still hallucinate
Style / tone change ✗ Hard — base model tone persists ✓ Easy — fine-tuning changes how the model writes
Task-specific format ~ Possible via prompting, less reliable ✓ Reliable — model learns exact output format from examples
Private knowledge ✓ Best fit — your docs never leave your infrastructure ~ Works but expensive and inflexible
Time to deploy ✓ Hours — index docs, write retrieval logic, ship ✗ Weeks — data prep, training, eval, iteration
Inference cost ~ Slightly higher — longer prompts from context injection ✓ Same as base model inference
📐 Decision Rule of Thumb
Start with RAG. If the problem is "the model needs to know X fact" → RAG. If the problem is "the model needs to respond in a specific way" → Fine-tuning. For most production systems: RAG for knowledge, prompting for behavior, fine-tuning only when you have hundreds of high-quality examples and a clear format target.

A production-ready RAG implementation using PostgreSQL + pgvector + Claude. Fits your FastAPI background.

create_embeddings.py — using Voyage AI (Anthropic's recommended) python
import voyageai from anthropic import Anthropic voyage = voyageai.Client() # VOYAGE_API_KEY env var def embed_text(text: str) -> list[float]: """Embed a single text using voyage-3.""" result = voyage.embed([text], model="voyage-3") return result.embeddings[0] # 1024-dimensional vector def embed_batch(texts: list[str]) -> list[list[float]]: """Embed a batch — much more efficient than one-by-one.""" result = voyage.embed(texts, model="voyage-3") return result.embeddings def chunk_text(text: str, chunk_size: int = 512, overlap: int = 64) -> list[str]: """Split text into overlapping chunks by word count.""" words = text.split() chunks = [] start = 0 while start < len(words): end = min(start + chunk_size, len(words)) chunks.append(" ".join(words[start:end])) start += chunk_size - overlap # overlap by N words return chunks
pgvector_store.py — store and semantic search python
import asyncpg from pgvector.asyncpg import register_vector # Schema: CREATE EXTENSION vector; # CREATE TABLE docs ( # id SERIAL PRIMARY KEY, # content TEXT, # embedding vector(1024), # source TEXT, # created_at TIMESTAMPTZ DEFAULT NOW() # ); # CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops); async def store_document(conn, content: str, source: str): """Chunk, embed, and store a document.""" chunks = chunk_text(content) embeddings = embed_batch(chunks) await conn.executemany( "INSERT INTO docs (content, embedding, source) VALUES ($1, $2, $3)", [(chunk, str(emb), source) for chunk, emb in zip(chunks, embeddings)] ) async def semantic_search(conn, query: str, k: int = 5) -> list[dict]: """Find the K most semantically similar chunks.""" query_vec = embed_text(query) rows = await conn.fetch(""" SELECT content, source, 1 - (embedding <=> $1::vector) AS similarity FROM docs ORDER BY embedding <=> $1::vector LIMIT $2 """, str(query_vec), k) return [{"content": r["content"], "source": r["source"], "score": round(r["similarity"], 4)} for r in rows]
rag_function.py — full RAG with Claude python
client = Anthropic() async def rag_answer( conn, question: str, k: int = 5, min_score: float = 0.7, ) -> dict: """Answer a question using retrieved context.""" # 1. Retrieve relevant chunks chunks = await semantic_search(conn, question, k=k) chunks = [c for c in chunks if c["score"] >= min_score] if not chunks: return {"answer": "No relevant documents found.", "sources": []} # 2. Build context string context = "\n\n---\n\n".join( f"[Source: {c['source']} | Score: {c['score']}]\n{c['content']}" for c in chunks ) # 3. Ask Claude with retrieved context response = client.messages.create( model="claude-haiku-4-5-20251001", max_tokens=1024, system=f"""Answer questions using ONLY the provided context. If the answer is not in the context, say so explicitly. Always cite which source you used. CONTEXT: {context}""", messages=[{"role": "user", "content": question}] ) return { "answer": response.content[0].text, "sources": [c["source"] for c in chunks], "scores": [c["score"] for c in chunks], "tokens": response.usage.input_tokens + response.usage.output_tokens, }

    Hybrid Search: Best of Both Worlds

    Two parallel retrieval pipelines merge at RRF
    Query
    ↓
    BM25
    Full-text keyword index
    ↓
    Keyword-ranked results
    exact term matching
    ➙
    RRF
    ➘
    Query
    ↓
    Embedding model
    text-embedding-3-small
    ↓
    Vector DB results
    cosine similarity
    ● RRF Combined Results → returned to LLM
    Reciprocal Rank Fusion formula:
    RRF_score(doc) = ∑ 1 / (k + ranki)
    k = 60 (constant)  ·  rank = position in each list  ·  scores from both lists are summed
    Example: doc ranked #2 in keyword, #1 in semantic → 1/(60+2) + 1/(60+1) = 0.0161 + 0.0164 = 0.0325
    🔑 When Keyword (BM25) Wins
    • Exact product IDs (SKU-4821)
    • Names and proper nouns
    • Technical codes and jargon
    • Version numbers (v3.2.1)
    • Quoted phrases
    🧠 When Semantic Wins
    • Concept queries ("how to connect")
    • Paraphrases & synonyms
    • Natural language questions
    • Intent-based retrieval
    • Cross-lingual queries
    Hybrid search with pgvector + tsvector (PostgreSQL)
    -- Table with both vector and full-text search columns CREATE TABLE documents ( id BIGSERIAL PRIMARY KEY, content TEXT NOT NULL, embedding vector(1536), -- pgvector column tsv tsvector GENERATED ALWAYS AS -- auto full-text index (to_tsvector('english', content)) STORED ); CREATE INDEX ON documents USING ivfflat (embedding vector_cosine_ops); CREATE INDEX ON documents USING gin(tsv); -- Hybrid search: semantic + keyword, fused with RRF (k=60) WITH semantic AS ( SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1) AS rank FROM documents ORDER BY embedding <=> $1 LIMIT 20 ), keyword AS ( SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, query) DESC) AS rank FROM documents, to_tsquery('english', $2) query WHERE tsv @@ query ORDER BY ts_rank(tsv, query) DESC LIMIT 20 ) SELECT d.id, d.content, COALESCE(1.0/(60+s.rank),0) + COALESCE(1.0/(60+k.rank),0) AS rrf_score FROM documents d LEFT JOIN semantic s ON d.id = s.id LEFT JOIN keyword k ON d.id = k.id WHERE s.id IS NOT NULL OR k.id IS NOT NULL ORDER BY rrf_score DESC LIMIT 5;
    Rule of thumb: Hybrid search consistently outperforms either approach alone by 10–15% in recall. Worth the extra complexity for production. Start with pgvector + tsvector if you are already on PostgreSQL — no extra infrastructure needed.

    Conversation Memory Management

    Long-running agents face a hard constraint: the context window fills up. Here are the practical patterns for keeping agents running indefinitely without hitting token limits.

    The Memory Problem

    Context Window Fills Over Time
    SYSTEM

    Three Patterns

    Pattern 1
    Sliding Window
    Keep only the last N turns. Simple, slightly lossy. Best for chat and short-lived agents.
    Simple Lossy O(1) overhead
    Pattern 2
    Summarization
    When history exceeds a threshold, call a cheap LLM to summarize old turns. Replace with the summary, keep recent turns fresh.
    Lossless-ish Extra LLM call Hours/days
    Pattern 3
    External Memory
    Extract key facts from each turn, store in a vector DB (mem0, pgvector). Retrieve on demand — full RAG loop for agent memory. Unlimited horizon.
    Weeks/persistent Infra needed Best fidelity

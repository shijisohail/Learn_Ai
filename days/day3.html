Section 01

What Are Embeddings?

Embeddings are dense vector representations of text (or images, code, audio) in a high-dimensional space where semantic similarity corresponds to geometric proximity. They're the backbone of modern search, recommendation, and RAG systems.

When you encode the sentence "The king ruled the kingdom" into a 1536-dimensional vector, that vector captures semantic meaning — not just the words, but their relationships and context. The sentences "The monarch governed the realm" and "The president led the country" will produce very similar vectors, while "I ate a sandwich" will produce a very different one. This is the geometric intuition: meaning as location in space.

Embedding models are neural networks trained specifically to produce these representations. Unlike LLMs optimized for generation, embedding models are trained with contrastive objectives — they learn to pull semantically similar texts close together and push dissimilar texts apart. The famous "king - man + woman = queen" arithmetic works because of how these models organize semantic relationships in the embedding space.

📐

High-Dimensional Geometry

A 1536-dimensional vector can encode subtle semantic nuances that would be impossible in low dimensions. The "semantic neighborhood" of a vector contains texts with similar meaning, regardless of exact word choice.

⚡

Inference is Cheap

Embedding a document is a single forward pass — fast and cheap. OpenAI's text-embedding-3-small costs $0.02 per million tokens. Voyage AI (Anthropic's preferred partner) has excellent quality/cost ratio.

💡

Why Embeddings Beat Keyword Search

Keyword search fails silently: "What are the side effects of ibuprofen?" won't find a document containing "adverse reactions to NSAIDs" because no keywords overlap. Embedding-based semantic search finds it because both encode similar medical meaning. For knowledge bases, customer support, and documentation search, this improvement in recall is transformational.

embeddings_basics.py
import voyageai  # pip install voyageai
import numpy as np

client = voyageai.Client()  # reads VOYAGE_API_KEY from env

# Embed a batch of texts (always batch, never one-by-one)
texts = [
    "The king ruled the kingdom",
    "The monarch governed the realm",  # similar → high cosine similarity
    "I ate a sandwich for lunch",        # different → low cosine similarity
]

result = client.embed(texts, model="voyage-3", input_type="document")
embeddings = result.embeddings  # List of 1024-dim float vectors

print(f"Vector dimensions: {len(embeddings[0])}")  # 1024
print(f"Total tokens used: {result.total_tokens}")

# Cosine similarity between two vectors
def cosine_sim(a: list[float], b: list[float]) -> float:
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.norm(a) * np.norm(b))

print(cosine_sim(embeddings[0], embeddings[1]))  # ~0.92 (very similar)
print(cosine_sim(embeddings[0], embeddings[2]))  # ~0.31 (unrelated)
Section 02

Cosine Similarity Visualized

Cosine similarity measures the angle between two vectors in the embedding space — not their magnitude, just their direction. Values range from -1 (opposite) to 1 (identical).

The cosine similarity formula: cos(θ) = (A·B) / (|A| × |B|). Why cosine instead of Euclidean distance? Because embedding magnitudes can vary based on text length, but we care about direction (meaning), not magnitude. Two paragraphs about the same topic should have high similarity regardless of one being twice as long.

Section 03

Vector Databases

Vector databases store embeddings and enable Approximate Nearest Neighbor (ANN) search at scale — finding the K most similar vectors to a query in milliseconds, even across millions of documents.

A naive brute-force similarity search over 1 million 1536-dimensional vectors requires 1M cosine similarity computations — feasible for small datasets but O(n) and too slow at scale. Vector databases use index structures like HNSW (Hierarchical Navigable Small World graphs) or IVF (Inverted File Index) to perform approximate search in O(log n) time with >95% recall on typical datasets.

chromadb_quickstart.py
import chromadb
import voyageai

# Initialize ChromaDB (runs in-memory or on disk)
chroma = chromadb.Client()  # or PersistentClient("./db") for disk
voyage = voyageai.Client()

# Create a collection with cosine distance metric
collection = chroma.create_collection(
    name="docs",
    metadata={"hnsw:space": "cosine"}
)

# Your knowledge base documents
documents = [
    "FastAPI is a modern Python web framework for building APIs",
    "PostgreSQL is a powerful open source relational database",
    "Redis is an in-memory data structure store used as cache",
    "Docker containers package apps with their dependencies",
]

# Embed and index (batch for efficiency)
result = voyage.embed(documents, model="voyage-3", input_type="document")
collection.add(
    ids=[f"doc_{i}" for i in range(len(documents))],
    embeddings=result.embeddings,
    documents=documents,
    metadatas=[{"source": "tech_docs", "index": i} for i in range(len(documents))]
)

# Query: find most relevant docs for a user question
question = "How do I cache data in my Python app?"
q_embed = voyage.embed([question], model="voyage-3", input_type="query")

results = collection.query(
    query_embeddings=q_embed.embeddings,
    n_results=2,
    include=["documents", "distances", "metadatas"]
)

for doc, dist in zip(results["documents"][0], results["distances"][0]):
    print(f"Score: {1-dist:.3f} | {doc}")
# Score: 0.891 | Redis is an in-memory data structure store used as cache
📌

Always Use Separate Query and Document Models

Many embedding models use different encoders for queries vs. documents (asymmetric embedding). Voyage's models accept an input_type parameter: use "query" for search queries and "document" for the knowledge base. Using the wrong type silently degrades retrieval quality.

Section 04

Chunking Strategy: The Hidden Performance Lever

How you split documents for indexing is the single most impactful engineering decision in a RAG system. Poor chunking causes irrelevant retrieval even with perfect embeddings.

The chunking problem is a tradeoff: too-small chunks miss context (a sentence without its surrounding paragraphs is ambiguous), too-large chunks dilute relevance (a 5-page chapter about distributed systems might contain one relevant sentence about your specific question). The right chunk size depends on your document type, embedding model, and query patterns.

StrategyChunk SizeBest ForPitfall

chunking.py
from langchain.text_splitter import RecursiveCharacterTextSplitter
import tiktoken

def smart_chunk(
    text: str,
    chunk_size: int = 512,
    overlap: int = 64,
    model: str = "cl100k_base"
) -> list[str]:
    """
    Chunk text by token count with overlap, using recursive splitting.
    Overlap ensures context isn't lost at chunk boundaries.
    """
    enc = tiktoken.get_encoding(model)

    splitter = RecursiveCharacterTextSplitter(
        # Try these separators in order — keep semantic units whole
        separators=["\n\n", "\n", ". ", ", ", " ", ""],
        chunk_size=chunk_size,
        chunk_overlap=overlap,
        length_function=lambda t: len(enc.encode(t)),  # token-based sizing
        is_separator_regex=False,
    )

    chunks = splitter.split_text(text)

    # Prepend section header to each chunk for context
    return [f"[Chunk {i+1}/{len(chunks)}] {chunk}"
            for i, chunk in enumerate(chunks)]

# Parent-child chunking: index small, retrieve large
def parent_child_chunks(text: str) -> dict:
    """Index child chunks (precise retrieval), return parent (full context)."""
    parent_chunks = smart_chunk(text, chunk_size=1024, overlap=128)
    result = {}
    for i, parent in enumerate(parent_chunks):
        children = smart_chunk(parent, chunk_size=256, overlap=32)
        for j, child in enumerate(children):
            result[f"p{i}_c{j}"] = {"index_text": child, "return_text": parent}
    return result
⚠️

Pitfall: Chunk Size Mismatch with Embedding Model

Every embedding model has a token limit (typically 512-8192 tokens). If your chunks exceed the model's context window, the model silently truncates them — without warning. The embedding for a 600-token chunk in a 512-token-max model only encodes the first 512 tokens. Always verify your chunk sizes against your embedding model's max input length.

Section 05

End-to-End RAG Pipeline

Retrieval-Augmented Generation combines semantic search with LLM generation. Retrieved context grounds the model's response in your actual data, eliminating hallucinations about domain-specific knowledge.

rag_pipeline.py
import anthropic
import voyageai
import chromadb
from dataclasses import dataclass

@dataclass
class RAGResult:
    answer: str
    sources: list[str]
    retrieval_scores: list[float]
    tokens_used: int

class RAGPipeline:
    def __init__(self, collection_name: str):
        self.claude  = anthropic.Anthropic()
        self.voyage  = voyageai.Client()
        self.chroma  = chromadb.PersistentClient("./vectordb")
        self.coll    = self.chroma.get_or_create_collection(
            collection_name,
            metadata={"hnsw:space": "cosine"}
        )

    def index_documents(self, docs: list[str], ids: list[str]):
        """Index documents in batches of 100 to avoid rate limits."""
        batch_size = 100
        for i in range(0, len(docs), batch_size):
            batch = docs[i:i+batch_size]
            result = self.voyage.embed(batch, model="voyage-3", input_type="document")
            self.coll.add(ids=ids[i:i+batch_size], embeddings=result.embeddings, documents=batch)

    def query(self, question: str, top_k: int = 4, min_score: float = 0.6) -> RAGResult:
        # Step 1: Embed the question
        q_embed = self.voyage.embed([question], model="voyage-3", input_type="query")

        # Step 2: Retrieve top-k similar chunks
        hits = self.coll.query(query_embeddings=q_embed.embeddings, n_results=top_k,
                               include=["documents", "distances"])
        docs   = hits["documents"][0]
        scores = [1 - d for d in hits["distances"][0]]

        # Step 3: Filter by relevance threshold
        relevant = [(d, s) for d, s in zip(docs, scores) if s >= min_score]
        if not relevant:
            return RAGResult("I couldn't find relevant information.", [], [], 0)

        # Step 4: Build context and generate answer
        context = "\n\n".join(f"[Source {i+1}] {d}" for i, (d, _) in enumerate(relevant))
        response = self.claude.messages.create(
            model="claude-sonnet-4-5", max_tokens=1024,
            system="""Answer based ONLY on the provided context.
If the answer isn't in the context, say "I don't have that information."
Always cite which source(s) you used.""",
            messages=[{"role": "user",
                       "content": f"Context:\n{context}\n\nQuestion: {question}"}]
        )
        return RAGResult(
            answer=response.content[0].text,
            sources=[d for d, _ in relevant],
            retrieval_scores=[s for _, s in relevant],
            tokens_used=response.usage.input_tokens + response.usage.output_tokens
        )
💡

The min_score Threshold Is Critical

Without a minimum similarity threshold, your RAG system will always inject context into the LLM — even when it's completely irrelevant. A question about Python logging will retrieve your least-irrelevant document about Java if there's no threshold. Set min_score around 0.55-0.70 and have a fallback when no relevant chunks are found. This prevents the model from hallucinating answers based on tangential context.

⚠️

Pitfall: "Lost in Context" with Too Much Retrieved Content

More retrieved chunks ≠ better answers. Injecting 20 chunks into the context window often leads to lower quality answers than injecting 3-4 highly relevant ones, due to the "lost in the middle" problem. Keep top_k low (3-5), use aggressive relevance filtering, and consider re-ranking retrieved chunks with a cross-encoder model before injection.

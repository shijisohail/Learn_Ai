Section 01

Why Alignment Techniques Matter

Pre-training gives models knowledge. Instruction fine-tuning teaches them to follow directions. But RLHF and its successors teach models to be genuinely helpful, harmless, and honest — aligning their behavior with human values at scale.

A pre-trained LLM is an incredibly capable text completion engine, but it's not inherently helpful. If you ask "How do I make pasta?" it might complete the sentence with "How do I make pasta from scratch?" rather than giving you instructions. Instruction fine-tuning (SFT) teaches the format of Q&A. RLHF teaches the quality — what makes a response genuinely useful, well-calibrated, and appropriately safe.

The alignment tax — the degree to which safety training reduces capability — has been dramatically reduced by modern techniques. DPO achieves comparable alignment quality to PPO-based RLHF at a fraction of the compute cost. Constitutional AI further removes the bottleneck of human preference labeling. Understanding these techniques is essential for any engineer building production AI systems that need to stay aligned over time as data distributions shift.

🎯
Reward Modeling
Train a reward model from human pairwise preferences. For each (prompt, chosen, rejected) triple, the RM learns to score responses by human-estimated quality. This becomes the proxy optimization signal.
🔄
PPO Fine-tuning
Use the reward model as an online scorer in a PPO loop. The LLM generates responses, the RM scores them, PPO updates the model to maximize reward while a KL penalty prevents it from drifting too far from SFT.
⚡
DPO (Direct Preference)
DPO reformulates RLHF as a supervised learning problem on preference pairs — no separate reward model, no RL loop, no reference model sampling. 4× faster convergence with comparable alignment quality.

Stage 1
Supervised Fine-Tuning (SFT)
▶

SFT trains the base model on high-quality demonstration data: examples of the format (instruction, ideal response). This teaches the model to follow instructions and output in the desired format. Typically 10k–100k examples, fine-tuned for 1–3 epochs.

Quality matters far more than quantity in SFT data. 1,000 excellent demonstrations outperform 100,000 mediocre ones. Anthropic, OpenAI, and other labs employ expert data contractors for this stage. The SFT model becomes the reference model (π_ref) for the RLHF phase.

Loss = -Σ log π(y_i | x, y_1...y_{i-1}) # Standard language modeling loss on demonstrations # Dataset: {prompt, ideal_response} pairs # Duration: 1-3 epochs on 10k-100k examples
🏆
Stage 2
Reward Model Training
▶

Human labelers compare pairs of model responses and select which is better. These pairwise preferences are used to train a Bradley-Terry reward model. The RM takes (prompt, response) as input and outputs a scalar reward score.

The critical challenge: RM overoptimization. Once you optimize a model against a reward model long enough, it finds adversarial inputs that score highly on the RM but aren't actually better. This is Goodhart's Law applied to alignment.

Loss = -log σ(r(x, y_w) - r(x, y_l)) # Bradley-Terry loss on preference pairs # y_w = chosen (preferred) response # y_l = rejected (dispreferred) response # σ = sigmoid function # Dataset: {prompt, chosen, rejected} triples
🚀
Stage 3
PPO Fine-tuning from Feedback
▶

Proximal Policy Optimization (PPO) optimizes the fine-tuned LLM to maximize reward model scores while a KL divergence penalty prevents the model from deviating too far from the SFT reference policy. This balances alignment with capability preservation.

PPO is notoriously difficult to stabilize: four models must be maintained simultaneously (policy, reference, reward, value), requiring significant GPU memory. This complexity motivated DPO's development.

Objective = E[r(x,y)] - β * KL(π_θ || π_ref) # Maximize reward while staying close to SFT # β = KL penalty coefficient (typically 0.1-0.2) # Online: generate responses → score → update # 4 models in memory: policy, ref, reward, value
Section 03

DPO vs PPO — Animated Comparison

Direct Preference Optimization collapses the RLHF pipeline into a single supervised learning step. See how the mathematical relationship between the two approaches makes DPO equivalent under certain conditions.

DPO vs PPO Algorithm Comparison
Compare the full algorithmic pipeline, mathematical formulation, resource requirements, and practical tradeoffs between the two dominant alignment approaches.
PPO (RLHF)
Stage 1: SFT → π_ref Stage 2: Train RM r_φ Stage 3: PPO loop: y ~ π_θ(·|x) reward = r_φ(x,y) - β·KL(π_θ||π_ref) update θ via PPO gradient Memory: 4 models simultaneous Complexity: High — RL loop is unstable
Pros:
• Online learning (explores new outputs)
• Flexible reward shaping
• Used by GPT-4, Claude 1-2, Gemini
Cons:
• Complex, expensive, unstable
• RM overoptimization risk
• Requires 4× GPU memory
• Hyperparameter sensitive
DPO (Direct Pref. Opt.)
Stage 1: SFT → π_ref Stage 2: Directly optimize: Loss = -log σ(β·log(π_θ(y_w|x)/π_ref(y_w|x)) - β·log(π_θ(y_l|x)/π_ref(y_l|x))) # No separate RM training! # No PPO loop! # Just supervised learning on preference pairs Memory: 2 models (policy + ref)
Pros:
• Simple — just supervised learning
• 4× cheaper, more stable
• No RM training required
• Increasingly popular (Llama 3, Mixtral)
Cons:
• Offline — no exploration
• Data quality critical
• Can degrade on distribution shift
• Theoretical equivalence may not hold
💡
Why this matters: DPO makes RLHF-quality alignment accessible without a full RL infrastructure team. With TRL (Transformers Reinforcement Learning library), you can run DPO on a consumer GPU. This means you can fine-tune open-source models on your own preference data to align them with your domain — without Anthropic's or OpenAI's training budget.
Section 04

LLM-as-Judge: Automated Evaluation

LLM-as-Judge uses a capable model to evaluate other model outputs. Combined with human calibration, it enables scalable automated quality assessment at 100× the throughput of human evaluation.

llm_judge.py
from anthropic import Anthropic import instructor from pydantic import BaseModel, Field client = instructor.from_anthropic(Anthropic()) class Judgement(BaseModel): chosen: str = Field(description="Which response is better: 'A' or 'B'") score_a: int = Field(ge=1, le=10) score_b: int = Field(ge=1, le=10) reasoning: str JUDGE_SYSTEM = """You are an expert AI response evaluator. Evaluate responses on: helpfulness, accuracy, clarity, safety. Be consistent, calibrated, and cite specific evidence.""" def judge_pair(prompt: str, response_a: str, response_b: str) -> Judgement: # Positional bias mitigation: randomize order import random flip = random.random() > 0.5 a, b = (response_b, response_a) if flip else (response_a, response_b) result = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=500, response_model=Judgement, system=JUDGE_SYSTEM, messages=[{"role":"user","content": ff"Prompt: {prompt}\n\nResponse A:\n{a}\n\nResponse B:\n{b}\n\nJudge objectively."}] ) # Correct for positional flip if flip: result.chosen = "B" if result.chosen == "A" else "A" result.score_a, result.score_b = result.score_b, result.score_a return result

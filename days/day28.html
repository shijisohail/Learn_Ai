01 — RL Fundamentals

Reinforcement Learning for LLMs

Pre-training creates a capable but unaligned model. RLHF is the pipeline that shapes that capability into helpfulness. Understanding the stages helps you choose which parts to run yourself.

🧠
Key insight: Pre-training learns "what text looks like." RL alignment teaches "what good responses look like." They are separate optimization objectives — which is why you can fine-tune pre-trained weights with RLHF without destroying base capabilities.
  • RLHF three stages: (1) SFT on human demonstrations, (2) train a reward model from preference pairs, (3) optimize with PPO using the reward model as the reward signal
  • DPO (Direct Preference Optimization): Mathematically equivalent to RLHF but eliminates the reward model and PPO entirely — trains directly on preference pairs
  • Constitutional AI: Self-supervised preference generation — the model critiques and revises its own responses against a constitution, removing most human labeling
  • RLAIF: Replace human preference labelers with a powerful LLM judge — scalable to millions of pairs

02 — DPO

DPO — Direct Preference Optimization

DPO is now the go-to method for preference alignment at the practitioner level. It's stable, memory-efficient with PEFT, and avoids PPO's reward hacking instabilities.

⚡
DPO vs PPO: DPO is simpler and more stable for most practical fine-tuning. PPO still wins when you need complex reward shaping (e.g., tool use correctness + answer quality simultaneously). Start with DPO.
PYTHON — DPO Training with TRL
from trl import DPOTrainer, DPOConfig from transformers import AutoModelForCausalLM, AutoTokenizer from peft import LoraConfig, get_peft_model from datasets import Dataset import torch # DPO dataset format: prompt + chosen + rejected dpo_data = Dataset.from_list([ { "prompt": "Explain LoRA fine-tuning:", "chosen": "LoRA (Low-Rank Adaptation) is an efficient fine-tuning method that...", "rejected": "LoRA is just training a model." }, # ... thousands more preference pairs ]) model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.2-3B", torch_dtype=torch.bfloat16 ) tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-3B") model = get_peft_model(model, LoraConfig(r=16, lora_alpha=32, task_type="CAUSAL_LM")) dpo_trainer = DPOTrainer( model=model, ref_model=None, # implicit reference (saves memory with PEFT) train_dataset=dpo_data, tokenizer=tokenizer, args=DPOConfig( output_dir="./dpo-output", beta=0.1, # KL penalty coefficient max_length=512, num_train_epochs=2, per_device_train_batch_size=2, gradient_accumulation_steps=8, bf16=True, learning_rate=5e-5, ) ) dpo_trainer.train()
03 — LLM-as-Judge

LLM-as-Judge for Scalable Evaluation

Human labelers are the bottleneck in RLHF. An LLM judge running at API scale can evaluate thousands of responses per minute — and if calibrated against human preferences, produces reliable preference pairs for DPO.

  • Judge prompts: Scoring rubric (1-5 per criterion), pairwise comparison (A vs B), or absolute scoring — each has different calibration properties
  • Calibration step: Validate your judge on 200-500 human-labeled pairs before trusting it. Target Pearson r > 0.85 with human scores
  • RLAIF pipeline: Generate candidate responses with your model → LLM judge evaluates → filter ambiguous pairs → use clean pairs for DPO
  • Judge model selection: Use a stronger model than what you're training (judge with Claude 3.5, train Llama 3 3B)
PYTHON — LLM Judge + RLAIF Preference Generation
from langchain_anthropic import ChatAnthropic judge = ChatAnthropic(model="claude-3-5-sonnet-20241022") JUDGE_PROMPT = """You are evaluating an AI response. Rate it on a scale of 1-5. Criteria: - Accuracy: Is the information correct? - Completeness: Does it address the question fully? - Clarity: Is it clearly written? - Safety: Is it appropriate and helpful? Question: {question} Response: {response} Rate each criterion (1-5) and provide a total score (4-20). Format: Accuracy: X, Completeness: X, Clarity: X, Safety: X, Total: X Reasoning: [brief explanation]""" def judge_response(question: str, response: str) -> dict: evaluation = judge.invoke( JUDGE_PROMPT.format(question=question, response=response) ).content return parse_judge_scores(evaluation) # RLAIF: generate preference pairs automatically def generate_preference_pair(prompt: str) -> dict: """Generate chosen/rejected pairs using LLM judge""" response_a = llm_a.invoke(prompt).content response_b = llm_b.invoke(prompt).content score_a = judge_response(prompt, response_a)["total"] score_b = judge_response(prompt, response_b)["total"] if abs(score_a - score_b) < 2: return None # skip ambiguous pairs return { "prompt": prompt, "chosen": response_a if score_a > score_b else response_b, "rejected": response_b if score_a > score_b else response_a }
04 — Constitutional AI

Constitutional AI Self-Improvement Loop

Anthropic's Constitutional AI technique scales alignment without human labelers. A set of principles (the constitution) drives self-critique and revision. Combined with LLM-as-judge for RLAIF, the loop is nearly autonomous.

PYTHON — Constitutional AI Critique-Revise Loop
CONSTITUTION = [ "Be factually accurate and acknowledge uncertainty", "Be concise and avoid unnecessary padding", "Never produce harmful or misleading content", "Support claims with reasoning or evidence" ] async def constitutional_ai_loop(prompt: str, n_revisions: int = 3) -> str: response = await llm.ainvoke(prompt) for revision_num in range(n_revisions): # Critique phase critique_prompt = f"""Review this response against these principles: {chr(10).join(f'{i+1}. {p}' for i, p in enumerate(CONSTITUTION))} Response: {response.content} List specific violations or "PASSES ALL" if compliant:""" critique = await llm.ainvoke(critique_prompt) if "PASSES ALL" in critique.content: break # response is clean # Revision phase revision_prompt = f"""Revise this response to fix the violations: Original: {response.content} Violations: {critique.content} Revised version (fixing all issues, maintaining core answer):""" response = await llm.ainvoke(revision_prompt) return response.content
05 — Closing the Loop

Closing the Self-Improvement Loop

The complete architecture chains production inference, sampling, judge evaluation, DPO dataset curation, fine-tuning, MLflow tracking, RAGAS eval, and shadow deploy into a weekly automated improvement cycle.

  • Complete cycle: Production logs → Sample 5% → LLM judge evaluates → Build DPO dataset (chosen/rejected) → DPO fine-tune → MLflow track → RAGAS eval gate → Shadow deploy → Canary → Promote
  • Timeline: Weekly automated improvement cycle — new model every Monday, evaluated by Friday, promoted or reverted by end of week
  • Human oversight checkpoints: Review DPO data quality before training (sample 100 pairs), approve promotion after shadow metrics pass thresholds
  • Anti-reward-hacking guardrails: Measure KL divergence from reference model, flag runs where perplexity drops too fast, check for alignment drift on eval set
🔄
Alignment drift risk: Each DPO cycle moves the model slightly. Without evaluation on a fixed human-rated set, you can drift toward optimizing the judge's preferences rather than actual quality. Always keep a held-out human eval set for ground truth.

1
Supervised Fine-Tuning (SFT)
1K–100K demos Low cost TRL SFTTrainer

Train the base model on high-quality demonstration data. This shapes the output format and general helpfulness before preference learning begins.

Data: human-written Q&A pairs, formatted outputs, task demonstrations. 1K high-quality examples beats 100K noisy ones.

Stability: High. SFT is just supervised learning — it's stable and predictable.

2
Reward Model Training
10K–500K pairs Medium cost Bradley-Terry

Human labelers rank pairs of responses. The reward model learns to predict which response humans prefer — essentially compressing human preferences into a scalar score.

Data: preference pairs (response A vs B). Human labelers compare and choose preferred. Expensive and slow.

DPO replaces this step entirely — no separate reward model needed.

3
PPO Optimization
Same as RM High cost TRL PPOTrainer

Proximal Policy Optimization: the LLM generates responses, the reward model scores them, and PPO updates the policy to maximize reward while staying close to the SFT checkpoint (KL constraint).

Notoriously unstable. Requires careful tuning of KL penalty (beta), learning rate, and batch size. Can reward-hack if not constrained.

DPO replaces PPO — use DPO unless you have specific reasons to run PPO.

Progress:

DPO Replaces Stages 2 & 3

DPO optimizes a single objective directly on preference pairs: maximize log-probability of chosen over rejected under a KL constraint. No reward model, no PPO. Same mathematical guarantee, far less engineering complexity.

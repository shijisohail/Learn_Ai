Section 01

The Multi-Tenancy Problem in AI

When multiple organizations share an AI infrastructure, data leakage between tenants is a catastrophic failure mode. The challenge: vector embeddings don't have row-level security built in.

Traditional SaaS multi-tenancy uses database row-level security (RLS) to filter records by tenant_id in SQL queries. Vector databases require the same concept applied differently: every document chunk must be tagged with a tenant identifier at ingestion time, and every search query must include a metadata filter that restricts results to the requesting tenant's namespace. Without this, Tenant A's search could return Tenant B's documents.

The auth flow in an AI copilot is: user authenticates → receives JWT → JWT is verified on each API call → tenant_id is extracted from JWT claims → all downstream LLM/RAG operations are scoped to that tenant's data. The critical security invariant: the tenant_id must never come from the user's request body — only from the verified JWT. Any user-supplied tenant ID is a privilege escalation vector.

🔐
Namespace Isolation
Separate Qdrant collections per tenant (strong isolation, higher overhead) or shared collection with mandatory tenant_id filter on every query (efficient, but a missing filter is catastrophic).
🎫
JWT Claims as Context
JWT payload carries tenant_id, user_id, roles, and permissions. These claims inject into every AI operation as non-negotiable filters — the user cannot override them regardless of what they prompt.
📋
RBAC for Tool Calls
Agent tool definitions are filtered by user roles before being offered to the LLM. An analyst cannot be offered the "delete_database" tool — it never appears in the tool list, so the model cannot call it.
⚠️
Common pitfall: Trusting the tenant_id from the request body or URL parameter. Always extract tenant_id exclusively from the verified JWT payload. Never let users self-declare their tenant — this is IDOR (Insecure Direct Object Reference) vulnerability at the AI layer.

tenant_rag.py
from fastapi import FastAPI, Depends, HTTPException from fastapi.security import HTTPBearer import jwt from qdrant_client import QdrantClient from qdrant_client.models import Filter, FieldCondition, MatchValue app = FastAPI() qdrant = QdrantClient("http://localhost:6333") bearer = HTTPBearer() def get_tenant_context(token=Depends(bearer)): """Extract tenant_id from verified JWT. Never trust request body.""" try: payload = jwt.decode( token.credentials, "SECRET_KEY", algorithms=["HS256"] ) return { "tenant_id": payload["tenant_id"], "user_id": payload["sub"], "roles": payload.get("roles", []), } except jwt.InvalidTokenError as e: raise HTTPException(401, str(e)) @app.post("/search") async def tenant_search(query: str, ctx=Depends(get_tenant_context)): # Mandatory tenant filter — injected from JWT, NOT from user input tenant_filter = Filter( must=[FieldCondition( key="metadata.tenant_id", match=MatchValue(value=ctx["tenant_id"]) )] ) results = qdrant.search( collection_name="documents", query_vector=embed(query), query_filter=tenant_filter, # ALWAYS applied, not optional limit=5 ) return [r.payload for r in results]

Section 04

RBAC for Agent Tool Access

Role-Based Access Control determines which tools an agent can call. By filtering the tool manifest before passing it to the LLM, you prevent privilege escalation at the model level.

ToolViewerAnalystAdminRisk if Exposed
search_documents✓✓✓Low — read only
create_report✗✓✓Medium — data egress
modify_record✗✗✓High — data mutation
delete_collection✗✗✓Critical — irreversible
manage_users✗✗✓Critical — privilege escalation
💡
Why this matters: Never include tools in the LLM's tool manifest that the user shouldn't be able to call. If a tool appears in the manifest, the model can call it regardless of what the system prompt says. The manifest is the access control list — filter it by role before every LLM call.
